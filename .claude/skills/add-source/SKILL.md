# 소싱처 추가·업데이트 (add-source)

웹 "크롤링 가이드 > 소싱처 추가·업데이트" 화면이 만들어 준 프롬프트(소싱처명 + 유형별 URL)를 받아, **크롤링 가이드(정본)대로** 소싱처 크롤러를 신규 작성하거나 기존 크롤을 교정한다.

## 입력 (붙여넣은 프롬프트)

```
크롤링 가이드 > 소싱처 추가·업데이트 작업.
신규 소싱처: 29CM            (또는: 기존 소싱처 크롤 업데이트: 29CM)
URL:
- [단품] https://...
- [색상모음전] https://...
- [모델모음전] https://...
→ add-source 스킬대로 …
```

- **유형**이 옵션 구조를 결정한다: **단품**=단일 상품(색×사이즈 1세트) · **색상모음전**=여러 색이 한 상품에 묶임 · **모델모음전**=여러 모델이 묶임. 각 유형은 가이드의 옵션·재고 매칭 로직이 다르다(§1).
- 프롬프트가 없고 이름만 받은 경우 → 사용자에게 URL·유형을 먼저 요청.

## 🚨 절대 원칙 (먼저 읽기)

1. **정본 = 크롤링 가이드.** 손대기 전 반드시 정독(아래 Step 0). 추측 금지.
2. **무결성 4철칙**(가이드 §4): 폴백가 금지 · 크롤실패=표기(옛값 금지) · 단일 진실원천 · 누락에 경고.
3. **라이브 100% 일치 전 "완료" 선언 금지** (가격·재고 오류 = 금전 손실).
4. **작업은 origin/main 워크트리에서.** 현재 브랜치가 stale일 수 있음.

## Step 0 — 준비·정독 (모든 작업 공통)

1. **워크트리**: origin/main 기준 워크트리인지 확인. 아니면 `superpowers:using-git-worktrees`로 origin/main 분기 워크트리 생성.
2. **가이드 정독** (정본 = repo 파일, 라이브와 동일 내용):
   - 원문: `프로그램/_시스템/docs/크롤링-가이드.md` (전체 통독: §0 흐름·§1 재고·§2 가격·§4 무결성·§5 에러 이력)
   - 라이브 렌더(참고): `https://mou-m.com/sourcing-guide/map` (보기) — 단, 로그인 게이트라 WebFetch 불가할 수 있음 → repo 파일이 1차.
   - 에러 카탈로그: repo `프로그램/_시스템/webapp/static/error_catalog.json` (또는 공개 URL `https://mou-m.com/static/error_catalog.json` — 200 공개).
3. **과거 이력 자동 매칭 (핵심)** — `error_catalog.json`에서 **소싱처 이름과 무관하게** 이번 작업이 건드릴 메커니즘·공유코드에 해당하는 항목을 뽑아 **재발방지 체크리스트**로 만든다:
   - `rule:true` 4철칙 = 항상 포함
   - `g`(재고/가격) = 이번에 긁을 데이터 종류와 일치
   - `code.func` = 새 크롤러가 호출할 공유코드(`persist_crawled_options`·`_match_option_so`·`compute_breakdown` 등)와 일치 → **최우선 위험**
   - 다른 소싱처 에러라도 같은 메커니즘이면 반드시 포함 (예: S17 부분일치 오매칭, S1 재고합계 둔갑, S3 다중URL 중복행).
4. **진행 시각화**: `workflow-progress-widget`로 아래 스테이지 체크리스트를 띄워 사용자와 방향을 맞춘다.

## 단계적 실행 — 한 번에 다 하지 말 것

데이터 지도가 크다. **스테이지마다 [해당 § 정독 → 작업 → 가이드와 자기 대조 → 다음]**. 큰 가이드를 통으로 삼키지 말고 그 스테이지의 § 슬라이스만 본다.

**자동 OK/고쳐**: 각 스테이지 끝에서 결과를 해당 §와 **스스로 대조** → 일치하면 자동 진행, 어긋나면 자동으로 직전 스테이지로 돌아가 교정(사용자 개입 없이 루프). 진행은 위젯에 표시.

**사용자 게이트는 3곳만**: ① S5 라이브 100% 대조(사용자 실제 화면이 정답 기준) ② 자동 교정 3회 반복해도 못 맞추면 에스컬레이션 ③ 최종 완료 승인.

## 완료 게이트 + 에러 이력 선순환 (강제)

- ✅ 라이브 100% 일치 (§4).
- ✅ **새로 발견·해결한 문제는 반드시 `프로그램/_시스템/docs/크롤링-가이드.md` §5 + `error_catalog.json` 둘 다에 기재** (거버넌스: 같은 소싱처+근본+증상 = 기존행 rev / 다른 소싱처·같은근본 = 새 행, ID prefix S/P/O+다음번호, `code.func`·계층·상태 태그 필수). 이 기재는 **완료의 필수 조건**(안 하면 다음 소싱처가 같은 문제 재발).
- ✅ 공유코드 수정 시 같은 코드 쓰는 타 소싱처 회귀 검증 + 테스트 PASS.

## Claude가 만지는 곳 (repo → 배포)

- `프로그램/_시스템/lemouton/sourcing/crawlers/<key>.py` (+ `crawlers/__init__.py` 등록)
- `프로그램/_시스템/docs/크롤링-가이드.md` (§1·§2 표 행, §5 에러)
- `프로그램/_시스템/webapp/static/error_catalog.json`
- 커밋 → push main → AWS 자동배포 → 라이브 반영.

> ⚠️ 라이브 DB 쓰기(소싱처 카드의 6항목 필드 등)는 로그인 게이트라 Claude가 직접 못 함. Claude는 **repo(크롤러·가이드 표·에러카탈로그)**를 책임지고, 소싱처 카드 채움·전체보기 등록은 사용자가 웹 추가 화면/상세에서 수행(분석 결과 제공). 전체보기 행 자체는 웹 "추가" 버튼이 이미 생성.

## 마무리
- `superpowers:verification-before-completion`으로 라이브 대조 증거 확인 후에만 완료 선언.
- 작업 종료 = origin/main 머지 + 배포까지 (워크트리 머지 의무).
