# 데이터 코드 지도 최신화 (update-data-code-map)

"판매처관리(또는 소싱처관리) 업데이트 해줘" 한 문장 = **그 영역 지도를 전수 검토해 코드 현실과 일치시켜라**.
지도는 자동으로 안 늙는다 — 코드·기능이 먼저 가고 지도가 뒤처진다. 이 스킬이 그 격차를 정기적으로 0으로 만든다.

<HARD-RULES>
1. **되묻기 금지.** "판매처관리가 무슨 뜻인가요?" ❌ — 발화의 뜻은 지도 최신화로 확정. 바로 실행한다.
   (예외: 발화에 재설계·화면개편 등 지도 밖 요구가 명시적으로 섞여 있을 때만 그 부분을 분리 확인)
2. **전수(全數)다.** 일부 탭·일부 소싱처만 보고 끝내기 금지. 아래 체크리스트 전 항목을 TaskCreate로 만들어 하나씩 소거한다.
3. **날조 금지.** 지도에 쓰는 모든 값은 코드·커밋·실측 근거가 있어야 한다. 근거 없으면 "확인불가"/st=todo.
4. **개선·해결이 있었다면 과거이력은 필수다(사장님 직접 지시).** 스캔에서 버그 수정·함정 발견·방식 변경을 하나라도 찾았으면 **반드시 기록하고 넘어간다**: 판매처 = `incidents`(「해결」칸 = 코드로 어떻게 고쳤는지) / 소싱처 = §5 에러 이력(재발방지 체크 포함). 기록 없는 해결 = 다음 세션이 같은 함정을 다시 밟는다. "사소해서 생략" 금지 — 마무리 보고에 **과거이력 추가 N건**을 별도 수치로 명시한다.
</HARD-RULES>

## 공통 절차 (두 갈래 동일 뼈대)

1. **기준 시각** — 정본 파일의 마지막 갱신 커밋 시각 확인: `git log -1 --format=%ci -- <정본>`
2. **변경 스캔** — 그 이후 origin/main 커밋 중 해당 영역 커밋 전수 열거: `git log --oneline --since=<시각> -- <영역 경로들>`
3. **반영 판정** — 커밋 하나씩: 지도에 이미 반영됐나? (아래 갈래별 점검표) 누락이면 갱신 목록에 추가
4. **갱신 실행** — origin/main 워크트리에서 수정·추가·삭제 (⚠️ 본체 트리는 뒤처짐 — 반드시 새 워크트리)
5. **검증** — 갈래별 검증기/테스트 통과 + 화면 렌더 확인
6. **배포·라이브 대조** — main 머지 → 자동배포 → 라이브 화면에서 갱신분 확인
7. **마무리** — conversation-wrapup 형식 보고 (갱신 N건·삭제 N건·확인불가 N건 수치 명시)

## 갈래 A — "판매처관리 업데이트" (판매처 지도)

- **정본**: `프로그램/_시스템/webapp/data/marketplace_api_map.json` (검증기 `webapp/marketplace_api_map.py` — st·v enum, tabs 기준선, 완성게이트)
- **스캔 경로**: `프로그램/_시스템/shared/platforms/` `lemouton/uploader/` `lemouton/markets/` `webapp/routes/`(마켓 관련) `docs/markets/`
- **커밋별 점검표** (각 마켓 커밋에 대해):
  - [ ] 새 API를 배선했나 → 지도에 있나? 없으면 추가(st=code, req·res·fields·success 채움 — 완성게이트)
  - [ ] 기존 API를 라이브 검증했나 → st 승격(code→ok)?
  - [ ] **[필수·HARD-RULE 4]** 함정·사고·해결이 있었나 → `incidents`에 추가? (「해결」칸 = 코드로 어떻게 고쳤는지 필수)
  - [ ] 새 API에 업무탭 배정(`tabs`)했나? (미배정이면 기준선 `tabsUnassignedBaseline` 초과로 검증기가 울린다 — 배정이 원칙, 의도적 기타만 기준선 상향)
  - [ ] 주문상태 전환·정산 계산·속도 실측이 바뀌었나 → `autoConfirm`·`settleCalc`·`markets[].upload_limit` 갱신?
- **검증**: `python -m pytest tests/marketplace_guide -q` (⚠️ 인자 없는 pytest 금지) + `validate_map` 신규 오류 0 + `/marketplace-guide/map` 렌더 확인
- **삭제 판정**: 코드에서 사라진 API의 st=ok/code 잔존 → st 강등 또는 삭제(과거이력에 사유 기록)

## 갈래 B — "소싱처관리 업데이트" (크롤링 가이드)

- **정본**: `프로그램/_시스템/docs/크롤링-가이드.md` (**§8 동기화 규칙이 이 문서 자체의 갱신법 정본** — 먼저 읽고 따르라)
- **스캔 경로**: `lemouton/sourcing/` `lemouton/sources/` `extension/moum-crawler/` `webapp/routes/api_benefits.py` `webapp/routes/api_pricing.py`
- **커밋별 점검표**:
  - [ ] 크롤러·저장·계산·표시(§0 4층) 코드가 바뀌었나 → 해당 `💻` 블록(파일:라인) 갱신?
  - [ ] 소싱처별 수집 방식이 바뀌었나 → §1(재고)·§2(가격) 소싱처 표 갱신?
  - [ ] **[필수·HARD-RULE 4]** 새 에러·버그를 겪고 고쳤나 → **§5 에러 이력 카탈로그에 추가?** (재발방지 체크 포함)
  - [ ] 신규 소싱처가 추가됐나 → §1·§2 표 + §7 온보딩 체크 반영?
- **검증**: 가이드 렌더 `/sourcing-guide/map` 확인 + 인용한 파일:라인이 실코드와 일치하는지 표본 대조
- ⚠️ 크롤 검증이 필요해지면 `feedback_crawl_verification_protocol` 원칙(기준=실상품 URL) 준수

## 하지 말 것 (baseline 실측 실패 패턴)
