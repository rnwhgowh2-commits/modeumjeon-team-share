## ⚠️ 이 문서를 읽기 전 반드시 알 것

1. **라이브 실제 설정값은 이 문서로 확인할 수 없다.** 소싱처별 혜택의 실제 행(`SourceBenefitTemplate` / `OptionBenefitOverride`)은 라이브 Supabase PG 에 있고, 이 조사는 SQLite 폴백 워크트리에서 **코드 경로·시드·가이드 파일만** 보고 썼다. 따라서 "이 소싱처에 어떤 템플릿 혜택이 몇 %로 들어 있나"는 **전부 ❓미확인**이다. 확인된 것은 *코드가 무엇을 하드코딩으로 주입하는가* · *크롤이 무엇을 수집하는가* · *엔진이 어떤 순서로 계산하는가* 뿐이다.
2. **모르면 「❓미확인」으로 적었다. 추측을 사실로 쓰지 않았다.**
3. **줄 번호 기준 시점: 2026-07-19, git `4bb6cbfe` + 미커밋 워킹트리.** 조사 중에도 다른 작업자가 `api_benefits.py`(1788→1828줄) · `final_price.py` 를 편집하고 있었다. 줄 번호는 **최종 재검증한 값**이지만, 이후 편집으로 다시 밀릴 수 있다. **줄이 안 맞으면 옆에 인용한 코드/주석 문구로 찾을 것.**
4. ⛔ **`lemouton/sourcing/crawlers/pricing_policy.py` 의 `apply_*_policy` 함수들은 죽은 코드다.** 호출자가 0건이고(크롤러 어디서도 부르지 않는다) 계산 순서도 현행 엔진과 다르다. **진실 원천이 아니므로 이 문서는 그 내용을 인용하지 않는다.** 그 파일을 근거로 가격 로직을 판단하지 말 것.

## 0-2. 정렬 — 카테고리 3단

`final_price.py:19-24`

```python
def _benefit_priority(it):
    if (it.benefit_type or 'rate') == 'amount':  return 0   # 정액(원)
    if '적립' in (it.benefit_name or ''):        return 1   # 정률 적립
    return 2                                                # 정률 할인
```

**정액(원) → 정률 적립(%) → 정률 할인(%)** 순. 같은 등급 안에서는 `sorted` 가 stable 이므로 **조립된 순서 그대로**다.

> 🚫 **판정이 「이름에 '적립' 이 들어 있나」다.** 혜택 이름을 바꾸면 계산 순서가 바뀐다. 정률 항목의 이름에서 '적립' 두 글자를 빼면 그 항목이 뒤로 밀리고 최종매입가가 달라진다.

## 0-3. 차감 방식 — 직전 잔액 기준 순차 누적

`final_price.py:198-208` (tagged) / `final_price.py:295-305` (legacy) — 두 경로 동일

```
rate   항목:  차감 = int(잔액 × base_ratio × 값)
amount 항목:  차감 = int(값)
차감 = min(차감, 잔액)       # 잔액보다 크게 못 깎는다
잔액 = max(잔액 − 차감, 0)
```

즉 **%는 항상 「직전까지 깎고 남은 잔액」에 곱한다.** 표면가에 한꺼번에 곱하는 게 아니다.

## 0-4. 캐시백 기준금액 계수 `base_ratio` (사장님 확정 2026-07-19)

`final_price.py:97-133`

```
캐시백 적립 = 기준금액 × 0.9 × 적립율      ← 기본 (부가세 뺀 공급가 기준)
캐시백 적립 = 기준금액 × 1.0 × 적립율      ← SSG · 신세계쇼핑 · CJ (전액 기준 예외 3사)
```

- **캐시백 항목에만** 붙는다. 판정은 `_is_cashback` 재사용 (`final_price.py:122-123`).
- 적립율(`value`)에 0.9 를 미리 곱해 넣지 않는다 — 영수증에서 1.1% 가 0.99% 로 뭉개지면 근거가 사라진다. 계수는 **기준금액 쪽**에만 건다(`final_price.py:198-202` 주석).
- 값이 NULL·비숫자·0 이하·1 초과면 전부 **1.0** 으로 떨어뜨린다 (`final_price.py:126-132`). = 안 줄이는 쪽 = 매입가 과대 = 안전 방향.
- 시드된 값: SSG `1.0` / 롯데온 `0.9` (`source_benefit_seed.py:79-81`).
- **override 는 같은 이름의 템플릿에서 `base_ratio` 를 물려받는다** (`api_benefits.py:641-670`). override 생성 경로(`add_override` / 스냅샷 저장 / 일괄 추가)는 클라이언트 payload 로 행을 만들어 `base_ratio` 가 NULL 로 남을 수 있는데, NULL = 1.0 이라 **캐시백이 10% 과다 차감 → 매입가 과소 → 언더프라이싱** 이 된다. 읽기 전용 프록시 `_WithRatio` 로 덧씌우며 **원본 ORM 행은 건드리지 않는다**(공유 캐시 오염 방지).

# §1. 르무통 공홈 (`lemouton`, source_id **1**)

## 1-1. 검증된 정답 순서 (fx영수증 실측)

```
표면노출가                116,900
  ① 리뷰금 5,000 (정액)   − 5,000  → 111,900
  ② 네이버페이 1% (정률·적립)  − 1,119  → 110,781      ← 리뷰 차감 후 잔액 기준
  ③ 현대카드 2.73% (정률·할인) − 3,024  → 107,757
  백원 버림                        → 최종매입가 107,700
```

**엔진 규칙과 정확히 일치한다** — ①은 `amount`(우선순위 0), ②는 이름에 '적립' 포함(우선순위 1), ③은 그 외(우선순위 2). `final_price.py:19-24`.
③이 ②를 결제 택1로 잡아먹지 않는 이유: `_is_payment('네이버페이 적립금')` 이 '네이버' 때문에 **False** (`final_price.py:38-39`).

## 1-2. 혜택 목록

| 이름 | 유형 | 값 | 출처 | 활성 조건 |
|---|---|---|---|---|
| 리뷰금 | 정액(원) | 5,000 | ❓ 템플릿(DB) | ❓미확인 |
| 네이버페이 적립금 | 정률 % | 1% | ❓ 템플릿(DB) | ❓미확인. 이름은 2026-06-12 마이그레이션으로 `구매적립금 (네이버페이)` → `네이버페이 적립금` 통일 (`scripts/migrate_naverpay_reward_naming.py:31, 34`) |

## 8-4. 카드 청구할인 — `enabled=True` 인 이유

`api_benefits.py:841-861` 주석 전문 요약:
- M1-5 가 표면가를 「최대할인가(카드 포함)」 → 「표면노출가(카드 미적용)」 로 바꾸면서 카드 청구할인을 `lotteimall_card_discount` 로 **분리 보관**했다 (`lotteon.py:1489-1497, 1574`).
- **이중차감이 아닌 근거**: ① `auto_card_discount.included_in_sale_price` 를 **False 로 정정** → `api_pricing.py:626` 의 "카드 OFF 시 가격 환원"(`price/(1-rate)`) 이 no-op. ② 표면가에 카드분이 이미 빠져 있지 않다(`benefitPrc + 카드금액`).
- **`True` 인 이유**: M1-5 이전 매입가는 카드할인이 이미 반영된 가격에서 출발했다. `False` 로 두면 결과가 옛 매입가보다 **8,180원 비싸진다 = 조용한 회귀**. `True` 가 무회귀 값.

크롤러 주석 (`lotteon.py:222-226`):
> "★ 2026-07-18 — `included_in_sale_price` 를 True→**False** 로 정정. … 이 플래그가 True 로 남으면 `api_pricing.py` 의 '카드 OFF 시 가격 환원'이 걸려 **이미 카드가 빠진 가격을 한 번 더 부풀린다**. False 가 사실이자 안전값."

표면가 산출 (`lotteon.py:415-446`): `benefitPrc` 는 카드 포함가이므로 **표면가 = `benefitPrc` + `cardDiscountList[0].discountAmount`**. 필드가 없으면 **추정하지 않고 실패**시킨다 (`lotteon.py:437-439`).

# §12. 조사 중 발견한 의심스러운 동작 (고치지 않음 — 보고만)

## 12-1. `benefit_parse._PRODUCT_DYNAMIC_KEYS` 가 단일 진실 원천과 어긋난다

`lemouton/pricing/benefit_parse.py:105` 주석은 **"키 목록은 `service.py` `PRODUCT_DYNAMIC_KEYS` 와 동일(단일 진실 원천)"** 이라고 적었지만, 실제로는 **4개가 빠져 있다**:

| 빠진 키 | `service.py` | `benefit_parse.py` |
|---|---|---|
| `product_coupon_list` | `:616` | ❌ 없음 |
| `member_price` | `:626` | ❌ 없음 |
| `is_member_price` | `:626` | ❌ 없음 |
| `login_marker_present` | `:626` | ❌ 없음 |

`extract_dynamic_benefits_from_options` 는 **navGrab → `/api/sources/parse` 경로 전용**(SSF·SSG·르무통·스스)이고 그 4개 키는 무신사 전용이라 오늘 당장 영향은 없어 보인다. 다만 **주석이 사실과 다르고**, 무신사가 나중에 parse 경로를 타면 조용히 드롭된다.

## 12-2. SSG 카드혜택가 정액이 `sale_price` 기준으로 계산된다

`api_benefits.py:739`
```python
_amount = float(sale_price) - float(_cbp)
```
이 금액은 **원래 표면가 기준**인데, 실제 차감은 잔액에서 일어난다(`final_price.py:207` 이 `min(deduct, base)` 로 자르긴 한다). 카드혜택가는 정액이라 우선순위 0(가장 먼저)이므로 대부분 표면가 = 잔액이지만, **다른 정액 항목이 먼저 조립돼 있으면 순서에 따라 과다 차감이 가능하다.** SSG 는 현재 정액 항목이 카드혜택가 하나뿐으로 보이나(상품쿠폰이 amount 형이면 2개), **템플릿에 정액 혜택이 추가되면 재검토 필요.**

## 12-3. `_SITE_BY_SRC` 가 `source_registry.id` 를 하드코딩한다

`api_benefits.py:524` — `{1:'lemouton', 2:'ss_lemouton', 3:'musinsa', 4:'ssf', 5:'lotteon', 6:'ssg'}`

**라이브 id 배치가 다르면 현대카드 플로어·기프트포인트·동적혜택이 엉뚱한 소싱처에 걸린다.** 시드표 문서도 같은 위험을 기록해 뒀다(§6-1). 이 워크트리에서는 라이브 id 를 확인할 수 없다 → **❓미확인, 라이브 대조 필요.**

무신사 판정은 한술 더 떠 **문자열 비교**다: `if str(source_id) == '3'` (`api_benefits.py:915`).

## 12-4. SSG 네이버페이 차단이 코드가 아니라 "데이터에 없음"에 의존한다

§6-5 참조. 누군가 SSG 템플릿에 '네이버페이' 이름의 혜택을 추가하면 **`_is_payment` 가 '네이버' 때문에 False 를 주어 택1을 우회하고 조용히 적용된다.** 사장님 요구("SSG는 네이버페이 적용 안됨")를 코드가 강제하지 않는다.

## 12-5. SSF 적립 이중차감 방어가 없다

`ssf.py:237-238` 이 "DB 에 `구매적립금 (포인트)` 0.5% 고정 시드가 있다"고 기록하는데, 크롤 `point_rate` 항목(`api_benefits.py:701-706`)을 추가할 때 **그 시드 행을 끄는 코드가 없다.** 롯데아이몰 L.POINT 는 같은 상황에서 자동 비활성 코드가 있다(`api_benefits.py:829-833`) — **SSF 만 빠져 있다.**
라이브에 그 시드 행이 살아 있는지 ❓미확인. 살아 있다면 **적립이 두 번 깎여 매입가 과소 → 언더프라이싱** 이다.

## 12-6. 저장은 되는데 계산에 안 쓰이는 키

화이트리스트(`service.py:606-627`)에 있어 DB 에 저장되지만 `compute_breakdown` 이 읽지 않는 키:

| 키 | 소싱처 | 근거 |
|---|---|---|
| `review_point_max` | 스마트스토어 르무통 | `service.py:623` 에 있으나 `api_benefits.py` 에 참조 0건 |
| `lotteon_coupons` | 롯데온 | `service.py:622` 에 있으나 참조 0건 (UI 표시 전용으로 보임) |
| `point_amount` | SSF | `service.py:607` 에 있으나 `point_rate` 만 계산에 쓰임 |
| `ssg_money_amount` | SSG | `service.py:610` 에 있으나 `ssg_money_rate` 만 쓰임 |
| `product_coupon_max_discount` | SSG | `service.py:614` 에 있으나 `api_benefits.py` 참조 0건 — **쿠폰 최대할인 한도가 계산에 반영되지 않는다** |

> ⚠️ 마지막 항목을 주의 깊게 볼 것. SSG 상품쿠폰 가이드는 `베이스금액① × 12%(최대 2만)` 인데(`populate_ssg_guide.py:53`) 코드는 **rate 만 곱하고 상한을 씌우지 않는다** (`api_benefits.py:777-786`). 쿠폰이 활성(사용자 토글)되고 표면가가 높으면 **상한 초과분까지 깎여 매입가 과소** 가 될 수 있다.
> 다만 크롤러는 이 키를 `product_coupon_max_discount: int (최대 할인 원, **UI 표시용**)` 이라고 정의해 뒀다(`ssg.py:535`) — **의도적으로 표시 전용일 가능성**이 있다. 어느 쪽이 사장님 의도인지 ❓미확인. 기본이 `enabled=False` 라 자동 발생은 안 하고, 토글하는 순간에만 문제가 된다.

## 12-7. 크롤가이드 문서와 코드의 순서 표기가 다르다

| 소싱처 | 가이드 표기 | 코드 실제 순서 | 금액 영향 |
|---|---|---|---|
| 무신사 | 등급할인→등급적립→무신사머니→상품쿠폰 (`크롤링-가이드.md:141`) | 상품쿠폰→등급할인→등급적립→무신사머니 (`api_benefits.py:948-954`) | **없음** (전부 정액) |
| SSF | 기프트포인트 → 멤버십포인트 (`populate_ssf_guide.py:52-53`) | 멤버십포인트(적립,우선순위1) → 기프트포인트(우선순위2) | **있음** (둘 다 정률) |
| 롯데온 | 롯데오너스→스토어찜→L.POINT→OK캐시백 (`크롤링-가이드.md:142`) | 스토어찜(정액)→적립류→롯데오너스(정률할인) | **있음** |
| 무신사 | 등급할인·상품쿠폰 = `preapplied` (`populate_musinsa_guide.py:48-49`) | 실제로 **차감**한다 | **있음** |
| SSG | 카드혜택가 = `preapplied` (`populate_ssg_guide.py:52`) | 실제로 **차감**한다 | **있음** |

**전부 코드가 정본이다** — 가이드 쪽 표기가 stale 로 보인다(코드 주석에 더 최신 날짜의 사장님 확정 문구가 붙어 있다). 다만 **가이드는 `sync_templates_from_crawl_guide()` 를 통해 DB 템플릿으로 흘러들어간다**(`api_benefits.py:1382-1433`) — `apply` 값이 `apply_mode` 로 그대로 들어가므로(`:1376, 1382-1383`), **가이드에 `preapplied` 로 적힌 항목에 값을 넣으면 그 항목이 `apply_mode='preapplied'` 템플릿 행이 되어 차감이 사라진다.** 가이드 표기 정정이 실제로 필요하다.

## 12-8. `sync_templates_from_crawl_guide` 가 시드값을 덮는다

`api_benefits.py:1418-1425` — `benefit_name` 기준 upsert 로 기존 행의 `value`·`apply_mode`·`enabled` 를 **덮어쓴다**(`pay_method`·`category` 는 보존). 크롤가이드 혜택 카드에 이름 `OK캐시백` + 값을 입력하면 시드한 `base_ratio` 는 남지만 `value`·`apply_mode` 가 갈아엎힌다. (시드표 §6-2 에도 기록됨)

## 12-9. legacy 결제 택1 승자를 `sale_price` 근사로 고른다 (무신사 `base_override` 와 어긋남)

`final_price.py:252-255`
```python
def _approx_deduct(it):
    v = float(it.value or 0)
    return v if (it.benefit_type or 'rate') == 'amount' else float(sale_price) * v
```
`base_override` 를 쓰는 무신사에서는 실제 베이스가 `surface_price` 인데 근사는 `sale_price` 로 한다. 결제성 항목이 2개 이상이면 **승자를 잘못 고를 수 있다.** (tagged 경로는 실측이라 문제없음 — `card_candidates.py:26-30` 주석이 같은 문제를 지적한다.)
