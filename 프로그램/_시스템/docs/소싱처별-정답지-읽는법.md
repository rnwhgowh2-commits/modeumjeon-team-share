# 소싱처별 「정답지 읽는 법」 — 이 화면의 이 숫자가 표면노출가다

> **계층**: 프로젝트 문서 (검증 실무 정본)
> **책임**: 소싱처마다 **페이지의 어느 숫자를 ① 표면노출가로 읽어야 하는지**
> **단일 진실 원천**: 본 문서 (표면가 '읽는 법' 한정)
> **관련**: `docs/크롤링-가이드.md` (크롤 전반) · `lemouton/sourcing/price_verify.py` (3층 대조 판정)
> **화면**: 소싱처 관리 → 🧾 최종매입가 검증 (`/sourcing-guide/price-verify`)

---

## 왜 이 문서가 있나

2026-07-19 기준, 가격 오류 6건 중 **3건이 코드 검증으로 안 잡히고 라이브 화면 대조에서만** 드러났다.
원인은 하나다 — **소싱처마다 표면가의 위치가 다른데 그걸 적어둔 곳이 없었다.**

가장 비쌌던 사고: **롯데아이몰**. 페이지에 크게 뜨는 「롯데홈쇼핑 최대할인가」를 표면가로 집었는데,
그 숫자는 **카드할인이 이미 먹은 값**이었다. 그래서 카드할인이 두 번 차감됐다.

---

## 표면노출가(①)의 정의

**우리 계산엔진(`compute_breakdown`)에 `sale_price` 로 들어가는 값 = 베이스금액.**
여기서부터 우리가 혜택을 차감해 최종매입가를 만든다.

따라서 표면가는 **우리가 차감할 혜택이 아직 안 먹은 값**이어야 한다.
카드할인·적립이 이미 반영된 숫자를 표면가로 넣으면 **이중차감 → 언더프라이싱 → 금전 손실**이다.

> 🚫 **헷갈리면 추측하지 말고 「미확인」으로 두고 보고할 것.**
> 이 표에 틀린 값을 적으면 그대로 돈이 샌다.

## 소싱처별 표

| 소싱처 | 페이지에서 읽을 숫자 (①) | 절대 쓰면 안 되는 숫자 | 상태 |
|---|---|---|---|
| **롯데아이몰** (lotteimall) | **카드 미적용 할인가** — 할인율(%)이 붙어 있는 그 가격 (예 `149,000 → 22% → 116,900`) | ⚠️ **「롯데홈쇼핑 최대할인가」** = 카드 청구할인 **포함**된 값 | ✅ 확인됨 |
| **현대H몰** (hmall) | **깜짝할인 선반영가** (`itemPtc.bbprc`) — 카드할인 **미포함** | 카드 즉시할인 적용가 (카드할인은 별도 항목으로 분리돼 있음) | ⚠️ 값은 확인, **화면 라벨 미확인** |
| **SSG** (ssg) | **「최적가」** (`bestAmt`) — 상품 즉시할인 반영, 카드·쿠폰 제외 | 카드/쿠폰까지 먹은 최저가 | ✅ 확인됨 |
| **SSF샵** (ssf) | **「판매가」** (`em.price`) — 시즌 할인 선반영 | 멤버십 포인트·기프트포인트 반영가 | ✅ 확인됨 |
| **스마트스토어 르무통** (ss_lemouton) | **표시 할인가** (즉시할인 적용 후, 비회원 기준) | — | ⚠️ 값은 확인, **화면 라벨명 미확인** |
| **르무통 공홈** (lemouton) | — | — | ❌ **미확인** |
| **무신사** (musinsa) | — | ⚠️ **「나의 할인가」(회원가)는 아님** (이걸 표면가로 쓴 이중차감 사고 기록 있음) | ❌ **미확인 — 문서 간 충돌** |
| **롯데온** (lotteon) | — | ⚠️ **「나의 혜택가」는 아님** (롯데오너스 포함 → 이중차감 위험 기록 있음) | ❌ **미확인 — 정의 3개 충돌** |

## 상세

### ✅ 롯데아이몰 (lotteimall) — 이번 사고의 핵심 사례

페이지 가격 라인은 이렇게 쌓인다:

```
정가            149,000
  ↓ 상품 할인 −32,100 (22%)
표면노출가 ①    116,900   ← ★ 이걸 읽어야 한다
  ↓ 삼성카드 7% 청구할인 −8,180
「롯데홈쇼핑 최대할인가」 108,720   ← ✗ 이건 표면가가 아니다
```

- **읽을 것**: 할인율(%)이 붙어 있는 가격 = 카드 미적용 할인가
- **읽지 말 것**: 크게 강조되는 「롯데홈쇼핑 최대할인가」 — **카드 청구할인이 이미 포함**
- 근거: `lemouton/sourcing/crawlers/lotteon.py` `_resolve_surface_price` (2026-07-18 사용자 확정),
  `docs/sources/lotteimall/profile.yaml`, `tests/sourcing/test_lotteimall_surface_price.py`
- 페이지에 카드 미적용가가 단독으로 안 뜨면 `최대할인가 + 카드할인액` 으로 역산한다.
  **역산도 실패하면 크롤은 에러를 낸다 — 폴백 금지.**

### ⚠️ 현대H몰 (hmall) — 롯데아이몰과 정반대

H몰은 **카드할인이 처음부터 분리**돼 있다. 이게 우리가 지향하는 올바른 구조다.

- **표면가** = 깜짝할인 선반영가 (`itemPtc.bbprc`), 카드할인 **미포함**
- **카드 즉시할인** = `표면가 − 카드적용가` 로 따로 계산돼 별도 혜택 항목(`hmall_card_discount`)으로 들어감
- 근거: `lemouton/sourcing/crawlers/hmall.py`
- ❗ **미확인 부분**: `bbprc` 가 **화면에서 어떤 라벨로 뜨는지**는 코드·문서 어디에도 없다.
  `docs/sources/hmall/` 디렉터리 자체가 없다. 화면 대조 시 이 부분은 사장님 확인 필요.

### ✅ SSG (ssg)

- **표면가** = 「**최적가**」 (`bestAmt`) — 상품 즉시할인 반영, **카드·쿠폰 제외**
- `bestAmt` 가 0/누락이면 `sellprc` 로 폴백
- 근거: `lemouton/sourcing/crawlers/ssg.py`, `docs/크롤링-가이드.md`

### ✅ SSF샵 (ssf)

- **표면가** = 「**판매가**」 (`em.price`) — 시즌 할인(%)이 이미 반영된 값
- 멤버십 포인트·기프트포인트는 **표면가에 안 들어가 있고** 별도 혜택으로 차감된다
- 근거: `lemouton/sourcing/crawlers/ssf.py`, `docs/sources/ssf/profile.yaml`

### ⚠️ 스마트스토어 르무통 (ss_lemouton)

- **표면가** = **비회원 기준 표시 할인가** (즉시할인 적용 후)
- 근거: `lemouton/sourcing/crawlers/ss_lemouton.py`, `docs/sources/ss_lemouton/profile.yaml`
- ❗ **미확인 부분**: 네이버 PDP 는 이 숫자에 「판매가」같은 **캡션을 붙이지 않는다**.
  "화면에 크게 뜨는 할인가" 로 이해하되, 라벨로 특정할 수는 없다.

### ❌ 르무통 공홈 (lemouton) — 미확인

- 크롤이 집는 값은 확인됨 (`strong.price-number` → `span.ProductPrice` → meta 태그 순)
- ❗ 그러나 **그 셀렉터가 화면의 어느 라벨에 해당하는지 적힌 문서가 없다.**
  코드 주석이 "할인가" 라고 부르지만 그건 **셀렉터에 대한 코드의 자체 명명**이지 페이지 캡션이 아니다.
- **지어내지 않고 미확인으로 둔다.** 사장님이 실제 페이지를 보고 채워야 한다.

### ❌ 무신사 (musinsa) — 미확인 (문서 간 충돌)

리포지토리 안에 **서로 모순되는 두 정의**가 있다:

| 출처 | 주장 | 시점 |
|---|---|---|
| 크롬 확장 (실제 동작 경로) | 표면가 = 구조화 API `goodsPrice.salePrice`. **「나의 할인가」(회원가)는 base 가 아님** | 2026-06-13 |
| `docs/크롤링-가이드.md` | 표면가 = **회원가** | (더 오래됨) |

확장 쪽 주석에는 **회원가를 표면가로 썼다가 이중차감·언더프라이싱 사고가 났다**는 기록이 남아 있다.
따라서 `goodsPrice.salePrice` 가 맞을 가능성이 높지만, **문서가 충돌하는 상태를 '확인됨' 으로 적을 수 없다.**

👉 **조치 필요**: 실제 페이지 대조로 확정한 뒤 이 문서와 `docs/크롤링-가이드.md` 중 틀린 쪽을 고칠 것.

### ❌ 롯데온 (lotteon) — 미확인 (정의 3개 충돌)

| 출처 | 주장 |
|---|---|
| 크롬 확장 (실제 동작 경로, 2026-07-03 수정) | **판매가** (롯데오너스 **미**반영) |
| 서버 파이썬 크롤러 | `qty.immdDcAplyTotAmt` (즉시할인 적용가, 카드할인 미반영) |
| `docs/크롤링-가이드.md` | **나의 혜택가** (즉시·카드할인 선반영) |

확장 쪽 2026-07-03 수정 주석: 「나의 혜택가」(롯데오너스 포함)를 저장하다가 **롯데오너스 이중차감 위험**이 있어
판매가로 바꿨다는 기록. 가이드 문서 쪽이 낡았을 가능성이 크지만 **셋 중 무엇이 맞는지 확정할 근거가 없다.**

👉 **조치 필요**: 실제 페이지 대조로 확정 필요.

## 검증할 때 이렇게 쓴다

1. 소싱처 관리 → **🧾 최종매입가 검증** 열기
2. 소싱처 + 상품 URL 넣고 **불러오기** → ②(우리 수집)·③(우리 계산)이 자동으로 채워짐
3. 실제 페이지를 열고 **위 표대로** 표면가를 찾아 ① 에 입력
4. 가능하면 혜택 항목별 금액도 입력 (안 넣으면 그 층은 **확인불가** — 일치로 안 침)
5. **판정하고 저장** → 어느 층에서 갈렸는지 자동으로 짚어줌

| 판정 | 뜻 | 다음 행동 |
|---|---|---|
| **일치** | ①=②, ② 혜택=③ 차감 | 없음 |
| **불일치 (①↔②)** | 표면가가 다름 | **크롤 파싱 문제** — 크롤이 페이지의 다른 숫자를 집고 있다. 위 표 대조 |
| **불일치 (②↔③)** | 혜택이 빠졌거나 금액이 다름 | **계산·설정 문제** — 혜택 템플릿/오버라이드 확인 |
| **확인불가** | 크롤 데이터 없음 · 계산 실패 · 혜택 미입력 | ⚠️ **문제없음이 아니다.** 원인을 없애고 다시 검증 |

> ⚠️ **확인불가를 일치로 뭉개지 말 것.** 크롤 실패를 "문제없음" 으로 세면 그게 조용한 실패다.

## 미확인 항목 (채워야 할 숙제)

| 소싱처 | 무엇이 미확인인가 |
|---|---|
| 르무통 공홈 | 표면가의 화면 라벨 전체 |
| 무신사 | 표면가 정의 — 확장(salePrice) vs 가이드 문서(회원가) 충돌 |
| 롯데온 | 표면가 정의 — 판매가 / 즉시할인가 / 나의 혜택가 3개 충돌 |
| 현대H몰 | `bbprc` 의 화면 라벨 |
| 스마트스토어 르무통 | 표시 할인가의 화면 라벨명 (네이버는 캡션 없음) |

**이 표가 빌 때까지 검증은 반쪽이다.** 실제 페이지 대조로 하나씩 채울 것.
