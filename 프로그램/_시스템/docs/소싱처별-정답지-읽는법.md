# 소싱처별 「정답지 읽는 법」 — 이 화면의 이 숫자가, 이 상태에서 표면노출가다

> **계층**: 프로젝트 문서 (검증 실무 정본)
> **책임**: 소싱처마다 **① 페이지의 어느 숫자를 표면가로 읽는가** + **② 그때 무엇을 끄고/켜야 하는가**
> **단일 진실 원천**: 본 문서 (표면가 '읽는 법' + '캡처 조건' 한정)
> **관련**: `docs/크롤링-가이드.md` (크롤 전반) · `lemouton/sourcing/price_verify.py` (3층 대조 판정)
> **화면**: 소싱처 관리 → 🧾 최종매입가 검증 (`/sourcing-guide/price-verify`)

---

## 왜 이 문서가 있나

2026-07-19 기준, 가격 오류 6건 중 **3건이 코드 검증으로 안 잡히고 라이브 화면 대조에서만** 드러났다.
원인은 하나다 — **소싱처마다 표면가의 위치가 다른데 그걸 적어둔 곳이 없었다.**

가장 비쌌던 사고: **롯데아이몰**. 페이지에 크게 뜨는 「롯데홈쇼핑 최대할인가」를 표면가로 집었는데,
그 숫자는 **카드할인이 이미 먹은 값**이었다. 그래서 카드할인이 두 번 차감됐다.

**그리고 2026-07-19, 두 번째 축이 빠져 있었음이 드러났다.**

---

## ★ 표면가는 "숫자"가 아니라 "상태"다 (이 문서의 핵심)

표면노출가는 **페이지에 뜨는 숫자**가 아니다. **무엇을 끄고 켠 상태에서 뜨는 숫자**다.

무신사가 교과서적인 예다. 상품 페이지에는 이런 것들이 달려 있다:

| 페이지 요소 | 켜면 | 우리 정책 |
|---|---|---|
| **「적립금 사용」 체크박스** | 보유 적립금만큼 가격이 내려간다. **사이트 기본값이 ON** | **꺼야 한다** |
| **「적립금 선할인」 / 「구매 적립」 라디오** | 둘 중 뭘 고르냐에 따라 표시가·적립액이 달라진다 | **구매 적립 쪽** |
| **「나의 할인가」(회원가)** | 등급할인·적립이 모두 먹은 최종 표시값 | **표면가 아님** |

**이걸 켜둔 채로 표면가를 잡으면 → 혜택이 표면가에 이미 먹혀 있는데 우리가 또 뺀다 → 이중차감 → 언더프라이싱 → 금전 손실.**

그래서 이 문서는 소싱처마다 **두 가지**를 적는다:
1. 어느 숫자를 읽는가
2. **그 숫자를 읽을 때 무엇이 꺼져 있어야 하는가**

## 표면노출가(①)의 정의

**우리 계산엔진(`compute_breakdown`)에 `sale_price` 로 들어가는 값 = 베이스금액.**
여기서부터 우리가 혜택을 차감해 최종매입가를 만든다.

따라서 표면가는 **우리가 차감할 혜택이 아직 안 먹은 값**이어야 한다.

> 🚫 **헷갈리면 추측하지 말고 「미확인」으로 두고 보고할 것.**
> 이 표에 틀린 값을 적으면 그대로 돈이 샌다.

## 한눈에 보기 — 캡처 조건 요약

| 소싱처 | 읽는 값 | ★ 꺼야 하는 것 | 이미 선반영된 것 | 상태 |
|---|---|---|---|---|
| **무신사** | API `goodsPrice.salePrice` | **「적립금 사용」 체크박스 OFF** (기본 ON) · 「선할인」 대신 「구매적립」 · 결제수단 할인 무시 | 「쿠폰적용가」 라벨 뜨면 **쿠폰 선반영** | ⚠️ 정의 4개 충돌 |
| **롯데온** | 판매가 (`N% 가격원` 패턴) | 「나의 혜택가」를 잡지 말 것 (**롯데오너스 선반영**) | 스토어/롯데ON/무료배송 즉시할인 | ⚠️ 정의 3~4개 충돌 |
| **롯데아이몰** | 카드 미적용 할인가 (`benefitPrc` + 카드금액) | 「최대할인가」를 잡지 말 것 (**카드 선반영**) | 상품 일반할인만 | ✅ 확인 |
| **SSG** | 「최적가」 `bestAmt` | (조작 없음) | 즉시할인 + **SSG MONEY 즉시할인**(패턴 A/C) | ⚠️ 용어 충돌 |
| **SSF샵** | 「판매가」 `em.price` | (조작 없음) | 시즌 할인만 | ✅ 확인 |
| **현대H몰** | `itemPtc.bbprc` | (조작 없음) | 깜짝할인만 (**카드할인 미포함**) | ⚠️ 화면 라벨 미확인 |
| **스마트스토어 르무통** | `benefitsView.discountedSalePrice` | (조작 없음) | 즉시할인 (비회원 기준) | ⚠️ 화면 라벨 없음 |
| **르무통 공홈** | `strong.price-number` 계열 | (조작 없음) | 기본할인 %만 | ❌ 라벨 미확인 |

> **읽는 법**: 「꺼야 하는 것」이 **(조작 없음)** 인 소싱처는 크롤러가 페이지 토글을 건드리지 않는다는 뜻이지,
> **그 사이트에 토글이 없다는 뜻이 아니다.** 사람이 화면에서 대조할 때는 여전히 조심해야 한다.
> 실제로 토글을 조작하는 곳은 **무신사(적립금 사용 체크박스) 한 곳뿐**이다.

### ⚠️ 무신사 (musinsa) — 캡처 조건이 가장 복잡한 곳

- **표면노출가로 잡는 값**
  - **라이브(크롬확장) 경로** — 구조화 API `goodsPrice.salePrice` 직읽기
    · 근거 `extension/moum-crawler/background.js:1083-1108`
    · 검증 게이트: G1 `salePrice > 0` · G2 `salePrice ≤ normalPrice`. **통과 못 하면 `price=null`(크롤실패), 폴백 일절 금지** — `background.js:1114-1118`
  - **서버(Playwright) 경로** — DOM `[class*="CalculatedPrice"]` → `[class*="CurrentPrice"]`
    · 근거 `lemouton/sourcing/crawlers/musinsa_playwright.py:66-71`
    · ⚠️ 이 경로는 **화면 값을 읽으므로 아래 토글에 직접 영향받는다.**

- **★ 캡처 조건**
  | 항목 | 상태 | 크롤러가 실제로 하는 일 | 근거 |
  |---|---|---|---|
  | **「적립금 사용」 체크박스** | **OFF** ← 사이트 기본값이 **ON** | Playwright 가 `input[type=checkbox]` 중 라벨에 `적립금 사용` 이 있고 `checked` 면 **자동 클릭해 해제** | `musinsa_playwright.py:152-162` |
  | **「적립금 선할인」 / 「구매 적립」 라디오** | **구매 적립** 쪽 가정 | 끄지 않는다. **켜진 쪽을 감지해서 값을 잡는다** — "둘 중 하나라도 활성이면 적립%" | `musinsa_playwright.py:315-330` · `docs/sources/musinsa/profile.yaml:131` `purchase_radio_default: purchase_reward` |
  | **결제수단 즉시할인 (토스페이/카드)** | 무시 | 추출은 하되 매입가 산식에 **안 씀** | `musinsa_playwright.py:115` · `:812-815` · `profile.yaml:129` `include_card_discount: false` |
  | **결제수단 적립 (무신사 삼성카드 등)** | 무시 | 무신사머니만 인정 | `musinsa_playwright.py:116` |
  | **월 1회 정기쿠폰** | 제외 | 매입가 베이스에서 뺀다 | `docs/sources/musinsa/pricing_policy.yaml:131` |

  > 왜 「적립금 사용」을 꺼야 하나 — 켜져 있으면 **모든 적립 금액이 (적립금 차감 후 베이스) × % 로 표시**된다.
  > 즉 표면가뿐 아니라 그 아래 혜택 금액이 전부 오염된다. (`musinsa_playwright.py:153-155`)

  > 🔴 **이 해제는 "시도"일 뿐이다.** `docs/sources/musinsa/pricing_policy.yaml:128` 이 명시한다 —
  > 「적립금 사용 (코드가 체크박스 OFF 시도, **★ 셀렉터 미적중 알려진 이슈**)」.
  > 즉 **꺼지지 않은 채 크롤될 수 있다.**

- **이미 선반영된 것**
  - **「쿠폰적용가」 라벨이 뜬 상품** — 그때의 표시가는 **이미 특별쿠폰이 적용된 값**이다.
    · 감지: `musinsa_playwright.py:376-380` `is_already_coupon_applied`
    · 분기: `musinsa_playwright.py:838-848` — 라벨 있으면 쿠폰을 **또 빼지 않는다**
    · 실사례: 상품 4677240 — 220,320 이 「쿠폰적용가」로 노출, 이미 −55,080 반영 (`musinsa_playwright.py:378-379`)
  - 「나의 할인가」(회원가) = 등급할인·적립이 전부 먹은 값. **표면가 아님** — 확장은 참고용 `member_price` 로만 담는다 (`background.js:1086-1087`, `:1109-1112`)

- **로그인 의존**
  - **표면가(API salePrice) = 로그인 불필요** — `background.js:1086`
  - **혜택(등급할인·등급적립·무신사머니) = 로그인 필수** (비로그인 시 「나의 할인가」 영역 미노출) — `musinsa_playwright.py:381`
  - 서버 디스패처는 **로그인 못 하면 크롤을 막는다** (`LoginExpiredError`) — "비로그인 가격 = 잘못된 매입가 = 금전 손실" `crawlers/musinsa.py:273-275`

- **경유 의존**: 코드·문서 어디에도 없음 → **해당 없음 (전 저장소 `ckwhere` 검색 결과 SSG 전용)**

- **확인 상태**: ⚠️ **문서 4개 충돌** (아래 「문서 충돌」 §1)

### ✅ 롯데아이몰 (lotteimall) — 사고의 원점, 지금은 확정

페이지 가격 라인은 이렇게 쌓인다:

```
표면노출가 ①    116,900   ← ★ 이걸 읽어야 한다
  ↓ 삼성카드 7% 청구할인 −8,180
「롯데홈쇼핑 최대할인가」 108,720   ← ✗ 이건 표면가가 아니다
```

- **표면노출가로 잡는 값**: **카드 미적용 할인가** = `dataBenefit.data.commonDiscountObj.benefitPrc` **+** `fullDiscountObj.cardDiscountList[0].discountAmount`
  · `crawlers/lotteon.py:410-480` (`_resolve_surface_price`), 카드 있을 때 분기 `:456-458`, 카드 없을 때 `:474-479`
- **★ 캡처 조건**: 페이지 토글 조작 없음. **카드 청구할인이 이미 먹은 값을 받아 카드금액을 되더해 복원**하는 방식
  - ⚠️ **나눗셈 역산 금지** — 사이트가 10원 절사를 하므로 `price/(1-rate)` 는 원 단위가 어긋난다. 금액이 없으면 **실패를 택한다** (`lotteon.py:438-439`, 테스트 `tests/sourcing/test_lotteimall_surface_price.py:170-185`)
  - **폴백 금지** — 역산 실패 시 `RuntimeError` (`lotteon.py:1478-1486`)
- **이미 선반영된 것**: **상품 일반할인까지만.** 카드 청구할인은 선반영 아님(되더해 뺐음)
  - 이중차감 방지 근거 `api_benefits.py:846-849`, 테스트가 `included_in_sale_price is False` 를 잠금 (`test_lotteimall_surface_price.py:140`)
- **로그인 의존**: 비로그인 서버 크롤 (`크롤링-가이드.md:55` SAMEORIGIN_FETCH). 등급별 차이 ❓미확인
- **경유 의존**: ❓미확인 (근거 없음)
- **확인 상태**: ✅ **확인** (코드·`pricing_policy.yaml`·테스트 3자 일치). 단 `크롤링-가이드.md:144` 만 stale

## 검증할 때 이렇게 쓴다

1. 소싱처 관리 → **🧾 최종매입가 검증** 열기
2. 소싱처 + 상품 URL 넣고 **불러오기** → ②(우리 수집)·③(우리 계산)이 자동으로 채워짐
3. 실제 페이지를 열고 **위 표대로** 표면가를 찾아 ① 에 입력
   - ★ **무신사라면 「적립금 사용」 체크박스를 먼저 끄고** 숫자를 읽을 것
   - ★ **롯데온·롯데아이몰이라면 「나의 혜택가」·「최대할인가」가 아닌 위쪽 숫자**를 읽을 것
4. 가능하면 혜택 항목별 금액도 입력 (안 넣으면 그 층은 **확인불가** — 일치로 안 침)
5. **판정하고 저장** → 어느 층에서 갈렸는지 자동으로 짚어줌

| 판정 | 뜻 | 다음 행동 |
|---|---|---|
| **일치** | ①=②, ② 혜택=③ 차감 | 없음 |
| **불일치 (①↔②)** | 표면가가 다름 | **크롤 파싱 문제** — 크롤이 페이지의 다른 숫자를 집고 있다. 위 표 대조 |
| **불일치 (②↔③)** | 혜택이 빠졌거나 금액이 다름 | **계산·설정 문제** — 혜택 템플릿/오버라이드 확인 |
| **확인불가** | 크롤 데이터 없음 · 계산 실패 · 혜택 미입력 | ⚠️ **문제없음이 아니다.** 원인을 없애고 다시 검증 |

> ⚠️ **확인불가를 일치로 뭉개지 말 것.** 크롤 실패를 "문제없음" 으로 세면 그게 조용한 실패다.
