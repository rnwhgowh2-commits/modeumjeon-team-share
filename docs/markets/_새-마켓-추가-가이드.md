# 새 판매처(마켓) 추가 가이드 — 정본

> **계층**: docs/markets 정본 (프롬프트가 이 문서를 지목해 읽게 함)
> **책임**: 신규 마켓 "업로드 연동"을 새 세션이 맥락 없이도 완성하도록 하는 단일 진실 원천
> **상위 의존**: CLAUDE.md 데이터 정합성 3대 원칙 · docs/markets/_schema.yaml
> **짝**: `webapp/templates/marketplace_guide/add.html` 위저드가 생성하는 프롬프트가 이 문서를 가리킴

판매처 관리 화면의 「판매처 추가·업데이트」 위저드가 만드는 프롬프트를 새 Claude Code 세션에 붙여넣으면, 그 세션은 **이 문서를 그대로 읽고** 아래 순서대로 신규 마켓 업로드 연동을 완성한다.
`docs/markets/{slug}.yaml` 하나만 쓰는 게 아니다 — **코드 배선까지 해야 실제로 작동**한다.

경로는 모두 저장소 루트 기준. 코드 파일은 `프로그램/_시스템/` 아래. file:line 은 origin/main 기준(2026-07-05 확인)이며 드리프트 시 실제 코드를 신뢰한다.

## 진행 방식 (가장 먼저 읽기) · 사용자를 단계별로 인도하며 점진 구현

이 연동은 대부분 **사용자에게서 받아야만 하는 정보(마켓 API 스펙·키)** 에 의존한다. 그리고 **사용자는 비개발자이고 API를 모를 수 있다**("난 정보 모르고 시키는 대로 할게"). 따라서 이 작업은 "혼자 다 만들고 끝"이 아니라 **사용자를 한 단계씩 인도하는 대화형 컨덕터**로 진행한다.

**핵심 규칙**
- **"스펙 주세요" 한 번 던지고 멈추지 말 것.** 사용자가 무엇을·어디서·어떻게 가져오는지 모른다고 가정하고, **정확한 화면·버튼·복사할 대상**을 짚어 지시한다.
  - (예) "롯데ON 스토어센터에 로그인 → 좌측 **판매자정보 > OpenAPI관리 > [정보설정]** 화면을 여시고, 그 페이지 텍스트를 통째로 복사해 붙여주세요."
  - 단, **화면 경로는 예시(작성 시점 기준)일 뿐 확정 사실로 단정하지 말 것.** 마켓 UI는 바뀐다 → "못 찾으시면 어디서 막히는지 알려주세요, 같이 찾겠습니다"로 폴백을 함께 준다.
- **한 번에 한 가지만 요청한다.** 여러 개를 한꺼번에 요구해 사용자를 압도하지 말 것.
- **받은 조각으로 그 슬라이스만 구현**하고 → "여기까지 됐습니다. 다음은 ○○이 필요합니다" 로 **요약 보고 + 다음 요청 1개** → 다음 단계로. 사용자를 막연히 기다리게 두지 말고 **능동적으로 이끈다.**
- **키의 실제 값(시크릿)은 채팅에 받지 않는다.** env 키 "이름"만 확인하고, 실제 값은 사용자가 직접 `.env`(또는 계정 키 입력 폼)에 넣도록 안내한다.

**단계 체크포인트** (각 단계 끝 = 요약 + 다음 요청 1개)
1. **슬러그·메타 확인** — `MARKET_METADATA` 의 기존 키 확정(STEP 0), 사용자에게 어떤 마켓인지 재확인.
2. **인증 스펙 받기 → `auth` 구현** — 인증 방식·헤더 형식·토큰 발급 경로를 사용자가 붙여준 스펙에서 확정 후 `shared/platforms/{slug}/auth.py` 작성.
3. **옵션 조회 스펙 받기 → `connect`(기존 상품 연동) 구현** — 상품번호로 옵션 조회·자동매칭.
4. **가격·재고 스펙 받기 → `update` 구현** — 어댑터 `update_price_and_stock`.
5. **키 입력 안내** — 키 이름 확정 후 사용자가 값을 직접 입력(채팅 금지). 키저장 UI가 열리도록 `MARKET_KEY_SUFFIXES` 배선이 선행돼야 함.
6. **실계정 검증** — 아래 "검증 프로토콜"대로. 키 없으면 Mock 단위테스트까지 + `라이브 미검증` 표기 + 중단·보고.
7. **`status` → ready** — 검증 통과 후에만 노출 전환.

**금지** — 사용자가 모른다는 이유로 값(base_url·엔드포인트·키)을 지어내기 / 스펙을 못 받았는데 그 단계를 건너뛰고 앞서 나가기.

## STEP 0 · 필수 선행 (건너뛰면 전부 어긋남)

1. **슬러그(영문 키)는 새로 짓지 말고 기존 것을 그대로 쓴다.**
   - `webapp/routes/accounts.py` 의 `MARKET_METADATA` 에 이미 등록된 키를 쓴다. (예: 롯데온 = `lotteon`, 11번가 = `eleven11`, 옥션 = `auction`, G마켓 = `gmarket`)
   - 새 슬러그를 지어내면 모든 딕셔너리 조회(`MARKET_SCHEMAS[slug]`, `MARKET_KEY_SUFFIXES[slug]`, `c_output[slug]`)가 **조용히 빗나가** 아무 것도 안 되고 에러도 안 난다.
   - `default_prefix`(예: `LOTTEON_MAIN`)도 그 항목에서 확인해 env 키 접두로 쓴다.
   - **목록에 아예 없는 완전 신규 마켓(임의의 판매처)이면** → 먼저 `MARKET_METADATA` 에 새 항목(`label`·`icon`·`api_type`·`guide_url`·`default_prefix`·`status:"coming_soon"`·`sort_order`)을, `MARKET_KEY_SUFFIXES`·`KEY_LABELS` 에 그 마켓 키 목록을 **직접 추가**한 뒤 그 슬러그를 쓴다. 슬러그는 영문 소문자 한 단어(예: 위메프=`wemakeprice`). 이걸 먼저 안 하면 계정·키저장·데이터지도가 그 마켓을 인식하지 못한다.

2. **이름 충돌 주의 — 소싱처 vs 판매처.**
   - 같은 슬러그가 소싱처(크롤·읽기)에도 있을 수 있다. 예: `lemouton/sourcing/crawlers/lotteon.py`, `lemouton/auth/scrapers/lotteon.py` 는 **롯데온을 "구매자 입장에서 크롤"** 하는 코드다.
   - 이 가이드는 **판매처(셀러 업로드 API)** 쪽이다. 소싱 크롤 코드는 업로드 API 근거로 쓰지 말 것(방향이 반대).

3. **API 문서·키가 없으면 추측하지 말고 사용자에게 요청한다.**
   - 위저드 프롬프트의 "문서: (조사 필요)" / "필요 키: (조사 필요)" 는 대개 비어 있다.
   - 마켓의 셀러센터 OpenAPI **문서 URL · 인증 방식 · env 키 이름**을 저장소에서 얻을 수 없으면 → **사용자에게 직접 요청**한다. `api_type: "셀러센터 API"` 같은 라벨은 인증 방식이 아니다.
   - CLAUDE.md 원칙: 추측·폴백 금지. base_url·엔드포인트·키를 지어내지 않는다.

4. **문서 URL을 받았는데 기계로 못 읽으면(SPA·로그인 게이트) — 그래도 추측 금지, 사용자에게 스펙을 붙여넣어 달라 요청한다.**
   - 셀러 API "문서"는 대개 JS로 렌더되는 API 센터거나 로그인 뒤에 있다. WebFetch 로 열면 본문 없이 `Loading…` 만 온다(예: 롯데온 `https://api.lotteon.com/` = "롯데ON API 센터" SPA).
   - 이때 화면에 보이는 조각으로 엔드포인트·인증을 **유추하지 말 것**. 사용자에게 **해당 API 스펙 페이지(인증 헤더 형식 · 옵션조회/가격수정/재고수정 엔드포인트 경로 · 응답의 옵션ID·색·사이즈 필드)를 텍스트로 붙여넣어 달라**고 요청한다.
   - 키 발급 경로도 대개 로그인 안쪽이다(롯데온 예: 스토어센터 로그인 → 판매자정보 > OpenAPI관리 > [정보설정] > [키발급]). env 키 이름을 여기서 확인해 사용자에게 확인받는다.
   - 스펙을 확보하기 전에는 `{slug}.yaml` 의 auth/endpoints/fields 를 `TODO(조사필요)` 로 두고, 코드 배선(어댑터 등)은 **뼈대만** 만들지 실제 호출 로직은 채우지 않는다.

## 데이터 코드 지도 채우기 (그 마켓 공식 API 전수)

이 마켓이 **공식 개발문서로 제공하는 API 전부**를 지도에 넣는다. 우리가 구현한 것만이 아니라 **미구현까지 카탈로그**로 — 그래야 무엇이 되고 무엇이 남았는지 보인다.

- **위치**: `map.html` 의 `MKT_CATS[<슬러그>]` 배열. 왼쪽 영역탭 = **그 마켓 공식 문서의 카테고리 그대로**(임의 분류 금지).
- **항목 구분자(`code`)** = 그 마켓의 **공식 API 식별자**(네이버=operationId 슬러그 `find-settle-by-case-pay-settle`, 롯데온=`apiNo=94`, 쿠팡=엔드포인트 경로). 이래야 "정확히 들어갔는지" 검증된다.
- **상태(`st`) 4단계**: `ok`(됨=실사용) / `code`(연결됨=우리 코드 있음·검증대기) / `off`(공식제공=마켓 문서엔 있으나 우리 코드 없음) / `todo`(미정의=확인 안 됨).
- **방향(`dir`)**: 프로그램→마켓 `send`, 마켓→프로그램 `recv`.

**수집 방법(문서가 로그인 SPA라 서버가 못 읽음 — 사용자 브라우저로 추출):**
1. 사용자에게 그 마켓 개발문서를 열게 한다.
2. **콘솔(F12) 추출 스니펫**을 만들어 준다 — 사이드바를 전부 펼친 뒤 `카테고리+API이름 || 슬러그`를 뽑아 클립보드 복사(스크린샷 금지·텍스트). 붙여넣기 길이 제한으로 잘리면 **콘솔 출력 뒷부분만** 이어서 받는다.
3. 받은 텍스트를 `MKT_CATS[<슬러그>]` 에 카테고리별로 채운다. 아직 항목 못 받은 카테고리는 빈 배열(→ 화면에 "공식 문서에서 가져오는 중" 자리표시).
4. **추측 금지** — 못 받은 엔드포인트를 지어내지 않는다.

## 검증 프로토콜 ("완료" 선언 조건)

- **실계정 키가 있으면**: 토큰 발급 → 옵션 조회 → 가격·재고 라운드트립을 **실계정 100%** 로 확인. `scripts/verify_smartstore_read.py` 패턴으로 `scripts/verify_{slug}_read.py` 를 만들어 조회부터 실증한다.
- **키가 없으면(정상적인 경우 — 쿠팡도 라이브 미검증 상태)**:
  1. 어댑터 + 등록/조회 래퍼를 구현하고,
  2. `Mock{Slug}Adapter` 로 `tests/uploader/` 단위테스트를 통과시키고,
  3. yaml 에 `evidence: 라이브 미검증(키없음) — 코드 근거만` 을 표기하고,
  4. `LEMOUTON_LIVE_UPLOAD` 는 **OFF 유지**(`uploader/runtime.py` 게이트 — 켜면 실제 전송),
  5. **중단하고 사용자에게 보고**한다. **절대 "완료" 선언 금지**(가격·재고 오류 = 금전 손실, CLAUDE.md 3대 원칙).
