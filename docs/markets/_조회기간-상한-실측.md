# 판매처 조회기간 상한 — 라이브 실측 결과

> **계층**: 판매처 API 문서 / **책임**: 주문·클레임 조회기간 상한의 단일 진실 원천
> **상위 의존**: `webapp/data/marketplace_api_map.json` (공식문서 접수분)
> **측정일**: 2026-07-20 · **방법**: 라이브 실계정 단발 프로브(`lemouton/markets/period_probe.py`)
> **재측정**: 저장소 변수 `PERIOD_PROBE_ARMED=1` → 재배포 → `/api/period-probe/sweep`

## 왜 이 문서가 필요했나

「최대 얼마나 조회되나」에는 서로 다른 두 질문이 섞여 있었다.

| 구분 | 뜻 | 문서 근거 |
|---|---|---|
| **① 1회 조회 창** | 한 번 호출에 며칠치 | 대부분 있음 |
| **② 과거 상한** | 얼마나 옛날까지 (= 샵마인 1년치) | **378개 API 전체에 0건** |

②는 API 지도에도 코드에도 근거가 없어 **실측 외에 확인 방법이 없었다.**
기존 `iter_*` 들은 윈도우가 하드코딩이라 경계 측정에 쓸 수 없다(쪼갠 창은 항상 상한 이내).

## ① 1회 조회 창 — 전부 실측 확정

| 마켓 | 주문 | 클레임 | 초과 시 반응 |
|---|---|---|---|
| 쿠팡 | **31일** | 반품·취소 **31일** / 교환 **7일** | 400 에러 (명시) |
| 스마트스토어 | **1일(24시간)** | 전용 API 없음(주문 피드가 겸함) | 400 `104140` |
| 11번가 | **7일** | 7일 (문서는 30일이나 **늘리면 안 됨** ↓) | 🔴 **에러 없이 0건** |
| 롯데온 | **1일** | **30일** | `returnCode 2003 / 2004` |
| 옥션 | **180일** | **7일** | `ResultCode 3000 / 2000` |
| G마켓 | **31일** | **7일** | `ResultCode 3000 / 99999` |

### 실측이 뒤집은 것

- **옥션 180일 = 맞다.** 181일에서 `3000 "180일 이하의 기간만 조회 할 수 있습니다"`.
  API 지도의 옥션 엔트리(31일)는 G마켓 문서를 복사해 넣은 것이라 **틀렸다.**
  같은 엔드포인트인데 siteType 별로 상한이 다르다(옥션 180 / G마켓 31).
- **쿠팡 반품·취소 = 31일.** v4·v6 둘 다 32일에서 `검색기간은 최대 31일입니다`.
  코드의 `claims.py:_MAX_WINDOW_DAYS = 7` 은 **교환(7일)의 값이 혼입된 것**으로 보인다.
  31일로 올리면 호출 수가 약 1/4.4 로 준다(정합성 문제는 아니고 낭비).
- **스마트스토어 = 24시간.** 공식문서엔 상한 문구가 없고 "생략 시 24시간"만 있었는데,
  실제로는 24시간이 **상한**이다. 2일부터 `104140 조회 날짜가 유효하지 않습니다`.

### 🔴 11번가 — 상한을 에러가 아니라 「빈 응답」으로 알린다

판매완료 조회 실측(같은 시각, 창 크기만 변경):

| 창 | 6일 | 7일 | 7.5일 | 8일 | 9일 | 14일 | 31일 |
|---|---|---|---|---|---|---|---|
| 건수 | 14 | **15** | 15 | **0** | 0 | 0 | 0 |

문서상 `-3904 "최대 조회기간은 일주일"` 이 떠야 하는데 **에러가 없다.**
따라서 누군가 `_MAX_WINDOW_DAYS` 를 문서값(30일)으로 올리면
**에러 없이 주문이 통째로 사라진다.** 게다가 `order_export.py` 는 클레임 조회 실패를
조용히 스킵하므로 엑셀은 정상으로 보인다. **11번가 창은 절대 넓히지 말 것.**

## ② 과거 상한 — 하드 컷은 롯데온 클레임 하나뿐

| 마켓 | 결과 | 근거 |
|---|---|---|
| **롯데온 클레임** | 🔴 **정확히 1년** | 350일 전 7건 → **365일 전 `3000` 거부** |
| 롯데온 주문 | 365일까지 거부 없음 | 건수 판정 불가(응답 구조상) |
| 쿠팡 주문 | 1,095일(3년)까지 거부 없음 | 180일 이전 0건 — 판매 이력 문제로 보임 |
| 쿠팡 클레임 | 365일까지 거부 없음 | 90일 이전 0건 |
| 11번가 | 730일까지 거부 없음 · **300일 전까지 데이터 확인** | 330일 이후 0건 연속 |
| 스마트스토어 | 730일까지 거부 없음 | 1일 창이라 이력 측정 불가 |
| 옥션 | 730일까지 거부 없음 · 180일 전까지 데이터 확인 | 365일 이후 0건 |

**결론: 롯데온 클레임(1년) 말고는 어느 마켓도 과거 상한을 걸지 않는다.**
0건 구간은 대부분 그 시기에 판매가 없었던 것이다 — 11번가에서 178~185일 전이 0건이었다가
**200일 전에서 다시 4건**이 나온 것이 그 증거다(컷오프였다면 되살아날 수 없다).

> ⚠️ **해석 규칙**: `accepted + 0건` 은 「마켓이 거부하지 않았다」는 뜻일 뿐
> 「그 시기 데이터가 나온다」는 뜻이 아니다. 이 둘을 섞으면 없는 상한을 만들어낸다.

## 🔴 진짜 병목은 마켓이 아니라 우리 앱

`webapp/routes/orders.py:160` 이 **마켓 능력과 무관하게 전 마켓 90일로 강제 절단**한다.

```python
if (d2 - d1).days > 90:            # 상한 90일
    d1 = d2 - timedelta(days=90)
```

11번가 클레임은 30일 창을 12번 돌리면 1년치가 나오고, 롯데온 클레임도 1년까지 열려 있다.
「샵마인은 1년치도 나온다」의 답은 마켓 API 제약이 아니라 **이 90일 클램프**다.

## 남은 미확정

- 11번가 **클레임** 창의 실제 상한 — 조회 기간에 클레임 데이터가 0건이라 판정 불가.
  주문과 같은 「조용한 0건」 동작이면 문서값 30일은 함정이다. 클레임이 쌓인 시점에 재측정 필요.
- 롯데온 **정산** 29일 — 클레임 문서값에서 상속받은 값이고 정산 API 자체 근거는 없다.
- 쿠팡·스마트스토어·옥션의 과거 상한 — 거부는 없으나 계정 판매 이력이 짧아 하한 미도달.
