2. 롯데온 → `lrtrNo`(하위거래처번호) 파라미터를 안 봐서, 정산 엑셀의 **하위거래처번호**
   `LO10161082` 를 `trNo`(거래처번호) 자리에 넣었다 → `9999 인증정보 불일치`.
   당연한 거부인데 "우리가 그 거래처 키를 안 갖고 있다"고 오진했다.
3. 쿠팡·스마트스토어 → 우리 DB 연동 이력으로 상품을 구해서, **목록 API 를 아예 안 찾았다**.
   실제로는 둘 다 있다.

**지도(`marketplace_api_map.json`)는 처음부터 답을 갖고 있었다.**
새 작업은 반드시 `req.params` + `fields` 를 열어보고 시작할 것.

## 🔴 방법론 함정 (반드시 지킬 것)

1. **순차 호출로는 한계점에 못 닿는다.** 쿠팡 200회 연속에 429 가 0건이었는데, 관측 속도가
   5.1 콜/s 였다 — 마켓 한도가 아니라 **왕복지연(~196ms)이 만든 천장**이었다.
   동시 호출(`load`)을 넣자 9.6 콜/s 에서 429 가 나왔다. **「한도 없음」으로 오판할 뻔했다.**
2. **429 만 한도로 센다.** 403(IP 미등록)·401·500 을 429 로 읽으면 **없는 상한을 날조**한다.
3. **`data: null` 을 「0건」으로 읽지 말 것.** 실제로 롯데온 인증 오류(9999)를
   "상품 0건"으로 읽고 "뚫렸다"고 잘못 보고했다. `raw=1` 로 원본을 확인할 것.
4. **함수 안 import 는 테스트가 안 잡는다.** `fetch_stocks`·`_site_qty_of`·`fetch_orders`
   (진짜 이름 `iter_orders`) 를 없는 이름으로 적었다가 실재 검사 테스트로 잡았다.
   `tests/markets/test_upload_rate_probe*.py` 의 심볼 실재 테스트를 유지할 것.

## 안전 규칙 (코드로 강제됨)

- **무변화 갱신** — 현재 재고를 읽어 그 값 그대로 다시 쓴다. 재고가 안 바뀐다.
- 현재 재고를 못 읽으면 **쓰지 않는다**(원복 보장 불가)
- **가격은 어떤 인자로도 안 넘어간다**(스스는 `sale_price=None` = 현재값 유지)
- 매 요청이 끝나고 재고를 **다시 읽어** 확인 → 다르면 `restored=false` + 경고
- 요청당 호출 400, 동시 32, 지속 30초 상한 / 킬스위치 `STOP` 파일 / 429 연속 20회 중단
