2. 핸드오프 — B(대기 큐) 2단계

프로그램이 Claude를 호출하지 않음. URL은 큐가 운반, 사용자는 이름만 말함.

🧑 1단계 — 웹 (즉시)

→ 전체보기에 바로 등장(미정의·0/6)

🤖 2단계 — Claude Code

사용자: "29CM 신규 추가해줘"
→ 큐에서 URL + 첨부 체크리스트 회수

→ 완료 시 카드 채움 + "정의됨 ✓"
💡 큐 = 거창한 테이블 없이 "URL은 있는데 가이드 카드가 빈 소싱처 = 분석 대기"로 식별.

3. 신규 추가 vs 기존 업데이트 — 방향이 다름

➕ 신규 = 발굴 + 생성

출발점 지침서 §1 재고 · §2 가격
일 사이트 첫 분석(API/HTML) → 6항목 위치 발굴 → 매입가 로직 → 크롤러 신규 작성 → 검증 → §1·§2 표에 행 신설
결과 새 crawler.py · 새 행 · 새 카드

🔄 기존 = 진단 + 교정

출발점 지침서 §5 에러 이력(증상→원인)
일 증상 확인 → 라이브와 diff → 바뀐 셀렉터·로직만 수정 → 재검증 → 기존 행·카드 갱신
결과 기존 crawler.py 수정 · 값만 교체
공통 마지막 관문 = mou-m.com 라이브 화면과 100% 일치 (§4 무결성).

📄 지침서 = 프로그램/_시스템/docs/크롤링-가이드.md (데이터·코드 지도 SSOT, 배포 포함)

⚠️ 함정 — 지금 로컬 브랜치엔 옛 사본(루트 docs/, 207줄)이 있고 라이브는 442줄(에러 41건). 작업은 반드시 origin/main 기준(워크트리)에서.
