# 소싱처 관리 — 아키텍처 & 레이아웃 설계

> 2026-06-06. 사용자 요청: 기능 기준으로 전체 아키텍처·레이아웃을 설계자가 결정해 구성.
> 확정 요구사항: 정확한 매입가(가격+혜택) · 조립식 신규추가 · 혜택 정규화+동기화 · 데이터연계 · 기능1(레시피)·기능2(가격룰→fx).
> 관계: [[project_crawl_chrome_extension]](크롤=확장), [[project_url_store_fragmentation]](기존 3중분열 통합 대상).

## 정보구조 (IA)
```
[소싱처 관리]  (신규 top-nav, 기존 소싱처 사전/운영/계정 흡수)
   ├ 소싱처 목록 = 건강 대시보드      ← 홈. 문제(⚠️)가 위로. [+신규추가]
   ├ 소싱처 상세 (한 화면 2단)
   │     좌: 기능1 크롤링 레시피 (상품명/가격/옵션·재고/썸네일/상세 + 테스트)
   │     우: 기능2 가격룰 (기준가 + 혜택블록 → 매입가 + fx 미리보기)
   │           └ 혜택 동기화는 우측 혜택블록에 인라인 배지(일치자동/신규확인/소멸경고)
   └ 신규 추가 = 모달 흐름 (URL → AI초안 → 빈칸확인 → 저장)

[모음전 편집]  (기존, 소싱처 관리 '밖')
   └ 옵션별 소싱처 선택 = 레지스트리에서 read (체크 + URL)
```

## 설계 결정 (왜)
1. 소싱처 = 목록+상세 2단계. 동기화·레시피·가격룰을 별도 화면으로 쪼개지 않고 상세 한 화면에 → 탭 지옥 방지.
2. 상세 2단 = 좌(수집=기능1) / 우(가격=기능2). 사용자 멘탈모델 "어떻게 긁고 → 얼마에 매입" 직선.
3. 동기화는 독립 화면 ❌ → 우측 혜택블록 옆 배지. 값 자동, 신규·소멸만 사람 확인.
4. 목록 = 건강 대시보드(⚠️ 위로). 매일 여기만 봐도 됨.
5. 역할 분리: 소싱처 관리=정의 / 모음전=사용. 데이터연계는 레지스트리 단일 진실원천으로.

## 데이터 모델(예정)
- source_registry(레시피 JSON + 가격룰 JSON + 혜택블록 JSON + 동기화상태)
- 모음전 옵션 ↔ source_registry 참조(옵션별 선택 URL)
- 크롤 결과 + 크롤된 실제혜택 → 동기화 엔진이 혜택블록과 매핑
