# 신규 소싱처 추가 가이드 (Claude 실행 SOP)

> **계층**: 본 문서 = "신규 소싱처 추가" 절차의 **단일 진실원천(SOP)**.
> 화면(`/sourcing-guide/how-to`, 시안 E 순서도)은 이 문서의 **요약 시각화**일 뿐 — 정식 절차는 항상 본 문서를 따른다.
> **상위 의존**: [[소싱처-크롤링-가이드-design.md]](스펙), [[소싱처관리_아키텍처.md]](IA), [[project_crawl_chrome_extension]](크롤=로그인 확장).

---

## 0. 사용법 (사용자 → Claude)

사용자는 다음 **한 문장**만 말하면 된다:

> **"신규 소싱처 추가 가이드 참고해서 〇〇 소싱처 추가해줘"**

그러면 Claude는 본 문서의 **6단계**를 순서대로 자동 수행한다.
사용자가 제공할 것은 **STEP 1의 샘플 URL 1개**뿐. 나머지(2~6)는 Claude가 처리한다.

---

## 1. 6단계 절차 (시안 E = 분기 순서도)

```
STEP 1  샘플 URL 1개                         [사용자]
   │
   ▼
STEP 2  ◇ 어떻게 수집? — API 있나 / 로그인 필요한가   [Claude]
   │     → 4분면 중 하나로 분류 (아래 §3)
   ▼
STEP 3  항목별 위치 추출 (6항목 셀렉터/엔드포인트)     [Claude]
   ▼
STEP 4  매입가 로직 정리 (혜택 순차 차감)             [Claude]
   ▼
STEP 5  검증 크롤 → 정답값 대조                       [Claude]
   ▼
STEP 6  가이드(정답) 카드 저장 → 복제 가능            [Claude]
```

### STEP 1 · 샘플 URL 1개 (사용자)
- 사용자가 대표 상품 **상세 페이지 URL 1개** 제공. (가능하면 검증용 1~2개 더)

### STEP 2 · 수집 방식 분석 (Claude) — 분기점 ◇
- 개발자도구 **Network 탭**으로 그 사이트가 데이터를 어떻게 내려주는지 확인.
- 두 질문으로 **4분면**(§3) 중 하나로 분류:
  1. **API 있나?** → 내부 API(JSON)로 받는가, 아니면 HTML을 직접 파싱해야 하는가.
  2. **로그인 필요한가?** → 회원가·혜택을 보려면 로그인 세션(쿠키/토큰)이 필요한가.

### STEP 3 · 항목별 분류·위치 추출 (Claude)
- 6항목 각각을 **2축으로 분류** + 위치 확정 (상세 기준 = §2.C):
  - **수집 방식**: `html`(HTML 스크래핑) / `api`(API 크롤링) / `manual` / `none`
  - **인증**: `open`(비인증) / `auth`(인증=로그인 세션 필요)
  - **위치(셀렉터/엔드포인트)** + 보조로 목록/상세.
- 항목: 썸네일 · 상품명 · 가격(표면가) · 혜택 · 옵션·재고 · 상세이미지.

### STEP 4 · 매입가 로직 정리 (Claude)
- 표면 노출가에서 혜택을 **순차 차감**하는 규칙 정의.
- 공식: **최종 매입가 = 표면가 − Σ(혜택)**. 각 정률/적립/결제 %는 **직전 차감 후 잔액**에 곱한다.
- 혜택 방식 enum: `rate`(정률%) · `accrue`(적립%) · `amount`(정액원) · `amount_or_rate`(정액·정률) · `payment`(결제).
- 혜택 적용 enum: `always`(상시) · `conditional`(조건부) · `optional`(선택) · `planned`(예정).

### STEP 5 · 검증 크롤 → 정답 대조 (Claude)
- 샘플 URL을 실제 크롤(로그인 세션/확장 경로 재사용)해서 **표면가 / 혜택합 / 최종가 / 옵션·재고**를 뽑는다.
- 사용자가 실제 화면에서 본 값과 **100% 일치**하는지 대조. 불일치 시 STEP 2~4로 되돌아가 셀렉터·로직 수정.

### STEP 6 · 가이드 카드 저장 (Claude)
- 위 결과를 `SourceRegistry.crawl_guide` JSON(스키마 = design.md §4)에 저장 → "정답 카드" 완성.
- 이후 다른 소싱처도 동일 절차로 복제.

## 2. 항목별 수집 양식 (A~E) — Claude가 채워 보고

신규 소싱처를 추가할 때 Claude는 아래 A~E를 채워 사용자에게 보고한다.

### A. 기본 정보
- 소싱처명 / 대표 샘플 URL / 추가 샘플 URL

### B. 수집 분류 (4분면 위치) — §3 중 하나

### C. 항목별 수집 방법

각 항목을 **2축(수집 방식 + 인증)** 으로 분류해 기록한다. 단순 "crawl"로 뭉뚱그리지 않는다.

| 항목 | 수집 방식 | 인증 | 위치(셀렉터/엔드포인트) | 비고(목록/상세 등) |
|---|---|---|---|---|
| 썸네일 | HTML/API | 비인증/인증 | | |
| 상품명 | HTML/API | 비인증/인증 | | |
| 가격(표면가) | HTML/API | 비인증/인증 | | |
| 혜택 | HTML/API | 비인증/인증 | | |
| 옵션·재고 | HTML/API | 비인증/인증 | | |
| 상세이미지 | HTML/API | 비인증/인증 | | |

**수집 방법 분류 기준 (전문 용어 — Claude가 채움):**
- **수집 방식** (`mechanism`):
  - `html` = **HTML 스크래핑** — 화면(DOM)을 직접 파싱. 셀렉터 사용.
  - `api` = **API 크롤링** — 내부 API(JSON) 엔드포인트 직접 호출. 빠르고 안정적 → 선호.
  - `manual` = **수동** — 자동 수집 불가, 사람이 직접 입력.
  - `none` = **없음** — 수집 안 함.
- **인증** (`auth`):
  - `open` = **비인증** — 로그인 없이 값이 보임.
  - `auth` = **인증** — 로그인 세션(쿠키/토큰) 있어야 값을 줌 (회원가·혜택 등).
- **범위** (혜택 전용, 기존): `per_product`(상품별 크롤) · `uniform`(일괄).
- **위치**(보조, 비고에 표기): `목록` / `상세` — 정확도가 아닌 속도/부담 정보라 별도 칸 없이 비고로.

> 예) 무신사 혜택 = **`api` · `auth` · `per_product` · 상세** → "API 크롤링 · 인증 · 상품별 · 상세 진입".

### D. 최종 매입가 로직
| 혜택명 | 혜택 방식 | 계산 규칙 | 적용 |
|---|---|---|---|
| | rate/accrue/amount/payment | | always/conditional/optional |

### E. 검증 정답값 (정답 카드)
- 표면 노출가 / 적용 혜택(−) / 최종 매입가 / 옵션·재고  ← 실제 화면값과 일치 확인

---

## 3. 4분면 분류 (어떻게 긁나 × 로그인 필요한가)

| | 비인증(공개) | 인증 필요(로그인) |
|---|---|---|
| **API 방식** | 가장 쉬움. 로그인 없이 내부 API(JSON). | 로그인 세션 + 내부 API. (예: **무신사** = 회원가·혜택형) |
| **HTML 방식** | API 없음. 공개 화면 파싱. | 가장 까다로움. 로그인 + 화면 파싱. |

- **API 크롤링** = 데이터를 나르는 내부 요청(엔드포인트)을 직접 호출 → 빠르고 안정적 → **선호**.
- **HTML 스크래핑** = 사람이 보는 화면(DOM)을 파싱 → 느리고 화면 바뀌면 깨짐.

## 4. 전문 용어 사전

| 용어 | 뜻 |
|---|---|
| HTML 스크래핑 / DOM 파싱 | 화면(HTML)을 그대로 읽어 추출 |
| API 크롤링 (내부 API 호출) | 데이터를 나르는 요청을 직접 호출 |
| 엔드포인트 (Endpoint) | 데이터를 받아오는 주소 (예: `/api/goods/{id}`) |
| 네트워크 요청 (XHR/Fetch) | 페이지가 뒤에서 서버에 보내는 데이터 요청 |
| JSON 응답 | 그 요청이 돌려주는 데이터 덩어리 |
| 개발자도구 Network 탭 | 주고받는 요청을 들여다보는 곳 (숨은 API 찾는 장소) |
| 인증 (세션·쿠키·토큰) | 로그인 상태 증명 — 있어야 회원가·혜택을 줌 |
| SSR / 정적 렌더링 | HTML에 값이 이미 박혀서 옴 → 긁기 쉬움 |
| CSR / 동적 렌더링 | 데이터가 나중에 따로 불려와 채워짐 → API를 잡아야 함 |
| 셀렉터 (Selector) | HTML 안에서 값의 위치 지정 (예: `.price-area .amount`) |

## 5. 무결성 규칙

- 본 문서가 절차의 **단일 진실원천**. 화면(순서도)·design.md 스키마와 모순되면 본 문서·design.md를 우선하고 사용자에게 보고.
- 검증(STEP 5)에서 **실제 화면값과 100% 일치** 전에는 "추가 완료" 선언 금지.
- 가격·재고 데이터 오류 = 금전 손실 → 정답 대조 필수.
