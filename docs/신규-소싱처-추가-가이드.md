# 신규 소싱처 추가 가이드 (Claude 실행 SOP)

> **계층**: 본 문서 = "신규 소싱처 추가" 절차의 **단일 진실원천(SOP)**.
> 화면(`/sourcing-guide/how-to`, 시안 E 순서도)은 이 문서의 **요약 시각화**일 뿐 — 정식 절차는 항상 본 문서를 따른다.
> **상위 의존**: [[소싱처-크롤링-가이드-design.md]](스펙), [[소싱처관리_아키텍처.md]](IA), [[project_crawl_chrome_extension]](크롤=로그인 확장).

## 0. 사용법 (사용자 → Claude)

사용자는 다음 **한 문장**만 말하면 된다:

> **"신규 소싱처 추가 가이드 참고해서 〇〇 소싱처 추가해줘"**

그러면 Claude는 본 문서의 **7단계**를 순서대로 자동 수행한다.
사용자가 제공할 것은 **STEP 1의 샘플 URL 세트**뿐. 나머지(2~7)는 Claude가 처리한다.

## 1. 7단계 절차 (시안 E = 분기 순서도)

```
STEP 1  샘플 URL 세트 (대표 1개 + 변형 2~5개)         [사용자]
   │
   ▼
STEP 2  ◇ 어떻게 수집? — API 있나 / 로그인 필요한가   [Claude]
   │     → 4분면 중 하나로 분류 (아래 §3)
   ▼
STEP 3  항목별 위치 추출 (6항목 셀렉터/엔드포인트)     [Claude]
   ▼
STEP 4  혜택 종합 — 여러 URL 교차분석                 [Claude]
   │     상시(always) / 조건부(conditional) 분류
   ▼
STEP 5  매입가 로직 정리 (혜택 순차 차감)             [Claude]
   ▼
STEP 6  검증 크롤 → 정답값 대조                       [Claude]
   ▼
STEP 7  가이드(정답) 카드 저장 → 복제 가능            [Claude]
```

### STEP 1 · 샘플 URL 세트 (사용자)
- 사용자가 대표 상품 **상세 페이지 URL 세트**(대표 1개 + 변형을 덮는 2~5개)를 제공한다.
- URL 세트가 필요한 이유 — 소싱처 안에서도 상품마다 혜택·재고·옵션 구조가 다를 수 있기 때문에, 여러 URL을 교차 분석해야 패턴 전체를 잡을 수 있다. 세트를 고를 때 아래 3가지 축을 의식한다:
  - **혜택 변형** — 세일·쿠폰·후기적립이 상품마다 켜지거나 꺼질 수 있으므로, 혜택이 다르게 붙는 상품을 최소 1개씩 포함한다.
  - **재고 3상태** — 품절 상품·한정수량(N개 남음) 상품·충분 상품을 각 1개씩 커버해 3상태 파싱이 실제로 동작하는지 확인한다.
  - **옵션 구조** — 단일색·멀티색·딜 묶음처럼 상품마다 옵션 표시 방식이 다를 때, 구조별로 최소 1개를 포함한다.

### STEP 2 · 수집 방식 분석 (Claude) — 분기점 ◇
- 개발자도구 **Network 탭**으로 그 사이트가 데이터를 어떻게 내려주는지 확인.
- 두 질문으로 **4분면**(§3) 중 하나로 분류:
  1. **API 있나?** → 내부 API(JSON)로 받는가, 아니면 HTML을 직접 파싱해야 하는가.
  2. **로그인 필요한가?** → 회원가·혜택을 보려면 로그인 세션(쿠키/토큰)이 필요한가.

### STEP 3 · 항목별 분류·위치 추출 (Claude)
- 6항목 각각을 **2축으로 분류** + 위치 확정 (상세 기준 = §2.C):
  - **수집 방식**: `html`(HTML 스크래핑) / `api`(API 크롤링) / `manual` / `none`
  - **인증**: `open`(비인증) / `auth`(인증=로그인 세션 필요)
  - **위치(셀렉터/엔드포인트)** + 보조로 목록/상세.
- 항목: 썸네일 · 상품명 · 가격(표면가) · 혜택 · 옵션·재고 · 상세이미지.
- ⚠️ **옵션·재고는 위치만 적고 끝내지 않는다.** 재고는 **3단계(품절 / 한정수량 / 표식없음)** 를 각각 어떤 신호로 잡는지 §2.C-재고 규칙으로 명시한다. (이 단계 누락이 "한정재고가 '재고있음'으로 둔갑"하는 조용한 실패의 원인.)

### STEP 4 · 혜택 종합 — 여러 URL 교차분석 (Claude)
- STEP 1에서 받은 URL 세트 전체를 훑어 각 URL에서 발견된 혜택 항목을 **합집합(전체 혜택 카탈로그)**으로 정리한다.
- 그런 다음 URL 간 출현 빈도를 비교해 두 가지로 분류한다:
  - **상시(always)** — 확인한 **모든 URL**에 등장하는 혜택. 특별 조건 없이 항상 적용된다.
  - **조건부(conditional)** — **일부 URL**에만 등장하는 혜택. 어떤 조건(세일 기간·쿠폰 코드·결제 수단 등)이 갖춰질 때만 켜진다. 이 경우 트리거 키워드(예: "기간한정", "신규고객", "특정카드")를 카드에 함께 기록한다.
- 합집합이 곧 STEP 5에서 순차 차감할 전체 혜택 목록이 된다. 단, 조건부 혜택은 카드의 `status=conditional`, `triggers=[키워드]`, `match=조건설명` 필드에 명시해 두고, 실제 계산 시 조건 충족 여부를 확인한다.
- ⚠️ 여러 URL 없이 단 1개 URL만 보면 조건부 혜택이 상시 혜택처럼 보이거나(과대계산) 반대로 놓칠 수 있다(과소계산). URL 세트가 필요한 핵심 이유.

### STEP 5 · 매입가 로직 정리 (Claude)
- 표면 노출가에서 혜택을 **순차 차감**하는 규칙 정의.
- 공식: **최종 매입가 = 표면가 − Σ(혜택)**. 각 정률/적립/결제 %는 **직전 차감 후 잔액**에 곱한다.
- 혜택 방식 enum: `rate`(정률%) · `accrue`(적립%) · `amount`(정액원) · `amount_or_rate`(정액·정률) · `payment`(결제).
- 혜택 적용 enum: `always`(상시) · `conditional`(조건부) · `optional`(선택) · `planned`(예정).

### STEP 6 · 검증 크롤 → 정답 대조 (Claude)
- 샘플 URL을 실제 크롤(로그인 세션/확장 경로 재사용)해서 **표면가 / 혜택합 / 최종가 / 옵션·재고**를 뽑는다.
- 사용자가 실제 화면에서 본 값과 **100% 일치**하는지 대조. 불일치 시 STEP 2~5로 되돌아가 셀렉터·로직 수정.
- **검증 체크리스트(§2.F)를 항목별로 통과**시킨다. "최종가만 맞으면 끝"이 아니라 **수집(collect) → 가공(process) → 전송(transmit)** 3단계를 항목별로 확인한다. 특히 재고는 한정수량 N이 끝까지 보존되는지(둔갑 없음)까지 본다.

### STEP 7 · 가이드 카드 저장 (Claude)
- 위 결과를 `SourceRegistry.crawl_guide` JSON(스키마 = design.md §4)에 저장 → "정답 카드" 완성.
- 이후 다른 소싱처도 동일 절차로 복제.

## 2. 항목별 수집 양식 (A~E) — Claude가 채워 보고

신규 소싱처를 추가할 때 Claude는 아래 A~E를 채워 사용자에게 보고한다.

### A. 기본 정보
- 소싱처명 / 대표 샘플 URL / 추가 샘플 URL

### B. 수집 분류 (4분면 위치) — §3 중 하나

### C. 항목별 수집 방법

각 항목을 **2축(수집 방식 + 인증)** 으로 분류해 기록한다. 단순 "crawl"로 뭉뚱그리지 않는다.

| 항목 | 수집 방식 | 인증 | 위치(셀렉터/엔드포인트) | 비고(목록/상세 등) |
|---|---|---|---|---|
| 썸네일 | HTML/API | 비인증/인증 | | |
| 상품명 | HTML/API | 비인증/인증 | | |
| 가격(표면가) | HTML/API | 비인증/인증 | | |
| 혜택 | HTML/API | 비인증/인증 | | |
| 옵션·재고 | HTML/API | 비인증/인증 | | |
| 상세이미지 | HTML/API | 비인증/인증 | | |

**수집 방법 분류 기준 (전문 용어 — Claude가 채움):**
- **수집 방식** (`mechanism`):
  - `html` = **HTML 스크래핑** — 화면(DOM)을 직접 파싱. 셀렉터 사용.
  - `api` = **API 크롤링** — 내부 API(JSON) 엔드포인트 직접 호출. 빠르고 안정적 → 선호.
  - `manual` = **수동** — 자동 수집 불가, 사람이 직접 입력.
  - `none` = **없음** — 수집 안 함.
- **인증** (`auth`):
  - `open` = **비인증** — 로그인 없이 값이 보임.
  - `auth` = **인증** — 로그인 세션(쿠키/토큰) 있어야 값을 줌 (회원가·혜택 등).
- **범위** (혜택 전용, 기존): `per_product`(상품별 크롤) · `uniform`(일괄).
- **위치**(보조, 비고에 표기): `목록` / `상세` — 정확도가 아닌 속도/부담 정보라 별도 칸 없이 비고로.

> 예) 무신사 혜택 = **`api` · `auth` · `per_product` · 상세** → "API 크롤링 · 인증 · 상품별 · 상세 진입".

#### C-재고. 옵션·재고의 3단계 캡처 규칙 (필수 — 누락 시 금전 손실)

재고는 "있다/없다"가 아니라 **3단계**다. 소싱처마다 표기·필드가 달라서, **각 단계를 어떤 신호로 잡는지** 카드에 적어야 한다 (`fields.option_stock.stock_rules`).

| 단계 | 의미 | 저장값 | 카드 필드 | 소싱처별 예시 |
|---|---|---|---|---|
| ① 품절 | 못 산다 | `0` (품절) | `soldout_markers[]` | 무신사 `품절`/`재입고 알림` · SSF `statcd=SLDOUT` · SSG `usablInvQty=0` · 르무통 `is_selling=F` · 롯데 `disabled` |
| ② 한정수량 | N개 남음 | `N` (실수량) | `qty_patterns[]` | 무신사 `잔여 N개`·`N개 남음`·`마지막 N개` · SSF `품절임박 (N)` · SSG·르무통 = 실수량 필드 |
| ③ 표식 없음 | 충분 | 센티넬→`재고있음` | `no_marker_means` | `in_stock`(충분) / `unknown`(수량 못 주는 사이트=롯데) |

**철칙 — ②와 ③을 절대 혼동하지 말 것.**
- 사이트에 한정수량 **표기가 있는데** 크롤러가 그 표기를 못 읽어 ③(충분)으로 떨어지면 = **버그**(한정재고가 "재고있음"으로 둔갑 → 오발주 → 손실).
- 그래서 `qty_patterns[]` 에 그 소싱처가 쓰는 **한정수량 표기를 빠짐없이** 적는다. 한 가지(`잔여 N개`)만 적고 `N개 남음`을 빠뜨리면 그게 곧 둔갑 버그다 (실제 무신사 사례, 2026-06-13).
- 수량을 **구조적으로 안 주는** 사이트(롯데온·롯데홈쇼핑)는 `no_marker_means="unknown"` 으로 명시 → "수량 미상"으로 정직하게 표시(가짜 충분 금지).

> 예) 무신사 옵션·재고 = `api`·`auth` / `soldout_markers=["품절","재입고 알림"]` / `qty_patterns=["잔여 N개","N개 남음","마지막 N개"]` / `no_marker_means="in_stock"`.

|---|---|---|---|

### E. 검증 정답값 (정답 카드)
- 표면 노출가 / 적용 혜택(−) / 최종 매입가 / 옵션·재고  ← 실제 화면값과 일치 확인

### F. 검증 체크리스트 — 수집 · 가공 · 전송 (필수)

"최종가만 맞으면 끝"이 아니다. 중요 데이터(특히 재고)는 **동시·무결성(integrity) → 수집(collect) → 가공(process) → 전송(transmit)** 어느 단계에서든 조용히 틀어질 수 있다. 신규 소싱처는 아래 항목을 **하나씩 pass** 로 만들어야 "추가 완료". (`verification.checklist`, 화면에도 체크리스트로 노출.)

핵심 3대 CHECK 요약 — **[동시]** 빠르게 많이 긁어도 정확·어긋나면 실패·재크롤 시 옛값 리셋 / **[가격]** 모든 가격 수집 + 로직 정확 반영 / **[재고]** 없음·한정·표시없음 3단계 정확(표시 없는 게 정말 없는 건지 확인).

| 단계 | 항목 | 확인 내용 |
|---|---|---|
| **동시** | 동시·고속 정확 | 여러 상품 빠르게 긁어도 상품 간 값 안 섞이고 정확 |
| **동시** | 로직 불일치 = 실패 | 수집값이 가이드 로직과 안 맞으면 **크롤실패** 처리 (엉뚱한 값 저장·성공 위장 금지 = **손실 방지**) |
| **동시** | 재크롤 시 옛값 리셋 | 모음전 재크롤 시 그 상품 가격·재고 **먼저 리셋** (옛 데이터 잔존 → 그 값으로 판매되면 **오발주 치명적 손실**) |
| **수집** | 상품명 / 표면가·정가·혜택금액 / 혜택 / 옵션(색×사이즈) | 모든 가격 정보 포함 6항목이 실제로 긁히는가 |
| **수집** | 품절 | 품절 마커 인식 → 재고 `0` |
| **수집** | 한정수량 | 표기(잔여·N개 남음·마지막·품절임박 등) **전부** → 실수량 `N` |
| **수집** | 표식없음 | **표식이 정말 없을 때만** 충분(재고있음) |
| **가공** | 순차 차감 | 표면가 − Σ혜택 (직전 잔액 기준) |
| **가공** | 센티넬 해석 | 999·cap 을 '재고있음'/'N개'로 올바로 구분 |
| **가공** | 폴백가 금지 | 매칭/크롤 실패 시 평균·최저 폴백가 금지 → 가격없음 표면화 |
| **전송** | 재고 보존 | 크롤 수량이 DB→매트릭스 표시까지 둔갑 없이 보존 |
| **전송** | 비활성 누출 | OFF 옵션에 가격·재고 누출 없음 |
| **전송** | 최종가 일치 | 매트릭스 표시 최종가 = 실제 화면값 100% |

> 이 체크리스트의 단일 진실원천 = `crawl_guide.py`의 `_CHECKLIST_TEMPLATE`. 표 항목과 코드가 어긋나면 코드를 우선하고 본 문서를 정정.

## 3. 4분면 분류 (어떻게 긁나 × 로그인 필요한가)

| | 비인증(공개) | 인증 필요(로그인) |
|---|---|---|
| **API 방식** | 가장 쉬움. 로그인 없이 내부 API(JSON). | 로그인 세션 + 내부 API. (예: **무신사** = 회원가·혜택형) |
| **HTML 방식** | API 없음. 공개 화면 파싱. | 가장 까다로움. 로그인 + 화면 파싱. |

- **API 크롤링** = 데이터를 나르는 내부 요청(엔드포인트)을 직접 호출 → 빠르고 안정적 → **선호**.
- **HTML 스크래핑** = 사람이 보는 화면(DOM)을 파싱 → 느리고 화면 바뀌면 깨짐.

---

## 4. 전문 용어 사전

| 용어 | 뜻 |
|---|---|
| HTML 스크래핑 / DOM 파싱 | 화면(HTML)을 그대로 읽어 추출 |
| API 크롤링 (내부 API 호출) | 데이터를 나르는 요청을 직접 호출 |
| 엔드포인트 (Endpoint) | 데이터를 받아오는 주소 (예: `/api/goods/{id}`) |
| 네트워크 요청 (XHR/Fetch) | 페이지가 뒤에서 서버에 보내는 데이터 요청 |
| JSON 응답 | 그 요청이 돌려주는 데이터 덩어리 |
| 개발자도구 Network 탭 | 주고받는 요청을 들여다보는 곳 (숨은 API 찾는 장소) |
| 인증 (세션·쿠키·토큰) | 로그인 상태 증명 — 있어야 회원가·혜택을 줌 |
| SSR / 정적 렌더링 | HTML에 값이 이미 박혀서 옴 → 긁기 쉬움 |
| CSR / 동적 렌더링 | 데이터가 나중에 따로 불려와 채워짐 → API를 잡아야 함 |
| 셀렉터 (Selector) | HTML 안에서 값의 위치 지정 (예: `.price-area .amount`) |
