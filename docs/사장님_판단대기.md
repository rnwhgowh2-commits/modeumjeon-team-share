## 🔴 지금 막고 있는 것

### 1. 크롤 계수 제안을 실제로 적용할까

**상황** — ① 데이터수집 탭이 「이렇게 바꾸는 게 좋겠습니다」를 보여주기만 합니다.
사장님 결정 5-B(확인 후 적용)대로 만들었는데, **적용 버튼이 아직 없습니다.**

**정할 것** — 적용 버튼을 붙일까요, 아니면 데이터가 더 쌓인 뒤에 붙일까요?

> **추천 — 데이터가 쌓인 뒤.** 지금 라이브는 구성별 관측이 10~27회라 최소 30회에 못 미쳐
> 등급이 하나도 안 나옵니다. 버튼을 먼저 만들어도 누를 게 없습니다.

## 🟡 판단이 필요하지만 급하지 않은 것

### 2. 추천기가 둘인데 하나만 남겨야 합니다

같은 「계수를 몇으로 할까」를 두 방식이 서로 다르게 답합니다.

| | 방식 | 어디 |
|---|---|---|
| 기존 | 변동률 → 계수 (경계 2/5/10/20%) | `recommend_weight` · 랩 보고서 화면 |
| 신규 | 강도 → 6등급 → 하루 N회 | `crawl_grade` · ① 데이터수집 탭 |

같은 아이디어를 다른 단위로 쓴 것이라 충돌은 아닌데, **둘을 같이 띄우면 서로 다른 숫자를
권해 혼선**이 납니다.

> **추천 — 신규(등급)로 통일.** 사장님이 7항목을 직접 정하신 체계이고,
> 기존 경계값(2/5/10/20%)은 코드 주석에도 "실측 없이 정한 출발점"이라 적혀 있습니다.
> 다만 기존 화면(랩 보고서)을 고쳐야 하니 손이 좀 갑니다.

### 3. 상품별 기록을 추가할까 (500MB 한도와 맞바꿈)

시안 11 의 **「상품별 기간 산포」**(상품 하나하나의 추세)는 **지금 데이터로 못 만듭니다.**
`CrawlChangeStat` 이 (랩, 소싱처, 브랜드) 집계라 상품별 행이 없습니다 —
Supabase 무료 티어 500MB 때문에 **일부러** 그렇게 설계된 것입니다.

**정할 것** — 상품별 기록을 새로 쌓을까요?

> **추천 — 일단 보류.** 구성 단위로 한 달 돌려보고, 그래도 상품별이 필요하면 그때.
> 쌓기 시작하면 되돌리기 어렵고 용량이 빠르게 찹니다.

### 3-1. 🔴 「1초에 10개」는 지금 구조로 설정할 수 없습니다

**③ 데이터전송 탭을 만들며 드러났습니다.**

속도 제한은 **이미 있습니다.** 다만 저장 방식이 사장님이 말씀하신 것과 다릅니다.

```
지금:  계정별 「1개당 몇 초」  (AccountUploadPolicy.seconds_per_item, 최소 1초)
       → 한 계정은 초당 1개가 최대
       → 마켓 처리량 = 켜진 계정 수 × 초당 1개
```

**「1초에 10개」를 하려면 계정이 10개 있어야 합니다.** 사장님이 원하신 「X초에 Y개」를
그대로 담으려면 저장 방식을 바꿔야 합니다.

| 선택지 | |
|---|---|
| **가.** 지금 방식 유지 | 계정 수로 조절. 코드 변경 없음. 다만 계정 없이는 못 빠르게 함 |
| **나.** 「X초에 Y개」 칸 신설 | 사장님 뜻대로. 컬럼 추가 + 페이서 수정 필요 |
| **다.** 최소 1초 제한만 풀기 | 0.1초/개 = 초당 10개. 가장 작은 변경 |

> **추천 — 다.** `max(1, int(...))` 를 실수 허용으로 바꾸면 「초당 N개」가 바로 됩니다.
> 크롤 계수 때와 같은 유형(정수 절삭)이라 고치는 방법도 같습니다.
> ⚠️ 다만 **라이브 업로드 속도**를 건드리는 일이라 승인이 필요합니다.

지금은 화면이 이 사실을 그대로 말합니다 — 「한 계정은 초당 1개가 최대」.

### 4. 마켓별 실제 API 한도 — 일부는 이미 우리 문서에 있었습니다

**조사 완료 (2026-07-19).** `marketplace_api_map.json`(API 610건)을 훑었습니다.
제가 설계서에 적었던 「스스 1초에 10개」 같은 숫자는 **임의값이었고, 근거가 없었습니다.**

**실제로 문서에 있던 한도**

| 마켓 | API | 한도 |
|---|---|---|
| 쿠팡 | 로켓그로스 주문 조회 | **분당 50회** |
| 옥션 · G마켓 (ESM) | 주문 조회 (esm.66 · 67) | **5초당 1회** ⚠️ 매우 빡빡 |

**아직 모르는 것** — 스마트스토어 · 롯데온 · 11번가, 그리고 **상품/가격/재고 업로드 API**
한도. 위 둘은 전부 **주문 조회** 한도지 업로드 한도가 아닙니다.

> ⚠️ **ESM 5초당 1회**는 지금 구조(계정당 초당 1개)로도 **너무 빠릅니다.**
> 옥션·G마켓에 주문 조회를 붙일 때 반드시 확인해야 합니다.
>
> **추천** — 남은 마켓은 공식 개발자센터 문서를 봐야 하는데, 대부분 **로그인이 필요**합니다.
> 사장님이 문서를 열어주시면 제가 읽고 정리하겠습니다.
> 그 전까지는 **설계서의 임의 숫자를 지우고 「미확인」으로 두었습니다** — 임의값이 남아 있으면
> 나중에 그게 근거인 줄 알고 쓰게 됩니다.
