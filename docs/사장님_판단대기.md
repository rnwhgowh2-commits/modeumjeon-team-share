## 🔴 지금 막고 있는 것

### 1. 계수 적용 — 「소싱처 × 브랜드」를 담을 자리가 없습니다

**버튼은 만들었습니다** (사장님 4번 = 나). 누르면 **무엇이 바뀌는지 먼저 보여주고**,
확인해야 저장됩니다. 다만 만들다 보니 틈이 하나 나왔습니다.

**제안은 `(소싱처 × 브랜드)` 단위인데, 계수 규칙 스코프는 넷뿐입니다.**

```
url · model · brand · source     ← 「소싱처 × 브랜드」 조합이 없음

brand 로 걸면   → 그 브랜드가 다른 소싱처에도 걸림
source 로 걸면  → 그 소싱처의 다른 브랜드까지 걸림
```

**지금 처리** — `brand` 로 걸되, 그 브랜드가 **여러 소싱처에 걸쳐 있으면 경고**를 띄웁니다.
「나이키는 SSG 에도 있어 같이 바뀝니다」처럼요. 막지는 않습니다 — 알고 누르시면 됩니다.

**정할 것** — 이대로 둘까요, 스코프를 새로 만들까요?

| | |
|---|---|
| **가** | **지금대로 (brand + 경고)** ← 추천 |
| **나** | `소싱처×브랜드` 스코프 신설 |

> **가를 추천합니다.** 나는 `resolve_crawl_weight` 의 **상속 순서를 건드려야** 하고,
> 그건 라이브 크롤 스케줄러를 또 고치는 일입니다. 경고로 충분히 막힙니다.
>
> 참고 — 지금은 관측이 10~27회라 **등급이 안 나와서 누를 게 없습니다.** 급하지 않습니다.

### 2. 🔴 가공 규칙 13항목 — 각 항목에 무엇을 넣을지

**② 데이터가공 탭의 마지막 조각입니다.** 정책을 만들고 URL·마켓을 붙이는 것까진 됩니다.
**항목별 규칙 편집칸**만 비어 있습니다 — 항목마다 설정 모양이 전부 달라서
사장님이 실제로 뭘 넣으실지 모르면 만들 수가 없습니다.

**제가 짐작한 것 (맞는지 봐주세요)**

| 항목 | 넣을 것 (짐작) |
|---|---|
| 상품명 | 치환표(A→B) · 금지어 목록 · 접두/접미 · 최대 글자수 |
| 카테고리 | 소싱처 카테고리 → 마켓 카테고리 코드 매핑표 |
| **판매가** | **마진율 %** 또는 고정 금액. ★ 최종매입가 기준 (확정) |
| 옵션 | 색상×사이즈 전개 방식 · 제외할 옵션 |
| 이미지 | 대표 1 + 추가 N장 · 순서 · 워터마크 제거 여부 |
| 상세설명 | 템플릿 HTML · 원문 삽입 위치 |
| 배송 | 배송비 · 무료배송 기준 · 출고지 |
| 고시정보 | 상품군별 기본값 (이미 4종 빌더 있음) |
| 브랜드 · 원산지 | 원문 그대로 vs 고정값 |
| KC인증 | 해당없음 / 인증번호 |
| 금지어 | 마켓별 금지어 목록 (상품명과 분리?) |
| 태그 | 자동 생성 개수 · 고정 태그 |

**정할 것 3가지**

1. **위 짐작이 맞습니까?** 빠진 항목·틀린 항목을 짚어주세요.
2. **어느 항목부터 필요합니까?** 13개를 한 번에 만들 필요는 없습니다.
   실제로 쓰실 순서대로 3~4개씩 나눠 만드는 게 안전합니다.
3. **마켓마다 다른 규칙이 필요합니까?** 지금은 정책 하나에 규칙 한 벌입니다.
   「스스는 상품명 100자, 쿠팡은 50자」처럼 갈라야 하면 구조가 달라집니다.

> **추천 — 상품명 · 판매가 · 옵션 3개만 먼저.**
> 이 셋이 없으면 등록 자체가 안 되고, 나머지는 기본값으로도 굴러갑니다.
> 특히 **판매가**는 금전 사고와 직결되니 제일 먼저 정확히 잡는 게 좋습니다.

## 🟡 판단이 필요하지만 급하지 않은 것

### 3. 추천기가 둘인데 하나만 남겨야 합니다

같은 「계수를 몇으로 할까」를 두 방식이 서로 다르게 답합니다.

| | 방식 | 어디 |
|---|---|---|
| 기존 | 변동률 → 계수 (경계 2/5/10/20%) | `recommend_weight` · 랩 보고서 화면 |
| 신규 | 강도 → 6등급 → 하루 N회 | `crawl_grade` · ① 데이터수집 탭 |

같은 아이디어를 다른 단위로 쓴 것이라 충돌은 아닌데, **둘을 같이 띄우면 서로 다른 숫자를
권해 혼선**이 납니다.

> **추천 — 신규(등급)로 통일.** 사장님이 7항목을 직접 정하신 체계이고,
> 기존 경계값(2/5/10/20%)은 코드 주석에도 "실측 없이 정한 출발점"이라 적혀 있습니다.
> 다만 기존 화면(랩 보고서)을 고쳐야 하니 손이 좀 갑니다.

### 4. 상품별 기록을 추가할까 (500MB 한도와 맞바꿈)

시안 11 의 **「상품별 기간 산포」**(상품 하나하나의 추세)는 **지금 데이터로 못 만듭니다.**
`CrawlChangeStat` 이 (랩, 소싱처, 브랜드) 집계라 상품별 행이 없습니다 —
Supabase 무료 티어 500MB 때문에 **일부러** 그렇게 설계된 것입니다.

**정할 것** — 상품별 기록을 새로 쌓을까요?

> **추천 — 일단 보류.** 구성 단위로 한 달 돌려보고, 그래도 상품별이 필요하면 그때.
> 쌓기 시작하면 되돌리기 어렵고 용량이 빠르게 찹니다.

### 5. 🔴 「1초에 10개」는 지금 구조로 설정할 수 없습니다

**③ 데이터전송 탭을 만들며 드러났습니다.**

속도 제한은 **이미 있습니다.** 다만 저장 방식이 사장님이 말씀하신 것과 다릅니다.

```
지금:  계정별 「1개당 몇 초」  (AccountUploadPolicy.seconds_per_item, 최소 1초)
       → 한 계정은 초당 1개가 최대
       → 마켓 처리량 = 켜진 계정 수 × 초당 1개
```

**「1초에 10개」를 하려면 계정이 10개 있어야 합니다.** 사장님이 원하신 「X초에 Y개」를
그대로 담으려면 저장 방식을 바꿔야 합니다.

| 선택지 | |
|---|---|
| **가.** 지금 방식 유지 | 계정 수로 조절. 코드 변경 없음. 다만 계정 없이는 못 빠르게 함 |
| **나.** 「X초에 Y개」 칸 신설 | 사장님 뜻대로. 컬럼 추가 + 페이서 수정 필요 |
| **다.** 최소 1초 제한만 풀기 | 0.1초/개 = 초당 10개. 가장 작은 변경 |

> **추천 — 다.** `max(1, int(...))` 를 실수 허용으로 바꾸면 「초당 N개」가 바로 됩니다.
> 크롤 계수 때와 같은 유형(정수 절삭)이라 고치는 방법도 같습니다.
> ⚠️ 다만 **라이브 업로드 속도**를 건드리는 일이라 승인이 필요합니다.

지금은 화면이 이 사실을 그대로 말합니다 — 「한 계정은 초당 1개가 최대」.

### 6. 마켓별 실제 API 한도 — 일부는 이미 우리 문서에 있었습니다

**조사 완료 (2026-07-19).** `marketplace_api_map.json`(API 610건)을 훑었습니다.
제가 설계서에 적었던 「스스 1초에 10개」 같은 숫자는 **임의값이었고, 근거가 없었습니다.**

**실제로 문서에 있던 한도**

| 마켓 | API | 한도 |
|---|---|---|
| 쿠팡 | 로켓그로스 주문 조회 | **분당 50회** |
| 옥션 · G마켓 (ESM) | 주문 조회 (esm.66 · 67) | **5초당 1회** ⚠️ 매우 빡빡 |

**아직 모르는 것** — 스마트스토어 · 롯데온 · 11번가, 그리고 **상품/가격/재고 업로드 API**
한도. 위 둘은 전부 **주문 조회** 한도지 업로드 한도가 아닙니다.

> ⚠️ **ESM 5초당 1회**는 지금 구조(계정당 초당 1개)로도 **너무 빠릅니다.**
> 옥션·G마켓에 주문 조회를 붙일 때 반드시 확인해야 합니다.
>
> **추천** — 남은 마켓은 공식 개발자센터 문서를 봐야 하는데, 대부분 **로그인이 필요**합니다.
> 사장님이 문서를 열어주시면 제가 읽고 정리하겠습니다.
> 그 전까지는 **설계서의 임의 숫자를 지우고 「미확인」으로 두었습니다** — 임의값이 남아 있으면
> 나중에 그게 근거인 줄 알고 쓰게 됩니다.

### 7. H몰·롯데아이몰 「보류」를 풀까요

**코드 문제가 아니었습니다.** 오늘 끝까지 따라가 확인했습니다.

사장님이 「H몰·롯데아이몰은 표면가와 최종매입가 구분이 안 보인다」고 하셨는데,
**화면도 엔진도 멀쩡합니다.**

| 확인한 것 | 결과 |
|---|---|
| 매트릭스 2단 표기 (표면 취소선 / 최종 굵게) | ✅ **이미 있음** (2026-06-24 반영) |
| 문자열 키 소싱처(`key:hmall`) 혜택 로딩 | ✅ **이미 고침** (M1-6) |
| **H몰·롯데아이몰 혜택 데이터** | ❌ **일부러 안 넣음** |

혜택이 0이면 최종매입가 = 표면가라 **2단이 뜰 이유가 없습니다.** 화면은 「같으면 한 줄」로
일부러 만들어져 있습니다(중복·모순 방지).

**왜 안 넣었나** — 코드에 사장님 지시가 그대로 적혀 있습니다.

```
H몰(현대H몰) 2.7% / 롯데홈쇼핑 2.5%(=롯데아이몰) — 사장님 보류 지시
카드 청구할인: 라이브 확인된 건은 롯데아이몰 삼성 7% 하나뿐인데
              그 소싱처가 보류 대상 → 시드 비워둠
```

**정할 것** — 이 보류를 풀까요?

> **추천 — 푸시는 게 맞습니다.** 지금은 두 소싱처의 매입가가 **실제보다 비싸게** 잡혀 있습니다.
> 매입가가 비싸게 잡히면 마진이 작게 보여 **팔 수 있는 걸 안 파는** 손해가 납니다.
> (반대로 싸게 잡히면 역마진이라 더 위험하니, 지금 방향 자체는 안전한 쪽입니다.)
>
> 풀기로 하시면 넣을 값 — H몰 캐시백 2.7% · 롯데아이몰 캐시백 2.5% ·
> 롯데아이몰 삼성카드 청구할인 7%. **맞는지 확인만 해주시면** 바로 넣겠습니다.
