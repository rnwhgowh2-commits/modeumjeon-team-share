## 1. 사용자 확정 결정 (그대로 지킬 것)

- **범위 = 원본 전 탭 1:1** — 요약·전체내역·일별·월별·브랜드별·금액대별·상품별·마켓별·소싱처별 + **블랙스팟** + **설정**. 화면·분류·표·버튼·색까지 원본과 동일.
- **데이터 = 마켓 API** (엑셀 아님). 원본은 브라우저에서 2엑셀로 계산 → 우리는 API 산 데이터를 **원본 JS 가 기대하는 형태로** 서버가 만들어 준다.

- **소싱처 자동확인 = 로컬 크롬확장 경유** (서버 Playwright 금지 — 크롤=로컬 원칙). UI·버튼은 원본 그대로, 실제 확인만 확장이 수행.
- **교환(교환완료/교환요청) = 정산O** (원본 classifier 정책 그대로). 셀러화면 대조는 후속.
- **팀 공유 DB 저장** — 원본의 flat JSON·localStorage 단일사용자 저장을 팀 DB 로 옮긴다.

## 2. ★핵심 전략 — "원본 JS 를 그대로, 서버가 원본 데이터 형태를 공급"

원본 프런트는 `window.analysisData` 하나를 읽어 모든 탭을 렌더한다(`/api/analyze` 결과). 그래서:

1. **원본 프런트 이식** — 원본 `templates/index.html` 의 렌더 함수 전부 + 그 CSS + 보조 JS 를 모음전 마진 탭 정적자산으로 그대로 옮긴다. **탭마다 손으로 다시 쓰지 않는다** — 원본 코드를 옮겨야 100% 동일.
2. **`/api/margin/analyze` 가 원본 `analysisData` 형태를 반환** — matched + **classified**(블랙스팟 분류 결과) + summary(+`_card_keywords`) + card_* 등. 이미 이식된 `matcher`·`aggregator` + 이식할 `classifier` 가 채운다.
3. 데이터 소스만 API (이미 됨).

이 방식이라 "각 탭 재현"이 아니라 "원본 프런트 통째 이식 + 데이터 계약 맞추기" 두 갈래로 단순해진다.
