# 마켓 정산금 계산 — 데이터맵 신설 + 롯데온 실검증 (설계)

> 계층: 스펙 (docs/superpowers/specs) · 2026-07-15
> 상위: 마진계산기 이식(2026-07-10) · 관련: 정산 엑셀 정답지(사용자 제공)

## 1. 목적

마진계산기의 **최종정산액**을 각 마켓이 실제 지급하는 정산금과 **오차 0**으로 맞춘다.
방법은 샵마인과 동일 — 주문 데이터 + 수수료율로 **계산**한다(미정산 주문 포함).

부수 산출물: 판매처 데이터·코드 지도(`marketplace_guide/map.html`)에 **'정산금 계산' 영역탭**을
신설해, "어느 API의 어느 필드로 어떤 값을 얻는지"를 완전 기록한다(프로그램 재구축 대비 데이터 지도).

## 2. 샵마인 공식 (4마켓 공통 · 스크린샷 실측)

```
총주문금액   = 단가 × 수량 + 옵션추가금액
마켓수수료금액 = 총주문금액 × 수수료율          ← 유일 미지수 = 수수료율
공급금액     = 총주문금액 − 마켓수수료금액
정산예상금액(추정)      = 공급금액 − 셀러부담할인 = 총주문금액 − 마켓수수료 − 셀러부담할인
정산예상금액(배송비포함) = 정산예상금액(추정) + 배송비정산
```

롯데온 실정산 엑셀로 검증한 정확 공식(117행 오차 0):
```
정산예정금액 = 정산대상판매가 − 중개수수료총합계
  정산대상판매가 = 상품가 − 롯데부담할인 − 셀러부담할인 + 배송비
  중개수수료총합계 = 상품가×13% + 배송비×3.3% + 상품가×2%(제휴) − 조정(=롯데부담할인)
  ⇒ 정산예정금액 = (상품가 − 셀러부담할인 + 배송비) − (상품가×13% + 배송비×3.3% + 상품가×2% − 롯데부담할인)
     (롯데부담할인은 상쇄되나, 셀러/롯데 분리 조회가 안전)
```

## 3. 필요 입력 (7종) 과 충분성

상품가 · 배송비 · 할인(셀러부담) · 상품수수료율 · 배송비수수료율 · 제휴/유입수수료 · 정산완료 실값.
→ 위 공식을 완전 재현하므로 **충분**. 예외는 '정산 이력 0인 신규상품의 직접 수수료율'뿐이며,
쿠팡은 `serviceFeeRatio` 직접 제공으로 해결, 롯데온·11번가는 셀러 계약율 고정이라 첫 정산 후 확정.

수수료율은 **정산완료분 API 실수수료에서 역산**해 학습(추측 아님) → 미정산 주문에 적용.
정산완료 주문은 역산 없이 실수수료를 그대로 쓴다(더 정확).

## 4. 마켓별 상세 API → 필드 지도 (데이터 지도 본문)

### 롯데온 (base https://openapi.lotteon.com · Bearer 인증키 · 계정별 trNo=client._cfg)
| 입력 | API (method · path) | 파라미터 | 응답 필드 |
|---|---|---|---|
| 상품가 | POST /v1/openapi/delivery/v1/SellerDeliveryOrdersSearch (209) | srchStrtDttm/srchEndDttm(yyyymmddhhmmss)·ifCplYN·trGrpCd·trNo | deliveryOrderList[].slPrc × .odQty |
| 배송비 | 〃 (209) | 〃 | deliveryOrderList[].dvCst |
| 할인(셀러부담) | GET /v1/openapi/order/v1/getSROrderList (245) | srchStrtDttm/srchEndDttm·odNo?·lrtrNo | data.orderItems[].fvrList[].prEntpShrAmt (prTypCd=PRD_DC 합) |
| 조정(롯데부담) | 〃 (245) | 〃 | fvrList[].prSfcoShrAmt |
| 상품 수수료율(13%) | POST /v1/openapi/settle/v1/se/SettleCommission | startDate/endDate(yyyymmdd)·trGrpCd·trNo·lrtrNo | data[].bseCmsnAmt ÷ 상품가 (역산) |
| 배송비 수수료율(3.3%) | 〃 SettleCommission | 〃 | dvCstCmsnAmt ÷ 배송비 |
| 제휴 수수료(2%) | 〃 SettleCommission | 〃 | pcsCmsnAmt (또는 cmsnAmt−bse−dvCst−pg) |
| 정산완료 실값 | 〃 SettleCommission | 〃 | cmsnAmt(중개수수료총합) → 정산예정 = 정산대상 − cmsnAmt |

※ 멀티계정 버그: 현재 commission_map 이 모듈 전역 trNo 사용 → 계정별 client._cfg 로 고쳐야 7계정 정산 정확.

### 11번가 (base https://api.11st.co.kr/rest · openapikey 헤더 · XML)
| 입력 | API | 응답 필드 |
|---|---|---|
| 상품가 | GET /rest/ordservices/{complete\|packaging\|completed}/{s}/{e} | order.ordAmt (또는 selPrc×ordQty) |
| 배송비 | 〃 | dlvCst / bmDlvCst(묶음) |
| 할인(셀러부담) | 〃 | ordAmt − ordPayAmt (도출) |
| 서비스이용율(8%) | 정산확정분 stlPlnAmt | (selPrc×qty − stlPlnAmt) ÷ (selPrc×qty) 역산 |
| 정산완료 실값 | GET /rest/ordservices/completed | stlPlnAmt (정산예정금액, 정산 후) |

※ 직접 카테고리 서비스이용율 API 는 셀러오피스(로그인) — 미배선. 역산으로 대체.

### 쿠팡 (base https://api-gateway.coupang.com · HMAC · vendorId)
| 입력 | API | 응답 필드 |
|---|---|---|
| 주문(미정산) | GET /v2/providers/openapi/apis/api/v5/vendors/{vid}/ordersheets | 상품가·할인·배송비 |
| 상품가 | GET /v2/providers/openapi/apis/api/v1/revenue-history | data[].items[].salePrice (수량반영) |
| 할인(셀러부담) | 〃 revenue-history | items[].sellerDiscountCoupon + downloadableCoupon |
| 상품 수수료율 | 〃 revenue-history | items[].serviceFeeRatio (%, VAT별도) — **직접** |
| 배송비 | 〃 revenue-history | deliveryFee.amount |
| 배송비 수수료율 | 〃 revenue-history | deliveryFee.feeRatio (3%) |
| 정산완료 실값 | 〃 revenue-history | items[].settlementAmount + deliveryFee.settlementAmount |

params: vendorId·recognitionDateFrom·recognitionDateTo(≤31일)·token·maxPerPage(≤50).
recognitionDate = 배송완료+7일 or 구매확정 → 미인식(최근)분은 미포함(추정).
총액 대사용: GET /v1/settlement-histories?revenueRecognitionYearMonth (월/주 finalAmount).
제휴/유입 수수료 = 없음.

### 스마트스토어 (base https://api.commerce.naver.com/external · Bearer)
| 입력 | API | 응답 필드 |
|---|---|---|
| 주문 | GET /external/v1/pay-order/seller/product-orders/last-changed-statuses | 상품가·할인·배송비 |
| 정산(직접) | GET /external/v1/pay-settle/settle/case | elements[].settleExpectAmount (productOrderType PROD/DELIVERY 별 행) |
| 수수료 | 〃 settle/case | totalPayCommissionAmount · paySettleAmount(정산기준) |

최종정산액 = settleExpectAmount 합(상품 by productOrderId + 배송비 DELIVERY by orderId).

## 5. API 정보 획득 방법 순서 (범용 · 데이터맵 상단에 기록)

1. 마켓 API센터 문서 URL 을 브라우저로 접속
2. WebFetch 로 본문 추출 시도(공개 문서면 성공)
3. WebFetch 차단(403/JS 렌더) → in-app 브라우저(mcp__Claude_Browser__ navigate + get_page_text/read_page)
4. in-app 정책 차단(예: openapi.11st.co.kr) → 로그인된 실제 크롬(claude-in-chrome) 또는 사용자 화면
5. **실제 API 호출**은 서버 IP 화이트리스트 → 서버 프로브(임시 라우트)로만 검증

## 6. 작업 (3파트)

### Part 1 — 롯데온 실검증 (오차 0까지)
- getSROrderList(245) 배선(셀러/롯데부담 할인 조인) + commission_map 멀티계정 trNo 버그 수정
- 서버 프로브: 정산 엑셀 117행 주문번호를 209+245+SettleCommission 로 불러와 정산예정금액 계산
- 엑셀 정산예정금액(AV열)과 **건별 대조** → 오차 0. 잔차는 근본원인까지 규명해 해소(부분검증 불가)

### Part 2 — 정산금 계산 영역탭 신설 (시안 확정)
- map.html 마켓별 '정산금 계산' 영역탭. 컬럼 = 필요입력 / 호출API / 응답필드 / 상태
- 상단 = 계산식 + §5 획득방법 + §3 충분성. 4마켓(§4) 채움. 상태 배지: 됨/신규배선/역산/검증완료

### Part 3 — 기존 API 코드 재점검
- map.html 기존 items 의 `code` 값이 실제 shared/platforms/* 코드·공식 문서와 맞는지 전수 대조 → 정정

## 7. 검증 기준

- 롯데온 117행 전수 오차 0 (Part 1). 통과해야 데이터맵 '검증완료' 표기.
- 이후 11번가·쿠팡·스스 동일 방식 순차 검증(마켓 하나하나).
- 배포는 검증 통과 후.

## 8. 정리 대상

- 임시 프로브 라우트 2종(_probe_lo_settle, _probe_lo_order) — 검증 완료 후 제거.
