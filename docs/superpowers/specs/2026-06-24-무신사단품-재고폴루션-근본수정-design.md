## 1. 목표 (한 문장)

무신사 단품 한 상품(SourceProduct)에 **자기 색·사이즈 행만** 존재하도록 만들어, 크롤이 항상 올바른 행을 갱신하고 매트릭스 재고가 실제 무신사와 일치하게 한다.

## 2. 근본 원인 (조사 확정)

| # | 메커니즘 | 코드 위치 |
|---|---|---|
| 주원인 | 서버사이드 크롤(`service.fetch_one_source`→`MusinsaCrawler.fetch`)이 `_discover_color_variants`로 **그 상품의 모든 색 변형 URL을 검색·병합** → 9색 옵션을 한 SP에 넣음. `save_crawl_result`엔 **색 필터 없음** → 오렌지 단품 SP에 '메이트 블랙'·'메이트 아이보리'… 전색 행 생성 | `musinsa.py`(_discover_color_variants, 멀티색 merge ~270-335) · `service.py`(save_crawl_result ~449-632, 필터부재) |
| 부원인 | **size_text 정규화 없음** → `'220mm'` vs `'220MM'` 별행 (unique 제약 우회) | `service.py` upsert_source_option ~135-180 |
| 부원인 | color_text 정규화/스코프 없음 → `'르무통…운동화'`(상품명), `'메이트 X'`(타색) 누적 | 동상 |
| 정상 | `bundle_url_crawl` 경로는 라벨색(`page_color`)으로 **필터해 정상** (`'오렌지'` 행만) | `bundle_url_crawl.py` ~207-229 |

확장 크롤(crawl-result→`_persist_option_stocks`)은 기존행 **갱신만** 하므로 폴루션을 만들진 않으나, 폴루션된 행 때문에 빈색(color='') size-only 매칭이 **엉뚱한 색 행**에 쓰여 화면과 영원히 불일치. (라이브 증거: 테스트값 818이 오렌지 SP의 `'메이트 블랙' 225mm` 행에 박힘.)

**불변 확인**: 배포·코드는 정상(원격진단으로 mou-m.com=새코드 확증). 문제는 **데이터 무결성**.

## 3. 설계 (4부분)

### A. 근본 — 서버 무신사 단품 크롤이 "자기 색만" 저장
- **A1 스코프**: 단품 SP 크롤 시 형제색 검색·병합을 끈다. `save_crawl_result`(또는 그 호출부)가 SP의 **등록색**(BundleSourceUrl.label `musinsa_{색}` → `{색}`, 없으면 SP가 단일색이면 빈색 허용)을 알아, 크롤 옵션 중 **그 색(또는 빈색)만** 남기고 나머지 색은 버린다 (`bundle_url_crawl`의 `pcns` 필터와 동일 규칙 이식).
  - 단품 판정: BundleSourceUrl.url_type=='단품' (또는 라벨 `musinsa_` 접두 + 단일색). 모음전(색상모음전/모델모음전) SP는 기존 병합 동작 유지.
- **A2 정규화(upsert_source_option, 일반·저위험)**: 저장·조회 전 `size_text`=숫자+소문자`mm`(`_norm_size`), `color_text`=strip. 같은 (sp, 정규색, 정규사이즈)면 1행으로 합쳐 갱신.

### B. 방어 — 단품 크롤이 올바른 행에 쓰도록 (확장 경로)
- crawl-result 저장(`_persist_option_stocks` / api_pricing 무신사 분기)에서, 단품 SP면 옵션 `color=''`를 **SP 등록색으로 채워** 매칭 → 빈색 size-only 모호성 제거. (정리 후 SP가 단일색이라 사실상 안전하지만 방어로 유지.)

### C. 정리 마이그레이션 (기존 폴루션 청소)
- 대상: 무신사 단품 SP들. 각 SP의 등록색(`reg_color`)·정규사이즈 행만 **canonical**로 남기고, **교차색·상품명·대소문자중복** 행은 soft-delete(`deleted_at`).
  - canonical 1행/사이즈 선정: `_stk_cnorm(color_text)`가 `reg_color`와 매칭 + 정규사이즈. 중복 시 current_stock 최신/비NULL 우선, 나머지 soft-delete.
  - reg_color 모르면(라벨 없음) 그 SP는 **건드리지 않음**(보수적).
- **안전장치**: ① 실행 전 대상 SP들의 전 행 **JSON 백업**(파일/로그) ② **드라이런**: 삭제·유지 행 수를 먼저 보고(SP별), 사용자 승인 ③ 실삭제는 soft-delete만(hard-delete 금지, 복구 가능) ④ 트랜잭션.
- 엔드포인트: `POST /api/admin/musinsa-dan/dedup?dry_run=1&bundle=<code>` (관리자, dry-run 기본).

### D. 검증
- 르무통_메이트 먼저: 정리(dry-run→실행)+ 무신사 단품 9색 재크롤 → 원격 DB 쿼리로 SP당 사이즈별 1행·등록색만 확인 + 실제 무신사 API 값과 9색 전수 일치 + 라이브 영수증.
- 통과 시 전체 무신사 단품 SP로 확장 적용.

## 4. 범위 / 비범위
- **A·B = 모든 무신사 단품 일반 적용**(이 모음전 한정 아님).
- **C = 르무통_메이트 우선 검증 → 전체 확장**.
- 비범위: 모음전(색상/모델모음전) SP 동작 변경, 타 소싱처(롯데온 등) 폴루션(동일 클래스 가능성은 메모만), 업로드 경로.

## 5. 리스크·무결성
- 정리는 **soft-delete + 백업 + 드라이런 + 승인** 4중 안전. hard-delete 금지.
- A 스코프 변경은 **단품만** 영향(모음전 SP 불변) — url_type 게이트.
- A2 정규화는 일반이나 매칭(`_stk_digits`/`_stk_cnorm`)과 정합(이미 숫자추출·정규화 사용) → 표시 무영향.
- 라이브 데이터(금전) → 단계마다 원격 DB 쿼리로 전후 대조.

## 6. 테스트
- 단위: `_norm_size`('220MM'→'220mm'), upsert 정규화 dedup, save_crawl_result 단품 색필터(타색 드롭·자기색 유지), dedup 마이그레이션(폴루션 픽스처→canonical 1행/사이즈).
- 라이브: 르무통_메이트 dry-run 리포트·실행·재크롤·9색 실대조.
