# 소싱처 추가·업데이트 카드 — 설계 (design)

> **작성**: 2026-06-29
> **상태**: 설계 확정 대기 (사용자 검토 → writing-plans)
> **관계**: 상위 = 글로벌 6원칙 · 프로젝트 CLAUDE.md / 정본 지침서 = mou-m.com 라이브 크롤링 가이드(보기/원문)

---

## 1. 목적 (한 줄)

크롤링 가이드 화면에 **"소싱처 추가·업데이트" 4번째 카드**를 신설하여, 사용자가 **소싱처명 + URL만 입력**하면 → Claude Code에 한마디 명령 → **Claude가 mou-m.com 라이브 크롤링 가이드(보기/원문)를 정독하고 그 원칙대로 크롤러를 구현·검증**하는 흐름을 만든다.

## 2. 핵심 원칙 — 정본은 mou-m.com 라이브

- **로컬 `.md` 지침서 파일을 직접 신뢰하지 않는다.** Claude는 작업할 때마다 **라이브를 정본으로 읽는다.**
  - 원문(Claude용): `https://mou-m.com/sourcing-guide/map.md`
  - 보기(사람용·6탭): `https://mou-m.com/sourcing-guide/map`
- 이유: 브랜치/로컬에 **옛 사본이 떠다녀(예: 루트 `docs/크롤링-가이드.md` 207줄 vs 라이브 442줄) 잘못된 방향**을 잡는 사고를 근본 차단.
- 라이브를 렌더하는 **유일 정본 파일** `프로그램/_시스템/docs/크롤링-가이드.md` + `webapp/static/error_catalog.json` 은 유지(이걸 고치면 배포 후 라이브가 갱신됨 = WRITE 경로). READ는 라이브, WRITE는 정본 파일 → 같은 것(origin/main 기준).

## 3. 전체 흐름 (end-to-end)

```
[웹: 소싱처 추가·업데이트 카드]
  탭1 신규 추가 / 탭2 기존 업데이트
        │  🧑 소싱처명 + URL(다중, 추가/수정/삭제, 바로가기) 입력
        ▼
  [📥 추가 + Claude 분석 요청 담기]
        │  ⑴ SourceRegistry 행 생성/갱신 → 전체보기에 즉시 등장(미정의·0/6)
        │  ⑵ "분석 대기" 큐 적재 (과거 이력은 Claude가 자동 매칭)
        ▼
[Claude Code]  🧑 "29CM 신규 추가해줘" (또는 "29CM 크롤 업데이트해줘")
        │  🤖 큐에서 URL + 첨부 체크리스트 회수
        │  🤖 Step0: mou-m.com 라이브 가이드(map.md 원문 + map 보기) 정독
        │  🤖 §1 재고·§2 가격·§4 무결성·§5 에러이력 원칙대로 구현
        │  🤖 검증 크롤 → 라이브 화면과 100% 대조 (불일치=중단·재수정)
        ▼
[완료]  전체보기 행 "정의됨 ✓" · 가이드 카드 채움 · (필요시 §1·§2·§5 정본 갱신)
```

## 4. 컴포넌트 (단위별 책임)

### 4.1 웹 — 4번째 카드 + 2탭 패널
- 위치: `/sourcing-guide/` 카드 줄(전체보기·데이터코드지도·크롤러설치) 다음.
- **탭1 신규 추가**: 소싱처명(1) + 샘플 URL 리스트(N, 행마다 `↗바로가기`·`✏️수정`·`🗑삭제` + `+URL 추가`, 대표 URL 표시) + `📥 추가+분석요청 담기`.
- **탭2 기존 업데이트**: 소싱처 드롭다운(기존 SourceRegistry) 선택 + 동일 URL 리스트 편집 + `🔄 크롤 업데이트 요청 담기`.
- URL 데이터는 **기존 `crawl_guide.sample_urls`(is_lead 포함)를 그대로 재사용** — detail.html에 이미 있는 구조. 새 저장소 만들지 않음.

### 4.2 핸드오프 — 대기 큐 (B, 별도 테이블 없음)
- 큐 = **"sample_urls는 있는데 가이드 카드가 비어있는(또는 재검증 플래그가 선) SourceRegistry 행"** 으로 식별.
- 신규: 카드 fields 전부 status=none + sample_urls≥1 = "분석 대기".
- 기존 업데이트: `crawl_guide` 에 `update_requested`(날짜·증상 메모) 플래그를 세움 = "업데이트 대기". (참고할 과거 이력은 Claude가 Step0에서 자동 판정)
- Claude는 이 목록을 읽어 처리. 사용자는 URL을 채팅에 재입력하지 않음(이름만 말함).
- **프로그램이 Claude를 호출하지 않음(API 연동 없음).** 큐는 운반만.

### 4.3 소싱처 표 단일화
- 신규 카드는 **`SourceRegistry` 에만** 기록(전체보기·지도·카드가 읽는 표). 옛 `SourcingSource` 경로는 사용하지 않음.
- 효과: 담는 즉시 전체보기 행 자동 등장 + 가이드 카드 자동 스켈레톤.

### 4.4 과거 이력 활용 — 메커니즘 매칭 + 단계적 패널
- **통찰**: 재발은 소싱처가 아니라 **수집 방식·공유코드**를 타고 옮음 → `error_catalog.json` 의 `g`(재고/가격)·`l/l2`(계층)·`code.func`(공유코드)·`rule`(4철칙)로 매칭.
- **⭐ 소싱처 이름은 매칭에서 무시한다.** 추가/업데이트하는 소싱처와 **다른 소싱처에서 난 에러라도, 같은 수집방식·공유코드·증상유형이면 반드시 참고 후보에 포함**한다(예: SSG 에러를 29CM 신규 추가 시 소환). "그 소싱처 자기 이력만 본다"가 아니다 — 오히려 **타 소싱처 이력이 신규 소싱처에 가장 큰 도움**이 된다(새 소싱처는 자기 이력이 없으므로).
  - 단, 기존 업데이트의 경우 **자기 소싱처 이력 + 같은 메커니즘 타 소싱처 이력**을 둘 다 본다(자기 것 우선 표시).
- **카드 패널 "⚠️ 과거 이력 참고" = 읽기 전용(정보 표시).** 사용자가 고를 체크박스 없음. **관련 항목은 Claude가 자동 판정**하며, 패널은 "Claude가 무엇을 참고하게 되는지"를 사용자가 눈으로 확인할 수 있게 보여줄 뿐(투명성).
  1. 처음(웹 표시) = **4철칙**(폴백금지·실패표기·단일진실·누락경고) + 재고·가격 고빈도 재발 패턴 미리보기.
  2. Claude가 Step0에서 라이브 가이드 정독 + 수집방식(API/HTML/DOM)·만질 공유코드 판정 → `g·l/l2·code.func·rule`로 **관련 에러를 자동 매칭**(예: 재고 999둔갑 S계열·부분일치 S17·다중URL S3). **사용자 입력 불필요.**
- **사용자 개입 = 0.** 과거 이력 선택·매칭·체크리스트화·구현·검증·에러이력 축적 **전부 Claude 자동**.
- **선순환(읽기→쓰기 양방향)**:
  - (읽기) Claude 자동 매칭 항목 → SOP Step0 강제 체크리스트 → 구현·검증 PASS(공유코드 수정 시 교차 회귀 검증).
  - (쓰기·강제) **이번 작업에서 발견·해결한 문제는 반드시 정본 §5(.md) + `error_catalog.json` 둘 다에 기재** → 다음 소싱처 작업이 자동으로 참고. 이 쓰기 단계는 **선택이 아니라 완료의 필수 조건**(§5.1 강제 게이트).

> ⭐ **이 양방향 선순환이 본 설계의 핵심.** 가이드(과거이력)는 작업할 때마다 **읽히고(참고) + 자라야(축적)** 한다. 한 번이라도 "고쳤는데 기재 안 함"이 생기면 다음 소싱처에서 같은 문제가 재발하므로, 기재를 **자동 강제**로 막는다(§5.1).

### 4.5 Claude 실행 SOP (라이브 정독 기반)
신규/기존 공통 시작:
0. **origin/main(워크트리) 확인** → `mou-m.com/sourcing-guide/map.md`(원문)·`/map`(보기) 정독 → 첨부 과거 이력 + 만질 공유코드로 재발방지 체크리스트 확정.

**신규 추가 = 발굴 + 생성** (출발점 §1·§2)
1. 수집 방식 분석(API/HTML/DOM × 인증) — §0·§1
2. 6항목 위치 추출(썸네일·상품명·가격·혜택·옵션재고·상세이미지) — §1 비교표 양식
3. 재고 3상태 규칙(0품절/실수량/999미상/-1불명/None미크롤) — §1 센티넬
4. 매입가 로직(혜택 4분류 순차차감) — §2
5. 검증 크롤 → 라이브 화면 100% 일치 — §4
6. `crawlers/<key>.py` 작성 + `crawlers/__init__.py` 등록 + has_adapter 활성 + 정본 §1·§2 표에 행 신설 + 카드 저장

**기존 업데이트 = 진단 + 교정** (출발점 §5)
1. 증상 확인(가격0·재고둔갑·혜택누락) — §5 에러이력/복귀 체크리스트
2. 라이브와 diff — 무엇이 바뀌었나(셀렉터·API·정책)
3. 바뀐 부분만 수정(소싱처 전용 우선, 공유코드면 교차 회귀)
4. 재검증 100% — §4
5. 기존 행·카드 값 갱신(행 추가 아님) + 관련 에러이력 rev 보강

### 4.6 단계적 실행 프로토콜 — 체크포인트 + 가이드와 소통 ⭐

> **문제**: 데이터 지도가 큼(442줄 + 에러 41건). Claude가 "한 번에 다 읽고 다 구현"하면 반드시 누락이 생긴다.
> **해결**: SOP를 **독립 스테이지**로 쪼개고, 스테이지마다 **가이드의 해당 § 슬라이스만 정독** → 작업 → 검증 → **사용자 체크포인트(보고+합의)** → 다음. 가이드 전체를 통으로 삼키지 않으니 놓치지 않는다.

**원칙**
- **스테이지 ↔ 가이드 § 1:1 대응** → "가이드와 계속 소통하며 진행"이 구조로 보장됨(각 스테이지가 특정 §만 인용·대조).
- **각 스테이지 끝 = 자동 자기검증(가이드 ↔ Claude 자동 소통).** Claude가 ⓐ 한 일 ⓑ 가이드 §가 뭐라 하는지 ⓒ 일치 여부를 **스스로 대조** → 일치하면 **자동 OK·다음**, 어긋나면 **자동 "고쳐"·직전 스테이지 롤백**(사용자 개입 없이 루프). 진행은 위젯에 실시간 표시(투명성).
- **사용자 게이트는 꼭 필요한 3곳만**: ⑴ **S5 라이브 100% 대조** — 사용자의 실제 화면이 "정답 기준"이라 확인 필수 ⑵ 자동 "고쳐"를 **N회(예: 3회) 반복해도 못 맞추면 에스컬레이션**(사람 판단 요청) ⑶ **최종 완료 승인**.
- 즉 OK/고쳐 판단은 **기본 자동**(가이드 자기대조 루프), 사람은 **라이브 진실·막힘·완료**에서만 개입.
- **실행 엔진** = superpowers `executing-plans` / `subagent-driven-development`(스테이지=태스크, 체크포인트=리뷰). **진행 가시화** = `workflow-progress-widget`(클릭 위젯 + 체크박스, 방향 변경 전 재확인).

**신규 추가 — 스테이지 (각 = 정독→작업→검증→합의)**
| # | 스테이지 | 정독할 가이드 § | 산출물·합의 포인트 |
|---|---|---|---|
| S0 | 준비·정찰 | 전체 인덱스 + 과거이력 자동매칭 | origin/main 워크트리 확인 · 라이브 정독 · "이 소싱처 접근법" 합의 |
| S1 | 수집 방식 | §0·§1 (API/HTML/DOM×인증) | 4분면 판정 보고 |
| S2 | 6항목 위치 | §1 비교표 양식 | 셀렉터/엔드포인트 6개 |
| S3 | 재고 3상태 | §1 센티넬·함정 | 0/실수량/999/-1/None 규칙 |
| S4 | 매입가 로직 | §2 혜택 4분류 | 순차차감 규칙 |
| S5 | 검증 대조 | §4 무결성 | 라이브 100% 일치(불일치→롤백) |
| S6 | 등록·축적 | §5.1 선순환 | 크롤러 등록 + 전체보기 노출 + 에러이력 기재 |

**기존 업데이트 — 스테이지**
| # | 스테이지 | 정독할 가이드 § | 산출물·합의 포인트 |
|---|---|---|---|
| U0 | 증상 접수 | §5 에러이력·복귀 체크리스트 | 증상→의심 원인 후보 합의 |
| U1 | diff 진단 | 해당 소싱처 §1·§2 + 라이브 | "무엇이 달라졌나" 보고 |
| U2 | 교정 | 바뀐 부분(전용 우선/공유면 교차회귀) | 수정안 합의 |
| U3 | 재검증 | §4 | 라이브 100% 일치 |
| U4 | 갱신·축적 | §5.1 | 행·카드 갱신 + 에러이력 rev |

> 진행 중 OK/고쳐는 **Claude가 가이드와 자동 대조해 판단**하고, 사용자는 **라이브 검증·막힘(에스컬레이션)·최종 완료**에서만 답하면 됨(개입 최소, 누락은 구조로 방지).

## 5. 완료 게이트 (무결성)
- ✅ 라이브 화면과 표면가·혜택합·최종가·옵션·재고 **100% 일치** (§4, 폴백가 금지).
- ✅ `guide drift` 동기화 — 코드 ↔ 정본 `.md` ↔ `error_catalog.json` 일치(compute_guide_drift).
- ✅ 공유코드 수정 시 같은 코드 쓰는 타 소싱처 회귀 검증 + 테스트 PASS.
- ✅ **에러이력 축적 완료** (§5.1).
- ❌ 위 미충족 시 "추가/업데이트 완료" 선언 금지.

### 5.1 에러이력 선순환 — 강제 게이트 ⭐ (본 설계의 심장)

> "문제 발견/해결은 **반드시** 과거이력에 기재되어 다음 작업에 참고된다"를 **사람의 선의가 아니라 자동 장치로** 보장한다.

**규칙 (governance — 라이브 §5에 이미 존재, 이를 강제):**
- **추가 vs 수정**: `같은 소싱처+같은 근본+같은 증상`이 이미 있으면 기존 행에 **rev 보강(수정)**, 새 행 금지. 같은 근본·다른 소싱처면 **새 행 추가**(ID = S/P/O prefix + 다음 번호, 재사용 금지).
- **양쪽 반영**: 정본 `.md` §5 표 + `error_catalog.json` **둘 다** 갱신. 각 변경에 rev(날짜·액션·내용·소싱처·커밋).
- **상태·태그**: 증상유형(재고/가격/기타)·계층(be/fe/기타)·상태(✅/⏳)·`code.func`(메커니즘 매칭 키) 필수 기입 — 안 그러면 다음 작업의 정밀 매칭이 안 됨.

**자동 강제 장치 (사람이 깜빡해도 막힘):**
1. **완료 판정 = 큐 상태 전이로만.** 요청은 Claude가 ⓐ 검증 100% + ⓑ 에러이력 rev 기록(또는 "신규 에러 없음" 명시 확인)을 마쳐야 "완료"로 전이. 둘 중 하나라도 없으면 "처리중"에 머묾.
2. **pre-commit / CI 가드** (setup-pre-commit): 커밋에 `crawlers/**` 또는 공유코드(`service.py`·`_match_option_so`·`compute_breakdown` 등)의 **수정(fix)** 이 포함되면, 같은 커밋(또는 직전)에 `error_catalog.json` 변경이 없으면 **차단** + 안내("이 수정의 근본/재발방지를 §5에 기재하세요"). 순수 신규 파일 추가(신규 소싱처 첫 크롤러)는 예외 허용하되 카드 저장 시 §1·§2 행 추가를 요구.
3. **drift 대시보드 연동**: `error_catalog.json`(보기) ↔ `.md`(원문) 불일치 시 `/sourcing-guide/map §5`가 🔄/❌로 표시 → 완료 게이트가 ✅ 아니면 통과 불가.

**효과**: 신규 추가·기존 업데이트를 할 때마다 가이드가 **읽히고(참고) → 자란다(축적)** → 다음 소싱처가 더 똑똑해지는 복리 효과.

## 10. 프로세스 정립 (2026-06-29 사용자 합의 — 구현됨)

> 사용자 핵심 요구: **신규 추가/기존 업데이트 시 사용자·프로그램·크롤링가이드·CLAUDE 가 어떻게 유기적으로 상호작용하는지**의 정본. hmall 중복 사건이 드러낸 빠진 게이트(신규/기존 미구분)를 메움.

### 10.1 4주체 상호작용
- **크롤링 가이드(mou-m.com 라이브) = 단일 진실 원천(SSOT).** CLAUDE 가 읽어서(정독) 구현하고, 새로 알아낸 건 다시 거기에 적는다(선순환·양방향).
- **사용자** = 카드에 소싱처명·URL 입력 + S5 라이브 검증 + 최종 승인. 그 외엔 결정만.
- **프로그램(웹)** = 신규/기존 분기, 존재검사 게이트, 붙여넣기 프롬프트 생성, 카드에 진행 표시.
- **CLAUDE** = 가이드 정독 → 과거이력 메커니즘 매칭 → 단계 구현 → 라이브 검증 → 에러이력 기재.

### 10.2 신규 8단계 / 기존 6단계 (요약)
- 신규: ①입력 → ②**존재검사 게이트** → ③"분석 대기" 등록+프롬프트 → ④붙여넣기 → ⑤가이드 정독+과거이력 매칭 → ⑥단계 구현·라이브 검증 → ⑦가이드 갱신+에러이력(선순환) → ⑧"분석대기→크롤지원" 전이.
- 기존: ①선택+URL → ②update_requested+프롬프트 → ③붙여넣기 → ④가이드 해당 § 정독 → ⑤진단·교정·검증 → ⑥가이드/코드 갱신+에러이력.

### 10.3 존재검사 게이트 (신규의 빠진 조각 — hmall 교훈)
- `api_add_source`: ①이름 정확일치 ②URL **도메인** 일치 — (a)이미 등록된 소싱처 sample_urls 도메인 / (b)빌트인 카탈로그(`SOURCE_CATALOG`) 도메인. 진짜 hmall 레지스트리 행은 URL 이 비어 (a)가 못 잡으므로 **(b) 카탈로그 도메인 매칭이 필수**.
- 걸리면 `409 {exists:true, existing:{kind, id?, name}}` → 프론트가 "기존 업데이트" 탭으로 유도(있으면 자동 선택). `force:true` 면 강행.
- 구현: `webapp/routes/sourcing_guide.py:_find_existing_by_domain`, `lemouton/sourcing/source_registry.py:domain_of`·`catalog_by_domain`.

### 10.4 단계적 소통 루프 (가이드가 커서 한 번에 못 삼킴)
- 스테이지마다 `[해당 § 정독 → 작업 → 자기검증(자동 OK/고쳐) → 체크포인트]`, S1~S6/U1~U4. 사용자 게이트는 S5 라이브 대조 1곳(+막힘 에스컬레이션·최종 승인).
- 단계 카탈로그 = `crawl_guide.STAGE_CATALOG` (단일 진실원천).

### 10.5 카드 진행 표시 (CLAUDE → 프로그램 소통, 옵션1)
- 스키마: `crawl_guide` JSON 에 `stage_progress {kind, stage, label, done[], note, updated_at}` (마이그레이션 불필요). 누적 헬퍼 `crawl_guide.advance_stage`.
- 통로: `POST /webhook/source-stage` (로그인 면제 `/webhook/` + `MOUM_STAGE_TOKEN` 인증) → CLAUDE 가 add-source 스킬에서 단계마다 찍음. 토큰 미설정 시 503=조용히 비활성(graceful).
- 표시: 전체보기 카드 이름 옆 "진행중 N/6 · S3 가격·혜택" / "분석 완료 N/6".
- ⚠️ 사용자 셋업 필요: 라이브 server `app.env` 에 `MOUM_STAGE_TOKEN` 1줄(+스킬이 읽을 로컬 env). 없으면 진행 표시만 생략.

### 10.6 중복 정리 (현재 상태 교정)
- `POST /<sid>/merge-into/<target_sid>` — 빈 카드(크롤 정의 없음)만 안전 병합·삭제. hmall ②"HMALL 분석대기" → ①"현대홈쇼핑(hmall)" 정리에 사용. 로그인 UI 에서 사용자 실행.
