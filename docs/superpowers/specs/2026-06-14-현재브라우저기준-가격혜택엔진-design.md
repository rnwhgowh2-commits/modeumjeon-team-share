## 1. 문제 정의

무신사 르무통 메이트 상품 크롤링에서 반복되는 가격 오류. 본질은:

> 가격·혜택은 수시로 바뀌고, **로그인(계정 LV)/비로그인에 따라 다 다르다.** 따라서
> **현재 열린 사용자 브라우저 기준으로만** 정보를 수집·가공해 표면노출가 / 혜택 /
> 최종매입가를 정확히 도출해야 한다.

### 사용자 보고 3대 문제 (코드 근거)

| # | 문제 | 코드 근거 | 진단 |
|---|---|---|---|
| 1 | 표면노출가 기준이 계속 바뀜 (표면가 = '나의 할인가' 위 금액, 현재 화면 122,900원) | `extension/moum-crawler/background.js:266-288` | 표면가를 화면 DOM이 아닌 별도 API(`goodsPrice.salePrice`)에서 읽음. 사용자 결정으로 **API 방식 유지**하되 화면값 일치 검증을 테스트 항목으로 둠 |
| 2 | 혜택이 로그인 상태별로 다른데 지난 크롤 혜택이 반영됨 | `background.js:301-311` (혜택 미수집) + `webapp/routes/api_benefits.py:510-535` (`compute_breakdown`) | **확장이 혜택을 전혀 안 긁음.** 혜택은 100% 저장된 템플릿/오버라이드/`dynamic_benefits_json`에서 옴 → 구조적으로 "지난 값·로그인 무관" |
| 3 | 현재 크롤 데이터 없으면 크롤실패가 떠야 하는데 기존 값이 도출됨 | `lemouton/pricing/unified.py:43-53` (`is_crawl_valid` — 표면가만 게이트) | 표면가엔 신선도 게이트가 있으나 **혜택엔 신선도 게이트가 없음** → 혜택은 항상 저장값 사용 |

### 근본 원인 한 줄

**표면노출가는 매 크롤에서 가져오지만(게이트 있음), 혜택은 크롤하지 않고 저장된 설정값을 쓴다(게이트 없음).**

## 2. 핵심 통찰 — 부품은 이미 있다, 배선이 안 됐다

사용자가 원하는 계산 흐름이 이미 **"미리보기/검증" 경로로 존재**한다. 다만 라이브
크롤→저장→계산 경로에 연결되어 있지 않다.

| 부품 | 위치 | 상태 |
|---|---|---|
| 혜택 키워드 게이트 (크롤 라인 → on/off) | `lemouton/pricing/benefit_gate.py` | ✅ 완성 |
| 순차 누적 차감 계산 (표면가−혜택, 베이스금액 누적) | `lemouton/pricing/final_price.py` | ✅ 완성 |
| 혜택 키워드 사전 (소싱처별 정의) | `crawl_guide.pricing.benefits` + `exclude_keywords` | ✅ 완성 |
| 위 셋을 묶어 "크롤 라인 → 최종매입가" | `webapp/routes/sourcing_guide.py:159-218` (`gate-preview`) | ⚠️ **미리보기 전용** |

**실제 라이브 계산(`compute_breakdown`)은 `gate_benefits()`를 호출하지 않는다.** 게이트는
미리보기 버튼에서만 돈다. 이 단절이 문제의 본질.

> 설계 방향: **미리보기 흐름을 라이브 경로로 끌어올리고, 확장이 혜택 라인+금액까지 긁게 한다.**

## 3. 확정된 결정 (사용자)

1. **혜택 출처**: 크롤 우선. 템플릿은 "어떤 키워드를 찾을지" 정의(합집합 사전)만 담당. 실제 값·유무는 매 크롤이 채움.
2. **표면가 출처**: 현재 API(`goodsPrice.salePrice`) 방식 유지. 단 로그인/비로그인 테스트에서 화면값(122,900)과 일치 대조.
3. **혜택 수집 방식**: A안 — 화면(DOM) 텍스트 스크래핑.
4. **범위**: 무신사 르무통 1개 완성 → 그 패턴으로 일반화.
5. **키워드 탐색(Phase 0)**: 로그인 3상태 동시 탐색 + 크롤가이드 탭의 소싱처별 다른 상품 URL 포함.

## 4. 가격 로직 (사용자 명세 반영)

```
표면노출가  : 정가에서 선반영 할인 적용된 상품 할인가 (= goodsPrice.salePrice)
최종매입가  : 표면노출가 − Σ(적용 혜택)

혜택 속성 (템플릿이 정의, 크롤이 값 채움):
  ㄱ. 혜택금액 방식 : 정률 / 정액 / 조건부(%, 단 XXX원 기준, 최대 XXX원)
  ㄴ. 베이스금액   : 누적. 예) 10,000 →(10%)→ 9,000 →(10%)→ 8,100
                    (compute_final_price 가 이미 순차 차감 처리)
  ㄷ. 상시 / 기간한정
  ㄹ. 횟수 : 1회성 / 무제한
```

## 5. 데이터 흐름 (변경점 ★)

```
크롬 확장 (로그인된 내 브라우저)
  ├ 표면가: goodsPrice.salePrice                          (현행 유지)
  └ ★ 혜택: 화면 혜택영역 DOM → [benefit_lines + benefit_amounts]   (신규)
        ↓
POST /api/sources/crawl-result
  └ ★ benefits_crawled_json 저장 (crawl 시각·is_logged_in·lines·amounts)  (신규)
        ↓
compute_breakdown (매트릭스·fx·업로드 공용)
  ├ ★ 신선도 게이트: 최근 성공 크롤 스냅샷만 사용, 없으면 "혜택 미수집"  (신규)
  ├ ★ gate_benefits(가이드 키워드사전, 이번크롤 lines) → 적용 혜택만     (배선)
  └ ★ compute_final_price(표면가, 적용혜택 + 이번크롤 amounts)          (배선, 폴백 금지)
        ↓
  표면노출가 / 혜택 / 최종매입가  (전부 현재 브라우저 기준)
```

## 6. 컴포넌트별 설계

### 6.1 확장 — `musinsaExtractor` 혜택 수집 (background.js)

- **입력**: 현재 열린 무신사 상품 페이지 (로그인 상태 그대로).
- **출력 추가 필드**:
  - `benefit_lines: string[]` — 화면 혜택/쿠폰 영역의 텍스트 라인. 펼침/아코디언 안 항목은 Phase 0에서 확인 후 필요 시 펼쳐 읽기.
  - `benefit_amounts: { [라인키]: { type: 'rate'|'amount', value: number } }` — 각 라인에서 추출한 %/원.
  - `is_logged_in` (기존 유지) — 어느 상태 화면이었는지 꼬리표.
- **불변식**: 혜택영역 자체를 못 찾으면 `benefit_lines=[]` + `benefits_ok=false`로 명시 (빈 배열을 "혜택 없음"으로 둔갑 금지 → 저장 단계가 "미수집"으로 처리).
- **표면가**: 현행 `goodsPrice.salePrice` + 검증 게이트 유지. 변경 없음.

### 6.2 저장 — `/api/sources/crawl-result` (api_pricing.py)

- 새 필드 `benefits_crawled_json`을 SourceProduct(또는 옵션 레벨)에 저장:
  ```json
  {
    "crawled_at": "ISO8601",
    "is_logged_in": true,
    "benefits_ok": true,
    "lines": ["쿠폰 5%", "등급할인 불가", "무신사머니 2,000원 적립"],
    "amounts": { "쿠폰": {"type":"rate","value":0.05}, "무신사머니": {"type":"amount","value":2000} }
  }
  ```
- **기존 무신사 `is_logged_in=False` 동적혜택 0 비우기 로직(api_pricing.py:1069-1084)은 폐기/대체** — 새 스냅샷 모델이 이를 자연 포섭(비로그인 화면엔 계정혜택 라인이 없으므로 게이트가 자동 off).
- 크롤 실패(status=error) 시 표면가와 마찬가지로 `benefits_crawled_json`을 **갱신하지 않음** (옛 스냅샷이 stale로 남되, 신선도 게이트가 걸러냄).

### 6.3 계산 — `compute_breakdown` (api_benefits.py)

- **변경 전**: template/override/dynamic 조립 → compute_final_price.
- **변경 후**:
  1. `benefits_crawled_json` 로드 → **신선도 게이트** (아래 6.4) 통과 못 하면 `benefits_status='미수집'` 반환, 최종매입가 추정 금지.
  2. `gate_benefits(가이드.pricing.benefits, snapshot.lines, 가이드.exclude_keywords)` → 적용 혜택 목록.
  3. 적용 혜택별 금액 = `snapshot.amounts` (이번 크롤값). 가이드는 계산속성(type/베이스순서/조건부)만 제공.
  4. `compute_final_price(표면가, 적용혜택+금액)` → 최종매입가 + steps.
- **폴백 금지**: 스냅샷 없음/실패 시 템플릿 금액으로 대체하지 않음 (메모리 `feedback_no_fallback_price_on_match_fail` 준수).

### 6.4 신선도 게이트 (신규, 혜택용)

`is_crawl_valid()`(표면가)와 동형의 혜택판:

```
benefits_fresh(snapshot, last_status) :=
    snapshot is not None
    AND snapshot.benefits_ok == True
    AND last_status != 'error'
```

- 통과 → 그 스냅샷으로 계산.
- 불통 → `benefits_status='미수집'`, 화면은 "혜택 미수집(재크롤 필요)" 표기, 최종매입가는 "산출불가" (표면가만 표시).

### 6.5 템플릿/오버라이드 역할 재정의

- **템플릿 = 키워드 사전 + 계산속성** (`crawl_guide.pricing.benefits[].{name, triggers, match, type, 조건부, 베이스순서, 상시/기간, 횟수}`). **라이브 금액 보유 ❌.**
- **OptionBenefitOverride(스냅샷)**: 의미 축소. 수동 강제 예외로만 유지하되, 새 크롤 스냅샷이 우선. 이중차감 지뢰(메모리 `project_benefit_snapshot_model`) 재발 방지 위해 마이그레이션·배포 순서 엄수.

### 6.6 크롤실패·미수집 UI 표면화 (문제 3)

- 표면가 없음 → 기존 ⛔판매중단(`crawl_blocked`, 메모리 `project_crawl_reset_block`) 유지.
- 혜택 미수집(표면가는 있음) → "혜택 미수집(재크롤 필요)" 배지 + 최종매입가 "산출불가". 추정치 금지.

## 7. Phase 0 — 키워드 탐색 + 로그인 3상태 테스트 (선행 게이트)

사용자가 직접 로그인 전환. 순서: **계정1(블랙다이아몬드 LV) → 비로그인 → 계정2(다이아몬드 LV).**

각 상태에서 수집 대상 = 르무통 메이트 + 크롤가이드 탭의 무신사 소싱처 다른 상품 URL들.

1. 상태별 페이지 HTML/스크린샷 수집.
2. 3상태 **합집합**으로 무신사 혜택 키워드 사전 확정 (정률/정액/조건부, 베이스순서, 상시/기간, 횟수 속성 포함).
3. 상태별 **정합성 대조표**: 표면가(API값) = 화면값(122,900) 일치 검증 + 표면가/혜택/최종매입가 손계산 대조.
4. 대조표가 맞아야 구현 진입 (선행 게이트).

## 8. 테스트 / 검증 계획

- **단위**: `gate_benefits`·`compute_final_price`는 기존 테스트 보유. 신규 = 확장 추출기 파서(혜택 라인/금액) + 신선도 게이트.
- **통합(라이브 실검증, 의무)**: `/live-browser-verify` → `/ui-verify`. 로그인 3상태 각각에서 매트릭스/fx 카드의 표면가·혜택·최종매입가가 그 상태 화면과 일치하는지, 크롤 안 한 옵션은 "미수집"으로 뜨는지.
- **거짓완료 금지**: 코드만 보고 PASS 선언 금지. 실 URL·실 로그인 전환으로 증거 캡처.

## 9. 리스크 / 주의

- **브랜치**: 현재 `feature/crawl-worker-queue`는 origin/main 대비 stale(메모리 `project_crawl_worker_queue_branch_stale`). 배포·검증은 origin/main 기준 (메모리 `project_deploy_isolated_change_via_worktree` 패턴).
- **이중차감 지뢰**: 혜택 모델 변경 시 라이브(같은 Supabase)의 구버전 코드와 충돌 주의. **코드 먼저 배포 → 데이터 마이그레이션 나중**, force-push 금지.
- **확장 버전 분열**: main(v0.4.x) vs feature(v0.3.x) (메모리 `project_musinsa_surface_price_fix`). 확장 변경은 단일 원본 기준으로.
- **무신사 혜택 숨김**: 일부 혜택이 펼침/결제단계에만 보일 수 있음 → Phase 0에서 확인, 못 긁으면 키워드 사전에 "DOM 미노출" 표시하고 그 혜택은 미적용(폴백 금지).
