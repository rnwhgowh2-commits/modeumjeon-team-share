## 1. 배경 — "소싱처는 했고, 이제 판매처"

지금까지 모음전은 **소싱처**(무신사·롯데온·SSF·SSG·스마트스토어·르무통공홈)에 URL을 붙여
**재고·매입가를 읽어오는** 데 집중했다. 이번 작업은 그 거울상(對)인 **판매처**다:
모음전 상품을 각 판매처 마켓에 **연동**하고, **재고·판매가를 마켓에 써넣는다(업데이트)**.

### 판매처 = 소싱처의 거울

| | 소싱처 (완성) | 판매처 (본 작업) |
|---|---|---|
| 연결 | 옵션에 **소싱처 URL** 붙임 | 옵션에 **판매처 상품(번호)** 붙임 |
| 방향 | 외부 → 우리 (재고·매입가 **읽기**) | 우리 → 외부 (재고·판매가 **쓰기**) |
| 매칭 | 색상·사이즈 매칭 | 색상·사이즈 매칭 (동일 alias 엔진) |
| 표시 | 매트릭스 소싱처 컬럼 | 매트릭스 판매처 컬럼 |

→ 소싱처에서 만든 **매칭 엔진·매트릭스·alias 시스템을 그대로 재활용**한다.

판매처 업로드 시스템의 **뼈대는 이미 코드로 존재**하나, 라이브 검증은 미완이다.

| 오케스트레이터(변동감지→dryrun→자동보류→push→DLQ) | `lemouton/uploader/orchestrator.py` | 코드 존재 |

### 점검으로 확정한 사실
- **쿠팡**: `.env`에 API 키 0개, DB 등록 0건 → **한 번도 연결된 적 없음**.
- **스마트스토어**: `.env`에 `CLIENT_ID/SECRET` 있음, `market_registrations`에 **smartstore `ok` 36건**(`last_success_at` 세팅됨) → 과거 어느 시점 동작 흔적. 단 **mock/test인지 라이브인지 미확정** → P0-2에서 읽기로 검증.
- V2 스키마(`BundleSet`)는 0건 미사용. 시스템은 V1(`Model`/`Option`) + `MarketRegistration` 위에서 돈다.

## 3. 목표 / 비목표

### 목표 (1차 범위)
- **[B] 기존상품 연결**: 모음전 상품에 마켓 상품번호를 붙여넣으면 → 그 상품의 옵션을 API로 조회 →
  색상·사이즈로 자동 매칭 → `MarketRegistration`에 (상품ID·옵션ID) 저장. *(이 시점엔 마켓에 쓰지 않음)*
- **[C] 업데이트**: 매핑된 옵션의 **재고**(1차) / **판매가**(2차)를 **미리보기 → 승인 → push**.
- **확장성**: 어댑터 패턴 유지 → 향후 수십 개 판매처를 같은 틀로 추가.

### 비목표 (이번에 뺌)
- **[A] 신규 상품 등록** — 사용자 결정으로 후속.
- 주문·정산·CS·배송 연동.
- 판매가 **자동** push (1차는 수동+미리보기만).

## 4. 데이터 흐름 (3단계)

```
[1 연결]  모음전 상품 → "쿠팡/스스 상품번호" 입력
            → API 옵션조회(fetch_product_options / coupang get_product)
            → 색상·사이즈 자동매칭(기존 alias 엔진)
            → MarketRegistration 에 (market_product_id, market_option_id) 저장 (status=pending)
            ※ 마켓에 쓰지 않음. 매칭 실패 옵션 = "연결 안 됨"으로 표면화(폴백 금지)

[2 미리보기]  변동 표 생성 (재고 5→8, 판매가 = 마켓 현재값 읽어 표시)
            → orchestrator dryrun + 자동보류(경고 N건·평균 가격변동 30%↑ 등) 안전판

[3 업데이트]  사용자 승인 시에만 push (1차 재고만)
            → 성공 ok 기록(last_synced_*) / 실패 DLQ
```

| `market_option_id` | ✅ 매칭된 옵션ID | — |
| `status` | `pending`(연결만) / `linked` | `ok` / `failed` |
| `last_synced_price`, `last_synced_stock` | — | ✅ push 성공값 |
| `last_attempt_at`, `last_success_at`, `sync_error` | — | ✅ |

## 7. 업무 순서도 — "지금 상품ID 못 줌" 제약 반영

사용자 입력 의존성으로 단계화한다. **P0는 사용자 부재에도 진행 가능.**

### [P0] 사용자 입력 0개 — 즉시 진행 가능
- **P0-1** 연결·매핑·미리보기 엔진 구축(코드) — 매칭→`MarketRegistration` 기록 함수, 연결 라우트, 미리보기 생성.
- **P0-2 ★스마트스토어 읽기경로 실검증** — **DB에 이미 저장된 상품ID(36건 중)** 로 `fetch_product_options` 호출.
  - 목적: "API가 진짜 되나?"를 **쓰기 없이 읽기만으로** 확정. 동시에 36건 ok가 라이브인지 가늠.
  - 산출: 실제 옵션목록(색상·사이즈·재고) 수신 증거 + 매칭 정확도 측정.
- **P0-3** 매트릭스 **판매처 컬럼 UI 골격**(소싱처 컬럼 거울) — 연결 상태/미연결 표시.

### [P1] 책상 복귀 시 — 가벼운 입력
- **P1-1** 쿠팡 API 키 발급(쿠팡윙: access/secret/vendorId) → `.env` 입력 → 연결 테스트.
- **P1-2** 미연결 상품에 **상품번호 붙여넣기 = 연결 실행**(B의 실데이터 적용).

### [P2] 명시적 GO + 감독 하 — 라이브 쓰기
- **P2-1** 재고 push 실검증 — 상품 1개로 미리보기→승인→라이브 1건, 마켓 화면 대조.
- **P2-2** 확대 적용 + (2차) 판매가 덮어쓰기 모드.

## 8. 안전 · 무결성 (돈·계정 직결)

- 마켓에 **쓰기**라 잘못된 가격/재고 = 금전 손실 + 계정 리스크.
- **수동 트리거 + 미리보기 승인 필수** (1차 자동 push 금지).
- 연결 시 마켓 **현재 판매가는 읽어 표시만**, 덮어쓰지 않음. 판매가 덮어쓰기는 명시적 모드(2차).
- **폴백가 금지**(기존 원칙): 매칭/연결 실패 옵션에 대표가(평균·최저) 절대 주입 금지 → "연결 안 됨"으로 표면화.
- `orchestrator`의 자동보류(경고 임계·평균 가격변동률) 게이트 유지.
- 라이브 쓰기 검증은 1건부터, 마켓 실화면 대조 후 확대.

## 12. 추가 설계 — 판매처별 옵션 선택 · API 조사 · UI 배치 (2026-06-26 추가)

사용자 추가 요구 4건을 업무 설계에 반영한다.

### 12-1. 판매처별 옵션 선택 연동 (일괄 ❌ → 선택 ✅)
모든 옵션을 모든 판매처에 일괄 연동하지 않는다. **판매처마다 연동할 옵션을 선택**한다.
- 데이터 토대 **이미 존재**: `Option.market_visible_ss`(models.py:280) · `Option.market_visible_coupang`(models.py:281) — 옵션×마켓 노출 플래그.
- 설계 변경: `link_bundle_market` 을 **노출 ON(선택된) 옵션만** 연결하도록 확장(매칭은 전체, upsert는 선택분만) + 옵션별 토글 UI.
- 무결성 유지: 선택 안 한 옵션은 `MarketRegistration` 미생성(폴백 금지 일관).

### 12-2. 판매처별 API 조사 — 마켓마다 필수
마켓마다 **옵션 구조·엔드포인트·인증이 다르다**:
- 스마트스토어 = `optionCombinations`(색상×사이즈 2축), `originProductNo` 단위.
- 쿠팡 = `sellerProduct → items[] → vendorItemId`, 구조 상이.
- 29CM·W컨셉·롯데 등 = 각기 다름(미조사).
→ 소싱처를 마켓별 어댑터로 만든 것과 동일하게, **판매처도 마켓별 조사 → `docs/markets/<market>.yaml` 프로필**(옵션 축·조회/수정 엔드포인트·인증·재고/가격 필드)을 산출한다.
- 절차: 기존 `/external-site-integration`(판매처 docs/markets/ 커버) → 어댑터 작성. **새 판매처 추가 = 이 조사가 진입점.**
- `market_fetch.py` 의 `_fetch_<market>` 어댑터가 각 프로필을 코드화(현재 smartstore 완료, coupang 미구현).

### 12-3. UI 배치 결정 — ②옵션 탭 vs ③마켓 탭
현재 상품 편집 화면(`bundles/edit.html:167-180`)은 3탭: **①기본 · ②옵션(매트릭스+소싱처 컬럼) · ③마켓**.
중요: **③마켓 탭이 이미 존재**(스마트스토어/쿠팡 상품번호 등록 + 옵션 매칭 + 업로드, edit.html:261-507).
- **결정(추천=하이브리드)**:
  - **상품번호 연결(상품 단위)** → ③마켓 탭(기존 위치 유지).
  - **옵션별 선택·판매가·연동상태(옵션 단위)** → ②옵션 매트릭스에 **판매처 컬럼**(소싱처 컬럼 `.d2-src` 거울). 한 행에서 매입가(소싱처) ↔ 판매가(판매처) 한눈에.
- 대안 A: 전부 ②옵션 탭(매트릭스 한 그리드). 대안 B: 전부 ③마켓 탭. → **시안으로 비교 후 확정**.

## 13. 다중 "구성(세트)" 레이어 — 자유 구성 (2026-06-26 추가, 핵심 모델 변경)

### 13-1. 요구
하나의 모음전에서 **여러 판매 구성(세트)** 을 만들어 각각 별도 상품으로 연결·판매한다(사용자: "자유 구성").
- 같은 모음전 → 여러 구성(예: 단품 / 2개세트 / 블랙 색상만 묶음 / 다른 제품과 다품 묶음).
- 각 구성 → **판매처에 별도 상품(상품번호 따로)**.
- **같은 판매처에도 한 모음전에서 여러 구성**이 올라갈 수 있음.
- "자유 구성" = 수량 세트 + 옵션 부분집합 + 다품 묶음 **전부 조합 가능**.

### 13-2. 계층 (모델 변경)
```
모음전 → 구성(세트) N개 → 각 구성:
            ├ 상품(다품 가능) 1..N
            │    └ 선택 옵션(수량 가능) — 옵션 부분집합
            └ 판매처별 상품번호·연동
```
즉 관계가 **모음전 1 : 구성 N : (판매처) 상품 N** 으로 깊어진다. 현 P0 모델
`MarketRegistration`(PK=`canonical_sku`+`market`, 평면)은 **"옵션 1 = 마켓당 상품 1"** 가정 →
같은 옵션이 같은 판매처의 두 구성(단품·2세트)에 동시 소속 **불가** → 구성ID 필요.

### 13-3. 기존 V2 스키마 매핑 (이미 이 모양으로 설계됨)
`lemouton/sourcing/models_v2.py` 의 `BundleSet → BundleProduct → BundleOption` 가 정확히 일치:
- `BundleSet`(models_v2.py:184) = 구성 1개 = **1 업로드계정(=판매처 계정) 등록 단위**, `market_product_id` 보유 → 같은 판매처 여러 구성 = 행 여러 개. ✅
- `BundleProduct`(:229) = 구성 내 상품(다품이면 N개), `profile_code`. ✅
- `BundleOption`(:254) = 선택 옵션, `market_visible`(부분집합 선택)·`market_option_id`. ✅
- **갭**: **수량(2개세트)** 필드 없음 → `BundleProduct.quantity`(또는 BundleOption) 신설 필요.

### 13-4. ⚠️ V1/V2 현실 — 전면 V2 전환은 대공사
V2 `BundleSet`은 **V2 소싱 레이어**(`sourcing_profiles`/`sourcing_options`, 현재 비어있음)에 FK 의존.
그러나 **라이브(크롤·재고·가격) 전부 V1**(`Model`/`Option`) 위에서 동작. 전면 V2 전환 = 소싱 레이어 통째 마이그레이션 = 대규모·고위험.
- **✅ 결정(2026-06-26, 사용자 확정): A = V1 위 경량 구성 레이어**. V2의 검증된 모양(set→product→option)만 차용하되 `canonical_sku`로 **V1 옵션 참조**. V2 소싱 마이그레이션 회피. 라이브 돈 파이프라인(크롤·재고·가격) 무손상이 핵심 근거.
- (기각) B 전면 V2 전환: 돈 직결 파이프라인 이사 = 고위험. 필요 시 후일 별건.

### 13-5. P0 영향 (이미 구현된 부분)
- P0 매칭 코어(`linker.py`)·조회 어댑터(`market_fetch.py`)·읽기검증(`verify_smartstore_read.py`) = **그대로 재사용**(옵션↔마켓옵션 매칭은 불변).
- `link_service.link_bundle_market`(모음전 단위) → **구성(set) 단위로 승격** 필요(`link_set_market`). `MarketRegistration`에 `set_id`(또는 신규 `set_registration`) 추가.
- 라우트 `POST /api/bundles/<code>/link/<market>` → 구성 단위 `POST /api/sets/<set_id>/link` 로 재정의.

### 13-6. UI 영향 — '구성 관리' 표면 신설 (결정 1~4 재정의)
연동 UI는 **구성(세트) 단위**가 된다. 새 표면 필요:
- **구성 목록·생성·편집**: 모음전 안에서 구성 만들기(옵션 부분집합 선택 + 수량 + 다품).
- **구성별 판매처 연동**: 각 구성 → 판매처 상품번호·옵션 선택.
- → 앞서 만든 결정 1(연동 배치) 시안들은 **구성 레이어 위에서 다시** 잡는다(구성 선택 → 그 구성의 옵션·연동).

### 13-7. 업무 순서도 갱신 (구성 레이어 = 별도 워크스트림)
- **신규 결정 게이트**: 13-4 V1경량 vs 전면V2 → 사용자 확정.
- **구성 레이어 spec/plan**(별도): 모델(set/product/option + quantity) → 구성 CRUD → 구성별 연동(P0 코어 재사용) → 구성별 재고/판매가 push(P2).
- 기존 P0 백엔드는 "구성 단위로 승격" 리팩터 대상(폐기 아님, 매칭·조회·검증 코어는 보존).

## 14. 판매처 연동 워크플로 — 확정 흐름 (2026-06-26, 사용자 정의)

사용자가 화면 2장(① 기존 탭바 기본/옵션/마켓, ② **조합 매트릭스 102/126** 색×사이즈 ON/OFF 그리드)과 함께 정의한 실제 작업 순서. **"검색 우선"의 전용 흐름**(옵션 구성은 한 모음전이 아니라 여러 모음전 대상이라 검색으로 시작)이며, 결과는 ③마켓 탭에 판매처 상품 매핑으로 반영된다.

### 14-1. 6단계 흐름
1. **(목표) ③마켓 탭에서 판매처 상품 매핑** — 연동 결과(상품번호↔구성)는 마켓 탭에 표시·관리.
2. **판매처별 API 환경 상이** — 단계 6의 입력 폼·전송 필드가 판매처마다 다름(§12-2 마켓별 API 조사 산출이 이 폼을 정의).
3. **모음전 검색** — 흐름의 시작점. 여러 모음전을 대상으로 하므로 먼저 모음전을 검색·선택.
4. **옵션 구성(선택)** — 검색된 모음전의 **조합 매트릭스(색×사이즈 ON/OFF 셀, "전체 선택/해제")를 그대로 재사용**해 옵션 부분집합 선택. **CRUD**(추후 상품 구성 변경에 따라 추가/수정/삭제 가능).
5. **판매처 선택** — 이 구성을 올릴 판매처 선택. **CRUD**(추가/수정/삭제 가능).
6. **판매처별 API 전송 정보 입력** — 선택한 판매처마다 API 전송에 필요한 필드 입력(마켓별 상이).

### 14-2. 설계 함의
- 이 흐름이 **V1 경량 구성(set)을 생성·편집**한다: (모음전 + 조합매트릭스 선택 옵션 + 수량/다품) = 1 구성 → 선택 판매처별 연동·API필드.
- **옵션 선택 UI = 기존 조합 매트릭스 재사용**(신규 격자 만들지 않음. 메모 [[feedback_matrix_ui_prefs]] 준수). 비활성('·') 셀은 미판매 조합 — 선택 불가.
- **검색 우선 = 전용 화면/진입점** 필요(모음전 edit 탭 안에 가두지 않음). 결과 매핑은 ③마켓 탭에서 확인.
- 판매처별 API 필드 폼은 **마켓별 프로필(docs/markets/<market>.yaml)** 로 정의 → 폼 동적 렌더.
- CRUD 필수(구성·판매처 모두) → 구성/연동은 영속·재편집 가능한 엔티티.

### 14-3. 결정 갱신
- 결정 1(연동 배치) 재정의: **전용 "판매처 연동" 검색우선 흐름**(4단계 마법사형: 모음전 검색 → 옵션구성(조합매트릭스) → 판매처 선택 → API필드) + 결과는 ③마켓 탭 매핑. 앞선 구성-관리 시안 1~5 중 "마법사형(시안4)+검색우선"에 수렴.
- 옵션 부분집합 선택 UI 결정: **조합 매트릭스 재사용** 확정(별도 시안 불요).
