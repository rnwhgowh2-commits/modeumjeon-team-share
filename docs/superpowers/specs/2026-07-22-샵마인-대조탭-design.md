## 2. 엔진 (lemouton/markets/shopmine_recon.py 신설 권장, TDD)

### 2.1 파싱
- 구형 .xls(OLE2) — pandas.read_excel(dtype=str). 컬럼명 유니코드 전각공백(　) 제거.
- ⚠️ pandas **itertuples 금지**(괄호 컬럼 '정산예상금액（배송비포함）' 깨짐) → `to_dict("records")`.
- 쇼핑몰 접두 매핑: 01.지마켓→G마켓 / 02.옥션 / 03.11번가 / 04.스마트스토어 / 06.쿠팡 / 18.롯데온.

### 2.2 계정 매핑 (확정 규칙 — 이름표 무시, 주문번호가 진실)
마켓별 (샵마인계정 × 우리계정) 주문번호 교집합 → 최다 계정으로 자동 확정.
교집합 0 = 미등록(제외 목록으로 표시). 복수 계정에 분산되면 「중복 등록 의심」 표시.
확정 매핑 24계정은 세션에서 검증 완료(11번가 박스↔브랜드위시 이름 교차·바이(스스)=브랜드트립 포함).

### 2.3 행 페어링
1. (마켓, 오픈마켓주문번호) 정확 일치.
2. 다품 주문: 옵션 정규화(공백 제거·HTML unescape·소문자) 일치 → 없으면 (단가,수량) 일치.
3. 남으면 **판정불가**(별도 표 — 현재 153라인). 일치/불일치/판정불가 3분류만 존재.
- 우리 쪽은 `_kind!="change"` 주문행 우선, 없으면 클레임행.

### 2.4 필드 대조 — ★A안 재현식 (2026-07-22 전건 실측으로 확정)
**샵마인 돈 숫자 = [정가→실결제→정산] 자체 항등식 체인(계산값). 우리 = 마켓 원본.**
비교는 우리 원본을 그대로 두고, 샵마인 정의를 재현해 대조한다:

| 필드 | 샵마인 재현식 | 증거 |
|---|---|---|
| 실결제(스스·쿠팡·G마켓 일부) | `(단가+옵션추가금)×수량` (정가총액) | 스스 326/326·쿠팡 109/111·G마켓 38 |
| 실결제(롯데온) | `정가 − 셀러부담할인` (이커머스부담 미차감) | 66/75 (나머지는 전체할인 차감과 동치) |
| 실결제(11번가·옥션·G마켓) | `샵 정산예상 + 수수료` (일부 `−고객배송비`) | 11번가 141/143·옥션 13/13·G마켓 94/94 |
| 정산(전 마켓) | `샵 실결제 − 수수료` (일부 `+고객배송비`) | 스스 292/292·쿠팡 288/289·롯데온 205/209 |
| 정산(ESM 우리 쪽) | 우리 정산(real)에 **배송비 가산** 후 비교 | sell_source._settlement_for 배포됨 |
| 주문일 | 날짜(YYYY-MM-DD) 추출 후 비교 | 99.5%+ |
| 단가·수량 | 정수 정규화 비교 | 99%+ |
| 송장 | 대조 제외(샵마인이 대부분 공란 — 우리가 더 풍부) | 실측 |
| ±6원 끝수 | 「허용 차이(반올림)」 별도 분류 | 스스331·쿠팡580 |
- 재현식으로도 안 맞으면 「불일치」로 정직 표시(개별 확인 대상 — 현재 ~30건 수준).
- 롯데온 상품명 HTML(`&lt;`) unescape 정규화.
