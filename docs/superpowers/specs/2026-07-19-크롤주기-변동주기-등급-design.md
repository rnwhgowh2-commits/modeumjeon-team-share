## 0. 이 문서가 정하는 것

> 소싱처 상품 3,000개를 **하루에 몇 번 크롤할지**, 그리고 **바뀐 걸 언제 마켓에 올릴지**.

크롤은 돈이 아니라 **시간**을 씁니다. 전부 하루 1회씩 돌면 자주 바뀌는 상품은 놓치고,
안 바뀌는 상품에는 헛돕니다. 상품마다 **실제 변동 주기**를 재서 계수를 차등하는 게 목적입니다.

---

## 1. 잣대 — 강도(%)

```
강도(%) = (기간 내 변동 횟수 ÷ 기간 일수) × 100
평균 주기(일) = 기간 일수 ÷ 변동 횟수
```

**하루에 1회 바뀜 = 100%** 입니다.

| 예시 | 계산 | 결과 |
|---|---|---|
| 30일에 5회 | 30 ÷ 5 | 평균 6일에 1회 · **16.7%** |
| 30일에 30회 | 30 ÷ 30 | 평균 1일에 1회 · **100%** |
| 30일에 75회 | 30 ÷ 75 | 하루 2.5회 · **250%** |

측정 기간은 **1 / 3 / 7 / 14 / 30일** 다섯 벌을 모두 유지합니다.
30일만 보면 「어제만 난리였던 상품」을 못 가립니다.

### 1-1. 화면 표기 방향

가로축은 **왼쪽이 과거, 오른쪽이 최근** — `30일전 → 14일전 → 7일전 → 3일전 → 1일전`.

> ⚠️ 각 점은 「그날 하루」가 아니라 **「그때부터 지금까지의 평균」**입니다.
> 오른쪽으로 오를수록 = 최근에 심해졌다. 평평 = 원래 그렇다.
> 시계열 그래프가 아니므로 화면에 이 문구를 명시합니다.

## 2. 🔴 크롤 기준축 = 「하나라도 변동」 (합집합)

**결정 1-A.**

크롤은 **URL 하나를 한 번 도는 행위**입니다. 표면가가 바뀌든 재고가 바뀌든
그 URL 은 어차피 다시 가져와야 하고, 한 번 가져오면 4축을 모두 얻습니다.

따라서 계수 기준은 축별이 아니라 **합집합**입니다.

```
union_intensity = (넷 중 하나라도 바뀐 날 수 ÷ 기간) × 100
```

| 방식 | 하루 크롤 (3,000개 예시) | 문제 |
|---|---|---|
| **합집합** ✅ | 2,980회 | — |
| 최댓값 축 | 2,560회 | 축이 서로 다른 날 바뀌면 놓침 |
| 축마다 따로 | 9,400회 | 같은 걸 4번 가져옴 |
| 4축 합산 | 4,200회 | 같은 날 둘 바뀌면 2로 셈 (중복) |
| 재고(주요)만 | 2,100회 | **가격만 바뀌는 상품을 놓침 → 역마진** |

> **합집합 ≠ 합산.** 같은 크롤 회차에 표면가와 재고가 같이 바뀌었으면 **1회**로 셉니다.
> 그래서 `max(축별) ≤ 합집합 ≤ Σ(축별)` 이 항상 성립해야 하고, 이걸 불변식 테스트로 둡니다.

**축별 강도는 계속 저장합니다** — 크롤 계수에는 안 쓰지만 §5 업로드 게이트와 화면 분석에 씁니다.

### 2-1. 크롤 방식 = **상세 전수** (전략 1 확정)

**사장님 결정 2026-07-19.** 목록 페이지 스캔은 쓰지 않습니다. 등급 계수대로 **상세 페이지만** 돕니다.

> 목록 감지(전략 2·3)를 안 쓰므로 **「목록가 == 상세가」라는 미검증 가정을 지지 않습니다.**
> 대가 — 조용한 등급(주간·월간·없음) 상품은 가격도 **3일에 1회**만 확인합니다.
> 그 사이 소싱처 가격이 오르면 최대 3일간 역마진으로 팝니다. **하한(§3-2)이 이 위험의 상한선**입니다.

### 2-2. 🔴 같은 소싱처 안에서 상품 중복 제거

**사장님 결정 2026-07-19.**

같은 상품이 한 소싱처의 **여러 URL 구성**에 걸치면 한 랩에 두 번 크롤됩니다.

```
무신사 > 나이키    ▸ 덩크 로우 팬다      ┐
무신사 > 스니커즈  ▸ 덩크 로우 팬다      ┘  같은 소싱처 · 같은 상품 → 1회로 합침
SSG   > 나이키    ▸ 덩크 로우 팬다         다른 소싱처 → 그대로 크롤 (가격이 다름)
```

- **중복 판정 키 = (소싱처, 소싱처 상품 id)**
- 계수가 다르면 **높은 쪽**을 따름
- **타 소싱처의 동일 상품은 합치지 않음** — 소싱처마다 가격·혜택·재고가 다르므로 각각 필요
- 통계는 **원래 구성마다 각각 기록** — 어느 구성이 바쁜지 따로 봐야 하므로

## 3. 등급 — 6칸

**결정 2-A.** 30일 합집합 강도로 자동 분류합니다.

| 등급 | 경계 | 제안 계수 |
|---|---|---|
| 하루 2회 이상 | 200% 이상 | 2회/일 |
| 하루 1회 | 100 ~ 200% | 2회/일 |
| 2~3일에 1회 | 33 ~ 100% | 1회/일 |
| 주간급 | 14 ~ 33% | 2일 1회 |
| 월간급 | 3 ~ 14% | 3일 1회 |
| 변동 없음 | 0% | 3일 1회 |

### 3-1. 상한 = 하루 2회

**결정 4-A (수정).** 강도가 500% 여도 **하루 2회**를 넘지 않습니다.
→ 최대 12시간마다. 소싱처 차단 위험과 크롤 총량을 우선합니다.

> 상위 두 등급이 같은 계수(2회/일)로 수렴하지만, **등급은 그대로 둡니다.**
> 상한을 나중에 올릴 때 등급이 이미 나뉘어 있어야 하고, 화면에서 「상한에 걸린 상품」을 보여줘야 하기 때문입니다.
> 화면에 `제안 3회 → 상한 2회로 조정됨` 을 표기합니다.

### 3-2. 하한 = 3일 1회

**결정 3-A.** 30일 내내 안 바뀐 상품도 **3일에 1회**는 돕니다.

> 강도가 10% 라고 10일에 한 번만 돌면 안 됩니다 — **언제 바뀔지는 모르기 때문**입니다.
> 강도는 「아예 안 바뀌는 건 뜸하게, 심한 건 자주」를 정하는 근거일 뿐, 예측이 아닙니다.

## 4. 🔴 모든 수치는 제안값 — 최종 결정은 사장님

**결정 1·3·4·5 전체를 관통하는 원칙.**

프로그램은 통계를 근거로 **제안만** 합니다. 실제 적용되는 값은 사장님이 화면에서 설정합니다.

| 항목 | 프로그램 | 사장님 |
|---|---|---|
| 등급 경계값 (200/100/33/14/3%) | 기본값 제시 | **직접 수정** |
| 등급별 계수 | 통계 기반 제안 | **직접 수정** |
| 하한 (3일 1회) | 기본값 제시 | **직접 수정** |
| 상한 (하루 2회) | 기본값 제시 | **직접 수정** |
| 상품 개별 계수 | 등급값 자동 상속 | **개별 덮어쓰기 가능** |

### 4-1. 계수 적용 = 확인 후

**결정 5-B.** 통계가 갱신돼 등급이 바뀌어도 **자동으로 적용하지 않습니다.**

```
통계 갱신 → 「이렇게 바꾸는 게 좋겠습니다」 제안 목록 → 사장님 확인 → 적용
```

제안 목록에는 **왜 그런지**가 함께 나와야 합니다 — `30일 75회 · 하루 2.5회꼴 — 지금 1회로는 놓침`.

> 제안을 **무시한 이력**도 남깁니다. 같은 제안을 매일 다시 띄우면 피로해집니다.
> 무시한 제안은 강도가 한 등급 이상 더 움직이기 전까지 다시 올리지 않습니다.

### 4-1-1. 🔴 단위를 잇는 다리 — 「하루 N회」 ↔ crawl_weight

**구현하며 드러난 문제.** 두 단위가 다릅니다.

| | 단위 |
|---|---|
| 등급 제안 (`crawl_grade.py`) | **하루 N회** — 실수, `1/3` 같은 값 포함 |
| 기존 스케줄러 (`crawl_schedule.py`) | **crawl_weight** — 유효간격 = 기준주기 ÷ weight × 완화배수 |

환산 자체는 됩니다. 기준주기가 하루면 `하루 2회 → weight 2`, `3일 1회 → weight 1/3`.

**그런데 함정이 있습니다.**

```python
# crawl_schedule.effective_interval_seconds
weight = int(crawl_weight)      # ← 절삭
if weight <= 0:
    return float("inf")         # ← 계수 0 = 크롤 제외
weight = min(5, weight)         # ← 상한 5
```

`weight 1/3` 을 그대로 넣으면 `int(0.33) == 0` 이 되고, **0 은 「크롤 제외」**입니다.
「3일에 1회로 뜸하게 돌자」는 제안이 **「아예 돌지 말자」로 뒤집힙니다.**

**대응** — `crawl_grade_apply.py` 가 표현 가능 여부를 **먼저 판정**합니다.

| 계수 | 적용 |
|---|---|
| 1~5 사이 **정수** | ⭕ 그대로 적용 |
| 1 미만 (`1/3`, `0.5`) | ❌ **막음** — 소수점이 잘려 크롤 제외가 됨 |
| 5 초과 | ❌ 막음 — 5로 깎임 |
| 정수 아님 (`1.5`) | ❌ 막음 — 소수점 잘림 |

막힌 제안도 **조용히 버리지 않고** 이유를 달아 화면에 보여줍니다.
안 그러면 사장님이 「왜 안 내려가지」 하게 됩니다.

### 4-1-2. ✅ 해결 — 손잡이를 둘로 나눔 (사장님 승인 2026-07-19)

`crawl_weight` 는 **Integer 컬럼**이라 타입을 바꾸려면 ALTER COLUMN 이 필요한데,
이 프로젝트는 Alembic 이 없고 `_apply_lightweight_migrations()` 가 **ADD COLUMN / CREATE INDEX 만**
지원합니다. 그래서 **기존 컬럼은 그대로 두고 새 컬럼을 붙였습니다.**

```
유효간격 = 기준주기 ÷ 계수 × 느리게배수 × 완화배수
```

| 손잡이 | 컬럼 | 방향 | 범위 |
|---|---|---|---|
| `crawl_weight` | INTEGER (기존) | **자주** | 1~5 정수 |
| `crawl_slowdown` | **FLOAT (신규)** | **뜸하게** | 1.0 이상 |

```
하루 2회  →  계수 2 · 느리게 1.0
3일 1회   →  계수 1 · 느리게 3.0
```

**왜 방향을 갈랐나** — 느리게배수가 1 미만이면 「더 자주」가 되는데 그건 계수가 할 일입니다.
두 손잡이가 같은 방향을 조절하면 **어느 쪽이 이겼는지 알 수 없게** 됩니다. 1 미만은 거부합니다.

**하위호환** — `DEFAULT 1.0`, 그리고 코드에서 `None → 1.0`. 기존 행은 동작이 하나도 안 바뀝니다.
`계수 0 = 크롤 제외` 규칙도 그대로 유지됩니다(느리게배수가 뭐든 0이면 제외).

> **아직 안 한 것** — `CrawlWeightRule.slowdown` 컬럼은 만들었지만 **캐스케이드에 안 태웠습니다.**
> `resolve_crawl_weight` 가 계수 하나만 돌려주는 구조라 같이 꺼내려면 반환형을 바꿔야 하고,
> 그건 라이브 스케줄러를 더 크게 건드립니다. 지금은 **상품 단위 컬럼만** 반영됩니다.

### 4-2. 새로 생기는 화면 — 위치는 **대량등록 모드**

> 🔴 **위치 = 대량등록 모드** (사장님 확정 2026-07-19).
> 본 문서의 결정 7항목은 **전부 대량등록 탭 이야기**입니다.
> **모음전 모드는 지금 그대로 둡니다** — 모음전 쪽 기존 자동화 설정은 건드리지 않습니다.
>
> ⚠️ 이 문서 초안은 「모음전 탭」으로 잘못 적었다가 정정했습니다.
> 혼선의 원인 — 모음전에도 자동화 설정 화면이 이미 있어서, 사장님이 그걸 가리켜
> 「자동화설정은 모음전 탭이야」라고 하신 걸 제가 **이 결정들의 목적지**로 잘못 읽었습니다.
> 사장님 말씀은 「그건 모음전 거고, 우리는 지금 대량등록을 다루는 중」이라는 뜻이었습니다.

**들어갈 곳** — 대량등록 8탭 중 ①데이터수집 · ③데이터전송 · ⑧설정에 나눠 붙습니다(§6).

- 등급 경계값 6칸 입력
- 등급별 계수 6칸 입력
- 하한 · 상한 입력
- 업로드 속도 제한 (§5-2)
- 상품 개별 계수 덮어쓰기 목록

## 5. 마켓 업로드 — 크롤과 분리

**결정 6-A.** 크롤은 자주 해도, 업로드는 거릅니다.

| 축 | 업로드 |
|---|---|
| 표면가 변동 | **항상** (재고 무관) |
| 혜택 변동 | **항상** |
| 재고 — 주요 | **올림** = 0이 됨 · 품절이었다 생김 · 2개 이하로 떨어짐 |
| 재고 — 그 외 | 올리지 않음 (5→3 등) |

> 가격이 틀리면 곧바로 돈이 샙니다(역마진 또는 경쟁력 상실). 재고는 의미 있는 것만.
> 예시 기준 하루 업로드가 2,900건 → **1,100건**으로 줄어듭니다.

### 5-1. 🔴 즉시 업로드 + 상품별 하루 2회 상한

**사장님 확정 2026-07-19.** 정시 배치가 **아닙니다.**

```
기본        변동이 잡히면 → 바로 업로드
상한        상품 하나당 하루 최대 2회   ← 여유가 없을 때만 걸림
```

- **여유가 되면 바로바로 올립니다.** 지연을 일부러 두지 않습니다.
- 올릴 게 너무 많을 때를 대비한 **안전장치**가 상품별 상한입니다.
- 상한은 **마켓 전체 속도 제한(§5-2)과 별개**입니다. 이건 *한 상품이* 하루에 몇 번 마켓을 건드리냐입니다.

> 근거 — 소싱처 가격이 바뀌는 시간대가 대체로 정해져 있습니다(밤 12시경, 오전 11시~오후 1시경).
> 그래서 한 상품을 하루 2번 이상 갱신할 일이 실제로 거의 없습니다.
> 재고는 수시로 변하지만 **매번 올리는 건 의미가 없습니다.**

**상한에 걸리면 버리지 않습니다.** 마지막 값만 들고 있다가 다음 슬롯이 열릴 때 **최신 상태로** 올립니다.
중간 값들은 어차피 §5-3 합치기에서 하나로 접힙니다.

### 5-1-1. 🔴 품절은 상한 예외 — 무조건 즉시 (확정)

**사장님 확정 2026-07-19.** 「품절은 빠르게 무조건 빼야 함.」

```
재고 0 (품절)   →  상한을 이미 다 썼어도 즉시 업로드     ← 예외
그 외           →  상품당 하루 2회 상한 적용
```

| 변동 | 상한 |
|---|---|
| **재고 0 (품절)** | **면제** — 몇 번이든 즉시 |
| 가격 · 혜택 | 적용 |
| 재입고 · 재고 증감 | 적용 |

근거 — 품절 상품을 계속 팔면 **주문 받고 취소 → 마켓 페널티 · 고객 이탈**입니다.
호출 몇 번보다 이 손실이 큽니다.

> **구현 주의** — 상한 카운터를 **차감하되 막지는 않습니다.**
> 품절 때문에 카운터가 초과돼도 전송을 거르지 않고, 화면에 「상한 초과(품절 예외)」로 표시합니다.
> 카운터를 아예 안 세면 **얼마나 초과했는지 모르게** 됩니다.
>
> **마켓 속도 제한(§5-2)은 여전히 적용됩니다.** 품절이라도 큐에는 들어가고, 다만 §5-1 우선순위에서
> **가격 다음 순위**로 앞자리를 받습니다. 속도 제한까지 뚫으면 마켓이 차단합니다.

### 5-2. 🔴 단, 속도 제한 (신규 요구사항)

**사장님 지시.** 너무 빠르면 마켓 API 가 차단합니다. **초당/분당 건수 상한**을 둡니다.

**형식 = 「X초에 Y개 이내」** (사장님 확정). 초당 환산이 아니라 **두 값을 그대로 입력**받습니다.

```
마켓별 rate limit   window_seconds · max_count
```

**⚠️ 실제 한도 (2026-07-19 조사 · `marketplace_api_map.json` 610건 기준)**

| 마켓 | 확인된 한도 | 출처 |
|---|---|---|
| 쿠팡 | **분당 50회** (로켓그로스 주문 조회) | 공식 문서 인용 |
| 옥션 · G마켓 | **5초당 1회** (주문 조회) | 공식 문서 인용 |
| 스마트스토어 · 롯데온 · 11번가 | **미확인** | — |
| **업로드(상품·가격·재고) 한도** | **전 마켓 미확인** | 위 둘은 주문 조회 한도 |

> 🔴 이 문서 초안에 「스스 1초에 10개」 같은 숫자를 예시로 적었다가 **지웠습니다.**
> 근거 없는 임의값이었고, 남겨두면 나중에 그게 확인된 값인 줄 알고 쓰게 됩니다.
>
> 🔴 **ESM 5초당 1회**는 지금 구조(계정당 초당 1개)보다도 느립니다 —
> 옥션·G마켓 주문 조회를 붙일 때 반드시 별도로 다뤄야 합니다.

> 초당 1개로 환산해버리면 「10초에 30개(순간 5개 몰림 허용)」 같은 한도를 못 담습니다.
> 그래서 **창(초) + 개수** 두 칸으로 받고, 슬라이딩 윈도로 셉니다.

- 값은 **사장님이 화면에서 설정** (§4-2 자동화 설정) · 나중에 조절 가능해야 함
- 상한을 넘으면 **큐에 쌓고 순서대로** — 버리지 않습니다
- 대기가 길어지면 화면에 `대기 N건 · 예상 M분` 표시
- **우선순위** — 가격 변동 > 품절 > 재입고 > 나머지 (역마진 위험이 큰 순서)

> ⚠️ 마켓별 실제 한도는 **아직 확인 안 했습니다.** 위 숫자는 임의값입니다.
> 각 마켓 API 문서에서 실제 한도를 확인해 `marketplace_api_map.json` 에 넣어야 합니다.

### 5-3. 🔴 업로드 중복 제거 — 네 겹

**사장님 지적** — 「가격/재고 변동마다 다 업로드하면 이중 삼중」, 「크롤 후 변동 있는 걸 업로드해야 의미 있다」.

| 겹 | 무엇을 막나 | 예시 (하루) |
|---|---|---|
| **0. 게이트** (§5) | 올릴 가치 없는 재고 변동 | 2,850 → 1,046 |
| **1. 목표값 대조** 🔴 | 비교 대상을 **직전 크롤값이 아니라 「마지막으로 마켓에 성공적으로 올린 값」**으로 | 1,046 → 820 |
| **2. 합치기** | 상한에 걸려 대기 중인 같은 상품 변동을 **마지막 값 1건**으로 | 820 → 640 |
| **3. 상품 단위 묶기** | 옵션 여러 개가 바뀌어도 마켓 호출은 **상품당 1회** | 640 → 410 |

**1번 겹이 사장님 말씀의 핵심입니다.**

```
소싱처   100,000 → 110,000 → 100,000     하루 2번 변동
마켓 목표    112,400 →   —   →    —        바뀐 적 없음
→ 업로드 0회
```

직전 크롤값과 견주면 왕복 변동을 **두 번** 올립니다. `last_confirmed_snapshot`(마켓에 실제 올라간 값)과 견줘야 **0번**이 됩니다.

> 상한(§5-1)에 걸린 상품에서만 2번 겹이 작동합니다. 여유가 있으면 즉시 올리므로 합칠 일이 없습니다.
> 상한에 걸린 상품은 하루 6번 바뀌어도 **최대 2건**만 나갑니다.

> 프로젝트 최상위 원칙(중복·모순 금지)에 걸립니다.

### 6-1. 등급 상세 펼침

**결정 7-A.** 등급 카드를 누르면 **바로 아래에 상세가 펼쳐집니다.**

```
[축 고르기]  표면가 · 혜택 · 재고(전체) · 재고(주요) · 하나라도

[등급 카드 6개]  ← 누르면 아래가 바뀜
[요약 3칸]  지금 / 권장 / 놓치던 상품

[상세]  선택한 등급의 상품 목록
        상품 · 추세(30일전→1일전) · 판정 · 평균주기 · 뜻 · 마켓 업로드
```

- 기본 축 = **하나라도 변동** (크롤 계수의 기준이므로)
- 판정 = 최근 몰림 / 꾸준 / 잠잠해짐 / 조용
- 시안 = `바탕화면/모음전 시안 11_확정_등급 + 추세상세.html`
