# 전체 로컬 크롤 + 적응형 동시성 + 실시간 로그 대시보드 — 설계

> 단일 진실 원천: 본 문서(이 기능 한정). 도메인 룰 = 프로젝트 CLAUDE.md.

## 1. 목적 (Why)

현재 크롤은 **하이브리드**다 — 무신사·롯데온만 로컬 크롬 확장이 긁고, 나머지 4개(르무통·SSF·SSG·스스르무통)는 서버(AWS)가 긁는다. 사용자는 **전부 로컬 PC에서, 눈으로 보이는 창으로** 돌기를 원한다. 목적:

- 「전체 크롤」을 누른 **그 PC**가 6개 소싱처를 모두 로컬에서 수집(다른 PC가 눌러도 그 PC가 수행).
- **소싱처별로 보이는 새 창**을 띄워 진행을 눈으로 확인.
- PC 사양·상태에 맞춰 **동시 창 수를 자동 조절**(빠르면 늘리고, 느리거나 과부하면 줄임).
- **실시간 로그 대시보드**로 진행·가격·조절 결정·CPU/메모리를 실시간 확인.

비목표(YAGNI): 서버 완전 제거(저장·파싱은 서버 유지), 클라우드 분산 크롤, 다중 PC 동시 분배.

## 3. 아키텍처

```
[전체 크롤 클릭]
   └─ toss.js: 확장 설치 확인
        ├─ 설치됨 → ext_bridge.crawlBundleAll(code)  ← 신규(6개 전부, 적응형)
        │            ├─ 무신사·롯데온: 기존 JS 추출기(API/DOM)
        │            └─ 르무통·SSF·SSG·스스르무통: 창 HTML → /api/sources/parse
        └─ 미설치 → (폴백) 기존 서버 run-now 크롤 + 안내
   └─ 결과 전부 → POST /api/sources/crawl-result → SourceProduct 갱신 → 매트릭스 reload
```

### 3.1 컴포넌트 & 책임

| 컴포넌트 | 책임 | 의존 |
|---|---|---|
| `extension/background.js` | 소싱처별 **창**(chrome.windows.create) 열기·HTML 수집·무신사/롯데온 추출·CPU/메모리 측정(chrome.system.*) | chrome.windows/system API |
| `extension/manifest.json` | 권한 추가: `system.cpu`, `system.memory`. (windows는 권한 불필요) 버전 0.4.0 | — |
| `webapp/static/ext_bridge.js` | **적응형 컨트롤러**(동시성 결정) + 로그 이벤트 방출 + 서버 호출(/parse, /crawl-result) | window.MoumExt |
| `webapp/static/crawl_log.js` (신규) | 대시보드 로그 UI 렌더(게이지·소싱처 진행·로그 스트림) | ext_bridge 이벤트 |
| `webapp/routes/api_sources_parse.py` (신규) | `POST /api/sources/parse {source_key,url,html}` → 크롤러 `parse_html` 호출 → 구조화 결과 | crawlers |
| `lemouton/sourcing/crawlers/*.py` | 각 크롤러에 **`parse_html(html, url) -> CrawlResult`** 노출(기존 `_parse_*` 재사용, fetch와 분리) | 기존 파서 |
| `webapp/routes/api.py` run-now | 확장 경로일 때 서버 크롤 **스킵**(미설치 시에만 폴백) | — |

### 3.2 데이터 흐름 — 비로그인 소싱처 (A안)

```
ext_bridge: 소싱처별 URL 목록 수집(option-matrix)
 → background: chrome.windows.create({url, focused:false/true})  ← 보이는 창
 → 탭 로드 완료 대기 → executeScript: document.documentElement.outerHTML 반환
 → 창 닫음
 → ext_bridge: POST /api/sources/parse {source_key, url, html}  (로그인 세션 = 페이지 컨텍스트)
 → 서버: crawlers[source_key].parse_html(html, url) → {price, stock, options, benefits...}
 → ext_bridge: 누적 → 끝나면 POST /api/sources/crawl-result
```

무신사·롯데온은 창 안에서 기존 추출기(executeScript)로 직접 가격 추출(서버 parse 불필요).

## 4. 적응형 동시성 컨트롤러 — 처리량 언덕오르기 (시간이 주 신호)

**주 신호 = 1건당 크롤 시간(latency).** 이걸로 **유효 처리량 = concurrency ÷ 평균 latency**(분당 완료 건수)를 계산해 가장 빨리 끝나는 창 수를 찾는다. CPU·메모리는 **2차 보조**(안전 브레이크 + 원인 설명)로만 쓴다 — 늘릴지/줄일지는 시간(처리량)이 결정.

> 왜 단순 "1건 시간"이 아니라 처리량인가: 창을 늘리면 1건당 시간은 자연히 조금 늘지만 동시에 여러 건이 끝나 **총 속도(처리량)는 오를 수 있다.** 1건 시간만 보면 못 늘리므로, 측정은 시간으로 하되 판단은 처리량(창수÷시간)으로 한다.

상태: `concurrency`(동시 창), `cap`(상한), `floor=1`, `best`(지금까지 최고 처리량과 그때의 concurrency).

### 4.1 초기화 (크롤 전 1회)
- `cap = clamp(navigator.hardwareConcurrency − 1, 1, 5)`
- `deviceMemory < 4` → `cap = min(cap, 2)` (저사양 보호)
- 사용자 설정(자동/1~5)이 있으면 그 상한 우선
- `concurrency = 1` 로 시작

### 4.2 주 루프 — 처리량 언덕오르기 (시간 기반)
- 각 창 크롤 완료 시 `latency` 기록 → 현재 concurrency의 **유효 처리량 = concurrency ÷ (최근 latency 중앙값)** 갱신.
- **탐침(+1)**: `concurrency < cap` 이고 안전(§4.3)하면 창 +1 → N건(쿨다운) 측정.
  - 처리량이 직전보다 **개선** → 채택(`best` 갱신), 다시 +1 탐침.
  - 1건 시간이 너무 늘어 처리량이 **하락/정체** → 직전 concurrency로 **되돌림(−1)** 후 고정.
- **주기적 재탐침**: 일정 건수마다 다시 +1 시도(사이트·PC 상황 변화 추적).
- 진동 방지: 변경 후 최소 N건(예 3건) 재판정 보류. latency는 **중앙값**으로 노이즈 완화.

### 4.3 보조 신호 — CPU·메모리 (2차, 역할 2가지뿐)
1. **안전 브레이크**: 처리량이 더 오를 것 같아도 `cpu ≥ 90%` 또는 `mem ≥ 90%` 면 +1 금지, `≥ 95%` 면 강제 −1. (PC 멈춤·사용자 작업 보호용 — 조절의 주체 아님)
2. **원인 설명(로그)**: 1건 시간이 뚜렷이 늘면 로그에 "느려짐 — CPU 92%가 원인" 식 주석. 조절 판단은 처리량이 하고, 이건 사람에게 *왜* 느려졌는지 알려주는 용도.
- `system.cpu`/`system.memory` 권한 없거나 측정 불가 → 보조 신호 생략, **처리량(시간)만으로 정상 동작.**

### 4.4 안전장치
- 하드 상한 5창(사이트 부하·차단 방지). 소싱처 1곳당 동시 1창(같은 소싱처 병렬 금지 — 차단 위험↓).
- 창 열기 실패/타임아웃(25s) → 해당 URL 에러 기록 후 다음 진행, `concurrency −1`.
- 사용자 수동 정지 버튼 → 진행 중 창 닫고 누적분만 저장.

## 5. 실시간 로그 대시보드 (시안 8)

위치: 모음전 옵션탭 우측(또는 본문 상단) 대시보드 영역. 구성:
- **게이지 3종**: 동시 창(n/cap), CPU(%), 메모리(%) — 실시간 갱신.
- **소싱처별 진행 행**: 6개, 상태(대기/진행/완료)·진행바·N/M.
- **로그 스트림**(자동 스크롤, 시각·아이콘·메시지): 창 열림, 소싱처 완료+가격, 적응형 조절(⚡+1 / ⬇−1 / 🔥자원), 완료 요약.
- 로그 레벨 색: 성공(파랑)·증가(초록)·감소·경고(주황/빨강).

데이터 경로: ext_bridge가 이벤트(`{type, ts, source, msg, level, metrics}`) 방출 → crawl_log.js 구독·렌더. 별도 서버 불필요(클라이언트 이벤트).

## 6. 에러 처리

| 상황 | 처리 |
|---|---|
| 확장 미설치 | 폴백: 서버 run-now(기존) + "확장 설치 시 전체 로컬 가능" 안내 |
| 무신사/롯데온 비로그인 | 추출기가 비회원가 반환 + 로그에 "⚠️ 비로그인" 경고(회원가 누락 방지) |
| /parse 실패(셀렉터 불일치) | 해당 소싱처 error 기록, 다른 소싱처 계속, 로그 경고 |
| 창 차단/CAPTCHA | 해당 URL skip + 로그, 절대 자동 우회 안 함 |
| CPU/메모리 권한 없음 | 가드 비활성 → latency 신호만으로 조절(권한 없어도 동작) |

## 7. 검증 게이트 (가격 정확도 — 필수)

각 소싱처(르무통·SSF·SSG·스스르무통) 최소 1 URL에 대해:
- **브라우저 렌더 HTML → /api/sources/parse 결과** == **현행 서버 크롤(fetch+parse) 결과** (표면가·옵션·재고 일치).
- 불일치 시: 렌더 DOM과 raw HTML 차이 분석 → 파서 보정 또는 수집 방식 조정. **일치 전 출시 금지.**
- 무신사·롯데온은 기존 추출기 유지라 회귀만 확인.

라이브 실검증: [[live-browser-verify]] 절차로 실 PC에서 6개 전부 창 뜨고·로그 흐르고·매트릭스 반영 확인.

## 8. 권한·배포·재설치

- manifest 권한 추가(`system.cpu`, `system.memory`) + 버전 0.4.0 → **사용자 확장 재설치 1회 필요**(zip 재다운로드).
- 서버 변경(/parse, run-now 폴백, crawlers parse_html)은 배포로 반영.
- 단일 원본 확장 폴더: `프로그램/_시스템/extension/moum-crawler`.

---

## 9. 리스크 / 미해결

- 렌더 HTML ≠ raw HTML(서버 파서가 raw 기준으로 튜닝됨) → §7 게이트로 소싱처별 확인 필수.
- 보이는 창 6개가 사용자 작업을 방해할 수 있음 → 창을 `focused:false`(백그라운드 비활성)로 띄우되 보이게, 또는 별도 윈도우 그룹. (구현 중 UX 조정)
- 같은 소싱처 다중 URL 동시 차단 위험 → 소싱처 내부는 순차 유지.
- chrome.system.cpu는 누적값 → 델타 샘플링 필요(부정확 시 latency 신호 우선).
