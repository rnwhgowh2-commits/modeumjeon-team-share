# 판매처 추가·업데이트 팝업 + 데이터 코드 지도 (판매처판) — 설계

> **상태**: 설계 확정 대기 (사용자 리뷰 게이트)
> **작성**: 2026-07-04
> **브랜치**: `feature/marketplace-add-card` (origin/main 기준)
> **관계**: 소싱처 가이드(`/sourcing-guide/`)의 판매처 짝. 구조·기법 미러링, 입력만 판매처 개발 환경으로 교체.

---

## 1. 목적·배경

판매처 관리(`/accounts/upload`)에는 계정 목록만 있고 매뉴얼이 없다. 소싱처 관리처럼 두 매뉴얼을 얹는다:

1. **➕ 판매처 추가·업데이트** — 새 마켓(판매처) 업로드 연동을 프로그램에 심는 팝업. 소싱처 추가의 짝.
2. **📖 데이터 코드 지도** — 프로그램 가격·재고가 마켓까지 가는 흐름 + 마켓별 정밀 세팅 SOP + 코드 위치.

### 판매 전체 흐름에서의 위치

```
소싱처 추가 → 모음전 상품 구성 → 소싱처 크롤 → 프로그램 재고/가격 갱신
    → [판매처] 신규 상품 마켓 등록  OR  기 등록 상품 연동  (계정 연결 필요)
    → [판매처] 가격/재고 업데이트 반복
```

판매처 관리 탭이 담당하는 영역 = **파는 쪽 통합 허브** 3층:
- ① 어떤 마켓에 올릴 수 있나 = **마켓 연동** (스스·쿠팡 done, 롯데온·11번가… 추가) ← 본 팝업
- ② 그 마켓에 로그인/키 연결 = **계정 연결** (이미 있는 「계정 추가·🔑키」)
- ③ 상품 올리기·동기화 = 신규 등록 / 기존 연동 → 가격·재고 반복 ← **데이터 코드 지도**가 흐름 설명

### 핵심 설계 판단: "정밀 조사는 위저드가 아니라 Claude Code가 한다"

소싱처와 동일. 사용자는 최소 정보(마켓명·문서 링크·지원 동작 의도)만 넣고, **마켓별 셀러 API 정밀 조사(인증·엔드포인트·필드 매핑·옵션 구조)는 프롬프트를 받은 Claude Code가 수행**한다.

```
① [위저드]      사용자   — 마켓명 + (아는 문서 링크) + 지원 동작 체크 → 「📋 프롬프트 생성」
② [Claude Code] Claude   — 그 마켓 셀러 API·개발환경 정밀 조사 → 업로더 어댑터 작성 + 마켓 등록
③ [검증]        실계정   — 신규 등록·재고/가격 100% 일치 확인 (틀리면 ②로 복귀)
④ [계정 연결]   사용자   — 「계정 추가·🔑키」에 셀러 키 입력 → 실사용
```

위저드는 ①단계 **접수 폼**일 뿐 — 가볍게 유지한다. ②단계 정밀 조사 절차(마켓별 SOP)는 데이터 코드 지도에 참고 문서로 둔다.

## 2. 배치·진입 (위상)

- **위치**: `/accounts/upload` 페이지 상단에 소싱처 가이드와 동일한 **카드 버튼 2개**
  「➕ 판매처 추가·업데이트」 · 「📖 데이터 코드 지도」 (기존 계정 목록 위, 최소 변경).
- **열림 방식**: 소싱처와 동일하게 `_page_modal.html`(same-origin iframe 모달) 재사용.
  버튼에 `data-guide-modal="/marketplace-guide/add?bare=1"` / `.../map?bare=1` 부여.
- 위저드 팝업 레이아웃 = **좌측 세로 레일 위저드**(시안 #4).

> ⚠️ **X-Frame-Options 필수** — `?bare=1` 라우트는 응답에 `X-Frame-Options: SAMEORIGIN`을
> 직접 박아야 한다(전역 기본값 `DENY`가 same-origin iframe 까지 차단하므로). `/sourcing-guide/add`가
> 이걸 빠뜨려 모달이 깨졌던 선례(2026-07-04 PR #84) — 처음부터 넣는다.

## 3. 팝업 내용 구성 — 「판매처 추가·업데이트」

### 탭 1 · 신규 판매처 추가 (4단계 위저드, 좌측 세로 레일)

| 단계 | 담는 것 | 비고 |
|---|---|---|
| **1 · 마켓** | 마켓명 입력 + **준비된 목록 선택** | 목록 = `MARKET_METADATA` 중 `status=coming_soon` (롯데온·11번가·옥션·G마켓·위메프·인터파크). 고르면 아이콘·기본 API 방식 자동 채움. 없으면 직접 입력 |
| **2 · 인증·계정** | 인증/API 방식(OAuth·HMAC·셀러센터API·기타) · 셀러 API 문서 URL(키발급·상품등록·재고가격, 여러 개) · 필요 키(예: CLIENT_ID·SECRET) | 문서 행 = 소싱처 URL 행과 동일 UX(유형 태그 + ↗ + 삭제 + 추가) |
| **3 · 지원 동작** | ☑ 신규 상품 등록 ☑ 기존 상품 연동 ☑ 가격·재고 업데이트 | **체크 3종만**(간이). 동기화 세부 필드는 ②단계에서 Claude가 정밀 결정 |
| **4 · 프롬프트** | 위 입력 합친 **Claude Code 프롬프트** + 복사 버튼 | 하단 안내: "연동되면 「계정 추가·🔑키」에서 셀러 키 넣기" |

**생성 프롬프트 형식** (STEP 4):
```
판매처 추가 가이드 참고해서 "<마켓명>" 업로드 연동 추가해줘.
· 인증/API : <방식>
· 문서 : <키발급/상품등록/재고·가격 URL 들>
· 필요 키 : <키 목록>
· 지원 동작 : 신규 상품 등록 · 기존 상품 연동 · 가격·재고 업데이트  (체크된 것만)
※ 실계정 100% 검증 전 "완료" 선언 금지 (가격·재고 오류 = 금전 손실).
```

### 탭 2 · 기존 마켓 연동 업데이트

- 이미 연동된 마켓(스스·쿠팡 등 `status=ready`) 선택
- 증상/요청 입력 (예: "재고가 0으로 안 올라감 / 옵션 매칭 틀림")
- 「📋 저장 + Claude Code 프롬프트 생성」

---

## 4. 데이터 코드 지도 (판매처판) — 소싱처 지도와 동일 깊이

소싱처 `map.html`을 판매처 흐름으로 미러링. 탭/토글 구조 동일, 내용만 업로드 경로.

- **흐름 4층**: 프로그램(가격·재고) → 전송 대기(스냅샷) → 판매처 API(스스 OAuth·쿠팡 HMAC) → 마켓 반영.
  - **두 갈래 명시**: "신규 상품 마켓 등록" vs "기 등록 상품 연동" 을 흐름의 분기로 그림 → 이후 가격·재고 반복 동기화.
- **마켓별 정밀 세팅 SOP** (②단계 참고 문서): 인증 분류 → 엔드포인트 확정 → 필드 매핑 → 실계정 검증 100% 일치.
- **코드 위치 지도**: `uploader/runtime.py`(전송·플래그) · `platforms/smartstore/client.py`(OAuth) · `accounts.py _test_coupang`(HMAC) · `MARKET_METADATA`(마켓 정의).
- **리스크/무결성**: 실전송 기본 OFF(`LEMOUTON_LIVE_UPLOAD`) · 0원·재고없음 전송 금지 · 크롤 실패 값 폴백 금지.

## 5. 아키텍처·컴포넌트 분리

새 블루프린트 `marketplace_guide` (소싱처 `sourcing_guide` 미러, url_prefix `/marketplace-guide`):

| 유닛 | 파일 | 책임 |
|---|---|---|
| 라우트 | `webapp/routes/marketplace_guide.py` | `/add`·`/map` (bare 시 SAMEORIGIN), 마켓 목록 주입 |
| 추가 팝업 | `templates/marketplace_guide/add.html` | 좌측 레일 4단계 위저드 + 2탭 + 프롬프트 생성 JS |
| 지도 | `templates/marketplace_guide/map.html` | 판매처 데이터 코드 지도 |
| 진입 버튼 | `templates/accounts/upload.html` (수정) | 상단 카드 버튼 2개 + `_page_modal.html` include |
| 모달 기구 | `templates/sourcing_guide/_page_modal.html` (재사용) | 변경 없음 |

- 각 유닛은 소싱처 대응 유닛의 구조를 그대로 따르므로 독립적으로 이해·테스트 가능.
- 마켓 목록·메타데이터 단일 출처 = `accounts.py MARKET_METADATA` (중복 정의 금지).

## 6. 데이터 흐름

- 위저드 입력은 **DB에 쓰지 않는다** — 순수 클라이언트에서 프롬프트 문자열만 생성(머니 경로 무영향).
  (소싱처 `add`는 SourceRegistry에 등록하지만, 판매처 "마켓 추가"는 코드 레벨 작업이라 ②단계 Claude가 수행. 위저드는 프롬프트만.)
- 실제 마켓 등록·키 스키마·`MARKET_METADATA` 갱신 = Claude Code가 프롬프트대로 코드 수정.
- 계정/키 등록은 기존 `/accounts` 경로 그대로.

---

## 7. 에러/상태 디자인

- 위저드: 로딩 없음(정적). 4상태 중 **빈**(마켓명 미입력 시 다음 비활성) · **정상**(프롬프트 생성) 위주.
- 프롬프트 생성 전 필수값(마켓명) 없으면 토스트로 안내(소싱처 `submitNew` 게이트 동일).
- 마켓명이 이미 연동된 마켓이면 "기존 업데이트 탭으로" 유도(중복 방지, 소싱처 존재검사 게이트 미러).

## 8. 테스트·검증

- 라우트: `/marketplace-guide/add?bare=1`·`/map?bare=1` → 200 + `X-Frame-Options: SAMEORIGIN` (헤더 회귀 테스트 필수).
- 모달: `/accounts/upload`에서 두 버튼 클릭 → iframe 렌더(빈 상자·깨진 아이콘 없음). 라이브 브라우저 검증.
- 프롬프트: 입력 채우고 생성 → 마켓·인증·문서·키·동작·무결성 규칙 모두 포함 확인.
- 마켓 목록: `MARKET_METADATA`의 coming_soon 6개가 STEP 1 목록에 노출.

---

## 9. YAGNI (지금 안 하는 것)

- 위저드가 실제 어댑터 코드를 자동 생성/배포 ❌ (Claude Code ②단계 몫).
- 마켓별 동기화 필드 세부 UI ❌ (Claude 정밀 조사가 결정).
- 계정 키 입력 UI 중복 ❌ (기존 「계정 추가·🔑키」 재사용).

---

## 10. 오픈 이슈 (구현 중 확정)

- 블루프린트 이름 `marketplace_guide` vs `/accounts/guide` 하위 라우트 — 소싱처 대칭상 별도 블루프린트 권장.
- STEP 1 "준비된 목록"에 ready(스스·쿠팡)도 보일지 vs coming_soon만 — 신규 추가 탭이므로 coming_soon 위주, ready는 탭 2로.
