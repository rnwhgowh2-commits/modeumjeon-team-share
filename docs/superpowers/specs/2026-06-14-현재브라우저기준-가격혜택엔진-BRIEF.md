# 브리프 — "현재 브라우저 기준" 가격·혜택 엔진 (무신사 르무통 우선)

> **용도**: 다른 세션/다른 프롬프트가 그대로 읽고 이어받기 위한 자립형 요청 정리(개조식).
> **상태**: 가격편 진행 중. Phase 1 백엔드 완료. (재고편은 가격편 종료 후)
> **연관 문서**: 설계서 `...-design.md` · 구현계획 `plans/2026-06-14-현재브라우저기준-가격혜택엔진.md`
> **작업 위치**: worktree `feature/browser-price-engine` (origin/main 기준)

---

## 1. 목적 (한 줄)

- 무신사 르무통 메이트 등 상품의 **표면노출가 / 혜택 / 최종매입가**를, **"현재 열린 사용자 브라우저 기준으로만"** 수집·가공해 정확히 도출한다. 그 결과 우리 프로그램을 6소싱처로 일반화한다.

## 2. 대상·범위

- 1순위: **무신사**(source_id=3), 르무통 메이트. 현재 테스트 상품 = `musinsa.com/products/6020529` (가이드 샘플엔 4046672 등도 있음).
- 1개 완성 후 → 6소싱처(르무통·SSF·SSG·롯데온·스스르무통) 일반화.
- 본 작업 = **가격편**. **재고편**은 가격편 종료 후 별도.

## 3. 핵심 원칙 (절대 규칙)

- 3-1. **현재 브라우저 기준만**: 가격·혜택은 수시로 바뀌고 로그인(계정 LV)/비로그인마다 다르다. 반드시 지금 열린 브라우저에서 크롤된 정보로만 계산한다.
- 3-2. **자동화 우선 — "프로그램이 직접 수집"**: 사람(나/사용자)이 수동으로 들여다보거나 콘솔값을 붙여넣는 방식 ❌. 확장(프로그램)이 로그인된 브라우저에서 **자동 수집 → 서버 저장**해야 한다. 조사 도구와 생산 수집기는 **동일물**.
- 3-3. **폴백 금지**: 현재 크롤 데이터가 없으면 옛 값·템플릿·대표가(평균·최저)로 대체하지 말고 **"크롤실패/혜택 미수집"으로 표면화**한다. (잘못된 매입가 = 금전 손실)

## 4. 문제 3가지 (현상 → 코드 원인)

- 4-1. **표면노출가 기준이 계속 바뀜**
  - 현상: 표면노출가는 '나의 할인가(회원가)'가 아니라 그 위 금액(현재 화면 122,900원).
  - 코드: `extension/moum-crawler/background.js` `musinsaExtractor` — 표면가를 화면이 아닌 무신사 API `goodsPrice.salePrice`에서 읽음.
  - 결정: **API 방식 유지**(아래 6-2), 단 3상태 테스트에서 화면값과 일치 대조.
- 4-2. **혜택이 로그인 상태별로 다른데 지난 크롤 혜택이 반영됨 (가장 심각)**
  - 코드: 확장이 혜택을 **아예 안 긁음**(가격·재고만). 혜택은 100% 저장된 템플릿/오버라이드/`dynamic_benefits_json`에서 옴 → 구조적으로 "지난 값·로그인 무관".
- 4-3. **현재 크롤값 없으면 크롤실패가 떠야 하는데 기존 값이 도출됨**
  - 코드: 표면가엔 신선도 게이트(`is_crawl_valid`)가 있으나 **혜택엔 게이트가 없음** → 혜택은 항상 저장값 사용.

## 5. 가격 로직 (계산 규칙)

- 5-1. **표면노출가** = 정가에서 선반영 할인 적용된 상품 할인가.
- 5-2. **최종매입가 = 표면노출가 − Σ(적용 혜택)**.
- 5-3. **혜택 속성 4종** (템플릿이 정의, 값은 매 크롤이 채움):
  - ㄱ. 혜택금액 방식: 정률(%) / 정액(원) / 조건부(%, 단 XXX원 기준, 최대 XXX원)
  - ㄴ. 베이스금액(누적): 혜택1 차감 후 금액 기준으로 혜택2 적용. 예) 10,000 →(10%)→ 9,000 →(10%)→ 8,100
  - ㄷ. 상시 / 기간한정
  - ㄹ. 횟수: 1회성 / 무제한
- 5-4. **키워드 합집합**: 상품마다 혜택이 달라, 여러 URL을 합쳐 소싱처별 혜택 키워드 사전(합집합)을 만든다.

## 6. 확정 결정사항 (이번 세션)

- 6-1. **혜택 출처**: 크롤 우선. 템플릿 = "어떤 키워드를 찾을지"(합집합 사전) + 계산속성만. 실제 값·유무는 매 크롤이 채움.
- 6-2. **표면가 출처**: 현재 API(`goodsPrice.salePrice`) 방식 유지 + 화면값 대조 검증.
- 6-3. **혜택 수집 방식**: A안 — 화면(DOM) 스크래핑(견고한 키워드 스캔). 추후 안정 혜택만 API로 승격.
- 6-4. **신선도(N1)**: TTL 없음. **마지막 크롤값 유지 + "크롤 시각" 화면 표시**. 오래됐으면 사용자가 재크롤.
- 6-5. **범위/순서**: 무신사 1개 완성 → 일반화. **자동화 우선**이라 확장 수집기(Phase 2)를 먼저 만들어 그걸로 직접 수집·검증.

## 7. 해결 아키텍처 (부품 재활용 + 배선)

- 7-1. **핵심 통찰**: 원하는 계산 흐름이 이미 "미리보기" 경로로 존재 — 라이브 경로에 배선만 안 됨.
  - `lemouton/pricing/benefit_gate.py` `gate_benefits()` — 크롤 라인 → 혜택 on/off (✅)
  - `lemouton/pricing/final_price.py` `compute_final_price()` — 표면가−혜택 순차 누적차감 (✅)
  - `crawl_guide.pricing.benefits` + `exclude_keywords` — 혜택 키워드 사전 (✅)
  - `sourcing_guide.py` `gate-preview` 엔드포인트 — 위 셋을 묶은 계산 (⚠️ 미리보기 전용)
- 7-2. **데이터 흐름(목표)**:
  - 확장(로그인 브라우저): 표면가(salePrice) + **혜택 라인+금액 DOM 크롤**(신규) + 페이지 HTML 덤프
  - → `POST /api/sources/crawl-result`: `dynamic_benefits_json['_crawl']`에 스냅샷 저장(`crawled_at·is_logged_in·benefits_ok·lines·amounts`)
  - → `compute_breakdown`(무신사): **신선도 게이트** 통과 시 `gate_benefits(가이드 키워드, 크롤 lines)` on/off + 크롤 `amounts` 금액으로 계산. 통과 못 하면 **`final_price=None`, `benefits_status='미수집'`** (폴백 금지)
- 7-3. **조인 키 계약(중요)**: 크롤 `amounts`의 키 = `crawl_guide.pricing.benefits[].name`과 **글자 그대로 일치**. 확장 파서가 가이드 혜택 이름으로 금액을 내보낸다.
- 7-4. **저장 방식**: 새 컬럼 ❌(마이그레이션 회피, 프로젝트 관례). 기존 `dynamic_benefits_json`의 예약 키 `_crawl` 사용.

## 8. 테스트 (정합성 검증)

- 8-1. 브라우저 환경을 바꿔가며(로그인/비로그인) 수집·가공이 정확한지 확인. 표면노출가/혜택/최종매입가 정합성 대조.
- 8-2. 순서(사용자가 직접 로그인 전환): **계정1(블랙다이아몬드 LV) → 비로그인 → 계정2(다이아몬드 LV)**.
- 8-3. 각 상태에서 **프로그램(확장)이 자동 크롤** → 저장된 `_crawl` 스냅샷으로 키워드 사전·대조표 도출/튜닝.
- 8-4. 검증 포인트: ① 표면가 API값=화면값(122,900) 일치 ② 비로그인 시 계정 의존 혜택(등급할인·등급적립)이 사라지고 옛 계정1 혜택이 안 남음 ③ 크롤 안 한 옵션은 "미수집"(추정가 금지).
