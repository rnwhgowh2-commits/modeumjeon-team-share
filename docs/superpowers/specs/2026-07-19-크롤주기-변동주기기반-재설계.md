# 크롤 주기 재설계 — 「변동률」에서 「변동 주기」로 (제안)

> 2026-07-19. 사장님 제안: *"평균 1회 변경되기까지 며칠인지 파악하고 그걸 계수 근거로. 1주일에 1번 바뀌는데 매일 업데이트할 필요 없잖아."*
> 상태: **제안 — 검토 대기**. 구현 전 확정 필요.
> 관계: M5(변동성 통계) 위에 얹음. `lemouton/sources/crawl_schedule.py`(랩·계수) 기존 구조 유지.

## 1. 지금 방식의 결함 — 변동률은 자기 되먹임한다

M5 는 **변동률 = 변동횟수 ÷ 관측횟수** 로 계수를 권한다. 그런데 이 값은 **크롤 빈도의 함수**다.

```
하루 10번 크롤 → 매 크롤마다 바뀌었을 확률 낮음 → 변동률 ↓ → "덜 바뀌네, 계수 내려"
계수 내림      → 크롤 간격 벌어짐              → 변동률 ↑ → "자주 바뀌네, 계수 올려"
```

**측정하려는 대상(소싱처가 얼마나 자주 바뀌나)이 측정 행위(크롤 빈도)에 오염된다.** 계수가 진동하거나, 한 번 내려간 곳이 계속 내려간 채로 굳을 수 있다.

## 2. 대안 — 변동 주기(평균 며칠에 1번)

**변동 주기는 크롤 빈도와 무관한 물리량이다.** 소싱처가 주 1회 가격을 바꾸면, 우리가 하루 10번 보든 3일에 한 번 보든 주 1회다.

### 측정
```
변동 주기 = 관측 기간 ÷ 감지된 변동 횟수
  예) 30일간 4번 바뀜 → 7.5일에 1번
```
`CrawlDelta`(소싱처 기준선, M5 에서 확정)에 이미 `crawled_at` 이 있으므로 **추가 수집 없이 계산 가능**하다.

### ⚠️ 한계 — 크롤 간격보다 잦은 변동은 못 본다
3일에 한 번 크롤하는 곳이 하루에 한 번 바뀌면, 우리 눈엔 3일에 한 번으로 보인다(여러 변동이 1번으로 뭉침).
→ **측정된 변동 주기는 항상 "실제 주기 이상"이다.** 이걸 모르면 "천천히 바뀌네" 하고 계수를 더 내려 악순환이 된다.
→ **가드**: 측정된 변동 주기가 **현재 크롤 간격의 2배 이내**면 「더 짧을 수 있음」으로 표시하고 **계수를 내리지 않는다**. 올리는 건 허용(더 자주 보면 진짜 주기가 드러난다).

## 3. 변동 주기 → 크롤 주기

**목표: 소싱처가 바뀐 뒤 우리가 그걸 반영하기까지의 지연을 일정하게.**

변동 주기가 T일이면, 크롤 간격을 T의 1/2~1/3 로 잡으면 대부분 한 주기 안에 잡는다.

| 변동 주기 | 권장 크롤 | 계수(연속 모드 기준) | 뜻 |
|---|---|---|---|
| 1일 미만 | 하루 3~5회 | ×5 | 하루에도 몇 번 바뀜 |
| 1~2일 | 하루 2회 | ×4 | 거의 매일 바뀜 |
| 3~6일 | 하루 1회 | ×3 | 주 1~2회 바뀜 |
| 1~3주 | 2~3일에 1회 | ×2 | 가끔 바뀜 |
| 3주 이상 | 주 1회 | ×1 | 거의 안 바뀜 |
| 관측 부족 | — | 기본 유지 | 아직 판단 못 함 |

⚠️ **경계값은 실측 없이 정한 출발점이다.** 실제 분포가 쌓이면 다시 잘라야 한다(M5 의 2/5/10/20% 경계와 같은 성격).

### ★ 계수는 절대값이 아니라 배분이다
한 바퀴에 돌 수 있는 총량은 유한하다(현재 실측: 한 바퀴 ~9분, 하루 2바퀴).
**모든 구성의 계수를 올리면 바퀴가 길어져 오히려 전부 늦어진다.**
→ 권장 계수는 **총량 예산 안에서 상대 배분**으로 계산해야 한다. 화면에 「이대로 적용하면 한 바퀴가 9분 → 14분」처럼 **영향을 먼저 보여준다.**

## 4. 3,000개 문제 — 한 구성 안에서 상품마다 다를 때

「롯데온 > 나이키」에 상품이 3,000개면 그 안에서 변동 주기가 제각각이다.
- 평균을 내면 **자주 바뀌는 소수가 묻힌다**
- 자주 바뀌는 쪽에 맞추면 **3,000개를 다 자주 크롤해 낭비**

### 제안: 쪼개되, 사람이 보는 단위는 안 늘린다

**계수는 상품(URL) 단위로 붙이고, 사람은 「등급」만 본다.**

기존 `CrawlWeightRule` 에 이미 `scope_type='url'` 이 있다 — **상품별 계수는 지금도 가능하다.** 문제는 3,000개를 사람이 손으로 못 정하는 것뿐이다. 그러니 **자동 등급**이 답이다.

```
롯데온 > 나이키                     URL 3,000개   완성도 80%
  ├ 매일급   (1일 미만)      120개   ×5
  ├ 주간급   (1~6일)         800개   ×3
  ├ 월간급   (1~3주)       1,900개   ×1
  └ 미분류   (관측 부족)      180개   기본 유지
```

- **사람이 보는 행 = 구성당 4~5줄로 고정.** 3,000개가 3,000줄이 되지 않는다.
- 등급은 **변동 주기로 자동 배정**되고, 데이터가 쌓이면 상품이 등급 사이를 옮겨 다닌다.
- 사람은 **등급 단위로만 손댄다**("주간급을 ×3에서 ×4로"). 개별 상품을 만질 일이 없다.
- 개별 상품을 고정하고 싶으면 기존 URL 계수로 덮는다(수동 > 자동).

### 등급이 흔들리지 않게
- **이동에 이력(hysteresis)을 둔다** — 경계에 걸친 상품이 매 바퀴 등급을 오가면 계수가 진동한다. 한 단계 올릴 땐 즉시, 내릴 땐 **연속 N바퀴 조건 충족 시**에만.
- **신규 상품은 「탐색 등급」에서 시작** — 처음엔 자주 크롤해 주기를 빨리 알아내고, 안정되면 제 등급으로 내려보낸다. 안 그러면 새 상품이 「미분류」에 갇혀 영영 관측이 안 쌓인다.

## 5. 무엇을 기준으로 「바뀌었다」고 볼 것인가

변동 주기를 재려면 무엇이 변동인지 정의가 필요하다. **가격과 재고는 성격이 다르다.**

| | 성격 | 늦으면 |
|---|---|---|
| **가격** | 드물게, 크게 | 역마진 또는 경쟁력 상실 |
| **재고** | 자주, 작게 | 품절 미반영 = 주문 몰림 |

→ **주기를 따로 재고 둘 중 짧은 쪽을 쓴다.** 가격은 주 1회인데 재고가 하루 3번 바뀌면 하루 3번에 맞춰야 한다.
→ 단 **의미 있는 재고 변동만** 센다 — 이미 확정된 업로드 게이트 규약(재고 3개 이상 구간 내 이동은 무시)을 그대로 적용한다. 5→3 은 변동으로 세지 않는다.

## 6. 단계 계획

| 단계 | 내용 | 위험 |
|---|---|---|
| **1** | 변동 주기 계산 추가(`CrawlDelta` 재사용) + 화면에 **변동률과 나란히** 표시 | 낮음 — 표시만, 계수 안 건드림 |
| **2** | 권장 계수 근거를 변동률 → 변동 주기로 교체. **적용 전 「한 바퀴 몇 분 → 몇 분」 영향 표시** | 중간 — 권장값만, 자동 적용 없음 |
| **3** | 구성 안 자동 등급(매일급/주간급/월간급/미분류) + 등급 단위 편집 | 중간 |
| **4** | 신규 상품 탐색 등급 + 등급 이동 이력(hysteresis) | 중간 |
| **5** | 실제 분포 보고 경계값 재조정 | — |

**1단계는 지금 바로 가능하다**(수집 중인 데이터로 계산만 하면 된다). 2단계부터는 실제 데이터가 몇 바퀴 쌓인 뒤에 해야 경계값이 의미 있다.

## 7. 미확정 — 사장님 확인 필요

1. **변동 주기 경계값**(위 표의 1일/3일/1주/3주)이 감각에 맞는지. 실측 없이 정한 출발점이다.
2. **가격과 재고 중 짧은 쪽**을 쓰는 게 맞는지. 아니면 가격만 기준으로?
3. **등급 이름** — 매일급/주간급/월간급이 읽히는지, 다른 말이 나은지.
4. **자동 적용 여부** — 지금은 권장만 표시한다(M5 확정). 등급이 생기면 자동 적용을 허용할지.
5. **총량 예산** — 하루에 크롤을 몇 바퀴까지 돌 수 있는지 상한이 정해져 있는지. 없으면 "영향 표시"의 기준을 못 잡는다.
