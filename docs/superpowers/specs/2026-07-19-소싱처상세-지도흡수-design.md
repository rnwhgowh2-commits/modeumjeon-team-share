## 1. 목적

소싱처별 정보가 **세 화면에 흩어져** 있고 서로 중복된다.

| 화면 | 성격 | 문제 |
|---|---|---|
| 데이터·코드 지도 `/sourcing-guide/map` | 읽기 전용 설명 | 소싱처별 카드가 **HTML 하드코딩** |
| 소싱처별 상세 `/sourcing-guide/<sid>` | 편집 폼 (DB) | 같은 정보를 **DB 값**으로 또 보여줌 |
| 전체보기 `/sourcing-guide/` | 목록 + 소싱처 관리 | 상세로 가는 통로 + 이름·로고·숨김·삭제 |

**목표** — 지도 하나로 합친다. 소싱처 카드를 누르면 오른쪽 패널이 열리고, 거기서 전부 편집한다.
설명(지도)과 값(DB)이 **한 곳**에서 나오게 해 중복을 없앤다.

**비목표** — 「가격」·「재고」 탭의 기존 내용·배치는 **바꾸지 않는다**. 카드가 눌리게 되는 것 외 변경 없음.

## 3. 화면 구조

### 3.1 지도 「가격」·「재고」 탭 — 원본 그대로 + 카드가 눌린다

- 요약표·상세카드·문구·배치 **일절 변경 없음**
- 소싱처 카드에 마우스를 올리면 `✏️ 고치기` 가 나타나고, 누르면 패널이 열린다
- 카드 자체는 **하드코딩을 걷어내고 DB(가이드)에서 렌더** → 편집하면 설명도 같이 바뀐다
  - 이관 전까지는 하드코딩 유지 → 가이드가 채워진 소싱처부터 DB 렌더로 전환(점진)

### 3.2 오른쪽 패널 — 네 칸

```
① 예시 주소 · 계산 결과        ② 수집 구조 + 주소·구분자
③ 혜택 · 최종 매입가           ④ 재고 규칙
```

**① 예시 주소 · 계산 결과** (가로 전체)
- 주소 카드: 이름 · 「대표」 · 주소 · 메모 · `▶ 크롤` · `수정` · `삭제`
- 크롤 후 그 자리에 **표면 노출가 · 혜택 합계 · 최종 매입가 · 재고 · 크롤 시각**
- 아직 안 긁은 주소는 **빨간 글씨로 "아직 크롤하지 않았습니다"** (빈칸으로 두지 않음)
- 맨 아래 붙여넣기 줄 + `▶ 전부 크롤`

**② 수집 구조** — 6항목(방식·로그인·위치) + **주소·구분자**(기본 주소 / 엔드포인트별 경로)

**③ 혜택** — 이름 · 값출처(고정/크롤) · 값 · 유형 · 적용 · **제외 체크** · 포함/제외 키워드 · 영수증

**④ 재고 규칙** — 품절 표시 / 한정 수량 / 표시 없을 때(충분 ↔ 확인 불가)

### 3.3 소싱처 관리 이관 (D12)

전체보기에 있던 **이름 변경 · 로고 · 숨기기 · 삭제**를 패널 머리말로 옮긴다.
사이드바 `/sourcing-guide/` → `/sourcing-guide/map` 으로 변경.

## 4. 데이터 모델

`SourceRegistry.crawl_guide` (JSON, 스키마 v3) 를 확장한다. **DB 마이그레이션 불필요**(JSON 키).

### 4.1 신규 — 주소별 크롤 결과 (D5)

`sample_urls[]` 각 항목에 결과를 붙인다. 지금은 결과가 **세 곳에 흩어져** 있고 그중
`last_new_check` 는 **한 칸뿐**이라 마지막 1건만 남는다.

```
sample_urls[]: { url, is_lead, name?, memo?,
                 result?: { surface_price, benefit_total, final_price,
                            stock_label, status, crawled_at, job_id } }
```

→ `verification.last_new_check` · `saved_checks` · `examples` 3곳을 여기로 통합.

### 4.2 신규 — 주소·구분자 (D7)

```
api: { base, endpoints: { <name>: { path, method, response_fields? } } }
```

원천은 `프로그램/docs/sources/<key>/profile.yaml` 에 이미 있으나 **코드가 읽지 않는다**(사본).
가이드로 옮겨 **단일 진실원천**으로 삼는다. 현대H몰은 yaml 자체가 없어 신규 입력.

### 4.3 기존 — 이미 있으나 UI 가 없던 것

`fields.option_stock.stock_rules`(재고 규칙) · `exclude_keywords` · `benefit.apply/base/freq`
→ ④·③ 칸에서 편집 가능하게.

## 5. 선수정 — 머니버그 2건 (D11)

편집 기능 자체가 방아쇠라 **기능보다 먼저** 고친다.

**5.1 혜택 이름이 계산 순서를 바꾼다**
`final_price._benefit_priority` 가 **이름에 「적립」이 들어있는지**로 정렬한다.
→ 이름 대신 **분류(`apply`/`method`)** 로 정렬. 이름 편집이 금액을 바꾸지 않게.

**5.2 값을 넣으면 차감이 꺼진다**
가이드에 `선반영`으로 적힌 혜택(무신사 등급할인·상품쿠폰, SSG 카드혜택가)을 코드는 **실제로 차감**한다.
저장 시 `sync_templates_from_crawl_guide` 가 그 표기를 그대로 밀어넣어 차감이 꺼진다.
→ 표기와 동작을 **한쪽으로 통일**. 어느 쪽이 진실인지 라이브 값으로 먼저 확정한 뒤 수정한다.

> ⚠️ 두 건 모두 **금액이 바뀌는 수정**이다. 수정 전/후 최종 매입가를 소싱처별로 대조해
> 바뀐 셀을 전부 보고하고 사용자 확인을 받은 뒤 배포한다.

## 7. 단계

| 단계 | 내용 | 검증 |

| S1 | 머니버그 2건 수정 | 수정 전후 금액 대조표 → 사용자 확인 |

각 단계마다 라이브 확인 후 다음. **S1 은 금액이 바뀌므로 단독 배포**한다.

## 8. 열린 항목

- **5.2 의 정답 방향** — 「선반영」 표기가 맞는지, 차감 동작이 맞는지 라이브 값으로 확정 필요
- **지도 카드 DB 렌더 전환 순서** — 가이드가 빈 소싱처(르무통 공홈·스마트스토어)는 나중
- **롯데아이몰·현대H몰의 문자 식별자** — 혜택 템플릿을 만들 수 없는 구조라 별도 수정 필요
  (`SourceBenefitTemplate.source_id` 가 Integer). 사용자의 가격로직 입력 **전에** 처리해야 함
