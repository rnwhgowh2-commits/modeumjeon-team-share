# 소싱처 크롤링 가이드 — 설계 (Design Spec)

> 2026-06-06. 브레인스토밍 결과 확정본.
> 관계: [[소싱처관리_아키텍처.md]](상위 IA), [[project_url_store_fragmentation]](3중분열 통합), [[project_crawl_chrome_extension]](크롤=확장).
> 계층: 본 문서 = 이 기능의 단일 진실원천(스펙). 구현 계획은 writing-plans 로 별도 생성.

---

## 1. 문제 (Why)

사용자 통증: **"소싱처별 혜택 정리해도 뭔가 틀어지면 초기화된 듯 새로 구성한다."**

브레인스토밍에서 확인된 근본 원인 = **둘 다**:

1. **기록 없음** — "이 소싱처는 이렇게 긁고, 이렇게 매입가를 낸다"는 정의가 한 곳에 적혀 있지 않아, 매번 머릿속으로 재구성한다.
2. **값 불안정** — 혜택 값이 크롤/다른 작업 후 0원이 되거나 날아가는 등 실제 손실/오염이 일어난다 (메모리: URL 저장소 3중분열, 혜택 별도 시스템).

→ 해결: **소싱처별 "진실원천 카드"** 하나를 만든다. 적는 곳이 곧 값의 기준이 되어, 기록 부재와 값 불안정을 동시에 잡는다.

## 2. 목표 / 비목표 (Scope)

**목표 (이번 1단계):**
- 소싱처별 카드에 (a) 크롤 구조, (b) 기준 샘플 URL, (c) 매입가 계산 정책을 **구조화해서 저장·표시·편집**.
- 카드 = 정의·도식 + **값의 기준**(틀어지면 여기와 대조해 바로잡음).
- 첫 템플릿 = **무신사** 1개를 끝까지 채워 "정답 카드" 완성. 나머지 소싱처는 같은 틀로 복제.

**비목표 (나중 단계 — 이번에 만들지 않음):**
- 2단계: 크롤러가 섹션1 셀렉터를 읽어 실제 크롤 (현재는 크롬확장이 크롤).
- 3단계: 가격엔진이 섹션3 정책을 읽어 매입가 자동 산출 → fx 반영.
- 혜택 동기화 엔진(일치/신규/소멸 배지) — 아키텍처 문서의 우측 배지.

비목표는 카드의 데이터 모델이 **나중에 그대로 읽힐 수 있게** 설계하되, 이번엔 배선하지 않는다.

## 3. 사용자 흐름 (협업 방식)

1. 사용자가 소싱처별 **대표 상품 URL**을 제안.
2. Claude와 함께 그 URL의 구조(썸네일/상품명/가격/혜택/옵션재고/상세이미지)를 분석.
3. 분석 결과를 카드 섹션1·2에 채우고, 매입가 정책(섹션3)을 합의해 저장.
4. 무신사가 "정답 카드"가 되면, 다음 소싱처도 같은 절차로 복제.

## 4. 저장 방식 — 결정: A

기존 `SourceRegistry` 모델에 **`crawl_guide` 컬럼 1개 추가** (JSON).

- 마이그레이션 1회 (Alembic) → Supabase 적용.
- 아키텍처 문서가 그린 `source_registry(레시피 JSON + 가격룰 JSON ...)` 와 일치.
- 항목을 나중에 늘리기 쉬움(스키마 변경 없이 JSON 키 추가).
- 타입: SQLAlchemy `JSON`(로컬 SQLite + Supabase Postgres 양쪽 호환). nullable, 기본 `NULL`(미작성 = 빈 카드).

대안 기각: 별도 테이블(B, 지금 과함) / markdown 파일(C, 실제 크롤·가격과 분리되어 ③ 진실원천 목표 깨짐).

### crawl_guide JSON 스키마 (v1)

```json
{
  "version": 1,
  "fields": {
    "thumbnail":    {"method": "crawl|manual|none", "locator": "", "status": "ok|warn|none", "note": ""},
    "title":        {"method": "crawl|manual|none", "locator": "", "status": "ok|warn|none", "note": ""},
    "price":        {"method": "crawl|manual|none", "locator": "", "status": "ok|warn|none", "note": ""},
    "benefit":      {"method": "crawl_per_product|uniform|none", "locator": "", "status": "ok|warn|none", "note": ""},
    "option_stock": {"method": "crawl|manual|none", "locator": "", "status": "ok|warn|none", "note": ""},
    "detail_image": {"method": "crawl|manual|none", "locator": "", "status": "ok|warn|none", "note": ""}
  },
  "sample_urls": ["https://www.musinsa.com/products/..."],
  "pricing": {
    "benefit_mode": "uniform|per_product",
    "uniform_rules": [
      {"label": "회원가", "type": "rate", "value": 5, "unit": "percent"},
      {"label": "카드할인", "type": "amount", "value": 3000, "unit": "won"}
    ],
    "note": ""
  },
  "updated_at": "2026-06-06T00:00:00Z"
}
```

- `fields.benefit.method` 는 `crawl_per_product`(상품별 크롤) / `uniform`(일괄) / `none` 으로, 섹션3 `pricing.benefit_mode` 와 의미를 맞춘다. (둘 중 하나가 진실원천이면 충분하나, UI 입력 편의상 둘 다 두고 저장 시 동기화한다 — `pricing.benefit_mode` 를 정본으로 삼고 `fields.benefit.method` 를 따라 세팅.)
- 최종 매입가 = `판매가 − Σ(혜택)`. `uniform_rules` 의 `rate` 는 판매가 대비 %, `amount` 는 원 단위 차감.

## 5. 화면 설계 — "소싱처 카드"

위치: 기존 **소싱처 사전 `/source-registry`** (`source_registry/list.html`). 새 상단 탭 ❌ (최소 변경 선호).

각 소싱처 행(`.src-row`)에 **`가이드 ▾` 토글 버튼** 추가 → 클릭 시 그 행 아래로 인라인 패널이 펼쳐짐(아코디언). 패널 = 3섹션 카드. 한 번에 한 행만 펼침.

### 섹션 1 — 크롤 구조 분석 "어디서 무엇을 긁나"
6행 표. 컬럼: 항목 | 수집 방법(드롭다운) | 위치(셀렉터/설명, 텍스트) | 상태(✅/⚠️/—).
행: 썸네일 · 상품명 · 판매가 · **혜택** · 옵션·재고 · 상세 이미지.
- 혜택 행의 수집 방법 = `상품별 크롤 / 일괄 / 없음`.

### 섹션 2 — 기준 샘플 URL
이 구조를 정의·검증할 때 쓴 대표 상품 URL 목록(추가/삭제). 각 URL 옆 `↗`(새 탭 열기).
(향후 "이 URL로 구조 재확인" 버튼이 크롤 테스트와 연결 — 이번엔 링크 열기까지만.)

### 섹션 3 — 최종 매입가 계산 도식
`소싱처 판매가 − 혜택 = 최종 매입가` 시각 도식(텍스트/박스 다이어그램).
혜택 처리 정책 토글:
- ⦿ **일괄 적용** → 규칙 목록 입력(라벨 + 율/고정값). 예: 회원가 −5%, 카드 −3,000원.
- ⦿ **상품별(크롤 필요)** → "크롤로 가져온 혜택 사용" 안내 + (향후) 마지막 크롤값 표시.

도식 하단에 **예시 계산 미리보기**: 가상의 판매가(예: 100,000원)에 현재 규칙 적용 결과를 즉시 보여줌(입력 검증용, 저장 안 함).

### 저장 동작
패널 하단 `저장` 버튼 → `PUT /api/source-registry/<id>/guide` 로 crawl_guide JSON 전송. 성공 시 토스트, 행 상태 점(dot) 갱신. 자동저장 ❌(명시 저장으로 실수 방지).

## 6. 백엔드 변경

1. **모델**: `SourceRegistry.crawl_guide = Column(JSON, nullable=True)` (`models_pricing.py`).
2. **마이그레이션**: Alembic revision 1개 — `add column crawl_guide`. Supabase 적용.
3. **라우트** (`routes/source_registry.py`):
   - `GET /api/source-registry/<id>/guide` → `{ok, guide}` (없으면 빈 스켈레톤 반환).
   - `PUT /api/source-registry/<id>/guide` → body = crawl_guide JSON, 서버에서 스키마 화이트리스트 검증 후 저장. `updated_at` 서버 스탬프. admin 전용(기존 `_admin_only` 게이트 통과).
   - 목록(`page_list`)의 `items` 에 `has_guide`(bool) 추가 → 행에 "가이드 있음" 점 표시.
4. **검증**: 잘못된 method/mode 값, 음수 value, URL 형식 등 서버측 거부. 데이터 무결성(중복·모순 금지) 원칙 준수.

## 7. 무결성 / 엣지 케이스

- crawl_guide 미작성(NULL) → 카드는 빈 스켈레톤. 정상.
- `benefit_mode` 와 `fields.benefit.method` 불일치 → 저장 시 `benefit_mode` 정본으로 동기화.
- 소싱처 삭제 시 crawl_guide 도 함께 삭제(같은 행이므로 자동).
- JSON 파싱 실패/스키마 위반 → 400, 기존 값 보존(덮어쓰기 안 함). "혜택 0원 버그"류 재발 차단(값 보존 우선).
- 동시 편집(팀 2인) → 마지막 저장 우선(낙관적). 추후 updated_at 기반 충돌 경고는 비목표.

## 8. 테스트 / 검증

- 단위: 라우트 GET/PUT — 빈 스켈레톤 반환, 정상 저장/조회, 잘못된 값 거부(400 + 기존값 보존).
- 마이그레이션: 로컬 SQLite + Supabase 양쪽에서 컬럼 추가·롤백 확인.
- 브라우저 실검증(`/live-browser-verify` 게이트): 소싱처 사전 → 무신사 행 펼침 → 3섹션 입력 → 저장 → 새로고침 후 값 유지 확인 → 잘못된 입력 거부 확인. 7섹션 보고서.

## 9. 마일스톤

- **M1 (이번)**: 모델+마이그레이션 / GET·PUT API / 인라인 카드 UI 3섹션 / 무신사 카드 1개 완성 + 실검증.
- M2 (별도): 크롤러가 섹션1 셀렉터 소비.
- M3 (별도): 가격엔진이 섹션3 정책 소비 → 매입가·fx 산출.
