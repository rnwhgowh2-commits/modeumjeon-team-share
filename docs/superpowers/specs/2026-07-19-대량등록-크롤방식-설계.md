# 대량등록 크롤 방식 설계 (제안)

> 2026-07-19. 사장님: *"대량등록 크롤 방식은 모음전과 다르다. 목록 URL 을 주고 하나씩 들어가서 크롤한다."*
> 예: `https://www.musinsa.com/search/goods?...&keyword=나이키&gf=A`
> 상태: **제안 — 검토 대기**.

---

## 1. 모음전과 무엇이 다른가

| | 모음전 | **대량등록** |
|---|---|---|
| 입력 | **상품 URL** (이미 우리가 파는 것) | **목록 URL** (검색 결과) |
| 그 안에 | 옵션 몇 개 | **상품 수백~수천 개** |
| 크롤 목적 | 가격·재고 갱신 | **발굴** + 갱신 |
| 상품 수 | 고정 | **계속 생기고 없어짐** |
| 접근 | URL 하나 = 요청 하나 | **목록 1회 + 상세 N회** |

⇒ **모음전 로직을 그대로 쓰면 안 된다.** 앞서 제안한 「변동 주기 기반 계수」는 **이미 등록한 상품**에만 맞고, 발굴 단계엔 다른 설계가 필요하다.

## 2. 핵심 아이디어 — 목록에 이미 가격이 있다

무신사 검색 결과에는 **상품명·가격·할인율·썸네일·품절 여부**가 이미 나온다.

```
목록 1회 요청  →  상품 60개 정보    (싸다)
상세 1회 요청  →  상품 1개 정보     (비싸다, 옵션 로딩까지)
```

**⇒ 목록에서 가격이 안 바뀐 상품은 상세에 안 들어간다.**

3,000개 구성에서 오늘 가격이 바뀐 게 50개면 상세는 **3,000번이 아니라 50번**이다. 이게 가장 큰 절약이다.

### ⚠️ 단, 목록 가격을 그대로 저장하면 안 된다
이번 세션에서 **롯데아이몰이 목록/상세의 가격 의미가 달라 사고**가 났다(「최대할인가」에 카드할인이 이미 포함). 무신사도 목록가가 회원가인지 정상가인지 문서가 충돌한다.

→ **목록 가격은 「변동이 있었나」 신호로만 쓴다. 실제 표면노출가는 반드시 상세에서 확정한다.**
→ 목록가를 그대로 쓰면 조용히 틀린 가격이 마켓에 올라간다.

## 3. 크롤을 두 갈래로 나눈다

**같은 목록 URL 이라도 목적이 다르면 주기가 달라야 한다.**

### ① 유지 크롤 — 이미 등록해서 팔고 있는 상품
- **목적**: 가격·재고 변동을 빨리 마켓에 반영
- **급하다.** 늦으면 역마진이거나 품절 주문이 몰린다 (대량등록의 존재 이유)
- 주기 = 앞서 제안한 **변동 주기 기반**(별도 문서)
- 대상은 3,000개가 아니라 **우리가 실제 등록한 일부**

### ② 발굴 크롤 — 아직 등록 안 한 후보
- **목적**: 팔 만한 신규 상품 찾기
- **안 급하다.** 늦어도 기회 손실일 뿐 손해는 아니다
- 목록만 돌면 된다. 상세는 **등록을 결정한 시점에** 한 번
- 주기 = 하루 1회 ~ 주 1회면 충분

**⇒ 유지가 발굴보다 항상 우선한다.** 크롤 총량이 부족하면 발굴을 미룬다.

## 4. 목록 크롤이 내놓는 것 — 신규·소멸 감지

목록을 돌면 직전 결과와 비교해 세 가지가 나온다:

| | 뜻 | 할 일 |
|---|---|---|
| **신규** | 소싱처에 새로 생김 | 등록 후보로 쌓기 |
| **사라짐** | 소싱처에서 내려감 | ⚠️ **우리가 팔고 있으면 판매중지** |
| **가격 바뀜** | 목록가가 달라짐 | 상세 크롤 대상으로 |

**「사라짐」이 특히 중요하다.** 소싱처에서 없어졌는데 우리가 계속 팔면 **주문을 이행할 수 없다.** 품절보다 나쁘다 — 품절은 재입고를 기다리지만 이건 영영 없다.

## 5. 페이지네이션 — 전체를 매번 돌지 않는다

검색 결과 3,000개면 페이지가 수십~수백 개다. 매번 전부 도는 건 비싸다.

| 스캔 | 범위 | 주기 | 목적 |
|---|---|---|---|
| **얕은 스캔** | 앞 3~5페이지 (최신순 정렬) | 자주 | 신규 발견 |
| **깊은 스캔** | 전체 페이지 | 드물게 (주 1회 등) | 소멸 감지·전수 대조 |

- 정렬을 **최신순**으로 두면 신규는 앞쪽에만 나온다 → 얕은 스캔으로 충분
- **소멸은 전체를 봐야 안다**(안 보이는 게 사라진 것) → 깊은 스캔이 필요
- 단, 우리가 **등록한 상품**은 목록에 안 보여도 상품 URL 로 직접 확인할 수 있다 → 소멸 감지를 깊은 스캔에만 의존하지 않아도 된다

## 6. 목록 URL = 「소싱처 &gt; 브랜드」 구성

사장님이 확인해주신 단위가 여기 딱 맞는다.

```
구성: 무신사 > 나이키
 └ 목록 URL: musinsa.com/search/goods?keyword=나이키&gf=A
    ├ 등록함    142개  ← 유지 크롤 (급함)
    └ 후보     2,858개  ← 발굴 크롤 (안 급함)
```

- 구성 하나 = 목록 URL 하나
- **가공정보(등록 템플릿)도 이 단위**로 붙는다 → 「무신사 > 나이키」 상품은 같은 템플릿으로 등록
- 완성도(E안)도 이 단위

### 중복 상품 처리
검색어가 겹치면 같은 상품이 여러 구성에 나온다(「나이키」와 「운동화」 둘 다에). **상품 ID 로 중복을 제거**하고, 이미 다른 구성에서 크롤했으면 다시 안 한다. 안 그러면 구성이 늘수록 크롤이 제곱으로 늘어난다.

## 7. 상세 크롤 우선순위

한 바퀴에 상세를 N개만 돌 수 있다면 이 순서로:

```
1. 등록한 상품 중 목록가가 바뀐 것        ← 가장 급함 (역마진 위험)
2. 등록한 상품 중 목록에서 사라진 것       ← 판매중지 판단
3. 등록한 상품 중 오래 안 본 것            ← 변동 주기 도달
4. 신규 후보 중 등록하기로 한 것           ← 사람이 고른 것
5. 나머지 후보                             ← 남는 여유에만
```

## 8. 단계 계획

| 단계 | 내용 | 왜 이 순서 |
|---|---|---|
| **1** | 목록 크롤 + 신규/소멸/가격변동 diff. **저장만, 상세 안 들어감** | 목록만으로 뭐가 나오는지 먼저 봐야 함. 비용도 낮음 |
| **2** | 「사라짐 → 판매중지 후보」 알림 | 실제 손해를 막는 것이라 가장 먼저 값어치가 남 |
| **3** | 목록가 변동 → 상세 크롤 트리거(등록한 상품만) | 유지 크롤 실효화 |
| **4** | 후보 목록 화면 + 등록 결정 → 상세 크롤 → 가공정보로 등록 | 발굴 흐름 완성 |
| **5** | 얕은/깊은 스캔 분리, 중복 제거, 총량 배분 | 규모가 커진 뒤에 필요 |

**1·2단계만으로도 값어치가 납니다** — 지금은 소싱처에서 상품이 사라져도 모르고 계속 팔고 있을 수 있습니다.

## 9. 미확정 — 사장님 확인 필요

1. **목록 URL 을 사장님이 직접 넣나요, 검색어만 넣고 우리가 URL 을 만드나요?** 무신사 URL 에 `type={componentKey}` 같은 템플릿 자리가 있어서, 그대로 붙여넣는 방식인지 조립하는 방식인지에 따라 설계가 갈립니다.
2. **한 구성에서 몇 개까지 후보로 볼 건가요?** 「나이키」 검색이 3,000개면 전부 후보로 쌓을지, 앞 N개만 볼지.
3. **소싱처에서 사라진 상품을 자동으로 판매중지할까요, 알림만 줄까요?** 자동이면 빠르지만 오검지 시 멀쩡한 상품이 내려갑니다.
4. **후보를 고르는 기준이 있나요?** 가격대·브랜드·카테고리 등으로 자동 걸러낼 수 있으면 상세 크롤을 크게 아낍니다.
5. **무신사 외 소싱처도 목록 URL 방식인가요?** 소싱처마다 검색 구조가 달라서, 지금은 무신사만 기준으로 썼습니다.

## 10. 이번 제안이 지키는 것

- **목록가는 신호로만, 실제 값은 상세에서** — 롯데아이몰 사고 재발 방지
- **유지 > 발굴** — 급한 것 먼저
- **크롤 실패를 「변동 없음」으로 세지 않기** — 기존 규약 유지
- **폴백 금지** — 목록에서 못 읽으면 추정하지 않고 상세로 확인
