# 대량등록 Phase 1B — 최종매입가·마진 엔진 (Design Spec)

> 2026-07-18. 대량등록(수기입력→마켓등록, Phase 1A 완료·라이브)에 **최종매입가·마진 계산**을 붙인다.
> 관계: [[2026-06-07-최종매입가-계산엔진-design.md]](기존 엔진 진실원천), [[2026-07-17-신규상품등록-가공템플릿-design.md]](대량등록 상위설계).
> 진실원천 = 기존 `compute_final_price`(`lemouton/pricing/final_price.py`) + 영빈 「대량위탁」 관리엑셀(카드/캐시백 데이터).
> 상태: **검토 대기** (사용자 확정 후 plan→구현).

## 0. 이 문서가 존재하는 이유 (경위)

소싱처 가격로직을 **2회 오진**한 끝에(네이버페이 "택1 버그" 오진, `pricing_policy.py` 죽은코드를 진실로 오인) 사장님이 두 번 바로잡아 확정한 모델을 문서로 고정한다. **금전 직결(가격·마진 오류=손실)** 영역이라 코드 전에 설계 합의를 남긴다.

**핵심 교훈(코드 착수 전 필독):**
- 소싱처 가격로직의 진실원천은 **하드코딩 python 이 아니라** 앱 데이터-주도 설정(SourceBenefitTemplate) + `compute_final_price` 엔진 + **모음전 상세페이지 가격셀/fx영수증 = 정답지**.
- `lemouton/sourcing/crawlers/pricing_policy.py` 의 `apply_*_policy` = **죽은 코드(호출자 0)이며 순서도 틀림** — 절대 참조·부활 금지.

## 1. 배경 / 조사 결과

- **기존 엔진 그대로 재사용**: `compute_breakdown`(`webapp/routes/api_benefits.py`)이 3계층(SourceBenefitTemplate + OptionBenefitOverride + `dynamic_benefits_json`)을 머지해 `compute_final_price`(순수함수) 호출 → 소싱처별 최종매입가. 모음전 매트릭스 셀·최저 소싱처 선정·fx영수증이 전부 이 함수 경유(blast radius 큼).
- 엔진은 이미 지원: 혜택 **5종**(선반영 preapplied / 차감 deduct / 적립 accrue / 결제 payment / 캐시백 cashback), **순차 누적 곱셈 차감**(각 %가 직전 잔액=베이스금액 기준), **세트제약**(결제 택1 `affiliate_card⟷naver_pay`, `channel=naver_via`⟹캐시백 off), **경로 최저가 자동선택**(`_compute_tagged` 경로열거).
- **한계(이번에 확장할 지점)**: 결제카드가 **현대카드 2.73% 하나만 하드코딩**(`api_benefits.py:870~886`, 롯데온·SSG·무신사). 카드 종류·소싱처별 청구할인·OK캐시백 사이트 적립율은 **데이터가 없다**.

→ **새 엔진 금지. 기존 `compute_final_price` 를 데이터·태깅으로만 확장.** 재구현·`pricing_policy.py` 사용 금지.

## 2. 목표 / 비목표

**★ 대량등록 프로그램의 본질 (사용자 확정 2026-07-18) — 모든 설계가 여기서 나옴:**
> 소싱처의 **가격·데이터 변동을 수집하면 가능한 빨리** 최종매입가로 가공해 **판매처(마켓)에 업로드**해야 한다.
> - 우리 가격이 **낮으면 → 역마진**(팔면 손해, 못 판다)
> - 우리 가격이 **높으면 → 경쟁력 상실**(안 팔린다)
> - 따라서 **① 변동점의 빠른 업로드**와 **② 주문 시점에 그 가격 그대로 주문되는 것**이 관건.
> - 주문 시점에 가격 차이가 있으면 **화면에 전후 가격을 전부 표시**한다(숨기지 않는다).

**목표:**
- 대량등록 각 주문(ProductDraft/주문행)에 대해 **최종매입가 = 표면노출가 − 혜택(순차차감)** 을 기존 엔진으로 산출.
- **가격 변동 → 재계산 → 마켓 업로드 지연 최소화**(§4-5).
- **주문 시점 가격 정합성 대조 + 차이 시 전후 가격 전부 표시**(§4-6).
- 기존 엔진에 **결제카드 다양화**(카드별 적립율 + 소싱처별 카드 청구할인, 둘 다 별개 차감, 카드 택1 최유리) + **유입경로 OK캐시백**(소싱처별 적립율) 데이터를 추가.
- **현대카드 2.73% 기본 플로어** — 대안(OK캐시백+타카드 등)이 현대카드를 못 넘으면 현대카드 채택(현행 동작 보존, 절대 후퇴 금지).
- 크롤로 어려운 값(유입경로·카드청구할인)은 **주문별 수기 입력/override** 가능.
- 대량등록 **마진 = 판매가 − 최종매입가**, fx영수증 재사용.

**비목표(이번 X):**
- 혜택 율/금액 자동 크롤 신규 작성(기존 크롤·수기·config 값 사용).
- 카드 청구할인율의 자동 수집(수기·config 시드로 시작).
- 실전송/실등록 로직 변경(Phase 1A 게이트 그대로).

**사용자 확정(2026-07-18):**
- 적립율·청구할인 = **별개 항목, 둘 다 차감**.
- 카드 = **택1 최유리**(현대카드보다 유리한 카드 있으면 그것). **없으면 현대카드 기본**.
- 유입경로·카드청구할인 **수기 수정 가능** 필수.
- 네이버페이 적립 = **누적**(택1 아님).

## 3. 계산 모델 (핵심, 확정)

입력: `BASE`(소싱처 표면노출가 = 크롤 or 수기), 혜택 목록(태그 포함).

```
잔액 = BASE
잔액 −= Σ 소싱처 정액혜택(원)                 # 리뷰금 등, 정액 먼저
── 이하 전부 정률, 직전 잔액 기준 순차 곱셈(잔액 ×= (1−r)) ──
잔액 ×= (1 − 소싱처 정률혜택)                  # 등급할인·쿠폰 등
잔액 ×= (1 − 네이버페이 적립율)                # 누적(택1 제외). 소싱처별 적용여부 상이
── [유입경로 택1] N쇼핑경유 ↔ OK캐시백 (세트제약) ──
잔액 ×= (1 − OK캐시백 적립율)                  # 캐시백 경로일 때만
── [카드 택1] 최유리 카드 1개 (현대카드 기본 플로어) ──
잔액 ×= (1 − 카드 적립율)                       # 카드 고유 적립 (엑셀 카드표)
잔액 ×= (1 − 카드 청구할인율)                   # 소싱처×카드별 청구할인 ★적립율과 별개, 둘 다
최종매입가 = min over 모든 유효경로            # 백원 단위 버림(floor)은 헤드라인만
마진 = 판매등록가 − 최종매입가
```

- **현대카드 기본 플로어**: 현대카드 경로(적립 2.73% 등)를 **항상 후보로 유지**. 경로열거 min 이 현대카드 포함 전 경로를 비교 → 대안이 현대카드보다 매입가를 못 낮추면 현대카드 자동 채택. (별도 if 분기 불필요 — min 이 곧 규칙.)
- **적립도 실질 비용 인하**라 같은 순차 곱셈에 포함(표시엔 '적립'/'청구할인' 구분).
- ※ 순차 누적 방식은 기존 `compute_final_price` 동작과 동일 → M1 회귀 테스트 그대로 성립.

> ### ★ 저장 방식 확정 (사용자 결정 2026-07-18) — **기존 소싱처 혜택 설정에 얹는다**
> **위 4-2·4-3 은 신규 테이블을 만들지 않는다.** 소싱처별 **카드 청구할인**과 **OK캐시백 적립율**은 **기존 `SourceBenefitTemplate` 행**으로 시드한다(`apply_mode=payment|cashback` + `pay_method=<card_key>` / `channel` 태그). 이유:
> - 신규 테이블 0 · 엔진(`compute_final_price`) 무수정 · 3계층 머지(base+override+dynamic) 그대로 탐.
> - **기존 소싱처 혜택 편집 UI(「가격계산로직」)에서 그대로 보이고 수정된다** → 수기 override 요구(§2)가 공짜로 충족.
> - `OptionBenefitOverride` 도 그대로 작동 → 모음전 옵션별 예외도 유지.
>
> **유일한 신규 테이블 = `PurchaseCard`(4-1, 카드 마스터)** — 카드 적립율은 소싱처 무관 공통값이라 소싱처 혜택 행에 중복 저장하면 N×M 중복·불일치가 생긴다. 카드 목록·적립율만 여기 한 곳(단일 진실원천).
> ⇒ **정리: 신규 테이블 1개(PurchaseCard) + 나머지는 SourceBenefitTemplate 시드.**

### 4-5. 가격 변동 → 재계산 → 업로드 파이프라인 (핵심)

대량등록의 존재 이유. **소싱처 변동 감지 → 최종매입가 재계산 → 판매가 가공 → 마켓 업로드**를 최단 지연으로.

```
[소싱처 크롤: 표면노출가·혜택·재고 변동 수집]
        │ 변동 감지(직전 스냅샷과 diff)
        ▼
[compute_final_price 재계산] → 새 최종매입가
        │ 변동 있을 때만(무변동 = 업로드 스킵, 불필요 API 호출 0)
        ▼
[판매가 가공: 최종매입가 + 마진정책 → 마켓 업로드가]
        ▼
[마켓별 가격 업로드] ★기존 4마켓 실전송 검증 완료분 재사용
        │
        ▼
[가격 스냅샷 저장: 어떤 값으로 언제 업로드했나 = 주문시점 대조 기준]
```

- **변동 diff 기준**: 표면노출가·혜택·(카드/캐시백/유입경로 설정)·재고 중 하나라도 바뀌면 재계산.
- **무변동 스킵**: 같은 값 재업로드 금지(마켓 API 낭비·rate limit 방지).

#### 4-5-1. 변동 우선순위 · 업로드 게이트 (사용자 확정 2026-07-18) ★

모든 변동을 다 올리지 않는다. **의미 없는 재고 업로드가 큐를 막으면 정작 급한 가격·품절 반영이 늦어진다** — 이 게이트는 절약이 아니라 **P0 지연을 지키는 장치**다.

| 우선순위 | 변동 | 판정 | 업로드 | 이유 |
|---|---|---|---|---|
| **P0** | **가격 변동** | 최종매입가·업로드가 달라짐 | ✅ **즉시 최우선** | 낮으면 역마진, 높으면 경쟁력 상실 |
| **P0** | **품절 전환** | 재고 있음 → **0** | ✅ **즉시** | 남들 다 품절이면 **주문이 우리한테만 몰림** → 이행 불가 → 고객 불만·CS 폭증. **우리도 즉시 품절 처리** |
| **P1** | **품절 임박** | 재고 ≥3 → **≤2** | ✅ | 곧 품절될 구간이라 주의 대상 |
| **P1** | 재입고 | 재고 0 → ≥1 | ✅ | 판매 재개(매출). ※최종 확인 필요 — §10 |
| **P2** | **재고 잔변동** | ≥3 구간 내 이동 (5→3, 3→4, 10→3) | ❌ **스킵** | **의미 없음.** 2개 이하로 떨어질 때만 주의하면 됨 |

**게이트 판정 순서(구현 규약):**
```
if 가격변동:                 → 업로드 (P0, 재고 무관)      # 가격이 최우선
elif 재고 == 0:              → 업로드 (P0, 품절 처리)
elif 이전재고 == 0 and 재고 >= 1: → 업로드 (P1, 재입고)
elif 이전재고 >= 3 and 재고 <= 2: → 업로드 (P1, 품절임박)
elif 이전재고 >= 3 and 재고 >= 3: → 스킵   (P2, 의미없음)
else:                        → 업로드 (≤2 구간 내 변동 = 주의구간)
```
- **가격 변동이 있으면 재고 조건과 무관하게 항상 업로드**(P0 우선). 재고 스킵 규칙은 **가격 무변동 + 재고만 변동**일 때만 적용.
- **큐 분리 권장**: P0 전용 레인 / P1·P2 일반 레인. P2 스킵으로 확보한 여유를 P0 지연 단축에 쓴다.
- **스킵도 기록**: 스킵한 변동은 로그·스냅샷에 남긴다(조용한 실패로 오인 방지, [[project_silent_failure_bug_class]]).
- **역마진 가드**(사용자 확정 2026-07-18): 기준은 **마진율이 아니라 마진금액(원)**, 그리고 **사용자가 화면에서 설정**한다. `min_margin_amount`(설정값, 기본 0원) 미만이면 **업로드 대신 경고·판매중지 후보로 표시**. 조용히 손해보는 가격 올리지 않는다.
  - 설정 위치: 대량등록/자동화 설정에 「최소 마진금액」 입력칸(원 단위). 소싱처·마켓별 개별 설정 여부는 M3 에서 결정(우선 전역 1개로 시작).
  - 예: 최소 마진금액 3,000원 → 마진 2,500원 상품은 업로드 보류 + 경고.
- **크롤 실패 시 폴백 금지**: 값 못 가져오면 **'확인불가'**로 두고 기존 가격 유지 + 경고. 추정가 업로드 절대 금지([[feedback_no_fallback_price_on_match_fail]]).
- **가격 스냅샷**(`PriceSnapshot`): `sku·source·표면가·최종매입가·업로드가·계산근거(steps)·uploaded_at`. 주문시점 대조의 기준값.

### 4-6. 주문 시점 가격 정합성 (전후 가격 표시)

주문이 들어온 순간, **그 주문이 걸린 가격**과 **현재 소싱처 실가격**이 다를 수 있다(변동~주문 사이 시차). 이때 **숨기지 않고 전부 보여준다.**

- **대조 대상 3개**: ① 업로드 당시 최종매입가(스냅샷) ② 주문 시점 마켓 판매가 ③ 현재 소싱처 실매입가(재조회).
- **차이 발생 시 화면 표시(필수)**: 전/후 값을 **둘 다** 표시 + 차액 + 마진 영향. 어느 쪽이 최신인지 명시.
- **정직성 규칙**: 차이를 자동 보정·평균·반올림해 하나로 뭉개지 않는다. **역마진 전환 시 강조 경고**(빨강) — 이 주문 그대로 매입하면 손해라는 사실을 사람이 즉시 알게.
- **시안 필요** — 전후 가격 표시 UI 는 별도 시안으로 확인받고 구현(§6, 사용자 요청).

### 4-7. 소싱처×브랜드 변동성 통계 + 랩 보고서 (계수 근거) ★

**목적**: 크롤 1바퀴(랩)를 돌 때마다 **소싱처×브랜드별로 가격·재고가 실제로 얼마나 자주 바뀌는지** 통계를 쌓고, 그 근거로 **크롤 계수**를 정한다. "감으로 1~5" → "데이터로 1~5".

#### 이미 있는 것 (재구현 절대 금지)
`lemouton/sources/crawl_schedule.py` 에 **가중 라운드로빈 랩**이 완성돼 있다:
- **한 랩 = 각 URL을 유효계수만큼 크롤**(×1=1번, ×2=2번, ×3=3번) — 사용자가 말한 "전체 크롤 1회 때 얘는 3~4번"이 정확히 이것.
- `crawl_weight` 1~5(0=제외) · `CrawlWeightRule` **scope_type = source|brand|model|url** (소싱처별·브랜드별 계수 이미 지원) · `resolve_crawl_weight` 5단계 상속 · `build_batch_weight_resolver`(N+1 장애 수정분).
- `lap_quota` · `record_crawl_served` · `start_new_lap` · **`CrawlLapRun`**(랩 완료 이력 → '오늘 몇 바퀴', '1바퀴 시간').
- `no_change_streak` — 무변동 연속 시 간격 확대(초보적 적응).
- 계수 편집 UI: `webapp/templates/automation/weights.html`.

#### 이번에 새로 만들 것
1. **변동 이벤트 기록** — 크롤 결과 저장 시(`save_crawl_result` 경로) 직전 값과 비교해 **변동 이벤트**를 남긴다: `{source, brand, sku, lap_id, 가격변동여부, 이전가→새가, 재고변동여부, 이전재고→새재고, ts}`. §4-5 의 diff 를 그대로 재사용(중복 계산 X).
2. **소싱처×브랜드 집계** — `SourceBrandVolatility`: `source_key · brand · 최근 N랩 가격변동횟수 · 재고변동횟수 · 품절전환횟수 · 관측 URL수 · 변동률(변동/관측)`.
3. **랩 보고서** — 랩 1바퀴 끝날 때(`start_new_lap` 시점) 그 랩의 통계를 확정·저장하고 화면에 보고서로 노출:
   - 소싱처×브랜드 **변동률 순위표**(많이 바뀌는 순)
   - 현재 계수 vs **권장 계수**(변동률 기반 산출) 나란히
   - 이번 랩 소요시간·오늘 바퀴 수(`CrawlLapRun` 기존값 재사용)
   - **P2 로 스킵된 재고변동 건수**(§4-5-1) — 스킵이 묻히지 않게
4. **계수 자동 산출** — 변동률 → 계수 1~5 매핑. **자동 적용 vs 추천만(사람 확정)** 은 M5 착수 시 결정(§10). 어느 쪽이든 **왜 그 계수인지 근거(최근 변동 횟수)를 화면에 같이 표시**.

#### 무결성
- 통계는 **관측 기반**만. 크롤 실패(`last_status='error'`)는 '변동 없음'으로 세지 않는다 — **실패를 안정으로 오독하면 계수가 잘못 내려간다**(조용한 실패 클래스).
- 계수 0(크롤 제외)은 통계·자동산출 대상에서 제외(사용자 의도 존중).
- 신규 브랜드·표본 부족(관측 N 미만)은 자동 조정 보류(기본 계수 유지).

## 5. 구현 위치 (단일 진실원천, 재구현 0)

- `api_benefits.py`의 하드코딩 현대카드 주입부(870~886)를 **카드 후보 다중 주입**으로 교체:
  - `PurchaseCard` × (해당 소싱처의 `SourceCardBillingDiscount`) → 각 카드를 `apply_mode='payment'`, `pay_method=<card_key>` 태그의 **혜택 2항목**(적립율·청구할인율)으로 append. **같은 pay_method 로 묶여** 경로열거 시 그 카드 선택 경로에서만 동시 활성.
  - 현대카드 후보는 항상 포함(기본 플로어).
  - OK캐시백은 `apply_mode='cashback'`(+`channel` 세트제약)로 소싱처 적립율 append.
- `compute_final_price` **무수정**(경로열거·최저가·floor 이미 지원). 후보만 늘어난다.
- 주문별 수기값은 위 append 단계에서 override(해당 카드/캐시백 enabled·값 치환).

## 6. 대량등록 UI (수기 입력)

- 수기 폼에 **유입경로(N쇼핑/OK캐시백/없음) · 결제카드(드롭다운, PurchaseCard) · 네이버페이 O/X · 캐시백사이트** 입력칸.
- 표면노출가·소싱처 선택 → 최종매입가·마진 **실시간 표시**(fx영수증 팝업 재사용, `smRenderFxPopBody`).
- 비워두면 소싱처 config 기본값 사용, 입력하면 override. **크롤 불가값 수기 우선.**
- **주문 시점 전후 가격 표시**(§4-6) — 업로드 당시 가격 vs 현재 실가격을 **둘 다** 표시 + 차액·마진영향, 역마진이면 빨강 경고. **→ 별도 시안으로 확인 후 구현**(사용자 요청).

## 7. 무결성 / 리스크

- **blast radius = 전 매트릭스 가격.** `api_benefits.py` 주입부 변경 = 모든 소싱처 매입가. → **회귀 테스트 필수**: 변경 전후 동일 SKU 다수의 `final_price` 스냅샷 == (현대카드만 있던 기존 결과 불변) 확인 후에만 머지.
- **후퇴 금지 불변식**: 카드 후보 확장 후에도 어떤 소싱처든 최종매입가 ≤ (현대카드만이던 기존값). min 선택이라 수학적으로 보장되지만 테스트로 고정.
- 값 없는 카드/캐시백(rate 0/None) → 차감 0(무시), 오류 X.
- 음수 매입가 방지(엔진 내 `max(base−deduct,0)` 유지).
- SQLite 폴백 worktree 는 실 config DATA 못 봄 → 시드·유닛테스트로 검증, 라이브는 배포 후 fx영수증 실검증.

## 8. 테스트

- **회귀(최우선)**: 카드 다중주입 전/후 기존 SKU 스냅샷 동일(현대카드만이던 결과 보존).
- 유닛(순수 `compute_final_price`): ①현대카드 단독 vs ②OK캐시백+타카드 합산 — 유리한 쪽 자동 채택 / ③대안<현대카드 → 현대카드 채택(플로어) / ④적립율·청구할인 둘 다 차감(같은 카드) / ⑤네이버페이 누적(택1 제외) / ⑥유입경로 택1(N쇼핑↔캐시백) / ⑦수기 override.
- **업로드 게이트(§4-5-1)**: 가격변동→항상업로드(재고무관) / 재고 있음→0 업로드 / ≥3→≤2 업로드 / 5→3·3→4·10→3 **스킵** / 0→≥1 업로드 / 가격변동+재고≥3구간 → **업로드**(가격 우선). 스킵 건이 **로그에 남는지**까지 검증.
- 값 검증: 엑셀 실제 주문행 몇 건의 최종매입가와 엔진 출력 대조(표면가·카드·네이버페이 입력 → 최종매입가 일치).
- 실검증: 라이브 fx영수증에서 카드 경로·최종가 표시 확인.
