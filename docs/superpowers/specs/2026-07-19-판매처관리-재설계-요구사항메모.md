> **이 문서의 목적**: 요구사항이 대화에만 남아 사라지는 걸 막기 위한 **받아적기**.

---

## 사장님 요구사항 (원문 취지)

> 판매처 관리에서 모음전탭과 대량등록탭 마켓 ID/API 는 **한 탭에서 관리**하되,
> 주문에 필요한 정보가 다르니 **공용 / 모음전용 / 대량등록용 구분**이 필요하다.
> 그리고 같은 아이디로 공유하더라도, 주문내역 끌어올 때 **브랜드별로 분류해서 구분해 끌어오는 기능**도 추가.

---

## 받아적은 항목 3가지

| # | 요구 |
|---|---|
| 1 | 마켓 ID/API 자격정보는 **한 탭에서 통합 관리** (지금처럼 흩어놓지 않음) |
| 2 | 항목마다 **용도 구분** — `공용` / `모음전 전용` / `대량등록 전용` |
| 3 | 같은 계정이라도 주문 조회 시 **브랜드별로 분류**해서 가져오기 |

---

## ⚠️ 설계 시 반드시 확인할 것 (지금 아는 범위)

**3번이 간단하지 않습니다.** 주문 내역 API 는 **브랜드를 안 줍니다.**

마켓이 주는 건 상품명·마켓 상품ID 정도입니다. 브랜드를 붙이려면
`ProductDraft.brand` 와 **마켓 상품ID 로 조인**해야 합니다.

| 주문 | 브랜드를 붙일 수 있나 |
|---|---|
| 우리가 대량등록으로 올린 상품 | ⭕ `ProductDraftMarket` 에 마켓 상품ID 있음 |
| 우리가 모음전으로 올린 상품 | ⭕ 같은 경로 |
| **수기 등록 · 과거 상품** | ❌ **못 붙임** — 조인할 근거가 없음 |

→ 「브랜드 미상」 이라는 상태가 반드시 생깁니다. 그걸 어떻게 다룰지가 설계의 핵심입니다.
→ 상품명 문자열로 브랜드를 추측하는 건 **하지 말 것** — 오분류가 정산·발주로 번집니다.

**2번도 확인 필요.** 지금 계정 관리는 마켓별 env prefix 방식입니다
(`project_marketplace_multi_account_per_market_prefix`). 용도 구분을 넣으면
기존 계정들이 어느 용도로 마이그레이션되는지 정해야 합니다.
