# 주문내역 재정의 #1 — 「주문일 = 신규 주문」 분리 (design)

> **계층**: 프로젝트 스펙 / **날짜**: 2026-07-15 (rev2 — "남긴다" 결정 + 마켓별 데이터 한계 반영)

> **본 스펙 범위**: **①번만**. ②·③은 별도 스펙(순서 B — 1→2→3).
> **관련 메모리**: [[project_marketplace_order_status_unification]], [[project_multi_account_orders_20260710]], [[project_delivery_inspection_tab_20260712]]

## 1. 목표 (한 줄)

주문내역의 **주문일 목록**을 「그 기간에 실제로 들어온 주문(**실주문일** 기준)」만으로 정리한다. **옛 주문이 이 기간에 상태만 바뀐 것**(취소/교환/반품 이벤트)이 주문일 목록을 오염시키지 않게 데이터 원천에서 갈라, 기능 #2(상태변경 별도 탭)가 소비할 태그를 심는다.

## 2. 확정된 정의 (사용자 결정)

- **신규 주문 = 그날(기간에) 주문이 들어온 건. 주문상태는 무관.**
  - 분리 기준 = **"실주문일이 이 기간 안인가"** (어느 API에서 왔나 아님).
  - 나중에 취소/반품/교환돼도, **주문일이 기간 안이면 주문일 목록에 그대로 남긴다**(상태 배지만 취소완료 등). → 사용자 결정: **"남긴다(상태 무관)"**.
  - 반대로 **옛 주문(주문일 기간 밖)이 이 기간에 취소된 것**은 신규 주문이 아니므로 주문일 목록에서 **뺀다** → 기능 #2가 변경일 기준으로 잡는다.
- **세는 단위 = 옵션 줄**(현재 방식 유지). KPI 「신규주문 N건」도 옵션 줄 기준.
- **상태변경(취소/교환/반품) 탭·워크플로**는 기능 #2. 기능 #1은 데이터 태그(`_kind`·`_change_date`)와 `new_order_rows`까지만.

## 3. 현재 구조 (as-is) — 실측

- `lemouton/markets/order_export.py`
  - 마켓별 빌더가 **주문 행 + 클레임 행**을 한 리스트에 섞어 만든다.
  - `combined_order_rows(…)` → `_fetch_combined(…)` → `_filter_by_order_date(rows, since, until)`(주문일 트리밍, **파싱 실패 행은 보존**).
  - 공유처 4곳: `/orders/preview.json`(화면), `/orders/export.xlsx`, `delivery/market_enrich.py`(배송검사), `margin/sell_source.py`(마진).
- **클레임 행의 주문일 실측 (마켓별 상이 — 이게 오염 원인):**

| 마켓 | 취소/반품/교환 등장 경로 | 클레임 행 `주문일` 값 | 오염 |
|------|--------------------------|-----------------------|------|
| 스마트스토어 | 주문 행에 상태로 흡수(`productOrderStatus`) | **실주문일**(orderDate/paymentDate) | 없음 |
| 롯데온 | 별도 `_claim_row` | 기존엔 `odAccpDttm` 없으면 `clmReqDttm`(클레임일) 폴백 → 옛주문이 클레임일로 오염 | 폴백 시 오염 |
| 쿠팡 | 별도 `_cp_claim_row` | `createdAt`(**클레임 생성일** — 실주문일 아님) | **항상 오염** |
| 11번가 | 별도 `_claim_row` | `""`(공란) → 파싱실패=**항상 보존** | **항상 오염**(기간 무시) |

→ 클레임 행이 **옛 주문의 상태변경**인데도 클레임일/공란 때문에 주문일 목록에 신규처럼 끼어든다. **★rev4 라이브 검증 정정: 롯데온 클레임 itemList는 실제로 `odAccpDttm`(실주문접수일)을 제공**한다(claims.py 문서가 불완전했을 뿐 — itemList 평탄화가 전체 필드 보존). 따라서 롯데온은 폴백만 제거하면 실주문일로 정확히 트리밍된다(rev3에서 "미제공"이라 한 건 문서만 보고 내린 오판, 라이브로 반증).

## 4. 설계 (to-be)

### 4.1 데이터 계층 — 성격 태그 + 주문일 위생 (seam)

**(a) 태그** — 모든 행에 기본 `_kind='order'`. 클레임 빌더에서만 override.

| 필드 | 값 | 의미 |
|------|-----|------|
| `_kind` | `'order'` | 주문/배송 API 유입 행. **상태가 취소완료여도(140 덮어쓰기·스스 흡수) `'order'` 유지.** |
| `_kind` | `'change'` | 클레임 API에서 온 취소/교환/반품 **이벤트** 행(롯데온·쿠팡·11번가 `_claim_row` 경로). |
| `_change_date` | `str` | `_kind='change'` 행에만. **변경일 = 클레임 접수일**(롯데온 `clmReqDttm` / 쿠팡 `createdAt` / 11번가 클레임 날짜필드). #2가 이 값으로 트리밍·정렬. |

**(b) 주문일 위생** — 클레임 행의 `주문일`을 **실주문일로만** 두어(추정·클레임일 금지) 트리밍이 정직해지게:
- 롯데온 `_claim_row`: `주문일 = odAccpDttm`만(**`clmReqDttm` 폴백 제거** → 없으면 공란). 클레임일은 `_change_date`로.
- 쿠팡 `_cp_claim_row`: `주문일 = ""`(기존 `createdAt` 제거). `createdAt`은 `_change_date`로.
- 11번가 `_claim_row`: `주문일 = ""` 유지. 클레임 날짜필드를 `_change_date`로.
- (근본원인 [[project_no_fallback_price_on_match_fail]] 규율과 동일 — 못 구하면 공란, 추정 금지.)

### 4.2 신설 함수 — `new_order_rows`

```
new_order_rows(markets, *, days/since/until/use_cache/…) -> list
```
규칙: **"실주문일이 기간 안인 행만"** 남긴다.
- `_kind='order'` 행 → **항상 유지**(활성 주문·스스 취소 포함, 실주문일로 이미 트리밍됨). → "남긴다" 충족.
- `_kind='change'` 행 → **실주문일이 기간 안일 때만 유지**, 아니면 제외(#2 소관).
  - 롯데온 클레임(실주문일 있음) 중 **기간 안** = 유지("남긴다" 충족). 기간 밖 = 제외.
  - 쿠팡·11번가 클레임(실주문일 공란) = **항상 제외** → #2에서 변경일 기준으로만 보임.

구현: `combined_order_rows(같은 인자)` 결과 위 얇은 후처리(재fetch 없음 → TTL 캐시·경고 전파 그대로 재사용). 기간(lo,hi)은 since/until, 없으면 days로 유도.
```
rows = combined_order_rows(...)              # order+change, 태그됨, order행은 주문일 트리밍 완료
lo, hi = _window(since, until, days)
out = []
for r in rows:
    if r.get('_kind') == 'change':
        d = _row_order_date(r)               # 실주문일 파싱
        if d is None or not (lo <= d <= hi): # 공란(쿠팡·11번가)·기간밖(롯데온 옛주문) → 제외
            continue
    out.append(r)
return out
```

### 4.3 화면 배선

- `/orders/preview.json`가 `combined_order_rows` → **`new_order_rows`** 호출로 교체(오직 이 엔드포인트만).
- **호출 불변**: `/orders/export.xlsx`, `market_enrich`, `margin/sell_source`는 `combined_order_rows`를 그대로 호출한다(함수 교체 없음).
- **단, 데이터 부수효과(rev3 정직화)**: 주문일 위생은 빌더 레벨이라 `combined_order_rows` 출력의 **클레임 행 주문일도 공란**이 된다. 따라서 엑셀·마진에서도 클레임(취소/반품/교환) 행 주문일이 공란으로 바뀐다. **금액 계산(정산·수수료·마진)은 주문일을 쓰지 않아 영향 없음.** 영향은 (a) 엑셀 클레임 행 주문일 표시 공란, (b) 마진 날짜 버킷에서 클레임 행 제외(취소/반품이 매출월에 안 잡힘 — 오히려 정확), (c) 정렬 시 공란 클레임 행이 하단으로. 라이브 육안검증(§9)으로 확인.
- 사용자 체감:
  - 주문 목록 = 그 기간 실유입 주문만(옵션 줄). **옛 주문의 취소/교환/반품 churn 행이 빠짐**.
  - 이 기간에 들어와 취소된 주문(**스스·롯데온** — 실주문일 확보, §5)은 **남고** 배지만 취소완료. 쿠팡·11번가 클레임은 #2.
  - KPI 「신규주문 N건」이 유입 건수로 정확.

### 4.4 `status_change_rows`(변경일 기준 수집)·상태변경 탭 = 기능 #2

`_kind`/`_change_date` 태그는 #1에서 심되, 변경일 기준 수집 함수와 탭 UI(3단계 워크플로)는 #2 소관.

## 5. 마켓별 데이터 실태 (rev4 — 라이브 검증 반영)

라이브(mou-m.com, 2026-07-15, days=7) preview.json 실측:
- **스마트스토어 · 롯데온 = "남길 수 있음" 버킷**(실주문일 확보):
  - 스스: 취소가 `productOrderStatus` 주문 행으로 옴(`orderDate` 있음).
  - 롯데온: 클레임 itemList가 **`odAccpDttm` 제공**(실측 확인 — 주문일 ≠ `_change_date`). 라이브 change 60건 전부 실주문일 창내(=오늘 유입 후 취소) → 주문일 탭에 정상 유지(남긴다 ✓). 옛주문 취소는 창밖이라 드롭.
- **쿠팡 · 11번가 = "클레임은 #2에서만" 버킷**(실주문일 미제공):
  - 쿠팡=`createdAt`(클레임일)만, 11번가=날짜필드 없음 → 주문일 공란 → `new_order_rows`가 드롭. 라이브 검증: 쿠팡·11번가 preview `_kind='change'` **0건**(전부 드롭), 순수 order 행만·취소/반품/교환 churn 0. 이 건들은 **#2에서 `_change_date` 기준으로** 노출 예정.
  - (한계이지 버그 아님 — 실주문일을 추정·날조하지 않고 공란, [[feedback_no_fallback_price_on_match_fail]] 규율.)
- **핸드오프(#2)**: `_change_date`는 원시 문자열(`YYYYMMDDHHMMSS`)로 `_norm_order_dt` 정규화를 안 거침 → #2 소비 시 정규화 필요.

## 6. 엣지 케이스

| 상황 | 처리 |
|------|------|
| 기간 안 주문이 나중에 취소(140이 롯데온 배송행을 취소완료로 덮음, 또는 스스 흡수) | `_kind='order'` → 주문일 탭 **남음**(배지 취소완료). |
| 기간 안 롯데온 주문이 발송전 취소(클레임행만, `odAccpDttm` 기간 안) | `_kind='change'` + 실주문일 기간 안 → **남음**. |
| 옛 주문(주문일 기간 밖)이 기간 안 취소 | `_kind='change'` + 주문일 기간 밖/공란 → 주문일 탭 **제외** → #2. |
| 쿠팡·11번가 발송전 취소(실주문일 미제공) | `_kind='change'` + 주문일 공란 → 주문일 탭 **제외** → #2. (§5 한계) |
| 클레임 조회 실패 | 기존 `try/except`로 활성 주문 유지(현행). 태그만 추가라 영향 없음. |

## 7. 테스트 (TDD)

1. **태그·위생**(마켓별 fixture): 주문/배송 행→`_kind='order'`(취소완료여도); 클레임 행→`_kind='change'`+`_change_date` 채움; 롯데온 클레임 `주문일=odAccpDttm`(폴백 없음), 쿠팡 클레임 `주문일=""`.
2. **`new_order_rows`**: (a) order 행 유지, (b) 롯데온 change 실주문일 기간 안 유지·기간 밖 제외, (c) 쿠팡·11번가 change(공란) 제외, (d) 스스 취소(order) 유지. 동일 인자·캐시 경로.
3. **`/orders/preview.json`**: `new_order_rows` 사용 → 응답에 "옛주문 상태변경" churn 없음. KPI 신규주문 = order+기간내 change 옵션 줄 수.
4. **회귀**: `combined_order_rows` 출력 계약 불변(태그 필드 추가만) → 기존 `test_order_export`·`test_market_enrich`·엑셀·마진 테스트 그린.

## 8. 비목표 (Non-goals)

- 상태변경 별도 탭 UI·3단계 워크플로·`status_change_rows` 변경일 수집 → **#2**.
- 마켓별 CS(문의) 탭·발송 API → **#3**.
- 주문일 목록 시각/레이아웃 재디자인(데이터만 교체, UI 구조 유지).
- 쿠팡·11번가 실주문일 역추정(§5 한계 수용).
