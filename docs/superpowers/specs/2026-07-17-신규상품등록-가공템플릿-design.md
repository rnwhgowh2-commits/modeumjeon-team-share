# 신규 상품등록 — 소싱처 크롤 → 마켓 등록 (가공 템플릿) 설계서

> **작성**: 2026-07-17 · **상태**: 설계 확정(사용자 승인) · **다음**: Phase 1 구현 플랜
> **계층**: 스펙 / **상위**: CLAUDE.md(프로젝트) / **관련**: `docs/markets/_새-마켓-추가-가이드.md`, `marketplace_api_map.json`

---

## 1. 목적

소싱처(무신사·SSF·SSG·롯데온·현대H·르무통메이트 등)에서 가져온 상품을 **판매처 마켓(스마트스토어·쿠팡·롯데온·11번가)에 신규 상품으로 등록**한다. 더망고 솔루션과 같은 기능.

**두 진입 경로가 하나의 그릇으로 수렴한다.**

```
(1) 완전 자동 : 소싱처 크롤 ──[가공 템플릿]──┐
                                            ├─▶ ProductDraft(마켓 공통 그릇) ─▶ 마켓별 컴파일러 ─▶ 등록
(2) 수기 등록 : 사람이 직접 입력 ───────────┘
```

그릇은 하나. **크롤이 자동으로 채우느냐(1), 사람이 채우느냐(2)** 의 차이일 뿐. 마켓이 늘어도 컴파일러만 추가한다.

## 2. 현재 코드 현실 (조사 결과)

| 마켓 | 가격·재고 수정 | 신규 상품등록 |
|---|---|---|
| 스마트스토어 | 있음 | **있음** — `shared/platforms/smartstore/create_product.py` (라이브 검증 2026-04-22) |
| 쿠팡 | 있음 | **있음** — `shared/platforms/coupang/products.py::create_product` |
| 롯데온 | 있음(`_wt_auto_engine`) | **없음** — API 문서만 (`lotteon.product.create`) |
| 11번가 | 스텁 | **없음** |

- 도메인 래퍼: `lemouton/registration/{smartstore,coupang}.py` · 라우트 `POST /bundles/<code>/register/<market>` (스스·쿠팡만 허용)
- 하위 `create_product`는 **Model(bundle) 없이 호출 가능** → ProductDraft에서 직접 컴파일 가능. **기존 코드 재사용.**

**크롤이 지금 주는 것**: 상품명 · 브랜드 · 색상/사이즈 옵션 · 가격 · 재고 · 동적혜택 (`CrawlResult`)
**크롤이 안 주는 것**: 이미지 · 상세페이지 · 카테고리 · 상품고시정보 · 재질/원산지
→ `crawlers/base.py`가 이미지 다운로드를 의도적으로 차단 중(`_BLOCK_RESOURCE_TYPES`). Phase 3에서 조건부 해제 필요.

## 3. 아키텍처 — 3개의 축

설정은 **세 축**으로 결정되며, 아래로 갈수록 우선한다.

```
1) 세트 (소싱처 × 마켓 조합)   ← 기본 규칙 묶음
2) 마켓별 설정                 ← 마진·수수료·이미지 장수·카테고리
3) 브랜드별 예외 (override)     ← 최우선. 이미지 정책·상세 후크 이미지
```

**우선순위: 브랜드 예외 > 마켓 설정 > 세트 기본값.**
브랜드 축은 사용자 요구(모델 지재권 브랜드는 이미지 제외 / 브랜드별 후크 이미지)로 도입됐다.

## 3-1. 화면 위치 — 「대량등록」 3번째 모드 신설 ★

이 기능은 **모음전·재고관리 어느 쪽도 아니다.** 사이드바 최상단 모드 전환에 **「대량등록」을 3번째 모드로 신설**한다(사용자 확정).

**현재 구조(조사 결과)**
- 모드 전환 = `partials/sidebar.html`의 `.sb-modeswitch` — `/`(모음전) · `/inventory/`(재고관리) 2개 하드코딩
- 판별 = `active_app` 컨텍스트 변수. `inventory/__init__.py::inject_active_app()`가 `/inventory/*`에 `active_app='inventory'` 주입
- `base.html`이 `active_app`으로 **사이드바를 통째 교체** (`sidebar_inventory.html` vs `sidebar.html`)
- 모드 아이콘 = `webapp/routes/__init__.py::_MODE_DEFAULTS = {'bundles':'📦','inventory':'🏷'}` (icon_store로 편집 가능)

**추가 작업**
1. 새 블루프린트 `webapp/routes/bulk/` (`/bulk` prefix) + `context_processor` → `active_app='bulk'`
2. `_MODE_DEFAULTS`에 `'bulk'` 추가 (기본 아이콘)
3. `base.html` 사이드바 분기에 `bulk` 추가 → `bulk/partials/sidebar_bulk.html` 신설
4. `toss.css`의 `.sb-modeswitch { grid-template-columns: 1fr 1fr }` → **3칸**으로

**⚠️ 반드시 고칠 것 (기존 결함)**
- **모음전 카드 조건이 `active_app != 'inventory'`** — "재고관리가 아니면 모음전 켜짐". 대량등록에 들어가도 **모음전이 켜진 것처럼 보인다.** → `active_app`이 `inventory`·`bulk` 둘 다 아닐 때만 켜지도록 수정.
- **모드 전환 카드가 사이드바마다 복제**(`partials/sidebar.html` + `inventory/partials/sidebar_inventory.html`). 3번째가 생기면 3벌 → **`partials/_modeswitch.html` 공통 조각으로 추출**해 세 사이드바가 include. (이번 작업에 필요한 범위의 정리)

**재사용 (새로 만들지 않음)** — 사용자 지시: "소싱처관리·판매처관리 등은 기존 프로그램 기조를 전부 따라간다"
- 소싱처 운영센터(`/sources`) · 판매처 계정 관리(`/accounts`) · 크롤 파이프라인 · 가격 템플릿(`PriceTemplate`) · fx 영수증 · 아이콘/사이드바 규약 · toss 디자인 토큰

## 3-2. 대량등록 모드 — 7탭 구성 ★

### ⚠️ 조사 결과: 요청 탭 대부분이 이미 존재한다 (중복 제작 금지)

라이브 `origin/main`의 `/orders` **SUBTABS**(모음전 「판매」 그룹):

| 기존 탭 | 내용 |
|---|---|
| `list` 📋 주문 내역 | 마켓별 주문 통합 조회 + 송장 입력 |
| `inspect` 🚚 배송검사 | 더망고 업로드·중복송장·배송흐름·배송방식 |
| `cs` 💬 CS | 취소·반품·교환 + 고객문의 조회·처리 |
| `register` 🆕 신규 상품 등록 | **"모음전 상품을 마켓에 신규 등록"** |
| `margin` 🧮 마진 계산기 | 가격·수수료·배송비 → 실마진 시뮬 |

(※ `sales` 정산·매출 탭은 2026-07-16 사용자 요청으로 삭제됨)
※ 이 조사는 반드시 **`origin/main` 기준**으로 할 것. 작업 브랜치 `feature/crawl-worker-queue`는 73커밋 이상 뒤처져 옛 파일을 보여준다.

### 탭 구성과 재사용 매핑

| # | 탭 | 신규/재사용 |
|---|---|---|
| 1 | **데이터수집** (소싱처 상품 수집) | 신규 화면 · 기존 크롤 파이프라인 재사용 |
| 2 | **데이터가공** (가공 템플릿 §7) | 신규 |
| 3 | **데이터전송** (판매처 업로드) | 신규 화면 · 기존 업로더/어댑터 재사용 |
| 4 | **상품관리** (수집상품 저장·가공·전송 현황) | 신규 |
| 5 | **주문관리** | **기존 `/orders` list·inspect 재사용** + 대량등록 관점 보강 |
| 6 | **CS관리** | **기존 `/orders` cs 재사용** |
| 7 | **통계** | **기존 `margin` 마진계산기 이관** |
| 8 | **설정** (기준값) | 신규 탭 · **화면은 한 벌만 만들고 모음전 쪽에서도 같은 화면 마운트**(수기등록과 같은 패턴). 카드할인·캐시백·네이버페이·마켓수수료율·까대기·가공템플릿 |

### 확정 원칙 (사용자 확정)

- **주문·CS·마진계산기 = 데이터는 한 곳에만.** 주문은 모음전으로 올렸든 대량등록으로 올렸든 **같은 마켓에서 들어오므로** 저장소를 나누지 않는다. 대신 **「등록경로」(모음전 / 대량등록) 필터**로 구분해 양쪽 모드에서 본다.
  → `ProductDraft.origin`(모음전|대량등록) 보관 → 마켓 상품ID로 주문에 조인해 필터.
  → **화면·데이터 2벌 금지** (본 프로젝트 최상위 원칙: 중복·모순 금지).
- **수기 등록 = 기존 `/orders?tab=register` 「신규 상품 등록」 탭을 확장·공유.**
  - 확장: "모음전 상품만" → **아무 상품이나**(ProductDraft 직접 입력)
  - 공유: **모음전 모드·대량등록 모드 둘 다 같은 화면을 마운트**(한 벌만 관리). 기존 인라인 마운트 패턴 재사용.

### 4) 상품관리 탭 — 더망고에서 가져올 것 (그대로 아님, 더 낫게)
- **상품별 「상품/가격/재고 업데이트 ON·OFF」 토글** (원문사이트 변동을 상품 단위로 끊을 수 있음)
- 원문 상품명 병기 · **대표가격 옆에 원가 표시** · 마켓전송가격 · 카테고리 매핑정보 · 적용정책 · 가격재고 변동이력 · 옵션 목록
- 우리 강점 추가: **fx 영수증**(§5) · 까대기 반영가 · 소싱처 실재고 3상태

### 5) 주문관리 탭 — 재설계 (사용자와 공동 설계 예정)
핵심(더망고 대비 우위): 주문 한 줄에 **`[마켓|계정] 주문번호` + `[소싱처] 상품명 · 소싱처가 → 변동가(가격변동 표시)` + 결제금액(정산예정) + 주문상태 + 송장전송**을 함께 → **매입가와 판매가가 한 줄에 보여 실마진이 즉시 보인다.**
필수: 소싱처 크롤 정보 · 판매처(마켓·계정) 정보 · **마진 계산내역(fx 영수증·까대기 포함)** · 주문상태 · 송장번호 전송 · **강력한 필터**.
→ 세부 화면·필터 항목은 **별도 공동 설계 라운드**에서 확정한다(사용자 지시).

## 4. 데이터 모델 (신규)

- **`ProductDraft`** — 마켓 공통 상품 1건. 상품명·판매가·브랜드·옵션[]·이미지[]·상세HTML·고시정보·배송·판매방식·인증·소스(크롤/수기)·상태 · **`origin`(모음전 | 대량등록) ← 주문·CS 「등록경로」 필터의 근거** · **업데이트 ON/OFF(상품·가격·재고 각각)**
- **`ProductDraftMarket`** — 드래프트 × 마켓. 마켓별 카테고리코드·판매가·등록결과(성공/실패·마켓상품ID·에러)
- **`TemplateSet`** — 가공 템플릿 세트 (소싱처×마켓 바인딩, 기본세트 플래그)
- **`TemplateRule`** — 세트 안 항목별 규칙 (13항목 JSON)
- **`BrandPolicy`** — 브랜드별 예외 (이미지 정책·후크 이미지)
- **`CategoryMap`** — 소싱처 카테고리 → 마켓 카테고리 학습표 (확정 시 축적)
- **`NameReplaceRule`** — 상품명 치환 규칙 (from·to·대소문자구분·엑셀 업/다운)
- **`BannedWord`** — 금지어 (분류: `collect`=수집 자체 금지 / `upload`=마켓별 업로드 금지)
- **`MarginTier`** — 금액 구간별 마진 (구간 추가/삭제/수정)
- **`ExtraCostTemplate`** — 추가 비용 템플릿 (까대기 등). 이름 · **금액(절대값)** · 사용여부
- **`ExtraCostTarget`** — 템플릿 × 적용 대상. `계정`(필수) + `브랜드`(선택) + `상품`(선택). 브랜드·상품이 비면 그 계정 전체
- **`CashbackSite`** — 캐시백 사이트 × 적립률 (신규, §5-1 표 이관)
- **`PayMethodRate`** — 네이버페이 등 결제수단 적립률 (신규, 네이버페이 1%)
- **`MarketFeeRate`** — 마켓별 수수료율 (신규 · 정산 API 자동 확인 병행)
- **카드 할인율** — **신규 아님. 기존 `CardDiscountUserPref` 사용**(`api_benefits.py`)
- **`OrderCostSnapshot`** — ★ 주문 시점의 기준값 박제(카드할인율·캐시백률·수수료율·쿠폰·환율). **기준값이 바뀌어도 과거 주문 마진은 불변**(사용자 확정)

## 5. 가격·마진 계산 명세 ⚠️ 금전 직결

**확정 규칙: 마진은 「판매가 대비」. 마켓 수수료도 「최종 판매가 대비」. 매입가 대비 아님. 부가세는 계산에서 무시(사용자 확정).**

```
실마진  = 판매가 − (판매가 × 수수료율) − 배송비 − 추가비용 − 최종매입가
마진율  = 실마진 ÷ 판매가

[역산] 목표 마진율 m
판매가 = (최종매입가 + 배송비 + 추가비용) ÷ (1 − 수수료율 − m)
```

- **배송비 조건부**: 무료배송 = 판매자 부담 → 뺀다. 유료배송(구매자 부담) = **빼지 않는다.**
- **추가비용** = 까대기 등 §7-14 템플릿 합산액. 원가와 같은 자리에 더해져 **판매가가 오르고 마진율은 유지**된다(사용자 확정).
- **끝자리**: 100원 올림
- **정가(할인 전)**: 판매가 × 1.2 기본, 사용자 수정 가능
- **안전장치**: 하한·상한 벗어나면 등록 보류

**검산 (구현 시 테스트로 고정)** — 매입가 50,000 / 배송비 3,000 / 수수료 10% / 목표마진 20%
→ 판매가 = 53,000 ÷ 0.70 = 75,714 → 100원 올림 = **75,800**
→ 수수료 7,580 · 실마진 = 75,800 − 7,580 − 3,000 − 50,000 = **15,220 = 판매가의 20.1%** ✅

**검산 2 — 까대기 3,000원 추가 시** (위와 동일 조건 + 추가비용 3,000)
→ 판매가 = (50,000 + 3,000 + 3,000) ÷ 0.70 = **80,000**
→ 수수료 8,000 · 실마진 = 80,000 − 8,000 − 3,000 − 3,000 − 50,000 = **16,000 = 판매가의 20.0%** ✅
→ 까대기 3,000원 때문에 판매가가 4,200원 오르고 **마진율 20%는 유지**된다.

**마진 방식 4종(마켓별 선택)**: 퍼센트 마진 / 남길 금액 고정 / 최소 남길 금액 보장 / 목표 판매가 고정
**금액 구간별 마진**: 매입가 구간마다 다른 마진율·마진금액. 구간 추가·삭제·수정 가능.
**옵션별 추가금**: 소싱처 옵션가 차이 반영.
**fx 영수증**: 최종매입가 → 마켓별 판매가까지 **계산 수식을 영수증식으로 표시** (기존 fx 위젯 패턴 재사용). **까대기 등 추가비용은 「까대기 +3,000」처럼 별도 줄로** 노출. 참고로 부가세 반영 실이익도 함께 표기.

## 5-1. 최종매입가(소싱처에서 실제 나가는 돈) — 사용자 엑셀 로직 반영 ★

사용자가 실제로 쓰던 구글시트(`주문관리 은순님`)의 **수식을 원문으로 해독**해 반영한다. (2026-07-17 xlsx 다운로드 후 openpyxl 로 수식 확인)

**시트 원본 수식**
```
E 정산예정금액 = 내림( D × (1 − 마켓수수료율[C 룩업]) )
L 최종매입가격 = 반올림(
      K × (AM=1 이면 1 − 카드할인율[H 룩업], 아니면 1)
      − 내림( K × 네이버페이율[F 룩업] )
      − ( K × 캐시백률[G 룩업] )
  ) + IF(주소에 "역골동로" OR 소싱처="abc", 2800, 0)
M 마진금액 = E − L
AD 마진율  = M ÷ D × 100
```

**해독 결과 / 확정 사항**
- `D` = 사용자가 **더망고 솔루션의 예상 정산가를 손으로 입력**한 값 → 그래서 마켓수수료율 표가 전부 0인 것은 **버그가 아니라 이중 차감 방지(의도)**.
  → **우리는 마켓 정산 API로 이 손입력을 자동화한다** (롯데온 정산 오차 0 검증 완료 · 스마트스토어 배선됨). 이것이 시트 대비 핵심 우위.
- **카드할인의 `AM=1`(할부 1개월) 조건은 제거**(사용자 확정) → 일시불도 카드할인 적용.
- **네이버페이는 1%**(시트의 0.5%는 오기 — 사용자 확인).
- 시트의 `+2,800`(주소 문자열 하드코딩)은 **§7-14 까대기 템플릿으로 대체**(계정+브랜드/상품 지정).
- 🔴 **시트 버그: `AD3 = M4/D4`** (3행이 4행을 참조 → 첫 줄 마진율이 항상 틀림). 우리 구현에는 옮기지 말 것.

**우리 계산식 (확정)**
```
최종매입가 = 반올림( 표면매입가 × (1 − 카드할인율)
                    − 내림(표면매입가 × 네이버페이율)
                    − (표면매입가 × 캐시백률)
                    − 소싱처 쿠폰·적립(크롤값) )
```
그리고 §5 의 `추가비용`(까대기)은 판매가 역산 쪽에 들어간다(원가에 가산).

**기준값 표 (사용자 시트에서 이관 — §3-2 ⑧설정 탭에서 관리)**
- **카드 할인율 17종**: 넥슨현대카드 2.7% · 롯데프로페셔널 2% · 삼성셀렉트 1% · BC바로 1% · 케이뱅크 1.1% · 나머지 0%
  → **`CardDiscountUserPref` 가 이미 존재**(`api_benefits.py`). 값만 채우면 됨.
- **캐시백 사이트 8곳**: 더현대 3.0% · H몰 2.7% · 롯데홈쇼핑 2.5% · SSG 2.0% · GS샵 1.6% · CJ 1.5% · 롯데백화점 1.1% · 롯데온 1.1% · 해당없음 0%  → **신규**
- **네이버페이 1%** → 신규
- **마켓별 수수료율** → 신규(정산 API 자동 확인 병행)

### 「어디서 살까」 비교 — 같은 상품, 소싱처·카드·캐시백 조합별 최종매입가
사용자는 주문 건마다 **그때그때 더 싼 곳**에서 산다(등록된 소싱처가 아닐 수도 있음). 조합을 자동 비교한다.
- 표면가만으로 판단하면 틀린다(예: SSG 51,900 vs 더현대 52,000 → 캐시백 3% 덕에 **더현대가 426원 더 남음**).
- **크롤 안 되는 소싱처는 「직접 추가」**(이름+가격+캐시백%) 입력 → 즉시 계산.
- 고르면 그 값이 **§7-14 구매내역 입력칸에 자동 반영**.

### ⚠️ 정직성 원칙 (프로젝트 최상위 규칙 적용)
- **같은 소싱처라도 상품마다 쿠폰·청구할인이 다르다.** 고정 공식 금지.
- 크롤이 잡은 혜택 = 체크박스로 표시(켜고 끄면 즉시 재계산) + **크롤 시각 표기**(쿠폰은 하루면 바뀜).
- **크롤이 못 잡는 것(SSG·롯데아이몰 청구할인 = 결제금액만 노출)은 추측 금지.** `?` 로 두고 **마진을 범위로 표시**(예: `22,238 ~ 23,639`), **판단은 항상 낮은 쪽**으로. → 폴백 금지 원칙과 동일.
- **실제 결제금액 되먹임**: 구매내역에 입력한 실결제금액 ↔ 예상값을 대조해 소싱처×카드 조합별 오차를 누적, 다음번 **제안**(자동 확정 금지 — 반드시 사용자에게 물음).

### 🔒 기준값 변경 시 과거 주문 (사용자 확정)
**과거는 그대로, 앞으로만 적용.** → **주문 시점의 기준값(카드할인율·캐시백률·수수료율·환율 등)을 그 주문에 스냅샷으로 박아둔다.**
설정을 바꿔도 지난달 장부가 변하면 안 된다.
⚠️ 과거 이 프로젝트에서 **혜택 스냅샷 모델을 시도했다가 마이그레이션 롤백**된 이력이 있음([[project_benefit_snapshot_model]]) — 이번엔 주문 단위로 확실히 박제.

- 세트 단위 = **소싱처 × 마켓 조합마다** · 미지정 조합은 기본 세트 · (+ §3 브랜드 예외가 최우선)

### 13) 등록 마무리
- **등록 상태: 초기엔 초안 → 가격·카테고리 검증되면 스위치로 「바로 판매중」 전환** (사용자 확정)
- 등록 후 재고·가격 자동 동기화 연결
- 자체 상품코드 = **소싱처 품번 사용**
- **중복 등록 방지**: 동일 계정 재등록 차단 · **타 계정은 별도 설정으로 허용** · 소싱처 동일 URL은 **1개만 관리**(두 번 크롤하지 않음)

### 14) 추가 비용 템플릿 (까대기) — 신설
기본은 **위탁판매**(소싱처가 고객에게 직접 발송, 추가비용 0). 그러나 **우리 사무실로 받아 재포장 후 발송**해야 하는 경우가 있고, 그 비용을 가격에 반영해야 한다.

- **템플릿 = 이름 + 절대 금액**. 사용자가 직접 입력(예: 까대기 3,000원 / 4,000원). **여러 개 생성 가능**(까대기·검수·라벨링 등)
- **적용 대상 = 계정 + 브랜드/상품별 지정**(사용자 확정)
  - `계정`만 지정 → 그 계정에 올리는 상품 **전부**
  - `계정 + 브랜드` → 그 계정의 그 브랜드만
  - `계정 + 상품` → 그 상품만
- **여러 템플릿이 겹치면 합산** (까대기 3,000 + 라벨 1,000 = 4,000). 한 템플릿 안에서 여러 규칙이 맞아도 **한 번만** 적용(중복 부과 방지)
- **가격 반영**: §5 식의 `추가비용` 자리 → **판매가에 얹혀 마진율 유지**
- fx 영수증에 별도 줄로 표시

## 9. 미결·리스크

- ⚠️ **지재권**: 소싱처의 대표·추가이미지도 브랜드 저작물. 이미지 재조합 방식이 100% 안전하진 않음 — 사업 판단으로 진행하되 브랜드별 예외로 위험 브랜드 회피.
- ⚠️ **스스 이미지**: 아무 URL이나 안 받고 **네이버 이미지 서버 업로드 후 CDN URL만** 허용 → 업로드 단계 필요. 쿠팡은 공개 URL(R2) 가능.
- ⚠️ **고시정보 하드코딩**: 현재 `_build_default_shoes_notice()` 신발 고정 → 4종류 편집 가능하게 분리 필요.
- ⚠️ **쿠팡 자동매칭**: WING에서 사전 **동의** 필요(동의 여부 확인 API 있음).
- ⚠️ **부가세**: 계산에서 제외하기로 확정 → 실제 손에 남는 돈은 표시 마진보다 약 10% 적음. fx 영수증에 참고 표기.
- **마켓 수수료율**: 세트의 마켓 설정에서 **사용자가 직접 입력**(마켓·카테고리마다 다르므로). 기본값은 제공하되 사용자 값이 우선.
- 미확정(구현 중 확보): 공휴일 달력 출처 · 마켓별 이미지 허용 장수 실제 수치.
