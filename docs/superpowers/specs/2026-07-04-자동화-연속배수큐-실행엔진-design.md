# 자동화 실행 엔진 — 연속 배수 큐 (Continuous Weighted Queue) 설계

> **작성일**: 2026-07-04
> **상태**: 설계 확정 (구현 전)
> **관계**: 프로젝트 도메인 설계 · 상위 = CLAUDE.md 데이터 정합성 3대 원칙 · [`docs/크롤링-가이드.md`](../../크롤링-가이드.md) 정본
> **선행 사실**: `/automation` 화면(자동화 설정)은 **저장만 되고 실행 엔진 미구현**. 별도 서버 스케줄러는 gunicorn(`create_app()`) 기동이라 애초에 안 뜸. 본 설계 = **그 실행 엔진을 붙이는 것.**

---

## 1. 목적 (한 줄)

자동화 설정 화면의 「크롤 자동 주기」·「자동 전송」이 **실제로 동작**하게 하는 실행 엔진을 만든다. 판매처 상품이 1,000개로 늘어도 소싱처를 막히지 않게, PC 한 대에서 최대치로 돌리고, 미래(팀 PC·외부 서버)로 일꾼만 늘릴 수 있는 구조로.

## 2. 배경 · 핵심 오해 정정

- **크롤 단위 = 판매처 상품(모음전)이 아니라 소싱처 URL.** 여러 모음전·옵션이 같은 URL을 공유 → **중복 제거한 유일 URL 목록**이 실제 작업 대상. (근거: `lemouton/sources/service.py::fetch_unique_sources`)
  - 예: 판매처 상품 1,000개 → 실제 크롤 URL은 그보다 훨씬 적음.
- 그래서 "상품1→상품2→상품3" 순차가 아니라 **유일 URL 큐**를 돈다.
- **한 번에 다 긁으면 소싱처가 막음** → 소싱처별 동시 상한 + 사이트끼리 병렬.

## 3. 확정된 사용자 결정 (본 설계의 근거)

| 갈림길 | 결정 |
|---|---|
| A. 주기 방식 | **오래된 것부터 연속** + **URL별 계수(1~5, 기본 1)**. 계수 2 = 남들 한 바퀴에 두 바퀴 |
| B. 실행 위치 | 지금 **PC 1대 최대치**. 큐↔일꾼 분리로 **팀 PC·외부 서버 워커** 미래 확장 (구조만 미리) |
| C. 업로드 제한 | **마켓별 속도 제한**, 사용자가 마켓마다 값 조절 (쿠팡·스마트스토어 각각) |
| 크롤↔업로드 | **분리하되 이어짐** — 크롤 끝나 변동 잡히면 그 URL만 업로드 큐로 |
| 화면 | **4번 시안(카운트다운 링 + 실시간 로그)**. 연속 방식이라 **링 = "이번 한 바퀴 진행률"**, 로그 = 지금 긁는 URL |

## 4. 전체 흐름

```
[실행/정지]  (사용자, 로컬)
   │
   ▼
① 연속 큐 ── 오래된 순 + 계수(1~5)로 "지금 갱신할 URL" 선정        ← 로컬 PC
   │                                             ▲
   ▼                                             │ 크롤 끝 → 마지막크롤시각 기록 → 다시 줄서기 (연속 순환)
② 소싱처별 병렬 크롤 (사이트별 동시 상한, 놀지 않는 큐)          ← 로컬 PC (일꾼 1대 → 확장)
   │
   ▼
③ 서버 저장 → URL별 변동 기록(재고/가격) → 변동 URL만 추림       ← 서버
   │                                └ 무변동 URL = 다음 주기 완화
   ▼
④ 마켓별 업로드 큐 (속도 제한, 사용자 조절)                      ← 서버
   │
   ▼
[판매처]  쿠팡 · 스마트스토어 …
```

> **원칙 경계**: ①② = 로컬 PC(크롤), ③④ = 서버(업로드). "크롤=로컬 / 업로드=서버" 절대 원칙 유지.

## 5. 구성 요소

### 5.1 데이터 (새로 기억할 것)

- **SourceProduct(소싱처 URL 행)에 추가**
  - `crawl_weight` INT (1~5, 기본 1) — 계수
  - `last_crawled_at` DATETIME — 마지막 크롤 시각 (오래된 순 판단). **현재 이 값이 갱신 안 되는 알려진 구멍이 있어, 크롤 성공 시 반드시 채운다.**
  - (파생) `next_due_at` = `last_crawled_at + (기본주기 ÷ crawl_weight)` — 저장하거나 조회 시 계산
  - `no_change_streak` INT (기본 0) — 연속 무변동 횟수 (자동 완화 근거)
- **GlobalSettings (기존 자동화 필드 재사용)**
  - `crawl_interval_hours`·`crawl_interval_minutes` = **기본 주기**(계수 1 기준)
  - `crawl_auto_enabled` = 실행/정지 상태
- **마켓별 업로드 정책 (신규)** — `MarketUploadPolicy(market, per_minute, enabled)` 또는 GlobalSettings 내 JSON. 쿠팡·스마트스토어별 분당 허용 개수. 사용자 조절.
- **URL별 크롤 변동 기록 (신규)** — `CrawlDelta(source_product_id, crawled_at, stock_changed BOOL, price_changed BOOL, detail)` — 직전 스냅샷 대비 무엇이 얼마→얼마로. **판매처 쪽 `ChannelChangeEvent`(전송 시점)과는 다른 층(크롤 시점).**

### 5.2 연속 큐 (순서 두뇌) — 로컬

- **선정 규칙**: `now >= next_due_at` 인 URL 중 **가장 오래 밀린 순**으로 꺼냄.
  - `next_due_at = last_crawled_at + (기본주기 ÷ crawl_weight)` → 계수 높을수록 자주 후보에 오름.
  - 한 번도 안 긁은 URL(`last_crawled_at` 없음)은 최우선.
- **소싱처별 동시 상한**: 사이트마다 상한(예 무신사 3 / SSF 2 …). 서로 다른 사이트는 병렬.
- **놀지 않음**: URL 1개 끝나는 즉시 다음 후보를 당김 → 일꾼이 항상 참.
- **기존 뼈대 활용**: `lemouton/sourcing/crawl_queue.py`의 `CrawlJob`/`CrawlWorker`(claim·lease·`reap_expired_jobs`·priority·attempts)를 URL 단위 선정과 연결. (현행 `enqueue_crawl`은 번들 단위 → URL 단위 due-선정 함수를 추가)

### 5.3 크롤 워커 (일꾼) — 로컬, "PC 한 대 최대치"

1. **놀지 않는 큐** — 대기 0, 끝나면 즉시 다음.
2. **소싱처 병렬 + 사이트별 상한** — 빠르되 안 막힘.
3. **로그인 세션 재사용** — 매번 로그인 X.
4. **실패 격리 + 재시도 백오프** — 한 URL 막혀도 전체 안 멈춤. 실패는 잠시 뒤 재시도. 확정 실패 = 해당 URL "확인 불가"(폴백 금지).
5. **일꾼↔큐 분리 (확장 이음새)** — 지금 워커 1개(내 PC). 미래: 팀 PC·외부 서버를 워커로 등록(`CrawlWorker`)하면 큐는 그대로, 일꾼만 증가. 한 대가 죽으면 `reap_expired_jobs`가 회수해 다른 일꾼이 이어받음.

### 5.4 서버 저장 · URL별 변동 기록 · 자동 완화

- 크롤 결과 저장 시 **직전 값과 비교** → `CrawlDelta` 기록 (재고 변동/가격 변동/상세).
- **변동 있는 URL만** 업로드 큐로 넘김 (④). 무변동은 안 넘김 = 업로드 시간·마켓 부하 절약.
- **무변동 연속 URL 자동 완화**: `no_change_streak`가 쌓이면 다음 `next_due_at`을 뒤로 미룸(총 크롤량 절약). 변동 발생 시 streak 리셋. (사용자 계수를 덮지 않는 선에서 보정)

> **크롤 시간 절약의 진실**: 변동 여부는 긁어봐야 알므로 **크롤 자체를 건너뛰어 시간을 줄이진 않는다**(안 긁으면 "확인 불가"). 절약은 **①업로드 스킵 + ②무변동 URL 자동 완화**에서 나온다. 소싱처가 가벼운 재고/가격 신호를 별도 제공하는 경우에 한해 "선(先) 경검사 후 무거운 파싱 생략"은 **신뢰 가능한 소싱처만** 선택적으로(원칙상 확장 신중).

### 5.5 업로드 (마켓별 속도 제한) — 서버

- 변동 URL → 마켓별 업로드 큐. **마켓마다 사용자 설정 속도(분당 N개)**로 천천히 배출.
- 오른쪽 「자동 전송」 카드 규칙 존중: `autosend_mode`(미리보기만/실제전송), 어떤 변동이면 보낼지(`autosend_on_stock/price/purchase`), `autosend_stock_threshold`.
- 마켓이 막으면 속도를 낮춰 재시도(사용자가 값 하향).

### 5.6 화면 반영 (4번 시안)

- **왼쪽 카드** — 「실행/정지」 = 연속 큐 켜기/끄기.
  - **링** = "이번 한 바퀴 진행률(전체 URL 중 갱신 %)" (연속이라 "다음 실행 시각"은 없음)
  - **로그** = 지금 긁는 URL 실시간 흐름
  - 정지 시 회색·멈춤, 실행 시 초록·되살아남 (앞서 확정한 추천값: 실행=초록/정지=회색, 실행중 카드 전체 초록)
- **계수(1~5) 설정 위치** — 소싱처 URL 목록(매트릭스)의 각 URL 줄.
- **마켓 속도 설정 위치** — 오른쪽 「자동 전송」 카드 안, 마켓별 칸.
- **변동 기록 표시** — 「자동화 로그기록」에 "이번 전체 크롤 변동 URL 목록"(재고/가격 변동 여부).

## 6. 원칙 준수 (절대 규칙)

- 크롤=로컬 / 업로드=서버 경계 유지. "서버 크롤 부활" 아님.
- 재고·가격 기준 = 실제 소싱처 상품 URL 데이터. 폴백가 금지. 확인 못 하면 "확인 불가".
- 데이터 무결성: 중복·모순 금지.

## 7. 테스트 (TDD, 서버측)

- 계수→주기 계산: `next_due_at = last_crawled_at + (기본주기 ÷ crawl_weight)` (계수 1/2/5 경계)
- 큐 선정 순서: 오래 밀린 순 + due 안 된 것 제외 + 미크롤 최우선
- 소싱처별 동시 상한 준수
- URL별 변동 기록: 재고만/가격만/둘 다/무변동 4케이스
- 무변동 자동 완화: streak 누적 → next_due 미룸, 변동 시 리셋
- 마켓별 업로드 속도 제한: 분당 N개 초과 배출 금지
- 변동 없는 URL은 업로드 큐로 안 감

## 8. 비목표 (YAGNI)

- 외부 서버 워커 **실제 배포**는 이번 범위 아님 — **구조(이음새)만** 준비.
- 팀 PC 다중 워커 실사용도 이번 범위 아님 — `CrawlWorker` 뼈대 유지·연결까지.
- 소싱처별 "가벼운 신호 선검사"는 이번 범위 아님(신뢰 소싱처 확인 후 후속).

## 9. 열린 항목

- 마켓별 속도 기본값(쿠팡·스마트스토어 초기 분당 개수) — 겪어보며 사용자 튜닝.
- 무변동 자동 완화의 상한(계수를 얼마나 덮지 않게 보정할지) — 구현 중 수치 확정.
- 계수 설정 UI의 매트릭스 배치 세부 — 구현 플랜에서 design-mockup 게이트로.
