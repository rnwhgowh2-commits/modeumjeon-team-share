# 판매처 데이터·코드 지도 — 단일 원천화 + 전수정독 방법 (설계)

> **계층**: 프로젝트 스펙 (도메인 지식 SOT 보강)
> **책임**: (1부) 판매처 API 지도 데이터의 진실원천을 하나로 통합, (2부) 마켓 개발 시 지도를 전수정독하는 방법(스킬+인앱 덤프) 설계
> **상위 의존**: 프로젝트 CLAUDE.md(데이터 정합성 3대 원칙) · `docs/markets/_API문서수집법.md`(문서 접수 플레이북)
> **최우선 제약**: **현재 검증된 마켓 동작(보내기/받기/쓰기/읽기) 절대 회귀 금지.** 탭 단위로 전/후 대조하며 진행.
> **작성일**: 2026-07-21

---

## 1. 배경 — 왜 필요한가

- 판매처(마켓) 관련 기능을 개발할 때, 인앱 「데이터 코드 지도」(`/marketplace-guide/map`) 팝업에 필요한 정보가 사실상 다 있는데도 **각 탭을 전수조사하지 않아** Claude가 계속 헤맨다(삥삥 돎). F12 반복 노가다, 이미 문서화된 정보를 재발견, 방향 오락가락.
- 근본 원인 2가지 (코드 실측으로 확인):
  1. **정보 산재**: 팝업은 렌더러일 뿐이고 정보가 여러 곳에 흩어져 있다.
  2. **진실원천 분열(카탈로그 2개)**: API 카탈로그가 **JSON(610개)** 와 **화면코드 인라인(약 446개)** 로 나뉘어 공존하며, 크기가 달라 **이미 어긋나 있다**.

## 2. 현황 — 코드 실측 (2026-07-21)

대상 파일(최신 코드는 origin/main 워크트리; 예: `_wt_reg`):
- `프로그램/_시스템/webapp/templates/marketplace_guide/map.html` — 팝업 렌더 템플릿
- `프로그램/_시스템/webapp/data/marketplace_api_map.json` — API 카탈로그 (610개, 단일 원천으로 지향)
- `프로그램/_시스템/webapp/marketplace_api_map.py` — 검증기(완성 게이트: st∈{ok,code} → req·res·fields·success 필수)
- 서빙: `/marketplace-guide/map-data.json` · 화면: `/marketplace-guide/map`
- `docs/markets/{coupang,smartstore,eleven11,auction,gmarket,lotteon}.yaml` — 어댑터 코드 근거
- `docs/markets/_API문서수집법.md` — 문서 접수 경로 플레이북

### 2.1 팝업 탭 구조

- 1행: `📚 마켓별 전체 API` · `📘 API 문서 수집법`
- 2행: `상품 조회` · `상품 관리` · `💰 정산` · `배송·송장` · `클레임·CS` · `⚡ 주문상태` · `🗂️ 과거이력`
- `📚 마켓별 전체 API` 하위: 마켓 가로탭(쿠팡·롯데온·옥션·G마켓·스스·11번가) × 보내기/받기 × 상태색(씀/검증대기/안씀) × 코드
- `⭐ 주요 데이터`(별표 담아 마켓 비교)

### 2.2 어느 탭이 어느 소스를 읽나 (★핵심 진단)

| 탭 | 읽는 소스 | 위치(코드) | 상태 |
|---|---|---|---|
| 마켓별 전체 API | `MKT_CATS_D` ← **JSON** `DATA.apis` | map.html:272,1298 | ✅ 좋은 원천(610) |
| 업무별 탭(상품조회·정산 등) | **JSON** `DATA.apis.filter(a.tabs∋tab)` | map.html:1111 | ✅ 같은 JSON |
| 주요 데이터(별표 비교) | `MKT_CATS`/`ESM_CATS` ← **인라인 하드코딩** | map.html:419,933(`dmFind`) | ⚠️ 낡은 별도 원천(≈446) |
| 정산 계산식 | 인라인(정산 필드코드 배열) | map.html:1094~ | JSON에 없는 보조 데이터 |
| 주문상태 전환 / 문서수집법 | 인라인(`DM_MARKETS`·`API_HOW`·`API_CALLS`·`TRANS`) | map.html:295,955,1044,1059 | JSON에 없는 보조 데이터 |

**실측 개수**: JSON 610(옥션117·G마켓117·스스129·롯데온115·쿠팡108·11번가24) vs 인라인 `MKT_CATS`≈365 + `ESM_CATS`≈81. → **크기 불일치 = 비동기 상태 확정.**

**진단 결론**: 메인 카탈로그는 이미 JSON으로 이관됨(마켓별전체·업무탭). 그러나 **주요 데이터(별표) 탭이 옛 인라인 카탈로그에 남아 있어** 두 원천이 공존·분열. 정산·주문상태·문서수집법은 "중복"이 아니라 JSON에 아직 없는 **보조 데이터**.

### 2.3 회귀 안전에 대한 핵심 사실 (사용자 최우선 관심사)

- 이 팝업은 **문서·지도(읽기 전용 표시물)** 이다. **실제 마켓 업로드/주문/정산 호출 코드**(`shared/platforms/*`, `lemouton/uploader/*`)와 **물리적으로 분리**되어 있다.
- 따라서 1부(팝업 데이터 원천 변경)의 최악 리스크는 **"팝업 화면이 다르게 보임"**(눈으로 즉시 확인·즉시 롤백)이지, **"업로드가 깨짐"이 아니다.** 검증된 보내기/받기/쓰기/읽기 동작은 이 작업 범위 밖 코드다.

## 3. 목표

1. **1부 — 단일 원천화**: API 카탈로그의 진실원천을 `marketplace_api_map.json` **하나**로 통합. 인라인 중복(`MKT_CATS`·`ESM_CATS`) 제거, 별표 비교 탭도 JSON을 읽게 전환. 보조 데이터(정산·주문상태·문서수집법)는 JSON(또는 인접 데이터 파일)로 이관해 인라인 하드코딩 제거.
2. **2부 — 전수정독 방법**: 마켓 API 개발 시작 시 강제 발동하는 **스킬** + 마켓 하나를 한 장으로 뽑는 **인앱 덤프 라우트**. 스킬이 덤프를 정독→브리핑→갭 선순환→라이브 대조→개발.

---

## 4. 1부 설계 — 단일 원천화 (회귀 안전 최우선)

### 4.1 절차 (탭 단위, 순차)

각 인라인 소비자를 **한 번에 하나씩** JSON 읽기로 전환한다. 전환마다 아래 게이트를 통과해야 다음으로 넘어간다.

1. **인라인 소비자 인벤토리 확정** — `MKT_CATS`/`ESM_CATS`/`DM_MARKETS`/`API_HOW`/`API_CALLS`/`TRANS` 및 정산식 배열이 각각 어느 탭·함수에서 소비되는지 코드로 전수 열거(추측 금지).
2. **데이터 매핑 판정** — 각 인라인 데이터가 JSON에 이미 있나:
   - **카탈로그(`MKT_CATS`/`ESM_CATS`)**: JSON이 상위집합(610⊇446). → 소비자(`dmFind` 등)를 `MKT_CATS_D`(JSON 파생)로 전환 후 인라인 삭제. **이관 전, JSON에 없는 인라인 항목이 있으면 먼저 JSON에 흡수**(누락 0 보장).
   - **보조 데이터(정산식·주문상태전환·문서수집법)**: JSON에 없음. → JSON 스키마에 필드 추가 또는 인접 데이터 파일 신설로 이관 후 소비자 전환. (범위가 크면 이 항목은 2부의 덤프가 "인라인도 읽기"로 잠정 처리하고, 이관은 후속 태스크로 분리 가능 — 플랜에서 결정.)
3. **전환 + 검증 게이트(탭 하나마다)**:
   - 검증기 `marketplace_api_map.py` 통과.
   - **전/후 브라우저 대조**: 해당 탭을 전환 전후로 열어 **항목 개수·상태색·보내기/받기 위치가 동일**한지 스크린샷·개수 대조. 다르면 중단·원인 규명.
   - 회귀 테스트: 렌더 함수 단위 테스트(가능 범위).
   - **원자 커밋**(탭 1개 = 커밋 1개) → 문제 시 즉시 되돌림.
4. **인라인 삭제** — 모든 소비자가 JSON을 읽는 것이 확인된 뒤에만 죽은 인라인 상수 제거.

### 4.2 안전장치 요약

- 표시 레이어만 변경, 업로드/주문/정산 호출 코드 불변(§2.3).
- 탭 단위 원자 커밋 + 전/후 화면 대조 + 검증기 + 누락0 흡수.
- 어느 단계든 화면이 달라지면 **정직하게 중단·보고**(가짜 완료 금지).

## 5. 2부 설계 — 전수정독 방법 (스킬 + 인앱 덤프)

### 5.1 인앱 덤프 라우트

- `GET /marketplace-guide/map-brief?market=<id>` → `text/markdown` (마켓 하나 = 마크다운 1장).
- 담는 것: 개발환경(인증·host·계정 prefix) → 업무탭별 전체 API(방향·상태·코드·요청/응답/필드/에러) → 정산 계산식(필드코드) → ID 모델 → 문서 수집 경로·접수량·등급 → 미확보/후속과제.
- 출처: 1부 완료 후 **`marketplace_api_map.json` 하나** + `docs/markets/<id>.yaml` + `_API문서수집법.md`. (1부 미완 항목이 있으면 잠정적으로 map.html 인라인도 병합.)
- 목적: Claude가 여러 곳을 헤매지 않고 **한 장만 Read**.

### 5.2 스킬 `consult-market-map` (판매처 지도 정독 게이트)

- **위치**: 프로젝트 `.claude/skills/consult-market-map/SKILL.md` (`add-source` 스킬과 동급).
- **트리거**: 판매처/마켓 API 개발 발화("쿠팡 가격 올려", "롯데온 클레임 연동", "스스 주문 조회", "○○ 마켓 API 붙여줘" 등). 관문 통과 전 코드 작성 금지.
- **6단계 프로토콜**:
  0. **멈춤(게이트)** — 판매처 API 작업 인식 → 스킬 강제 발동.
  1. **전수 정독** — 대상 마켓의 덤프(§5.1)를 Read + (1부 미완 시) map.html 인라인 보강. **체크리스트로 8개 탭 전부 커버 확인.**
  2. **브리핑 작성** — 이 기능에 필요한 것만: 필요 API의 method·URL·요청필드·응답필드·에러코드·ID모델·인증·정산필드·현재상태.
  3. **갭 판정 & 선순환** — 브리핑에 빈칸이 있으면 `_API문서수집법.md` 플레이북(A→A-2→robots→H→F→C→I)으로 확보 → **`marketplace_api_map.json`에 되채움(검증기 통과)** → 브리핑 완성. (다음번엔 지도에 이미 있음.)
  4. **라이브 대조** — `/marketplace-guide/map` 팝업을 열어 그 마켓 탭과 대조, **화면=데이터 일치** 확인.
  5. **개발 착수** — 그제서야 코드 작성(`build-source-adapter`/`tdd`로 연결).

### 5.3 완성 게이트 & 정직성

- 브리핑 "완성" = 필요한 각 API가 **채워졌거나 "확인불가"로 명시**됨. 빈칸 남긴 채 개발 착수 금지.
- 날조(등급⑤) 절대 금지. 없으면 "확인불가", **폴백 금지**(프로젝트 데이터 정합성 3대 원칙 그대로).

## 6. 구현 위치·배포·테스트

- **⚠️ 본체 트리(`feature/crawl-worker-queue`)는 origin/main보다 뒤처짐.** 최신 코드는 워크트리(`_wt_reg` 등)에 있음. → **스킬·라우트·1부 변경 모두 origin/main 워크트리에서** 구현 → 머지 → Fly.io 재배포.
- **테스트**:
  - 1부: 렌더 회귀 테스트 + 검증기 통과 + 전/후 화면 대조.
  - 덤프 라우트: pytest(마켓별 200 · 필수 섹션 존재 · 마크다운 형식).
  - 스킬: writing-skills TDD baseline(발동 트리거·6단계 준수·완성게이트).

## 7. 범위 밖 (YAGNI)

- 신규 마켓 추가 자체(별도 작업).
- 실제 업로드/주문/정산 **호출 로직 변경 없음** — 표시·문서 레이어만.
- 팝업 디자인 개편 없음(데이터 원천만 정리; 렌더 결과는 동일 유지가 목표).
