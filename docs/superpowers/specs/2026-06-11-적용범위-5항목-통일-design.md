# 설계 — 새 혜택 「적용 범위」 5항목 통일 + 옵션 매트릭스 직접 선택

> **계층**: 설계 문서 (spec)
> **책임**: 모음전 상세 매트릭스의 혜택 적용범위 UI/백엔드 재설계 단일 진실 원천
> **상위 의존**: 프로젝트 CLAUDE.md (데이터 무결성), [[project_fx_benefit_model]], [[project_url_store_fragmentation]]
> **작성일**: 2026-06-11
> **확정 시안**: 시안 2 (세로 라디오 카드)

---

## 1. 배경 · 목표

모음전 상세페이지 계산식 팝업에는 혜택을 추가/삭제/수정하는 세 진입점이 있고, 각각 "적용 범위"를 다르게 묻는다.

- **새 혜택 추가** 드롭다운(`b2-drop`): `이 옵션만 / 동일 컬러 / 모음전 전체 / 소싱처 전체 모음전`
- **🗑 삭제** 모달(`smOpenDeleteScopeModal`): `해당 옵션만 / 이 모음전 전체 / 소싱처 전체` (3개)
- **✎ 값 수정**: 범위 선택 없이 "이 모음전 고정" 인라인 편집만

목표: **세 곳을 동일한 5항목 세트로 통일**하고, 그중 2개(옵션 직접 선택·해당 모음전 소싱처 전체)를 새로 추가한다.

### 통일 5항목

| 항목(라벨) | scope 키 | 의미 | 대상 |
|---|---|---|---|
| **이 옵션만** | `option` | 이 옵션 × 이 소싱처 1칸 | sku 1 × source 1 |
| **옵션 매트릭스 직접 선택** | `select` | 팝업 그리드에서 켠(ON) 옵션들 × 이 소싱처 | 선택 sku N × source 1 |
| **모음전 전체** | `bundle` | 이 모음전 모든 옵션 × 이 소싱처 | bundle sku N × source 1 |
| **해당 모음전 소싱처 전체** | `bundle_all_src` | 이 모음전 모든 옵션 × **모든 소싱처 열** | bundle sku N × source M |
| **프로그램 소싱처 전체** | `source` | 이 소싱처 × 전체 모음전 (소싱처 공통 템플릿) | SourceBenefitTemplate 1건 |

- `동일 컬러`(기존 `color`)는 **UI에서 제거**. 같은 컬러 일괄은 `select` 팝업에서 컬러 열을 통째로 켜서 처리한다. (백엔드 `color` 분기는 레거시 호환 위해 남겨두되 UI는 노출 안 함.)

## 2. 데이터 모델 (현행 — 변경 없음)

- `SourceBenefitTemplate(source_id, benefit_name, ...)` — 소싱처 1개의 전 모음전 기본값. = `source` scope.
- `OptionBenefitOverride(canonical_sku, source_id, benefit_name, ...)` — 옵션×소싱처 단위 오버라이드. = `option`/`select`/`bundle`/`bundle_all_src` scope.
- 옵션↔모음전 매핑은 `Option.model_code` 기반 (`_options_by_bundle_code` / `_bundle_skus`). `bundle_options` 정션은 빈 상태라 미사용.

신규 scope도 **새 테이블 없이** 위 두 테이블에 N건 insert/delete로 구현 → 마이그레이션 불필요.

## 3. UI 설계 — 시안 2 (세로 라디오 카드)

5개 항목을 드롭다운 없이 세로로 펼친 라디오 카드. 각 카드 = `라디오 + 색 점 + 라벨 + 우측 카운트/액션`.

```
○  ● 이 옵션만                         +1
○  ● 옵션 매트릭스 직접 선택        [선택 →]
◉  ● 모음전 전체                      +134
○  ● 해당 모음전 소싱처 전체          다수
○  ● 프로그램 소싱처 전체             다수
```

- 선택 카드: 파란 테두리 + 연한 파란 배경(`#f4f9ff`), 라디오 채움.
- `옵션 매트릭스 직접 선택` 카드는 우측에 파란 `선택 →` 버튼 → 클릭 시 §4 팝업 오픈. 팝업에서 옵션 확정해야 이 카드가 "활성 선택" 상태가 됨(선택 0건이면 추가 버튼 비활성).
- 색 점: 이옵션=초록 / 매트릭스=청록 / 모음전=파랑 / 모음전소싱처=주황 / 프로그램소싱처=보라.
- 카운트 텍스트는 기존 `_bundleCount` 등 변수 재사용.

### 적용 위치 3곳

1. **새 혜택 추가** — `_matrix_v3.html` 의 `b2-drop` 블록(L3563~3595)을 세로 라디오 카드로 교체.
2. **🗑 삭제 모달** — `smOpenDeleteScopeModal`(L3716~)의 3카드를 동일 5카드로 교체. 삭제는 위험 동작이라 선택 시 빨간 강조 + 적용 버튼 라벨에 영향 건수 노출(기존 패턴 유지).
3. **✎ 값 수정** — 현행 순수 인라인 편집(`smCellEdit`, L3784~)을 **작은 모달**로 승격: `값 입력칸 + 5항목 라디오 카드 + 저장`. 사용자의 "삭제/수정도 추가와 동일하게" 요청 충족.

> 세 곳이 같은 마크업/CSS를 쓰도록 공통 헬퍼 `renderScopePicker(ctx)` 1개로 묶는다(중복 제거). `ctx` = {sku, srcId, bundleCount, srcName, sources}.

## 4. 「옵션 매트릭스 직접 선택」 팝업

사용자가 보낸 스크린샷의 **조합 매트릭스**(행=사이즈, 열=색상, 칸=ON/OFF, 상단 `N/전체` + `전체 선택/해제`)와 동일한 모양.

- **데이터**: 이미 로드된 전역 `DATA.options` 사용 (sku, color_display, size). 별도 fetch 없음.
- **구성**: 색상=열 헤더, 사이즈=행 헤더로 피벗. 존재하는 (색×사이즈) 조합 = 토글 셀, 미존재 = `·` 비활성.
- **상호작용**: 셀 클릭 토글 / 행·열 헤더 클릭 = 행·열 일괄 토글 / `전체 선택·해제`. 상단 카운트 실시간 갱신.
- **CSS 재사용**: `option_url_modal.js`의 조합 매트릭스 스타일(`.oum-mtx*`, `.oum-cell`, `toss.css`)을 그대로 차용 → 스샷과 동일 룩.
- **확정**: `적용` 클릭 → 켜진 셀의 `canonical_sku[]` 를 부모 picker로 반환. 부모는 `scope='select'` + `skus=[...]` 로 저장.
- 구현은 axis/cartesian 기반 기존 `renderMatrix`(조합 생성 전용, state 결합)를 재사용하지 않고, `DATA.options` 기반 **경량 빌더**를 신규 작성(결합도↓). 시각 스타일만 공유.

선택 컨텍스트는 **현재 소싱처(srcId) 1개**다(팝업은 옵션만 고르고 소싱처는 진입한 칸 기준).

## 5. 백엔드 변경

### 5.1 추가 — `POST /api/benefits/crud` (`api_benefits_crud.py`)

- scope 화이트리스트에 `select`, `bundle_all_src` 추가.
- **`select`**: payload `skus: string[]` 필수 → 각 sku × `source_id` 로 `OptionBenefitOverride` insert.
- **`bundle_all_src`**: payload `source_ids: int[]` 필수(프런트가 `DATA.sources`에서 전달) → bundle 옵션(`_options_by_bundle_code`) × 각 source_id 로 override insert.
- 응답 `applied_count` = 실제 insert 건수.

### 5.2 삭제 — `POST /api/source-benefits/delete-scoped` (`api_benefits.py`)

- scope 화이트리스트에 `select`, `bundle_all_src` 추가.
- **`select`**: `skus[]` 대상 override 삭제 + (소싱처 공통 템플릿 존재 시) 비활성 스텁 생성(기존 option/bundle 로직 재사용).
- **`bundle_all_src`**: bundle 옵션 × `source_ids[]` 각각에 대해 기존 bundle 삭제 로직을 source_id 별로 반복. 템플릿 비활성 스텁도 source_id 별 처리.

### 5.3 수정 — 값 적용 (`api_benefits.py`)

- `POST /api/source-benefits/overrides/set-value-scoped` **신설**(scope 인지). 기존 `set-value-bundle`은 호환 위해 남기되 내부에서 `scope='bundle'`로 신규 경로를 호출하도록 위임.
- scope별 대상 결정:
  - `option` → 단일 sku override upsert
  - `select` → skus[] upsert
  - `bundle` → bundle 옵션 upsert (현행 그대로)
  - `bundle_all_src` → bundle 옵션 × source_ids upsert
  - `source` → **SourceBenefitTemplate.value 직접 수정** (소싱처 전 모음전 반영)
- upsert 시 기존 `set-value-bundle`의 템플릿 연결·태그 복사 로직 유지.

## 6. 데이터 무결성 · 주의사항

- ⚠️ **`source` scope 수정의 광범위 영향**: 2026-06-11 직전 커밋(`✎`)에서 "공통 템플릿 수정 시 소싱처 전 모음전에 반영되는 사고"를 막으려 수정 범위를 "이 모음전"으로 고정했다. 본 설계는 사용자가 `프로그램 소싱처 전체`를 **명시적으로 선택**할 때만 템플릿을 수정하므로 의도된 동작이다. 단, 해당 카드는 **확인 다이얼로그**("전체 모음전 N개에 반영됩니다")로 한 번 더 막는다.
- 가격·재고 정확성 원칙(프로젝트 CLAUDE.md): 모든 scope insert/delete는 단일 트랜잭션, 실패 시 rollback. 영향 건수를 응답·UI에 노출.
- `select` 0건/`bundle_all_src` source_ids 0건 → 저장 거부(`_err`).

## 8. 범위 밖 (YAGNI)

- 표시가 vs 업로드가 통일([[project_pricing_display_vs_upload]])은 건드리지 않음.
- 매트릭스 팝업에서 소싱처(열)까지 동시에 고르는 기능은 미포함(옵션만). 다중 소싱처는 `bundle_all_src` 또는 `source`로 충분.
- 동일 컬러 전용 scope UI 재도입 안 함(매트릭스 열 선택으로 대체).

## 9. 검증 계획 (라이브 브라우저)

1. 새 혜택 추가에서 5항목 각각 저장 → 영향 옵션 수가 매트릭스 셀 가격에 반영되는지 전수 확인.
2. `옵션 매트릭스 직접 선택` 팝업: 셀/행/열/전체 토글 카운트 정확, 적용 후 선택 옵션에만 반영.
3. `해당 모음전 소싱처 전체`: 모든 소싱처 열에 동일 혜택이 붙는지.
4. `프로그램 소싱처 전체` 수정: 확인 다이얼로그 후 템플릿 값 변경, 다른 모음전에도 반영되는지.
5. 삭제 5항목 각각 → 비활성 스텁/행 삭제 동작 확인, 롤백 안전성.
