# 크롤링 가이드 ↔ 코드 자동 동기화 + 전체크롤 정합성 가드 — 설계

> **계층**: 프로젝트 스펙 (브레인스토밍 산출물)
> **책임**: 데이터지도(보기/원문)가 코드·전체크롤 실제 로직과 어긋나지 않게 유지하는 메커니즘 설계
> **상위 의존**: `docs/크롤링-가이드.md`(정본) · `[[project_crawl_guide_data_code_map]]`
> **작성일**: 2026-06-26 · **상태**: 설계 승인 대기

---

## 1. 배경 / 문제

데이터지도는 **두 곳을 따로 관리**한다 → 구조적 드리프트:
- **원문** = `프로그램/_시스템/docs/크롤링-가이드.md` (마크다운, `/sourcing-guide/map.md`가 직독)
- **보기** = `webapp/templates/sourcing_guide/map.html` (손코딩 HTML 7탭, 마크다운과 무관)

이번 세션 검증에서 드러난 어긋남 3종:
1. **추가 누락** — 새 소싱처가 늘면 보기·원문에 행·카드를 손으로 추가해야 하는데 빠뜨림.
2. **의미적 어긋남 — 구버전 참조** — §7 E섹션이 repo 확장(`extension/moum-crawler`, **v0.4.3**)을 인용했으나, **전체크롤 실제 로직은 데스크톱 로드 확장(현재 v0.7.6)** 에 있음. repo 확장에는 `crawl.enqueue` 큐러너조차 없음.
3. **의미적 어긋남 — 라인/심볼 표류** — `_detect_site_from_url` 1475→1620 등 인용 라인이 표류.

## 2. 결정된 방향 (브레인스토밍 합의)

| 항목 | 결정 |
|---|---|
| 범위 | **둘 다**(추가 누락 + 어긋남), 단계적 구현 |
| 근본구조 | **보기 디자인 유지**(보기/원문 통일 안 함) + **자동 감지로 보강** |
| 경고 위치 | **인앱 데이터지도(`/sourcing-guide/map`) 상단 배너** (CI 아님) |
| 정합성 범위 | **확장 버전 정합성 + 핵심 심볼 존재** |

명시적 범위 밖(YAGNI): CI 게이트, 보기↔원문 통일/코드젠, 마크다운 본문 파싱.

## 3. 전체크롤 실제 흐름 (정합성의 "기준")

```
[버튼] toss.js  enqueueCrawl(code)
  → ext_bridge.js  send("crawl.enqueue", {code})      ← 확장에 메시지
  → 확장 백그라운드 큐러너(데스크톱 로드본 v0.7.6)      ← 실제 전체크롤 로직(repo 밖)
     · 비로그인 4(르무통·SSF·SSG·스스) = navGrab  → 서버 /api/sources/parse
     · 로그인 2(무신사·롯데온)        = navExtract → 확장 추출기(EXTRACTORS)
  → /api/sources/crawl-result 저장 → 계산(compute_breakdown) → 매트릭스·위젯 표시
```
핵심 제약: **전체크롤 확장 로직은 repo에 없다**(버전드 데스크톱 확장). 따라서 정합성은 "실제 로드된 확장 버전"을 기준으로만 자동 확인 가능.

## 4. 아키텍처 — 4파트

### P0. 즉시: §7 E-섹션 정정 (보류분 + 새 절차의 첫 적용 사례)
- E1~E4를 위 §3 실제 흐름으로 재기술: 진입점 = `enqueueCrawl → crawl.enqueue → 확장 큐러너`. `crawlBundle`(구버전 스케줄러·무신사/롯데온만)·`crawlBundleAll`(미정의 폴백) 인용 제거.
- 경고문 추가: **"전체크롤 확장 실제 코드 = 데스크톱 로드본(현재 v0.7.6), repo `extension/`(v0.4.3)은 구버전 — 라인 신뢰 금지."**
- 보기(map.html 단계7)·원문(.md §7) 둘 다.

### P1. 자동 드리프트 배너 — 소싱처 존재 누락 (서버측)
- `data_code_map()` 라우트가 렌더 시 검사 → 배너 데이터 전달.
- **기준(단일 진실 원천)**: `source_registry.SOURCES` 의 빌트인 소싱처 `key`·`label`. (코드에 빌트인 추가 시 자동 등장.)
- **검사**: 각 소싱처 라벨이 **원문(.md)** 텍스트와 **보기(map.html)** 텍스트 양쪽에 등장하나.
- **배너**: `⚠️ 코드엔 있으나 가이드 누락: '현대H몰' (원문 ✓ / 보기 ✗) — §1·§2·§6·§7 갱신 필요`. 누락 0이면 배너 없음.

### P2. 신규 소싱처 추가 갱신 절차 (추가 누락 + 의미적 어긋남)
- **§7 체크리스트 게이트화**: 마지막 줄에 "**P1 배너·P3 정합성 0인지 확인**" 추가.
- **프로젝트 스킬 `build-source-adapter` 확장**: 마지막 단계 "가이드 §1·§2·§6·§7 **보기+원문** 갱신 → `/sourcing-guide/map` 배너 0 확인".
- §7 도리 유지: "심볼 우선 인용 · 확장은 데스크톱 로드본이 진실".

### P3. 전체크롤 정합성 가드 (버전 + 심볼) — 같은 배너에 합류
- **C1 확장 버전 정합성 (클라이언트측)**: map.html JS가 `MoumExt.ping().version` 호출 → 가이드가 명시한 **기준 버전**(`GUIDE_EXT_BASELINE`, 서버가 템플릿에 주입)과 비교. 다르면 배너 `⚠️ 전체크롤 확장 v0.8.0 로드됨 — 가이드 기준 v0.7.6, §7 전체크롤 설명 재검토`. 확장 없으면 조용히 skip.
- **C2 핵심 심볼 존재 (서버측)**: `SYMBOL_MANIFEST`(명시 목록)의 각 `{symbol, file}`이 코드에 실재하나 검사. 사라지면 배너 `⚠️ §7 인용 심볼 '_SITE_BY_SRC' 코드에서 사라짐 — 갱신 필요`.
  - 매니페스트 초기 항목: `build_crawlers`(crawlers/__init__.py), `_SITE_BY_SRC`·`compute_breakdown`(api_benefits.py), `_detect_site_from_url`·`_resolve_stock`·`_persist_option_stocks`(api_pricing.py), `PRODUCT_DYNAMIC_KEYS`(service.py), `SOURCES`(source_registry.py), `SOURCE_ORDER`(crawl_log.js), `EXTRACTORS`(extension/background.js).

## 5. 컴포넌트 (격리 · 각자 단일 책임)

| # | 컴포넌트 | 책임 | 인터페이스 |
|---|---|---|---|
| 1 | `webapp/routes/guide_sync.py` (신규) | 순수 검사 함수 | `missing_sources(md, html, sources)→list` · `missing_symbols(manifest)→list` |
| 2 | `SYMBOL_MANIFEST` (guide_sync.py 내 상수) | C2 심볼-파일 명시 목록 | `[(symbol, relpath), ...]` |

| 5 | `map.html` 배너 영역 | 표시 — 서버 데이터 배너 + JS 버전체크(C1) | — |

검사 함수(1)는 라우트·템플릿과 분리되어 **단위 테스트 가능**.

## 6. 데이터 흐름

```
[서버 렌더] data_code_map()
   ├ source_registry.SOURCES → guide_sync.missing_sources(md_text, html_text, sources)  → missing_sources[]
   ├ guide_sync.missing_symbols(SYMBOL_MANIFEST)                                          → missing_symbols[]
   └ GUIDE_EXT_BASELINE                                                                   → 템플릿
[클라이언트] map.html
   ├ (서버 데이터) missing_sources / missing_symbols → 상단 배너 렌더
   └ MoumExt.ping().version vs ext_baseline (C1) → 버전 불일치면 배너 한 줄 추가
```

## 7. 에러 처리

- 모든 검사는 **best-effort, 페이지를 깨지 않음**: .md 읽기 실패·grep 실패 → 해당 검사 skip(배너에 안 띄움).
- 확장 미설치(ping 불가) → C1 조용히 skip.
- 검사 결과가 비어 있으면 배너 자체를 렌더하지 않음(조용함).

## 8. 테스트

- `guide_sync` 단위 테스트(fixture):
  - 코드에 소싱처 있고 .md에 없음 → `missing_sources`에 포함.
  - 라벨이 보기에만 없음 → `(원문✓ 보기✗)`로 구분.
  - 매니페스트 심볼 존재 → 빈 리스트 / 심볼 삭제한 fixture → 포함.
- 회귀: 현재 6 빌트인 소싱처는 누락 0이어야(배너 안 뜸) — 라이브 대조.

## 9. 범위 밖 (YAGNI)

- CI/푸시 게이트(사용자: 인앱만).
- 보기↔원문 통일·코드젠(사용자: 디자인 유지).
- 마크다운 본문에서 심볼 자동 추출(취약) → **명시 매니페스트**로 대체.
- 커스텀(DB) 소싱처 가이드 강제 — 빌트인만 대상.

## 10. 미해결 / 리스크

- **C2 매니페스트도 드리프트 가능**: §7 인용이 바뀌면 매니페스트도 갱신해야 함(작지만 존재). → 매니페스트를 §7과 같은 PR에서 함께 고치도록 P2 절차에 명시.
- **C1 기준 버전 수동 bump**: 전체크롤 확장이 업데이트되고 §7 설명을 갱신할 때 `GUIDE_EXT_BASELINE`도 올려야 함(의도된 트리거 — 안 올리면 배너가 재검토를 요구).
- 배포는 `[[project_deploy_isolated_change_via_worktree]]` 패턴(origin/main 워크트리 → push main).
