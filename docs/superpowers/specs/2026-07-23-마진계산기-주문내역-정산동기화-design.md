# 마진계산기 정산액을 주문내역과 동기화 (2026-07-23)

> **결정 주체**: 사장님 (2026-07-23 대화)
> **정답지**: 주문내역 탭 — "이거 정확해"
> **한 줄**: 마진계산기가 정산액을 **다시 계산하지 않고**, 주문내역이 확정한 `정산예정금(배송비포함)` 을 그대로 쓴다.

⚠️ **표기 규칙**: 엑셀 열 문자(N열·M열·G열)로 부르지 않는다. 엑셀 구조는 계속 바뀌므로 **필드명**이 유일한 식별자다(사장님 지시).

## 1. 배경 — 하루 차이로 끊긴 배선

주문내역(`order_export`)은 2026-07-23 에 6마켓 공통 규약을 확정했다(샵마인 45건 전수 실측).

| 필드명 | 뜻 |
|---|---|
| `정산예정금액` | **상품 정산만** (배송비 제외) |
| `배송비` | 고객이 낸 배송비 — 배송건 첫 행에만, 나머지 행은 0 |
| `정산예정금(배송비포함)` | `정산예정금액` + `배송비` = 샵마인 `정산예상금액(배송비포함)` 과 같은 정의 |

마진계산기(`margin/sell_source._settlement_for`)의 마지막 수정은 2026-07-22 였다. 그래서
`정산예정금액`(상품분)을 읽어 **마켓별로 배송비를 손으로 더하는** 옛 코드가 남았고, 규약 전환을 못 따라가 조용히 어긋났다.

### 실측된 어긋남 (함수 직접 실행)

| 케이스 | 주문내역(정답) | 옛 마진계산기 | 결과 |
|---|---|---|---|
| 쿠팡 정상(상품정산 45,000 + 배송비 3,000) | 48,000 | 45,000 | 배송비만큼 마진 **과소** |
| 롯데온 취소완료 | 0 | **50,350 "추정"** | 취소 주문이 매출로 잡혀 마진 **부풀림** |
| 옥션·G마켓 취소완료 | 0 | 3,000 | 배송비가 정산으로 잔존 |

롯데온 취소건이 가장 비쌌다. 주문내역은 취소완료 행의 `마켓수수료` 를 0 으로 확정하는데,
옛 코드는 그 0 을 "아직 미정산"으로 오해해 `실결제금액 × 0.947` 추정을 만들어 냈다.

## 2. 설계

### 2-1. 정산액 — 단일 원천 = 주문내역

`_settlement_for(row)` 우선순위:

| 순위 | 조건 | 값 | 배지 |
|---|---|---|---|
| ① | `_settle_source == "zero_cancel"` 또는 `주문상태` 에 "취소완료" | **0** | `zero_cancel` |
| ② | `정산예정금(배송비포함)` 이 있고 배지가 real·store·estimated | **그 값 그대로** | 원래 배지 |
| ③ | 주문내역이 못 채운 11번가·롯데온 미정산 | 상품 추정 + `배송비` 전액 | `estimated` |
| ④ | 재료 없음 | 0 | `none` |

- ①에서 주문상태 문자열도 함께 보는 이유: 적재분에 `zero_cancel` 태그가 없던 시절 행이 남아 있어도 같은 판정이 나오게.
- ②에서 배지를 확인하는 이유: 배지가 `none` 인데 값만 남은 행(취소건의 배송비 잔존)을 정산으로 오인하지 않기 위해.
- ③의 추정 계수는 기존 그대로(`LO_FEE_FACTOR_*`, `EL_FEE_FACTOR_*`). 실수수료가 없다고 0 으로 두면 매출이 사라져 '손실'로 둔갑한다.

### 2-2. 매출 필드 동기화

마진계산기가 매출 금액을 스스로 만들면 주문내역과 조용히 어긋난다.
`matcher` 는 원본 바이트 동치(verbatim)라 `판매가 = 단가 × 수량` 만 계산하고 **옵션추가금을 못 담는다**.

- `SELL_COLUMNS` 에 `옵션추가금`·`상품금액`·`총주문금액` 추가 → 주문내역 값 그대로 전달
- `pipeline._attach_settle_source` 가 `배송비` 와 같은 방식으로 이들을 matched 행에 재부착
  (`_CARRY_FIELDS`). matcher 를 못 고치므로 이 경로가 유일하다.
- 재부착 값은 `_NUMERIC_FIELDS` 에 넣어 숫자 칸으로 유지 — 문자열로 새면 aggregator 의 `sum()` 이 TypeError 로 죽는다.

`판매가`(단가×수량)와 `총주문금액`(+옵션추가금)이 함께 실리므로, 옵션가가 붙은 주문에서 두 값의 차이가 화면에 드러난다.

### 2-3. 배지 신뢰도 순위

`pipeline._TAG_RANK` 에 `store`(=real 급)와 `zero_cancel` 추가.
`zero_cancel` 은 `estimated` 보다 아래 — 한 주문번호에 정상행과 취소행이 섞이면
"취소가 섞여 있다"는 쪽을 보여준다. 돈을 받은 것처럼 보이는 오표시가 반대 방향보다 훨씬 비싸다.

## 3. 범위 밖 (의도적)

- **스마트스토어 `정산예정금액` 정렬**: 스스만 아직 `정산예정금액` 에 배송비 정산액을 품고 있어,
  `정산예정금(배송비포함)` 이 배송비를 두 번 셀 수 있다(배송비 붙은 스스 주문에 한해).
  **사장님이 나중에 직접 업데이트하기로 결정**(2026-07-23). 주문내역 쪽 변경이라 마진계산기와 분리.
- 매입원가: `구매가격` 만 유지(`국제운송료` 제외) — 사장님 결정.
- `마진율` 분모(`단가`×`수량`), 매칭 로직, 마진탭 8개 화면 구성 — 손대지 않음.
