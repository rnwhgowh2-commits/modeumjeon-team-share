# 판매처(마켓)별 API 단일 진실 원천 — 설계 문서

> **계층**: 프로젝트 스펙 (docs/superpowers/specs)
> **책임**: 6개 판매처 마켓의 **모든 API 코드·구분자·실제 요청/응답**을 한 곳에 담고, 인앱 「데이터 코드 지도」가 그대로 렌더하게 하는 체계 설계
> **상위 의존**: 프로젝트 CLAUDE.md (데이터 무결성·중복금지)
> **작성일**: 2026-07-16 (최종 갱신: 목적·화면·데이터모델 확정)
> **관련 기존 산출물**: 인앱 「데이터 코드 지도」 `webapp/templates/marketplace_guide/map.html`(이 화면이 SOT 데이터를 렌더) · 구 `docs/markets/_schema.yaml`·`{coupang,smartstore}.yaml`(→ 데이터 파일로 이관 후 삭제)

## 1. 목적 (왜) — 반복 노가다 종식

**실제 겪는 문제**: 새 기능을 추가할 때 "데이터 코드 지도 봐" 라고 해도, 지도가 **얕아서**(엔드포인트 이름 + apiNo 만 있고 **실제 요청 파라미터·응답 JSON 이 없음**) Claude 가 구현을 못 한다. 결국 사용자가 **매번** 공식 API 문서로 돌아가 F12 콘솔로 실제 요청/응답을 캡처·복붙한다 → **동일한 중복 노동 무한 반복**.

**해결 원칙**: 문서에 **"다시 볼 것"을 남기지 않는다.** 각 API 마다 **실제 요청 + 실제 응답 JSON 통째 + 필드별 뜻**까지 박아넣어, 그 데이터만 보면 F12 재방문이 불필요하게.

> 앞으로 사용자는 **"이 기능 추가해줘"** 라고만 말하고, Claude 는 **원본 API 문서·F12 없이 이 데이터만 읽어** 구현한다.

**역할 분담** (한 API 당 1회):
- **사용자** = 원본 그대로 **노가다 붙여넣기** (F12 응답 JSON·공식 문서 캡처·복붙 — 가공 불필요).
- **Claude** = 붙여넣은 원본을 **데이터 파일에 구조화** (요청·응답·필드 뜻·성공판정 정리).

## 2. 진실 원천 = 「데이터 코드 지도」가 읽는 데이터 파일 1개

- **결정**: 별도 `.md` 문서를 두지 않는다. **인앱 화면이 렌더하는 데이터 파일**(예: `webapp/static/marketplace_api_map.json`)을 **유일한 진실 원천**으로 삼는다.
- **이유**: 사용자가 늘 보는 곳이 그 화면이다. 데이터가 화면 데이터면 → 내가 채우는 즉시 **화면에 그대로 박힌다**("완벽히 박혀?" = 예). 그리고 나도 같은 파일을 읽어 구현 → 화면과 나의 진실이 **1곳**.
- **형식**: JSON(화면이 fetch). 원본 응답 JSON 은 필드에 통째 보관, 사용자는 **렌더된 화면으로만** 보므로 JSON 편집 부담 없음(편집은 Claude).
- 570줄 HTML 안에 하드코딩하지 않고 **데이터 파일로 분리** → 6마켓 × 수백 API 규모를 감당.

## 3. 데이터 모델 (핵심)

납작한 **API 레코드 목록** 하나 + 부속 사전. 중복 정의 없음.

### 3-1. `apis[]` — API 1건 = 유일 id (마켓별)
| 키 | 뜻 |
|---|---|
| `id` | 유일 식별자 (예: `coupang.product.get_detail`) |
| `market` | 마켓 (coupang / smartstore / lotteon / eleven11 / auction / gmarket) |
| `fnKey` | **기능키**(마켓 공통 논리명, 예: `product.get_detail`) — 마켓 비교의 축 |
| `tabs[]` | 이 API 가 **노출될 업무 탭 목록** (겹침 허용, 예: `["클레임·CS","주문상태"]`) |
| `category` | 그 마켓 공식 문서상 카테고리 (좌측 레일용) |
| `nm` / `dir` | 표시명 / 방향(send=보내기·recv=받기) |
| `st` | 신뢰도 dot: `ok`(됨·실사용) / `code`(연결됨·검증대기) / `off`(공식제공·미구현) / `todo`(미정의) |
| `endpoint` | METHOD + 경로 |
| **`req`** | **실제 요청**(파라미터·헤더·바디, 원본 그대로) — 완성게이트 |
| **`res`** | **실제 응답 JSON 통째**(F12 원본, 자르지 않음) — 완성게이트 |
| **`fields[]`** | 응답 필드 뜻 `{key, 뜻, 예시}` — 완성게이트 |
| **`success`** | 성공판정("무엇을 보면 진짜 성공"; 거짓성공 함정 포함) — 완성게이트 |
| `idTraps[]` | 식별자 함정(예: 스스 `A.id≠productNo`) |
| **`persistIds[]`** | (조회 API 한정) 이 응답에서 **우리 주문 레코드에 저장해둘 마켓 원본 식별자** — 나중에 명령(전환·송장·클레임)을 그 주문으로 정확히 라우팅하기 위한 열쇠. 캡처만 하고 버리면 나중에 명령 불가. |
| `codeRef` | 우리 구현 파일:라인 (있으면) |

- **한 번만 정의**: API 는 `apis[]` 에 1건. **탭은 `tabs[]` 로 참조**만 → 여러 탭에 노출돼도 데이터는 한 벌(진실 원천 분열 0).

### 3-2. `codes[]` — 구분자(enum) 사전 ⭐
| 키 | 뜻 |
|---|---|
| `kind` | 구분(주문상태 / 클레임상태 / 배송사 …) |
| `market` / `code` / `label` | 마켓 / 마켓코드 / 마켓라벨 |
| **`meaning`** | **평문 뜻** (미래의 Claude 가 `-3313`·`odPrgsStepCd` 를 알도록) |
| `unified` | 우리 12 통일상태 매핑 |
| `st` | 신뢰도 |

### 3-3. `markets[]` — 마켓 메타
env키·base_url·인증방식·다계정 prefix·**병렬 규칙**(예: 11번가 순차조회 필수, 병렬 시 429 전체다운).

### 3-4. `unifiedStatuses[]` — 우리 통일상태 기준 (한 번만 정의)
우리 프로그램이 쓰는 유일한 상태 어휘. `{id, 이름, 뜻, 순서}`. `codes[].unified` 와 `transitions[].from/to` 가 이걸 **참조**만 한다(복제 금지). → "결제완료(롯데온)·결제완료(쿠팡)" 껍데기를 만들지 않고, 우리 결제완료 **1개**에 마켓 코드들이 매핑된다(N:1).

### 3-5. `transitions[]` — 우리 상태 전환 ↔ 마켓 액션 API (보내기)
"결제완료 → 배송준비중" 같은 **전환은 상태 글자 변경이 아니라 마켓의 실제 액션 API 호출**이다.
```
{ from:'결제완료', to:'배송준비중', requires:[],
  perMarket:{ coupang:'coupang.order.confirm', lotteon:'lotteon.order.confirm',
              smartstore:'unsupported' } }              # 미지원은 명시
{ from:'배송준비중', to:'배송중', requires:['송장번호'],   # 추가 입력 요구 가능
  perMarket:{ ... : '...invoice.send' } }
```
- `perMarket` 값 = `apis[]` id **참조**(액션 정의는 한 곳, 전환은 가리키기만 → 중복 0).
- **라우팅 원리(핵심)**: 명령 단위는 상태가 아니라 **주문 1건**. 각 주문이 든 `market`·`account`·마켓 원본ID(= 조회 때 `persistIds` 로 저장한 값)로 라우팅 → 같은 "배송준비중" 명령도 주문마다 정확한 마켓 API·ID·계정 키로 날아간다.
- `'unsupported'` = 그 마켓엔 그 전환 없음 → 전환 안 함 "미지원" 표시(추정·거짓성공 금지).

## 4. 화면 구조 (7탭 + 유틸)

데이터에서 **자동 렌더**(마켓 탭·카테고리 레일·목록·비교표·상세 드로어 모두 데이터 기반 — HTML 무수정으로 확장).

**최상위 탭 = 7개** (+ `API 문서 수집법` 유틸):
| 탭 | 성격 | 내부 레이아웃 |
|---|---|---|
| 마켓별 전체 API | 마켓별 전체 조망(coverage) | **마켓 우선** — 마켓 고르면 그 마켓 카테고리별 전 API |
| 상품 조회 | 업무 | **기능 우선 + 마켓 비교** |
| 상품 관리 | 업무(가격·재고·판매상태·상품관리·신규등록·상품속성 추가/수정/삭제) | 기능 우선 + 마켓 비교 |
| 정산 | 업무(구 「정산」 탭 흡수) | 기능 우선 + 마켓 비교 |
| 배송·송장 | 업무(배송정보·반품지·출고지·택배사코드·송장) | 기능 우선 + 마켓 비교 |
| 클레임·CS | 업무(반품/교환/취소 · 고객문의/상품Q&A) | 기능 우선 + 마켓 비교 |
| 주문상태 | 업무(구 「주문상태 전환」 흡수) | **우리 통일상태(앵커) × 마켓** 매핑표 + 상태 간 **전환 화살표**(클릭 시 그 마켓 액션 API 상세) |

- **최상위 탭은 고정**(데이터로 자동 증식 X). 새 업무 성격이 생길 때만 사람이 추가.
- **마켓 탭·카테고리·API 목록·상세는 데이터에서 자동**.

**업무 탭 = 기능 우선 + 마켓 비교** (Q2 결정):
- `fnKey`(기능) = 행, 마켓 = 열. 셀 = 신뢰도 dot / 제공여부. → **"어디는 주고 어디는 안 줌"이 한눈에**(예: 반품 시 오픈마켓주문번호로 고객정보 조회 — 주는 마켓 / 안 주는 마켓).
- 행(기능) 클릭 → **상세 드로어**: 마켓별 실제 요청·응답 JSON·필드 뜻·성공판정.

**상세 드로어(모든 탭 공통)** — API 행 클릭 시 그 자리에서 펼침:
```
요청   GET customer/v1/getSellerInquiryList?startDate=…
응답   { "code":"200", "data":{ "inquiries":[…] } }
필드   inquiryNo=문의번호 · status=NEW(미답변) · …
성공   code=="200" & data.inquiries 존재  (HTTP200 이어도 code 확인)
```

**완성 게이트**: `req`·`res`·`fields`·`success` 4종이 없으면 그 API 는 `todo/off` 로 남기고 "미완". 4종이 다 차야 "F12 다시 볼 필요 없음" 상태.

## 5. 운영 규약 (사용 방식)

**A. 데이터 채우기 (사용자 = 붙여넣기 / Claude = 구조화)**
1. 사용자가 **원본 그대로 붙여넣음** — 공식 문서 캡처, F12 실제 요청/응답 JSON. 가공 불필요.
2. Claude 가 데이터 파일에 채운다: `apis[]`(완성게이트 4종 포함) · `codes[]`(평문 뜻) · `markets[]` · 각 API 의 `fnKey`·`tabs[]` 태깅.
3. **우리 코드와 대조.** 충돌 시: **문서=의도 / 코드=현실 / 라이브=진실**. 임의로 고르지 말고 ⚠️ 플래그 + 사용자 보고.
4. 화면 자동 갱신(같은 데이터) — 별도 동기화 작업 없음.

**B. 기능 구현 (사용자 = 요청 / Claude = 이 데이터로 구현)**
1. 사용자: "X 기능 추가해줘".
2. Claude: 데이터 파일에서 해당 `fnKey`/`tabs` 의 API 들을 읽어 구현. **원본 API 문서·F12 재방문 불필요**.
3. 근거 신뢰도가 `ok`(라이브) 아니면 → "미검증 근거 위 구현" 경고 후 진행.

## 6. 성공 기준

- [ ] 데이터 파일 1개 = 유일 진실 원천, 화면이 그대로 렌더.
- [ ] 최상위 7탭(마켓별 전체 API + 6 업무탭) + 수집법 유틸. 구 정산·주문상태 흡수.
- [ ] 업무탭 = 기능 우선 + 마켓 비교, 행 클릭 → 상세 드로어.
- [ ] **완성 게이트**: 채운 API 마다 `req·res·fields·success` 존재 → **F12 다시 볼 것 0**.
- [ ] 모든 enum·응답 필드에 평문 뜻(빈칸 0), 모든 API 에 신뢰도.
- [ ] 한 API 는 데이터에 1건, 탭은 참조만(겹쳐도 중복 저장 0).
- [ ] `unifiedStatuses[]`(우리 기준 1벌) · `transitions[]`(전환→마켓 액션 참조) · 조회 API `persistIds` 존재 → "상태 공유·명령은 주문별 정확 라우팅" 성립.
- [ ] 구 `_schema.yaml`/`*.yaml` 이관 후 삭제.
- [ ] **최종 검증 시나리오**: "11번가 반품 승인 붙여줘" 를 **API 문서·F12 없이** 이 데이터만으로 구현 가능.

## 7. 범위 밖 (지금 안 함 — YAGNI)

- **소싱처(크롤)** — 별도 지도(`docs/sources/소싱처_API_수집지도.md`)가 담당. 이 체계는 판매처 전용.
- **런타임 import** — 참조 데이터이지 서버 설정이 아님(어댑터 코드는 별개).
- **최상위 탭 자동 증식** — 업무 탭은 고정, 사람이 결정.

## 8. 구현 순서 (개요 — 상세는 후속 플랜)

1. 데이터 스키마 확정 + `map.html` 을 **데이터 파일 fetch + 7탭 + 기능비교 + 상세 드로어** 로 개편(자동 렌더).
2. 쿠팡·스마트스토어(이미 연동) 데이터를 **완성본**으로 채워 **본보기(템플릿)** 확보 — 완성게이트 4종 시연.
3. 구 `_schema.yaml`·`*.yaml` → 데이터 파일 이관 후 삭제. 화면 footer 의 정본 경로 갱신.
4. 나머지 4개 마켓(롯데온·11번가·옥션·G마켓)은 골격만 → 사용자가 원본 붙여넣는 순서대로 채움.
