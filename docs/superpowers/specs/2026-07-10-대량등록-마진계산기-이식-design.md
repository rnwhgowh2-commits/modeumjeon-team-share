## 0. 배경 · 문제

`C:\dev\대량등록 마진계산기` 는 두 엑셀을 매칭해 마진을 계산하는 단독 Flask 앱이다.

- **매입** = 더망고 솔루션 주문내역 엑셀
- **매출** = 샵마인 통합주문관리 엑셀 ← **이것을 우리 API로 대체하려는 것**

모음전은 이미 `lemouton/markets/order_export.py` 로 판매처 마켓 API 에서 주문을 조회한다.
조사 결과 `order_export` 의 출력 컬럼이 샵마인 엑셀 컬럼과 거의 1:1 이다. 대체가 성립한다.

### 대체 시 드러난 6가지 간극

| # | 간극 | 이번 스펙의 처리 |
|---|---|---|
| 1 | 샵마인은 6마켓, 우리 API 실검증은 4마켓 (옥션·G마켓 미검증) | 옥션·G마켓만 **샵마인 엑셀 보조 업로드**로 병행 |
| 2 | 정산예정금액 신뢰도가 마켓마다 다름 (쿠팡 추정, 롯데온 실결제 대용) | `_settle_source` 필드로 real/estimated 구분 + 롯데온 계산 수정 |
| 3 | 블랙스팟 분류기가 샵마인 주문상태 40여 개 정확값에 의존 | ②로 이월. 스키마에 `주문상태` 자리만 남김 |

| 6 | 소싱처 자동확인 = Playwright 로컬 전용 | ③으로 이월 |

### 사용자 결정 (2026-07-10)

- 옥션·G마켓 → **샵마인 엑셀 보조 병행**

## 1. 아키텍처

검증된 계산 로직을 **한 줄도 고치지 않고** 이식한다. 이 코드의 가치는 UI 가 아니라
수백 건의 실주문으로 다듬어진 판정 로직에 있다.

```
lemouton/margin/
  __init__.py
  config.py          ← 마진계산기 config.py 이식 (MARKET_MAP·COUPANG_FEE_RATE·DEFAULT_PRICE_RANGES)
  buy_parser.py      ← data_loader.parse_buy + split_by_site_order_no
  sell_source.py     ★ 신규 — 매출 어댑터 2종 (API / 샵마인 엑셀)
  matcher.py         ← 그대로 이식. **수정 금지**
  aggregator.py      ← 마진계산기 app.py `_aggregate` 추출
  export.py          ← 마진계산기 app.py `/api/download` 로직 추출
  store.py           ★ 신규 — MarginAnalysis 저장/조회/정리

lemouton/margin/models.py        ★ 신규 — MarginAnalysis SQLAlchemy 모델
webapp/routes/orders.py          ← margin 탭 렌더 분기 추가 (기존 파일)
webapp/routes/api_margin.py      ★ 신규 — /api/margin/*
webapp/templates/orders/margin.html  ★ 신규 — UI (design-mockup 게이트 후)
static/js/margin_rules.js        ← 그대로 이식. 마진 판정 단일 원천
```

### 단위별 책임

| 모듈 | 무엇을 하나 | 무엇에 의존하나 |
|---|---|---|
| `buy_parser` | 더망고 엑셀 bytes → 매입 DataFrame + G열(사이트주문번호) 유무로 분할 | pandas, config |
| `sell_source` | (기간 or 엑셀) → **SellRow DataFrame** | `order_export`, pandas, config |
| `matcher` | (매입 DF, 매출 DF) → 매칭 결과 · 미매칭 2종 | pandas, config |
| `aggregator` | 매칭 결과 → 요약/마켓/일별/월별/브랜드/금액대 집계 | pandas |
| `export` | 결과 → xlsx bytes | pandas, openpyxl |
| `store` | 결과 ↔ DB (gzip JSON) | SQLAlchemy, R2 |

`matcher` 는 매출 DF 가 어디서 왔는지 모른다. 이것이 A안(얇은 어댑터)의 핵심이다.

## 4. 정산예정금액 정책 — 🔒 3대 원칙 준수 지점

순마진 = `정산예상금액_배송비포함 − 구매가격`. 분자의 출처를 숨기지 않는다.

| 마켓 | 정산예정금액 산출 | `_settle_source` |
|---|---|---|
| 스마트스토어 | 정산 API 실값 (`pay-settle/settle/case`) | `real` |
| 11번가 | 주문 응답 `stlPlnAmt` | `real` |
| 롯데온 | **`실결제금액 − 실수수료`** ← 수정 | `real` |
| 쿠팡 (정산 잡힘) | 정산 API 실값 (`revenue-history`) | `real` |
| 쿠팡 (미정산) | `(단가×수량 + 배송비) × 0.8845` — **현행 유지** (아래 참조) | `estimated` |
| 옥션·G마켓 | 샵마인 엑셀 값 | `real` |
| 위 어느 것도 불가 | **`0`** (빈칸 금지 — 아래 참조) | `none` |

### 수정 1 — 롯데온

현재 `order_export.lotteon_order_rows` 는 `정산예정금액` 자리에 `od.actualAmt`(실결제금액)를
그대로 넣는다. 수수료가 빠지지 않아 **마진이 과대 계상된다.**

롯데온은 `claims.commission_map()` (apiNo45 `SettleCommission`) 으로 **주문번호별 수수료 실값**을
이미 가져오고 있다. 따라서:

```
정산예정금액 = actualAmt − commission_map[odNo]
```

수수료를 못 가져온 주문은 `estimated` 가 아니라 **`none`** 으로 둔다 (추정 금지).

### 수정 2 — 쿠팡 추정치: 통일 불가. 현행 유지 + 스파이크

| 곳 | 현재 계산식 | 베이스에 할인 반영? |
|---|---|---|
| 샵마인 파서 (`data_loader.py:241`) | `실결제금액 × (1 − 0.1155)` | ✅ |
| `order_export` (`CP_FEE_FACTOR`) | `(단가×수량 + 배송비) × 0.8845` | ❌ |

`0.8845 == 1 − 0.1155` 로 계수는 같으나 **곱하는 베이스가 다르다.**

**당초 "샵마인 기준(`실결제금액`)으로 통일" 하려 했으나 구현 불가로 판명됐다.**
쿠팡 주문 API(발주서 `ordersheets`)는 `실결제금액`을 주지 않는다
(`order_export.py` 쿠팡 빌더가 `실결제금액`을 채우지 않고, `_finalize_rows` 가
`총주문금액`으로 대체한다). 코드 어디에도 쿠팡 할인 필드를 읽는 곳이 없다.

따라서:

1. **현행 계산식을 유지**하고 `_settle_source = "estimated"` 로 표시한다.
   쿠팡 쿠폰·즉시할인이 걸린 주문에서 정산 추정이 **과대**해질 수 있다는 사실을 숨기지 않는다.
2. **스파이크 태스크** — 쿠팡 발주서 실응답에 할인/실결제 필드(`orderPrice`, `discountPrice`,
   `paidAmount` 등)가 존재하는지 실계정으로 확인한다. 있으면 `실결제금액` 을 채우고
   샵마인과 동일한 베이스로 교체한다. **없으면 없는 대로 둔다 — 추측하지 않는다.**

   > **상태: 미실행 (2026-07-10).** 스크립트 `scripts/margin_spike_coupang_paid.py` 는 작성·커밋됐고
   > 문법·import·호출방식까지 검증했으나, **아직 돌리지 않았다.**
   > 마켓 API 는 서버 IP 허용목록(AWS 54.116.196.90)에 묶여 있어 로컬에서 호출이 거부된다
   > (실행 시 `COUPANG_VENDOR_ID 미설정` 에서 멈춤 — 코드 문제가 아니라 키·IP 문제).
   > **서버에서 실행한 뒤 결과를 이 자리에 붙이고 결론을 적을 것.**
   > 그전까지 쿠팡 미정산 정산액은 `estimated` 태그가 붙은 추정치이며, 할인이 걸린 주문에서
   > 과대 계상될 수 있다. 이 사실을 화면 배지로 노출한다.
3. 골든 테스트 2단계에서 **같은 주문에 대해 샵마인 값 vs API 값의 차이를 정량화**해 기록한다.

### 수정 3 — 배송비 이중계상 (2026-07-10 코드리뷰 C1)

마진 분자로 `order_export` 의 **`정산예정금액`** 을 읽는다. `정산예정금(배송비포함)` 이 아니다.

| 필드 | 뜻 (order_export `COLUMN_META`) |
|---|---|
| `정산예정금액` | **상품정산 + 배송비정산(수수료 차감)** |
| `정산예정금(배송비포함)` | 위 값 + **고객배송비 총액** |

후자를 쓰면 배송비가 두 번 들어간다. 배송건당 배송비만큼 마진이 부풀려진다.
모의 계산: 배송비 3,000원 주문 20건 → 정산 합계가 60,000원 과대
(실데이터 260704 총순마진 1,325,722원의 **4.5%**).

**샵마인 실파일 대조로 정의를 확정했다** (`데이터/260704/20260704_샵마인.xls`):

    단가 29,800 / 실결제 29,204 / 고객배송비 3,000
    정산예상금액 25,330  +  고객배송비 3,000  =  정산예상금액(배송비포함) 28,330

즉 샵마인의 `(배송비포함)` = **상품정산 + 고객배송비(총액)** — 배송비 수수료를 차감하지 않는다.
우리 `정산예정금액` 은 **상품정산 + 배송비정산(수수료 차감)** 이므로, 배송건에서
배송비 수수료만큼 샵마인보다 **작다**. 즉 우리 쪽이 **보수적이고 실수취액에 가깝다.**

> ⚠️ 두 정의의 정확한 차이는 **골든테스트 2단계**(`scripts/margin_api_parity.py`, 서버 실행)로
> 정량화해야 한다. 그 전까지 API 경로의 정산액을 "샵마인과 같다"고 말하지 않는다.

`order_export` 의 `정산예정금(배송비포함)` 컬럼 자체는 건드리지 않았다 — `/orders` 주문 엑셀이
그 값을 쓰고 있고, 그 컬럼의 정의가 맞는지는 별개 문제다(후속 조사 대상).

### 재계산 위치 = `sell_source`

`order_export` 의 계산식은 건드리지 않는다. 그 파일은 `/orders` 의 주문 엑셀 다운로드도
먹이고 있어, 계산식을 바꾸면 그 산출물이 함께 바뀐다.

다만 `sell_source` 는 "이 정산액이 실값인가 추정치인가"를 알 수 없다 —
`order_export` 가 추정치를 실값과 같은 칸에 채워 넣기 때문이다. 따라서
**`order_export` 에 `_settle_source` 키를 함께 실어 보내도록 얇게 확장한다.**

- `ALL_COLUMNS` 는 건드리지 않는다 → `rows_to_xlsx` / 대시보드 출력은 불변.
- `_finalize_rows` 는 여분 키를 제거하지 않으므로(`_shipkey`·`_odseq`만 pop) 그대로 통과한다.
- **이것이 `order_export` 에 가하는 유일한 변경이다.**

롯데온 수수료 차감(수정 1)은 `sell_source` 에서 수행한다:
`정산예상금액_배송비포함 = 실결제금액(actualAmt) − 마켓수수료`.
`actualAmt` 는 배송비를 이미 포함하므로 **배송비를 다시 더하지 않는다.**

### 정산 없음 = `0`, 빈칸이 아니다 (2026-07-10 실측으로 정정)

당초 "빈칸"으로 적었으나 **실측 결과 빈칸은 두 가지를 망가뜨린다.**

`matcher._make_result_row` 는 `pd.to_numeric(v, errors='coerce') or 0` 로 읽는다.
`pd.to_numeric("")` 은 `0` 이 아니라 **`NaN`** 이고, `NaN` 은 truthy 라 `or 0` 을 통과한다.

1. **JSON 이 깨진다** — `순마진 = NaN` → `json.dumps` 가 `NaN` 리터럴을 뱉고
   브라우저 `JSON.parse` 가 거부한다. 마진 탭 전체가 안 뜬다.
2. **손실이 총합에서 사라진다** — `aggregator` 의 `df['순마진'].sum()` 은 `NaN` 을
   건너뛴다. 매입 5만 원짜리 미정산 주문의 −50,000 이 총 순마진에서 **빠진다.**
   사용자는 실제보다 좋은 마진을 본다. 조용히 틀린 숫자.

옛 프로그램은 `parse_sell` 의 `_to_numeric_safe` 가 샵마인 빈칸을 `0` 으로 바꿔
이 문제가 없었다. `static/js/margin_rules.js` 도 `0` 을 "정산 없음" 센티널로 읽는다:

    정산 > 0 & 매입 = 0  → 고마진
    정산 = 0 & 매입 > 0  → 의심손실 (순마진 = −매입)
    정산 = 0 & 매입 = 0  → 계산불가

`0` 은 조작된 그럴듯한 숫자가 아니다 — **0원에 정산되는 주문은 없으므로 `0` 은
"정산 없음"을 뜻하는 확립된 부호다.** 정직성은 `_settle_source='none'` 태그가 보존한다:
**값은 `0`, 태그는 `none`, 화면은 "정산 확인 불가" 배지.** 숨기는 것이 없다.

### 화면 표기

- `_settle_source == "estimated"` → 해당 행 정산액에 `⚠ 추정` 배지.
- KPI 요약에 `추정 정산 N건` 을 별도 표기.
- **추정치를 실값처럼 보이게 하지 않는다.**

## 9. 실패 처리 — 조용한 실패 금지

관련: `project_silent_failure_bug_class` (무결성 제약 부재 + 진실원천 분열 + 누락 무경고)

| 상황 | 처리 |
|---|---|
| **마켓 4개 중 1개라도 API 조회 실패** | **분석 전체를 중단.** 저장하지 않고 `⚠ 롯데온 조회 실패 (사유)` 표시 후 재시도 안내 |
| 등록 계정 중 일부만 키 미등록/인증 실패 | `combined_order_rows(warnings=[...])` 가 담아 주는 사유를 `markets_failed` 에 기록하고 배너로 표면화. 나머지는 진행 |
| 정산예정금액 없음 | 빈칸. 마진 계산 안 함 → `계산불가` (기존 `margin_rules.js` 규칙 그대로) |
| 옥션·G마켓 주문이 더망고에 있는데 보조 엑셀 미업로드 | `⚠ 옥션 주문 N건 — 매출 미연동` 배너 |
| 더망고 엑셀 파싱 실패 | 기존 3단 fallback(openpyxl→xlrd→html5lib) 유지. frameset `.xls` 안내 메시지 그대로 |

### 왜 "부분 성공"을 허용하지 않는가

`order_export._fetch_combined` 는 이미 *"한 마켓 실패는 전파(부분 성공 숨김 금지)"* 로 설계돼 있다.
마진 탭에서는 이 선택이 더 중요하다 — 롯데온 조회가 실패하면 **롯데온 매입 행 전부가
"매출 미매칭"으로 둔갑**해, 마치 블랙스팟(매입했는데 안 팔림)처럼 보인다.
조용한 실패보다 나쁜, **적극적으로 틀린 신호**다. 따라서 전체 중단이 옳다.

## 10. 검증 — "100% 완벽"의 조작적 정의

골든 테스트를 **2단계**로 나눈다. 이것이 완료 기준이다.

### 1단계 · 회귀 동치 (이식이 무손실인가)

```
입력: 더망고 + 샵마인 엑셀 쌍 (실측 확인된 4벌)
      데이터/260629/ · 260630/ · 260704/ · 260706/
  옛 프로그램 POST /api/analyze  →  snapshot.json  (고정 픽스처로 커밋)
  새 코드, 매출 소스 = sell_source.from_shopmine_excel
  → matched · unmatched_buy · unmatched_sell · 모든 집계가 snapshot 과 완전 일치
```

매출 소스를 샵마인 엑셀로 고정하면 API 라는 변수 없이 **이식 로직만** 검증된다.
한 건이라도 어긋나면 이식이 잘못된 것이다. **이 테스트가 통과하기 전에는 API 경로를 손대지 않는다.**

### 2단계 · API 동치 (어댑터가 맞는가)

> **상태: 미실행 (2026-07-10).** 스크립트 `scripts/margin_api_parity.py` 는 작성·커밋됐고
> 샵마인 파싱 → `from_api` 호출 직전까지 실제로 실행됨을 확인했다. 마켓 API 가
> 서버 IP 허용목록에 묶여 있어 **로컬에서는 완료할 수 없다. 서버에서 실행할 것.**
> (부수 확인: 스마트스토어 키 부재만으로 `from_api` 전체가 예외를 던졌다 — 스펙 §9 의
> "마켓 하나라도 실패하면 분석 중단"이 실제로 그렇게 동작한다.)

```
같은 기간에 대해:
  샵마인 엑셀 SellRow   vs   from_api SellRow
  → 오픈마켓주문번호 단위 필드별 대조표 출력
     (정산예상금액_배송비포함 · 단가 · 실결제금액 · 마켓수수료 · 수량 · 주문상태)
```

여기서 롯데온 수수료 누락 같은 **실제 차이가 드러난다.** 차이가 나면 그 자리에서
"우리 API 가 틀림" / "샵마인이 틀림" 을 판정해 기록한다. **차이를 덮지 않는다.**

### 3단계 · 실화면 대조 (🔒 3대 원칙)

대조표 상위 불일치 건 + 무작위 표본을 **판매처 셀러 어드민 화면의 실제 정산예정금액**과 육안 대조한다.
어드민 화면이 최종 심판이다.

### 단위 테스트

- `sell_source.from_api` — `order_export` 를 모킹해 SellRow 컬럼·타입·`_settle_source` 검증
- `sell_source` 마켓명 역변환 (`smartstore` → `04.스마트스토어`)
- 롯데온 수수료 차감 (수수료 있음 → `real` / 없음 → `none`)
- 쿠팡 추정 계산식 (`실결제금액 × 0.8845`, `estimated` 태깅)
- 기간 자동 추론 (`min−3일`, `max+3일`)
- `store` — gzip 왕복, 20건 초과 시 오래된 것 삭제
- 실패 처리 — 마켓 1개 실패 시 `markets_failed` 에 기록되고 나머지는 진행
