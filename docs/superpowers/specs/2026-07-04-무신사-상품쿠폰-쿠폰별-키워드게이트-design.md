## 1. 배경 · 문제

무신사 상품쿠폰은 현재 **한 덩어리 금액**(`coupon_amount`) 하나로 저장되어, 최종매입가 계산에서 **무조건 차감**된다.

- 💻 수집→저장: 확장이 `benefit_lines`(혜택 텍스트)·표면가를 긁고, 서버가 최고 쿠폰 하나를 `coupon_amount`로 뭉쳐 저장 — `lemouton/sources/service.py`(`_dyn['coupon_amount']`).
- 💻 적용: `webapp/routes/api_benefits.py` `compute_breakdown` — 무신사(source_id=3)면 `_Inj('상품쿠폰', coupon_amount, enabled=bool(coupon_amount))` 로 베이스금액에서 차감.

**문제**: 무신사가 자동 적용하는 "최고 쿠폰"이 **내 등급 쿠폰(다이아몬드·플래티넘·골드·실버·브론즈)** 일 수 있다. 이건 나에게만 매달 주어지는 정기 쿠폰이라 상품값에 녹여 팔면 안 된다(언더프라이싱 → 금전 손실). 지금은 이걸 걸러낼 방법이 없다.

### main에 이미 있는 것 (재사용)

- ✅ **혜택별 포함/제외 키워드 UI** — ③ 혜택 카드마다 「적용 키워드」+「제외 키워드」(조건부 크롤 혜택). `webapp/templates/sourcing_guide/detail.html` · `detail.js`.
- ✅ **키워드 게이트가 계산에 배선됨** — `benefit_gate.gated_off_names` → `compute_breakdown` 에서 `status='conditional'` 혜택을 통째로 ON/OFF. 혜택별 `excludes`·공통 `exclude_keywords` 모두 사용(`line_excluded_by_benefit`).
- ✅ **게이트 미리보기** — `/sourcing-guide/api/<sid>/gate-preview`.

### 아직 없는 것 (이 스펙이 만드는 것)

- ❌ **쿠폰별 정밀 필터** — 상품쿠폰이 한 덩어리라, "등급쿠폰만 빼고 남은 일반 쿠폰 중 최고"를 못 고른다. 게이트는 `benefit_lines`(페이지 텍스트) 기준으로 **혜택 전체 on/off** 만 한다.

## 2. 목표 (확정 규칙)

> **상품쿠폰 = "제외 키워드에 안 걸린 쿠폰 중 최고 금액. 하나도 안 남으면 0원."**

- 폴백 금지: 남은 쿠폰이 없으면 0원(대표가·옛값 금지). 프로젝트 무결성 원칙 준수.
- 계산 시점 적용: 키워드만 바꿔도 **재크롤 없이 즉시** 가격 반영.
- 크롤=로컬(크롬 확장), 저장·계산=서버. (서버 크롤 부활 없음.)

### 브레인스토밍 확정 결정

| 항목 | 결정 |
|---|---|
| 제외 동작 | **B방식** — 쿠폰별 필터 후 남은 것 중 최고 재선택 |
| 남은 쿠폰 없을 때 | **0원** (폴백 금지) |
| 키워드 편집 범위 | 혜택별 포함/제외 (이미 있음, 재사용) |
| 화면(UI) | **10번 · 미니멀 라인(토스식)** 으로 ③ 혜택 편집 통째 재구성 |
| 키워드 문구 | "하나라도 있으면 / 모두 있어야" |
| 여러 단어 입력 | **㉮ 쉼표 붙여넣기** (칩 자동 분리). 키워드 **세트 기능은 미룸**(YAGNI) |
| 키워드 저장 | 평면 목록 (나중에 세트 얹어도 무리 없게) |

## 3. 아키텍처 (4층)

> **핵심 원칙 — 수집과 판단의 분리**: 크롤(확장)은 **판단 없이 쿠폰을 전량 원본으로** 수집만 한다. 어떤 쿠폰을 쓸지(포함/제외·최고 선택)는 **전적으로 서버 프로그램 로직**이 계산 시점에 결정한다. → 키워드를 바꿔도 재크롤 없이 fx(최종매입가)만 다시 계산돼 즉시 반영된다.


### ① 크롤 (크롬 확장) — 상품쿠폰을 "목록"으로

- 현재: 최고 쿠폰 1개를 금액만 뭉쳐 전송.
- 변경: `product_coupon_list = [{name, amount}, ...]` 를 함께 전송.
- **수집원(미지수) — 스파이크로 먼저 확정**: 무신사 PDP는 최고 쿠폰 1개만 자동 표시하고, 전체 목록은 "쿠폰 변경/다운로드"에 있다.
  - (우선) **무신사 쿠폰 API** — 가격(goodsPrice)·재고(inventory)를 이미 API로 읽는 것처럼, 다운로드 가능 쿠폰 목록 API가 있으면 그걸로(가장 견고).
  - (폴백) **쿠폰 목록 DOM** — API가 없으면 쿠폰 영역/모달에서 `{쿠폰명, 금액}` 파싱.
  - 참고: 폐기된 `crawlers/musinsa_playwright.py` 에 상품쿠폰 섹션·`cart_coupons` 파싱과 '월1회 정기쿠폰 제외' 하드코딩 선례가 있음 — 로직 참고만(서버 크롤 부활 아님).
- **실브라우저 검증 필수** — 로그인된 무신사 실페이지에서 쿠폰명·금액이 실제로 읽히는지 대조(프로젝트 크롤 검증 프로토콜). 못 읽으면 "확인 불가"로 표면화.
- 💻 대상: `extension/moum-crawler/background.js` (무신사 파싱), 반환 dict에 `product_coupon_list` 추가.

### ② 저장 (서버)

- `dynamic_benefits_json` 에 `product_coupon_list` 저장(평면 목록).
- 하위호환: 목록이 없으면 기존 `coupon_amount` 경로로 폴백(no-op 안전).
- 💻 대상: `lemouton/sources/service.py` (동적키 화이트리스트에 `product_coupon_list` 등록).

### ③ 계산 (compute_breakdown) — 쿠폰별 필터 + 최고 재선택

- 무신사 상품쿠폰 주입부를 다음으로 대체:
  1. `product_coupon_list` 를 읽는다.
  2. ③의 **상품쿠폰 혜택** 의 `triggers`(포함)·`excludes`(제외)·`match`/`exclude_match` 를 **각 쿠폰 `name` 에** 적용 — `benefit_gate` 의 `line_matches_triggers` + `line_excluded_by_benefit` 재사용(라인 대신 쿠폰명).
  3. 살아남은 쿠폰 중 **최고 `amount`** → 상품쿠폰 차감액. 없으면 0.
- **계산 시점 적용** → 키워드 수정 즉시 반영.
- 하위호환: 목록이 없으면 기존 단일 `coupon_amount` 로직 유지.
- 안전: `status != 'conditional'`·하드코딩 항목 불변(기존 게이트 불변식 유지).
- 💻 대상: `webapp/routes/api_benefits.py` `compute_breakdown` (무신사 상품쿠폰 주입 블록) + `lemouton/pricing/benefit_gate.py` (쿠폰명 필터·최고 선택 순수 함수 신설, 예: `pick_best_coupon(coupons, benefit) -> {name, amount} | None`).

### ④ 화면

**(a) ③ 혜택 편집 UI 재구성 — 시안 10번(미니멀 라인)**
- 구분선 기반, 색 최소, 넓고 깔끔. 혜택별 포함/제외 키워드는 라벨(포함=초록/제외=빨강) + 칩.
- 기존 기능 전부 유지: 값출처(고정/크롤)·유형·적용(상시/조건부)·"하나라도 있으면/모두 있어야".
- **㉮ 쉼표 붙여넣기** — 제외/포함 입력칸이 "다이아몬드, 플래티넘, 골드" 붙여넣기를 받아 칩으로 자동 분리 + 안내문 한 줄.
- 💻 대상: `webapp/templates/sourcing_guide/detail.html` · `detail.js`.

**(b) 계산식 영수증 투명성 — 어떤 쿠폰이 적용/제외됐나**
- 최종매입가 영수증(매트릭스 셀 팝업)에 "적용 쿠폰: OO / 제외됨: 다이아몬드(제외 키워드)" 표시. 무결성 원칙("누락에 경고") 충족.
- 💻 대상: 영수증 렌더(계산식 팝업) — `compute_breakdown` 결과에 쿠폰 판정 근거 포함.

## 4. 데이터 계약

```jsonc
// dynamic_benefits_json 추가 키
"product_coupon_list": [
  { "name": "다이아몬드 등급 쿠폰", "amount": 12000 },
  { "name": "무탠다드 상품쿠폰",   "amount": 5000 }
]

// ③ 상품쿠폰 혜택(가이드) — 이미 존재하는 스키마 재사용
{ "name": "상품 쿠폰", "value_source": "crawl", "status": "conditional",
  "triggers": [], "match": "any",
  "excludes": ["다이아몬드","플래티넘","골드","실버","브론즈"], "exclude_match": "any" }
```

- 필터 판정: 쿠폰 `name` 에 대해 `triggers`(비면 전체 통과) 통과 && `excludes` 미veto → 후보. 후보 중 `amount` 최대 선택.

## 5. 범위 밖 (YAGNI)

- **키워드 세트 관리 기능** — 지금은 등급쿠폰 1세트×1혜택×1소싱처뿐. 세트는 같은 묶음을 2곳+에서 재사용할 때 추가. 지금은 ㉮ 붙여넣기로 대체.
- **다른 소싱처 확장** — 무신사 검증 후 같은 패턴이면 별도 작업으로.
- 서버 크롤 부활 — 금지(설계 원칙).

## 8. 스파이크 결과 (2026-07-04 실브라우저 확인 — 르무통 상품 goodsNo 3728480, 로그인)

**수집원 확정 — API 존재 + DOM 둘 다 가능:**
- **API (권장·견고)**: `GET https://api.musinsa.com/api2/coupon/coupons/getUsableCouponsByGoodsNo?goodsNo={no}&brand={brand}&comId={comId}&salePrice={표면가}&specialtyCodes={...}` → 200, 사용가능 쿠폰 목록 반환. (부가: `api2/coupon/cart-coupons/goods-usable-list` = 장바구니 쿠폰.) 쿠폰변경 클릭 시 호출됨. 확장은 이미 goodsNo·salePrice 를 크롤하므로 직접 호출 가능. ⚠️ 응답 필드명은 라이브 크롤 테스트에서 console 로그로 확정(브라우저 검증 도구의 프라이버시 필터로 본문 직독 불가였음).
- **DOM (검증됨·폴백)**: 적용 쿠폰은 PDP 에 노출("상품 쿠폰 {쿠폰명} 쿠폰변경 -{금액}원"). 전체 목록은 "쿠폰변경" 모달("사용 가능한 쿠폰")에 행 단위로 노출. 파싱 가능 확인.

**실제 쿠폰 데이터 (이 상품 = 전부 등급쿠폰, 일반쿠폰 0):**
| 쿠폰명 | 금액 |
|---|---|
| [최대 할인 쿠폰] 7월 무신사 회원 **정기** 쿠폰 **블랙다이아몬드** 등급 | 6,390원 (자동적용) |
| 7월 무신사 회원 정기 쿠폰 블랙다이아몬드 등급 | 5,110 / 3,830 / 2,550원 |

**키워드 확정**: 실제 등급쿠폰명 = "7월 무신사 회원 **정기 쿠폰** … **블랙다이아몬드** 등급". → 제외 키워드 **"정기"** 가 가장 견고(모든 월간 등급쿠폰이 "정기 쿠폰"). "다이아몬드"도 "블랙다이아몬드"의 부분일치로 잡힘. 권장 제외 키워드 = `정기` + `다이아몬드·플래티넘·골드·실버·브론즈`(등급 백업). → 이 상품은 전부 제외 → **상품쿠폰 0원**(설계대로).

**Task 5 방향**: 확장이 쿠폰 API(`getUsableCouponsByGoodsNo`) 호출 → `product_coupon_list=[{name, amount}]` 구성(응답 필드명은 라이브 로그로 확정). API 실패 시 "쿠폰변경" 모달 DOM 파싱 폴백.
- 등급쿠폰의 실제 이름 표기(정확한 키워드 확정용).
