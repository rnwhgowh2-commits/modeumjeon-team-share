## 1. 배경 / 문제

§3 "옵션 통합 관리" 화면 상단의 **"소싱처 일괄 정책" 바**는 가격(판매가)을 일괄 설정하는 컨트롤이다. 현재 두 가지 문제가 있다.

1. **기능이 껍데기(stub)** — `매트릭스 재계산` 버튼은 `_matrix_v3.html:2182~2194` 에서 alert만 띄우고 끝난다. 칩을 바꿔도 화면 글자만 바뀔 뿐 실제 계산·저장이 없다.
2. **위상이 안 보임** — 칩 5개가 모두 같은 크기로 한 줄에 붙어 있어, 무엇이 중요한 설정인지·바가 무슨 역할인지 사용자에게 전달되지 않는다. 바 위치도 크롤 버튼 아래라 "설정 먼저 → 크롤" 작업 순서와 어긋난다.

비개발자 사용자(프로젝트 소유자) 입장에서 "이게 뭘 하는지, 왜 여기 있는지" 감이 안 잡힌다는 것이 출발점이었다.

## 2. 목표 / 비목표

**목표**
- 정책 바를 시각적으로 정리해 "가격을 정하는 핵심 컨트롤"임이 한눈에 보이게 한다.
- `매트릭스 재계산` 버튼이 실제로 동작하게 만든다 (stub 제거).

**비목표**
- 사입(박스히어로) 모드 옵션의 가격 로직은 건드리지 않는다. 이 바는 **소싱처 모드 옵션에만** 적용된다.
- 가격표(매트릭스) 자체의 행·열 구조는 바꾸지 않는다.
- 크롤 수집 메커니즘(크롤러·파이프라인)은 바꾸지 않는다.

## 3. Part A — UI 재설계 (가격 설정 통합 박스)

### 3.1 배치
- 정책 바를 §3 **맨 위**로 옮긴다 (현재는 크롤 버튼 아래).
- 정책 바 + `전체 크롤` 버튼을 **하나의 박스**(초록 테두리)로 통합한다.
- 박스 내부는 **좌우 분할** — 왼쪽 = 설정 영역, 오른쪽 = 큰 세로형 `전체 크롤` 버튼.
- 박스 헤더: `⚙ 가격 설정 → 크롤` 제목 + `↻ 가격만 재계산` 버튼(우측).

### 3.2 설정 영역 — 2그룹(위상 구분)
설정 영역은 기능별 2개 그룹으로 나눈다. 각 그룹은 작은 라벨 + 옅은 박스로 구분한다. **세로 구분선(세로바)은 쓰지 않는다** (사용자 명시 요청).

- **① 사올 때 — 매입**: `정책`(색상 통일 / 옵션별 cheapest) · `상한`(매입가 상한)
- **② 팔 때 — 판매 마진**: `마진 방식`(% 정률 / ₩ 정액) · `스토어 마진` · `쿠팡 마진`

② 그룹은 배경을 살짝 다르게 해 주인공으로 표시한다.

### 3.3 마켓 마진 크게
`스토어 마진`·`쿠팡 마진`은 **큰 숫자 칩**으로 표시한다 — 작은 라벨(브랜드 뱃지 포함) + 큰 값(약 30px). 나머지 칩(`정책`·`마진 방식`·`상한`)은 작은 칩으로 두어 위계를 만든다.

### 3.4 정리 4가지 (구조·위치 외 디테일)
1. **단위 일치** — `마진 방식`이 `% 정률`이면 마진 칩이 `%`, `₩ 정액`이면 `원`으로 단위가 따라온다. (현재 정액인데 `%`로 표시되는 모순 제거 — `_matrix_v3.html:2156` 부근)
2. **그룹 박스로 묶기** — 3.2 의 2그룹. 선 없이 옅은 박스 + 간격.
3. **재계산 버튼 톤다운** — `↻ 가격만 재계산`은 연한 테두리 버튼(ghost). 진초록 꽉찬 `전체 크롤` 버튼과 위계를 구분한다.
4. **라벨 다이어트** — 긴 초록 태그 "소싱처 모드 옵션에 자동 적용"을 작은 `ⓘ` 아이콘 + 툴팁으로 축소.

## 4. Part B — 재계산 엔진

### 4.1 트리거
- **`↻ 가격만 재계산` 버튼** — 이미 크롤된 매입가(캐시값)로 판매가만 다시 계산. 재크롤 안 함.
- **`전체 크롤` 버튼** — 크롤로 매입가를 새로 수집한 뒤, 이어서 재계산까지 수행.

### 4.2 계산 흐름 (소싱처 모드 옵션 전체 대상)
1. **매입가 결정** — 각 옵션의 소싱처 URL별 캐시 가격(`OptionSourceUrl.price_cached`)에서 정책에 따라 매입가를 고른다.
   - `색상 통일` — 같은 색상의 모든 사이즈가 같은 매입가(최저 기준). `lemouton/pricing/color_group.py` 의 색상 그룹 결정 로직 재사용.
   - `옵션별 cheapest` — 각 옵션(사이즈)별로 소싱처 최저가.
2. **판매가 산출** — 매입가 → 판매가. 기존 통합 공식 재사용: `lemouton/pricing/unified.py:compute_sale_price_unified()`
   - `% 정률`: `판매가 = 매입가 × (1+마진율) × (1+수수료율) + 배송비`
   - `₩ 정액`: `판매가 = (매입가 + 정액) × (1+수수료율) + 배송비`
   - 스토어·쿠팡은 각각의 마진율·수수료율로 별도 산출.
3. **상한 체크** — 매입가 ≥ 상한이면, 판매가는 그대로 계산·표시하되 해당 옵션을 **마켓 업로드 대상에서 제외**(플래그)한다.
4. **저장** — 바의 마진율을 대상 옵션들의 `OptionPriceConfig.margin_rate`(스토어/쿠팡)에 기록하고, 산출된 판매가를 반영한다.

### 4.3 적용 범위
- 자동(auto) 모드 + 소싱처 모드 옵션만. 수기가격(`auto_enabled=False`) 옵션·사입 모드 옵션은 건드리지 않는다.

### 4.4 신규/재사용 엔드포인트
- **신규** `POST /api/bundles/<code>/recalc-policy` — 본문 `{policy, margin_unit, ss_margin, cp_margin, ceiling}`. 4.2 흐름 수행 후 갱신된 매트릭스 데이터 반환.
- **재사용** — 일괄 마진 저장은 기존 `POST /api/options/price-config/bulk`(`api_pricing.py:763`) 로직을 내부에서 활용 가능. 크롤은 기존 `POST /api/bundles/<code>/run-now`(`api.py:2646`).

## 6. 데이터 모델

- `OptionPriceConfig` — 옵션별 `auto_enabled, margin_rate, ss_fee_rate, cp_fee_rate, manual_ss/cp_price, manual_stock`.
- `OptionSourceUrl` — 옵션×소싱처 매핑 + `price_cached, stock_cached`.
- **신규 필요** — "상한 초과 → 업로드 제외" 상태를 저장할 필드. `OptionPriceConfig` 에 `upload_excluded`(boolean) 추가가 후보. → **Alembic 마이그레이션 생성 → Supabase 적용** (프로젝트 규칙).

## 7. 위험 / 미해결

- **정액 마진** — 통합 공식이 정률 기준이라 정액 모드는 분기 구현 필요. 4.2 에 식을 명시했으나 기존 `compute_sale_price_unified()` 시그니처 확인 후 확정.
- **color_group 단독 호출** — 색상 통일 로직이 현재 "크롤 후 가격 결정 단계"에 묶여 있다. 재계산에서 단독 호출 가능한지 구현 계획에서 확인.
- **업로드 제외 플래그** — 기존에 옵션을 업로드에서 빼는 메커니즘이 있으면 그것을 재사용, 없으면 6절의 신규 컬럼.
- **데이터 무결성** — 가격·재고 오류는 금전 손실. 재계산은 트랜잭션으로 묶고, 매입가 캐시가 없는 옵션은 계산하지 않고 "미수집" 상태로 둔다.

## 8. 검증

- 소싱처 모드 옵션 N개에 대해 재계산 → 판매가가 공식대로 나오는지(스토어·쿠팡 각각).
- 정책 `색상 통일` ↔ `옵션별 cheapest` 전환 시 매입가 선택이 달라지는지.
- 상한 초과 옵션이 업로드 대상에서 빠지는지.
- 사입 모드·수기가격 옵션이 영향받지 않는지.
- 브라우저 실접속 UI 검증 (`/ui-verify`).

## 결정 요약 (브레인스토밍 경위)

- 가격 결정 방식: 기본 자동 + 일부 수정
- 마진 범위: 모음전마다 다름 → 이 편집 화면에 위치
- 작업 순서: 설정 먼저 → 크롤
- 방향: 전면 재설계 ❌, 기존 바 최소 정리 + 재배치
- 세로 구분선(세로바) 사용 안 함
- 마켓 마진: 크게(더 크게)
- 위상: 2그룹 (매입 ↔ 판매)
- 배치: 크롤 버튼과 한 박스 통합, 박스 내부 좌우 분할
- 범위: 디자인 정리 + 재계산 엔진 실제 구현
- 상한 초과: 업로드 대상에서 제외
