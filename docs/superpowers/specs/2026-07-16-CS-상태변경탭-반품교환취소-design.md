> **계층**: 프로젝트 스펙 / **날짜**: 2026-07-16
> **상위 발화**: 주문내역 3분리 (①주문일=신규주문 ②상태변경 별도탭 ③마켓별 CS) — 순서 B
> **선행**: #1 완료([[project_orders_new_order_split_kind_tag]]) — 클레임 행에 `_kind='change'`·`_change_date` 태그 심어짐
> **본 스펙 범위**: **②만**. ③(고객문의)은 같은 CS 탭의 다른 구역, 별도 스펙.
> **확정 시안**: `바탕화면/모음전_CS_칸반_마켓탭_시안_v2.html` — **A1(상단 가로 마켓 탭) + B5(로고+건수 배지 버튼)**.

---

## 1. 목표 (한 줄)

취소/교환/반품 클레임을 **CS 탭의 「반품·교환·취소」 구역**에서 3단계 워크플로(**신규요청 → 대응필요 → 대응완료**)로 관리한다. 「확인」만 수기, 「완료/철회」는 마켓 상태에서 자동 파생.

## 2. 화면 구조 (확정 시안 A1+B5)

```
CS 탭
├─ 상단 구분 토글: [반품·교환·취소]  [고객문의(#3)]
└─ 반품·교환·취소
   ├─ 상단 가로 마켓 탭 (로고+건수 배지):  전체·쿠팡·11번가·롯데온·스마트스토어
   └─ 3열 칸반:  🆕 신규요청  |  👀 대응필요  |  ✅ 대응완료
```

- 마켓 탭 = **로고색 + 마켓명 + 건수 배지**(B5). 「전체」 포함. 고른 마켓의 보드만 표시.
- 칸반 카드 = 유형 배지(취소=주황/반품=빨강/교환=보라) + 마켓 + 상품·옵션 + 수량 + 변경일 + 사유.
  - **신규요청 카드**: 「확인 ✓」 버튼 + 간단메모 입력칸.
  - **대응필요 카드**: 저장된 메모 표시(수정 가능).
  - **대응완료 카드**: 완료/철회 배지(회색), 읽기전용.
- 위치 = 기존 껍데기 `cs` 서브탭(💬 문의·반품)을 이 실제 CS 탭으로 전환. 「고객문의」 구역은 #3에서 채움(지금은 빈 상태/자리만).

## 3. 워크플로 (3단계 — 파생 규칙)

각 클레임의 단계는 **저장 안 하고 매번 파생**한다. 저장하는 건 **①확인 여부 ②메모** 둘뿐.

| 단계 | 조건 |
|------|------|
| **대응완료** | 마켓 상태가 **종결** = {취소완료, 반품완료, 교환완료, **철회**} 중 하나 → 자동(확인 여부 무관, 최우선) |
| **대응필요(확인)** | 종결 아님 + 사용자가 「확인」 누름(`acknowledged_at` 있음) |
| **신규요청** | 종결 아님 + 아직 확인 안 함 |

- 파생 우선순위: **종결 → 대응완료** 가 항상 이김(요청이 확인 전에 완료·철회되면 신규요청/대응필요 건너뛰고 바로 대응완료).
- **철회**(고객이 클레임 요청 취소): 롯데온 `odPrgsStepCd 22`, 11번가 `22` → 종결로 간주 → 대응완료.
- 메모는 단계 넘어가도 **유지**(대응완료에서도 보임).

## 4. 데이터 계층

### 4.1 `status_change_rows` (변경일 기준 수집) — #1에서 미뤄둔 함수

```
status_change_rows(markets, *, since, until) -> list[dict]
```
- `_kind='change'` 행만, **`_change_date`(변경일) ∈ [since,until]** 로 트리밍(주문일 아님).
- 클레임당 1행: `마켓`·`오픈마켓주문번호`·`유형`(취소/교환/반품)·`상태`(요청/완료/철회)·`변경일`·`사유`·`상품`·`옵션`·`수량`·`수령자`.
- **주의(#1과 다른 축)**: `combined_order_rows`는 주문일로 트리밍하므로 "옛 주문의 이번 기간 변경"이 탈락한다. 따라서 status_change_rows는 **클레임 조회창을 변경일 기준으로** 잡아 별도 수집한다(클레임 이터레이터는 이미 `[since, max(until,now)]` 폭으로 조회 → `_change_date`로 최종 트리밍).
- `유형`·`상태` 파생: `주문상태` 값에서(취소요청→유형 취소·상태 요청 / 반품완료→유형 반품·상태 완료 / 철회→상태 철회).
- **데이터 한계(#1 rev4에서 확인)**: 11번가 클레임 변경일(`clmDt`)이 없을 수 있음 → 변경일 트리밍 부정확 가능. 라이브 검증 필요(롯데온·쿠팡·스스는 변경일 확보). 없으면 조회창 그대로 노출(누락보다 과다 노출 우선).

### 4.2 처리상태 저장 모델 (신규 모듈 `lemouton/claims/`)

`lemouton/claims/models.py` — `ClaimHandling`:

| 필드 | 뜻 |
|------|-----|
| `id` | PK |
| `claim_key` | **안정 식별자**(unique) = `f"{market}:{order_no}:{claim_type}"` (+ 롯데온·쿠팡은 `clmNo/receiptId` 접미로 중복 클레임 구분) |
| `market` / `order_no` / `claim_type` | 조회·표시용 |
| `acknowledged_at` | nullable — 「확인」 누른 시각(있으면 대응필요) |
| `memo` | 간단메모(text) |
| `created_at` / `updated_at` | 타임스탬프 |

- 단계는 저장 안 함(§3 파생). `claim_key`로 status_change_rows 결과와 LEFT JOIN.
- **팀 공유**(멀티유저) 모델 — per-user 분리 없음(프로젝트 원칙).

### 4.3 서비스 (`lemouton/claims/service.py`)

- `list_claims(markets, since, until)` → status_change_rows + ClaimHandling 조인 → `{신규요청:[], 대응필요:[], 대응완료:[]}` (마켓별 건수 포함).
- `acknowledge(claim_key)` → `acknowledged_at=now` upsert.
- `save_memo(claim_key, memo)` → memo upsert.

## 5. 라우트 (`webapp/routes/orders.py` 또는 신규 `cs.py`)

- `GET /orders/cs/claims.json?markets=&since=&until=` → 3단계 그룹 + 마켓별 건수. (인증·게이트 기존 패턴)
- `POST /orders/cs/claims/ack` `{claim_key}` → 확인 처리.
- `POST /orders/cs/claims/memo` `{claim_key, memo}` → 메모 저장.
- `cs` 서브탭 렌더를 샘플 → 실데이터(claims.json fetch) 로 전환. 「고객문의」 구역은 빈 상태 placeholder.

## 6. 테이블 생성 (실제 프로젝트 패턴 — Alembic 아님)

- 이 프로젝트는 **Alembic 미사용**이 실태 — 신규 테이블은 `shared/db.py:init_db()`의 `Base.metadata.create_all`이 자동 생성한다(여러 모델 주석에 "Alembic 불요" 명시).
- 따라서 `ClaimHandling`(`shared.db.Base` 상속)만 정의하고 **`app.py`에 `import lemouton.claims.models` 등록**하면 create_all이 `claim_handling` 테이블을 만든다. 별도 마이그레이션 스크립트 불필요.

## 7. 엣지 케이스

| 상황 | 처리 |
|------|------|
| 요청이 확인 전에 완료/철회 | 종결 우선 → 바로 대응완료(신규요청/대응필요 안 거침). |
| 같은 주문·같은 유형 클레임 2회 | `claim_key`에 clmNo/receiptId 접미로 구분(가능한 마켓). 불가 마켓은 최신 1건 취급(라이브 확인). |
| 확인했는데 다시 요청 상태로(재요청) | `acknowledged_at` 유지 → 대응필요 유지(재요청은 드묾, 메모로 대응). |
| 11번가 변경일 없음 | §4.1 한계 — 조회창 노출, 라이브 검증. |
| 클레임 조회 실패(한 마켓) | 기존 try/except 유지 — 그 마켓만 비고, 나머지 정상. |

## 8. 테스트 (TDD)

1. **status_change_rows**: `_kind='change'`만·`_change_date` 트리밍·유형/상태 파생(취소요청→취소/요청, 반품완료→반품/완료, 철회→철회) fixture 검증.
2. **단계 파생**: 종결(취소완료·철회 등)→대응완료(확인 무관) / 확인함+미종결→대응필요 / 미확인+미종결→신규요청.
3. **service**: acknowledge·save_memo upsert(중복 claim_key 갱신), list_claims 조인·그룹핑·마켓별 건수.
4. **route**: claims.json 3그룹 구조 / ack·memo POST 왕복.
5. **회귀**: #1 `new_order_rows`·`combined_order_rows` 불변.

## 9. 비목표 (Non-goals)

- **고객문의(문의채널·API 전송)** → #3(같은 CS 탭 다른 구역).
- 마켓에 실제 클레임 처리 실행(환불 승인·교환 재발송 등 API 액션) — 본 기능은 **우리 내부 처리상태·메모 기록만**. 실행은 수기/후속.
- 클레임 자동 알림(토스트/텔레그램).

## 10. 구현 메모

- 구현 = **`main` 기준 새 워크트리**(현재 `feature/crawl-worker-queue` stale).
